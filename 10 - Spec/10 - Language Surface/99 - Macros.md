# Macros

## Summary

Spatial's app-level macro surface is declared twice, once in `spatial.libdsl` and once in `spatial.dsl`, because the two import scopes expose different alias/shadowing behavior but both need the same annotations (`src/spatial/dsl.scala:8-46`). Both scopes declare `@spatial`, `@struct`, and `@streamstruct` as `StaticAnnotation` classes whose `macroTransform` methods delegate to forge/argon/Spatial macro implementations (`src/spatial/dsl.scala:14-25`, `src/spatial/dsl.scala:34-45`). `@spatial` wraps `forge.tags.AppTag("spatial", "SpatialApp")`, `@struct` delegates to argon's `StagedStructsMacro.impl`, and `@streamstruct` delegates to Spatial's `StagedStreamStructsMacro.impl` (`src/spatial/dsl.scala:3-4`, `src/spatial/dsl.scala:14-25`, `src/spatial/dsl.scala:34-45`). The app entry type is `SpatialApp`, defined as `trait SpatialApp extends Spatial with DSLApp` (`src/spatial/SpatialApp.scala:1-5`). Macro mechanics for forge tags are documented in [[10 - Spec/30 - IR/00 - Argon Framework/C0 - Macro Annotations]].

## Syntax or API

```scala
import spatial.dsl._

@spatial object MyApp extends SpatialApp { ... }    // dsl.scala:34-38
@struct case class Point(x: I32, y: I32)            // dsl.scala:40-41
@streamstruct case class Ports(in: I32, out: Bit)   // dsl.scala:43-44

import spatial.libdsl._
@spatial object LibStyle extends SpatialApp { ... } // dsl.scala:14-18
```

`spatial.dsl` is the full view with shadowing, because it extends `SpatialDSL with StaticAPI_Shadowing` (`src/spatial/dsl.scala:28-45`). `spatial.libdsl` is the library view without shadowing, because it extends only `SpatialDSL` (`src/spatial/dsl.scala:6-26`). Both views redeclare annotation classes locally, so app authors can import either scope and still use the same annotation names (`src/spatial/dsl.scala:14-25`, `src/spatial/dsl.scala:34-45`).

## Semantics

`@spatial` is an application tag. In `libdsl`, `final class spatial` delegates `macroTransform` to `spatial.impl`, and the private companion object extends `AppTag("spatial", "SpatialApp")` (`src/spatial/dsl.scala:14-18`). The `dsl` object repeats the same `final class spatial` and private `AppTag("spatial", "SpatialApp")` companion (`src/spatial/dsl.scala:34-38`). The tag name is `"spatial"` and the expected application base is `"SpatialApp"` in both scopes (`src/spatial/dsl.scala:18`, `src/spatial/dsl.scala:38`).

`@struct` is Spatial-side syntax but argon-owned implementation. Both `libdsl.struct` and `dsl.struct` declare `macroTransform(annottees: Any*) = macro StagedStructsMacro.impl`, and `dsl.scala` imports `argon.tags.StagedStructsMacro` at the top of the file (`src/spatial/dsl.scala:3`, `src/spatial/dsl.scala:20-22`, `src/spatial/dsl.scala:40-42`). This means ordinary staged structs are generated by argon's struct macro, not by the stream-struct macro in `src/spatial/tags` (`src/spatial/dsl.scala:3-4`).

`@streamstruct` is Spatial-owned. `src/spatial/tags/StreamStructs.scala` declares a standalone `final class streamstruct extends StaticAnnotation` with `macroTransform` delegated to `StagedStreamStructsMacro.impl` (`src/spatial/tags/StreamStructs.scala:11-14`). The implementation builds Bits and Arith typeclass macro helpers, creates `MacroUtils`, and accepts either a class plus companion object or a class alone, synthesizing an empty companion object in the class-only case (`src/spatial/tags/StreamStructs.scala:29-45`). It reads constructor fields from the first constructor argument list, filters non-case-accessor methods, and records direct parent names (`src/spatial/tags/StreamStructs.scala:47-50`).

The macro rejects several shapes at expansion time. It aborts if the class has no fields, reports and aborts if it has disallowed non-accessor methods, aborts if it has type parameters, and aborts if any constructor field is a `var` (`src/spatial/tags/StreamStructs.scala:53-62`). It records field type metadata as field-name to `argon.lang.Bits[fieldType]`, records field names for construction, creates optional copy arguments with `null` defaults, and computes copy fallbacks through `Option(field).getOrElse{ this.field(ctx,state) }` (`src/spatial/tags/StreamStructs.scala:63-67`).

## Implementation

The generated stream-struct class is rebuilt as a zero-argument class extending `spatial.lang.StreamStruct[FullName]` (`src/spatial/tags/StreamStructs.scala:68-69`). For each constructor field, the macro injects a method that calls `field[fieldType](fieldName)(argon.lang.Bits[fieldType], ctx, state)`, so field access becomes the `StreamStruct.field` dequeue surface documented in [[40 - Streams and Blackboxes]] (`src/spatial/tags/StreamStructs.scala:70-75`, `src/spatial/lang/StreamStruct.scala:17-36`). The macro injects `lazy val fields = Seq(name -> Bits[type])`, mixes in `argon.Ref[Any, FullName]`, injects a `copy` method that calls the companion `apply`, and injects `box` evidence requiring `StreamStruct`, `Bits`, and `Arith` (`src/spatial/tags/StreamStructs.scala:76-87`).

The generated companion `apply` takes the original fields and calls `spatial.lang.StreamStruct[FullName](fieldName -> fieldValue, ...)`, passing `StreamStruct.tp[FullName]`, `ctx`, and `state` (`src/spatial/tags/StreamStructs.scala:89-95`). After class/object generation, the macro folds the Bits and Arith typeclass implementations over the class and companion, then calls `forge.tags.ref.implement` before returning `q"$cls4; $obj4"` (`src/spatial/tags/StreamStructs.scala:97-107`). This generated `Ref`/typeclass surface is why `SpatialCtrlBlackbox[A:StreamStruct,B:StreamStruct]` can use stream structs as typed controller ports (`src/spatial/lang/Blackbox.scala:23-30`, `src/spatial/tags/StreamStructs.scala:83-101`).

`SpatialApp` is the normal user app base, and it combines Spatial compiler behavior with argon's `DSLApp` (`src/spatial/SpatialApp.scala:1-5`). `SpatialTest` is the regression-test base and extends `Spatial with DSLTest with PlasticineTest`, while `SpatialTestbench` extends `argon.DSLTestbench` and overrides `initConfig` to return a `SpatialConfig` (`src/spatial/SpatialTest.scala:1-15`, `src/spatial/SpatialTestbench.scala:1-16`). These test variants are app-level entry points but not macro annotations in `dsl.scala` (`src/spatial/dsl.scala:14-25`, `src/spatial/dsl.scala:34-45`).

## Interactions

`@streamstruct` interacts directly with the stream/blackbox surface because the generated field methods call `StreamStruct.field`, and the generated companion calls `StreamStruct.apply` (`src/spatial/tags/StreamStructs.scala:70-75`, `src/spatial/tags/StreamStructs.scala:89-95`). `@struct` interacts with argon's ordinary struct machinery through `StagedStructsMacro.impl`, while `@spatial` interacts with forge's application tagging through `AppTag("spatial", "SpatialApp")` (`src/spatial/dsl.scala:3`, `src/spatial/dsl.scala:18`, `src/spatial/dsl.scala:38`). The two DSL objects interact with [[90 - Aliases and Shadowing]] because they expose the same macros through either the non-shadowing `libdsl` scope or the shadowing `dsl` scope (`src/spatial/dsl.scala:6-46`).

## HLS notes

`hls_status: rework`. These macros are Scala compiler plugins/macros, so an HLS or Rust port cannot reuse them directly (`src/spatial/dsl.scala:14-25`, `src/spatial/dsl.scala:34-45`). The behavior to preserve is the generated shape: app tagging to a `SpatialApp` base, ordinary staged structs through argon struct generation, and stream structs that reject methods, type parameters, and `var` fields while generating field-dequeue accessors and Bits/Arith/Ref evidence (`src/spatial/tags/StreamStructs.scala:53-107`). The port should encode those restrictions in its own derive/proc-macro layer or in a checked builder API (inferred, unverified).
