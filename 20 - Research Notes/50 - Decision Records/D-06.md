# D-06 — Iteration-Diff Analysis In The HLS Scheduler

## Question

Choose the role of Spatial iteration-diff analysis in the HLS scheduler.

Source: `src/spatial/traversal/IterationDiffAnalyzer.scala:17-169`; `src/spatial/traversal/InitiationAnalyzer.scala:23-41`.

Decision criteria: User decision: HLS scheduling reuses Spatial `iterDiff`, keeps it diagnostic-only, or replaces it with an HLS dependence/schedule model.

## Research findings

- `IterationDiffAnalyzer` is narrower than a general dependence analyzer. It runs inside banking analysis, looks for local-memory accumulation cycles, requires reader and writer affine matrices, uses Spatial counters and parallel-lane expansion, computes tick distances, writes `iterDiff`, and may also write `segmentMapping`. Its assumptions include Spatial local-memory metadata, conflict waivers, static counter bounds, affine matrices, retime gates, and Chisel-oriented consumers. See [[01-iteration-diff-algorithm]].

- `InitiationAnalyzer` turns `iterDiff` into II through a scalar formula. For inner controls it computes latency and interval, collects statement-level `iterDiff`s, forces II to `1.0` when any maximum diff is nonpositive, otherwise uses `ceil(interval / minIterDiff)` when the smallest positive distance exceeds one. Effective `II` then incorporates sequenced schedule and user-II overrides. See [[02-initiation-ii-semantics]].

- The metadata has multiple downstream lives. `iterDiff` directly affects `compilerII`; `II` and `bodyLatency` feed Chisel controller generation, pressure timing, Scala execution, runtime models, and reports. Separately, `segmentMapping` feeds banking conflict grouping, unrolling, latency/cycle modeling, and Chisel memory statistics. D-06 should not erase `segmentMapping` merely because `iterDiff` loses HLS scheduling authority. See [[03-pipeline-consumers]].

- Tests encode both performance expectations and correctness lessons. `InitiationIntervals` checks exact II in some compiler tests and contains cases labeled as iteration-diff edge cases; reductions, lane masks, and dynamic-programming apps depend on conservative loop-carried ordering. Exact Spatial IR specialization counts are legacy diagnostic parity, but loop-carried accumulation and lane-mask correctness must survive. See [[04-tests-app-usage]].

- HLS needs a richer schedule contract than a collapsed `iterDiff` scalar. Useful HLS-facing fields include requested II, Spatial-estimated II, per-edge dependence evidence, source/sink ops, memory, affine relation, distance vector, latency, segment/lane relation, explicit waivers, emitted pragmas, and backend accepted II. Spatial `iterDiff` can explain a requested II, but should not authorize dependence pragmas by itself. See [[05-hls-scheduler-implications]].

- Policy comparison rejects exact reuse as final authority. Exact reuse maximizes Spatial parity but imports the old assumptions into a backend whose partitioning, resource binding, and scheduler may disagree. Diagnostic-only reuse is safe but leaves the scheduler underspecified. Replacement with an HLS dependence/schedule model is the right destination, but expensive. A staged hybrid gives useful v1 behavior. See [[06-policy-options]].

- The specs already reveal the tension. [[60 - Use and Access Analysis]] tags this area as HLS rework; [[C0 - Retiming]] says retiming is Chisel-specific yet suggests `IterationDiffAnalyzer`/`InitiationAnalyzer` can emit pipeline II pragmas; [[70 - Timing Model]] says Rust should store compiler-requested and backend-accepted II separately. See [[07-spec-open-question-survey]].

- Cost/risk analysis favors the hybrid. Exact reuse risks miscompiles when old warnings become scheduling truth, and full replacement overlaps D-07, D-08, and D-21. The practical v1 path is to compute Spatial diagnostic/requested II, emit conservative requests and warnings, parse HLS accepted II when available, and reconcile reports before using final performance numbers. See [[08-cost-risk]].

- Precedent points toward dependence evidence plus backend feedback. HLS/DSL scheduling generally treats recurrence distance as a constraint in a broader scheduling problem, with user-requested II and backend-achieved II separated; external claims are marked unverified in the note. Spatial already has a `compilerII` versus `II` mismatch display in TreeGen, which is a local precedent for visible reconciliation. See [[09-precedent]].

- The recommendation matrix names the target architecture: hybrid requested-II/model-report reconciliation in v1, with a real HLS dependence model as the destination. It also names the fields later decisions should share: `spatial_iter_diff`, `spatial_compiler_ii`, `requested_ii`, `hls_accepted_ii`, `hls_report_source`, and `ii_reconciliation_status`. See [[10-recommendation-matrix]].

## Recommendation

**Adopt hybrid requested-II/model-report reconciliation for v1, with an HLS dependence/schedule model as the long-term authority.**

Spatial `iterDiff` should continue to run during the Rust/HLS transition, but it should not be the final HLS scheduler oracle. Use it to produce diagnostics, conservative requested II values, and compatibility reports. Treat the HLS backend's accepted II, once parsed from reports or returned by the tool flow, as the authoritative post-synthesis performance/scheduling result.

### V1 contract

For every HLS-scheduled controller, record these fields:

1. `spatial_iter_diff`
   - The Spatial-style per-statement/memory recurrence-distance evidence, including special values for `OpMemReduce` and `OpReduce`.

2. `spatial_compiler_ii`
   - The old `InitiationAnalyzer` estimate, including `ceil(interval / minIterDiff)`, force-II-1 behavior, and user/sequenced interactions as a diagnostic baseline.

3. `user_requested_ii`
   - The explicit `.II(...)` user request, if present.

4. `hls_requested_ii`
   - The II requested in emitted HLS pragmas/directives. In v1, derive this from the safer of user request, Spatial estimate, and conservative fallback policy.

5. `hls_accepted_ii`
   - The backend-achieved II from HLS schedule reports, synthesis reports, or tool APIs. Before reports exist, this is unknown, not silently equal to `hls_requested_ii`.

6. `ii_reconciliation_status`
   - One of `not-run`, `matched`, `accepted-higher`, `accepted-lower`, `tool-rejected`, or `unknown-report-format`, with a reason/source string.

7. `dependence_evidence`
   - Optional structured evidence: memory, reader, writer, affine relation, lane/segment relation, waiver provenance, and relevant source spans.

### What survives from Spatial

Keep `IterationDiffAnalyzer`-style logic for:

- warnings on unknown/static-bound failures,
- conservative fallback diagnostics,
- expected-II reports,
- regression comparisons against old Spatial tests,
- seed evidence for recurrence-distance explanations,
- preserving `segmentMapping` semantics until D-07 defines the HLS partition/banking replacement.

Do not use the collapsed `iterDiff` scalar alone to:

- prove an HLS dependence pragma is safe,
- override an HLS scheduler's accepted II,
- erase backend report discrepancies,
- decide final DSE latency after HLS reports are available.

### Long-term destination

Replace scheduler authority with an HLS dependence/schedule model that knows:

- HLS-visible loop structure,
- dependence edges and distance vectors,
- array partition/reshape/banking choices from D-07,
- operator and memory latencies from D-08,
- resource and memory-port limits,
- emitted pragmas/directives,
- backend accepted II and discrepancy reasons from D-21.

This is stronger than diagnostic-only carryover and safer than exact reuse. It preserves Spatial's hard-won recurrence lessons without letting old Chisel-era metadata masquerade as backend truth.

### Rejected alternatives

Reject exact reuse as the HLS scheduler authority. It would preserve old test expectations but imports Spatial's affine/tick/lane assumptions, incomplete dynamic-bound handling, and Chisel-oriented timing model into a different backend.

Reject pure diagnostic-only treatment as the final policy. It is useful as a fallback, but it leaves v1 without a concrete requested-II mechanism and punts too much to the HLS tool without explaining the compiler's intent.

Reject immediate full replacement as the v1 requirement. It is the right destination, but it depends on D-07 partition planning, D-08 latency sources, and D-21 accepted-II reporting, so requiring it immediately would block the rewrite on too many unresolved interfaces.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Add HLS scheduling metadata fields: `spatial_iter_diff`, `spatial_compiler_ii`, `user_requested_ii`, `hls_requested_ii`, `hls_accepted_ii`, `hls_report_source`, and `ii_reconciliation_status`.
2. Preserve existing Spatial `iterDiff` and `segmentMapping` computation as a diagnostic/requested-II input during v1.
3. Emit HLS pipeline II requests from the hybrid policy, with explicit diagnostics for unknown bounds, loose iteration-diff assumptions, and user-II overrides.
4. Keep `segmentMapping` available to D-07 until the HLS partition/banking model defines an equivalent or replacement.
5. Add report parsing hooks that can update `hls_accepted_ii` without mutating away the original requested/diagnostic values.
6. Update runtime/DSE summaries to prefer `hls_accepted_ii` when available and fall back to requested/Spatial estimates only when reports are absent.
7. Add mismatch reports modelled after the existing `compilerII != II` TreeGen display.
8. Add regression tests covering exact Spatial II baseline, user `.II`, unknown bounds, `OpMemReduce` diff zero, ordinary reduce diff one, segmented lane dependencies, and HLS report mismatch states.
9. Cross-reference this record in D-07, D-08, and D-21 so banking, latency, and accepted-II policy use the same field names.
