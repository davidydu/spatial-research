# D-16 Angle 7: Memory, Banking, and Addressing OOB Surfaces

## Scope

This angle answers where Rust must look for out-of-bounds behavior, not only what the policy should be after it is found. The existing contract is clear on the simulator side: Scalagen/emul logs OOB reads and writes, returns invalid on reads, discards writes, and does not abort (`Spatial Research/10 - Spec/20 - Semantics/40 - Memory Semantics.md:61-69`; `Spatial Research/20 - Research Notes/20 - Open Questions.md:2027-2037`). Rust should not rely on ordinary Rust indexing as the policy boundary: Rust slice/array indexing performs a runtime check that panics on failure, while `get` exposes an `Option`-style checked access path, so the simulator needs explicit checked access to preserve log/invalid/discard behavior ([Rust Reference: array index expressions](https://doc.rust-lang.org/reference/expressions/array-expr.html#array-and-slice-indexing-expressions); [Rust `slice::get`](https://doc.rust-lang.org/std/primitive.slice.html#method.get)).

## Local Memory Accesses

Every enabled SRAM banked read/write is an OOB candidate. The runtime shape is two-dimensional: `BankedMemory.apply` and `update` flatten the bank coordinate, then index the selected bank by offset (`/Users/david/Documents/David_code/spatial/emul/src/emul/BankedMemory.scala:23-43`). Therefore Rust diagnostics should cover both "bad bank coordinate" and "bad bank offset" per lane. Simulator mode should report the memory, lane, bank tuple, offset, enable, and source context; reads return invalid and writes are discarded. Synthesis mode should assert, gated by the lane enable, that the computed bank is in `0 until totalBanks` and offset is in that bank's depth.

Vector loads/stores are the same issue through a flat surface. RegFile reads/writes and LUT reads lower through `emitVectorLoad` / `emitVectorStore` to `ShiftableMemory.apply/update` (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenRegFile.scala:25-26`; `/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenLUTs.scala:14-16`; `/Users/david/Documents/David_code/spatial/emul/src/emul/ShiftableMemory.scala:34-64`). Rust should check the row-major flattened address for every enabled lane. RegFile shift-in is also address-bearing: even though the axis loop is internal, the non-axis indices and the derived source/destination addresses can still leave the flat array. Plain `Reg`/`Ptr` state is not an OOB surface because it has no user address (`Spatial Research/10 - Spec/50 - Code Generation/20 - Scalagen/30 - Memory Simulator.md:90-92`).

## Line Buffers and Transfers

Line buffers need their own diagnostic label because row addressing is circular but column addressing is still finite. Reads compute `(bank row + readRow) % fullRows` and then flatten column bank plus offset; writes use `posMod` for the row and advance `wrCounter`, then update a flattened column location (`/Users/david/Documents/David_code/spatial/emul/src/emul/LineBuffer.scala:20-59`). Rust should treat enabled read/write lane OOB in the column/offset plane as memory OOB, while separately reporting impossible line-buffer state if `fullRows`, `banks`, or `wrCounter` invariants are malformed.

DRAM transfers are OOB surfaces even when they are not ordinary local-memory nodes. `Memory.apply/update` directly index the host array (`/Users/david/Documents/David_code/spatial/emul/src/emul/Memory.scala:17-24`), and Scalagen wraps `SetMem`, `GetMem`, dense loads/stores, and sparse loads/stores in the generated `oob` helper (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenDRAM.scala:30-83`; `/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenMemories.scala:36-58`). Dense transfers should check `offset`, `size`, byte-to-word conversion, and burst padding; sparse transfers should check each supplied address after byte-to-word conversion. Dense/sparse aliases themselves do not index memory: `MemDenseAlias` and `MemSparseAlias` select a memory or return `null` when no condition matches (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenDRAM.scala:86-102`). Rust should diagnose unresolved alias selection separately from OOB, then apply OOB policy to the selected target.

## Addressing and Disabled Lanes

The most important implementation rule is to check the same coordinates that the backend uses. Bank choice comes from `bankSelects`, and within-bank offsets come from `bankOffset`, including block-cyclic modulo and intrablock arithmetic (`/Users/david/Documents/David_code/spatial/src/spatial/metadata/memory/BankingData.scala:49-65`; `/Users/david/Documents/David_code/spatial/src/spatial/metadata/memory/BankingData.scala:202-254`). Rust should keep these computations in signed/integer form until after diagnostics; casting a negative bank or offset to `usize` would hide the bug behind wraparound. Diagnostics should include abstract address, bank coordinates, offset, memory dimensions, bank counts, and lane id.

Disabled lanes should not be asserted as OOB. The emul wrappers receive the enable, but the actual array touch is guarded inside the thunk: disabled banked and shiftable reads return invalid without indexing, and disabled writes do nothing (`/Users/david/Documents/David_code/spatial/emul/src/emul/OOB.scala:19-38`; `/Users/david/Documents/David_code/spatial/emul/src/emul/BankedMemory.scala:28-43`; `/Users/david/Documents/David_code/spatial/emul/src/emul/ShiftableMemory.scala:39-64`). Therefore simulator diagnostics should suppress OOB logs for disabled lanes, and synthesis assertions should be `if en { assert(in_bounds) }`. Optional debug may still record disabled computed addresses, but it must not turn them into failures.

## Rust Policy

Recommended coverage: simulator mode uses checked helpers for every enabled SRAM, RegFile, LUT, LineBuffer, DRAM dense/sparse transfer, and selected alias target; it emits structured OOB diagnostics and preserves invalid/discard results. Synthesis mode emits gated assertions for the same addressable surfaces, plus non-OOB assertions for unresolved aliases and malformed line-buffer/banking metadata. This matches the current spec split: simulator parity for tests, stricter assertions or in-bounds assumptions for hardware (`Spatial Research/10 - Spec/50 - Code Generation/20 - Scalagen/30 - Memory Simulator.md:104-116`; `Spatial Research/30 - HLS Mapping/40 - Open HLS Questions.md:18`).
