# D-16 Angle 5: Tests/Apps Evidence for OOB and Invalid Values

## Claim

The tests/apps corpus gives narrow but real evidence for ScalaGen-compatible out-of-bounds behavior: OOB writes should not abort, and the in-bounds result should remain observable. It does not give equivalent direct coverage for OOB reads, OOB log contents, or synthesized hardware behavior. The adjacent bug corpus is much broader: invalid values appear in FIFO/stream empty-read fallbacks, disabled lanes, vector invalid-codegen, memory padding, and disabled/weak tests. Those should inform Rust's policy, but they are not all direct OOB evidence.

## Direct OOB Evidence

`OOBTest` is the only direct test/apps hit I found for `OOB`, `ArrayIndexOutOfBounds`, or `invalid`-by-name. It allocates `DRAM[Int](13)` and `SRAM[Int](13)`, loads 13 elements, then runs `Foreach(15 by 1){ i => x(i) = i }` and stores `x` back to DRAM. The golden is exactly `Array.tabulate(13){ i => i }`, so the test intentionally writes indices 13 and 14 beyond the SRAM extent and expects the run to complete with only the legal writes reflected (`test/spatial/tests/feature/unit/OOBTest.scala:8-18`). This proves "OOB write is tolerated/discarded" for the Scala simulator path; it does not prove OOB read invalid-value propagation.

The implementation backs that reading. `OOB.readOrElse` catches `java.lang.ArrayIndexOutOfBoundsException`, logs `[OOB]`, and returns the caller-provided invalid value; `writeOrElse` catches the same exception, logs, and performs no write (`emul/src/emul/OOB.scala:19-38`). ScalaGen also emits an access wrapper that catches `ArrayIndexOutOfBoundsException`, prints a warning, and emits `invalid(tp)` for reads (`src/spatial/codegen/scalagen/ScalaGenMemories.scala:36-50`). The open question records the policy split explicitly: Rust must choose Scalagen-compatible log/invalid behavior, synthesis assertions, or separate simulator/synthesis modes (`20 - Research Notes/20 - Open Questions.md:2027-2037`; `20 - Research Notes/40 - Decision Queue.md:71-73`).

## Adjacent Invalid-Value Evidence

FIFO, LIFO, and stream invalid behavior is adjacent, not identical, evidence. ScalaGen FIFO `peek` returns `invalid(A)` when empty, and banked dequeue returns `invalid(A)` when a lane is disabled or the queue is empty (`src/spatial/codegen/scalagen/ScalaGenFIFO.scala:30-39`). LIFO mirrors this for `peek` and banked `pop` (`src/spatial/codegen/scalagen/ScalaGenLIFO.scala:28-37`), and stream reads do the same per lane (`src/spatial/codegen/scalagen/ScalaGenStream.scala:84-90`). The spec notes this as reference simulation behavior: elastic queues, no enqueue back-pressure, invalid-on-empty (`10 - Spec/20 - Semantics/80 - Streaming.md:57-67`).

The tests mostly validate guarded use, not intentionally invalid reads. `FIFOBranch` drains `fifo1` until `isEmpty`, then drains `fifo2`, so it demonstrates empty checks before dequeue rather than relying on invalid-on-empty (`test/spatial/tests/feature/control/FIFOBranch.scala:21-35`). `FSMFIFOStack` similarly transitions from fill to drain and then to done using `isFull`, `isEmpty`, `isAlmostFull`, and `isAlmostEmpty` (`test/spatial/tests/feature/control/fsm/FSMFIFOStack.scala:20-48`). This matters because the newer Scala executor is stricter: `ScalaQueue.deq()` throws `SimulationException` on empty FIFO, and `ScalaTensor.flattenIndex` throws on OOB indices (`src/spatial/executor/scala/memories/ScalaQueue.scala:21-24`; `src/spatial/executor/scala/memories/ScalaTensor.scala:26-33`). So invalid-on-empty is a ScalaGen compatibility point, not a universal simulator invariant.

## Bug Corpus and Padding

`ScalaGenVec.invalid` is a direct invalid-value bug, but not direct OOB evidence. The emitted string is missing a parenthesis, so `invalid(Vec[A])` would generate malformed Scala (`src/spatial/codegen/scalagen/ScalaGenVec.scala:14-17`). The open question asks whether any tests reach vector invalid; it observes that invalid vectors may only appear through OOB or switch fallbacks (`20 - Research Notes/20 - Open Questions.md:1929-1949`). Nearby vector tests exercise normal values: `VecInStruct` asserts field extraction, while `FIFOVecType` uses a vector-containing FIFO but comments out its real assertion and ends with `assert(1.to[Bit])` (`test/spatial/tests/compiler/VecTypes.scala:3-30`; `test/spatial/tests/compiler/VecTypes.scala:33-69`).

Memory padding is also adjacent. `expandInits` pads initialized memories, but padded positions copy `inits.get.head` rather than zero or invalid (`src/spatial/codegen/scalagen/ScalaGenMemories.scala:69-87`). Q-126 flags this as unresolved and asks whether padded regions are guaranteed read-masked (`20 - Research Notes/20 - Open Questions.md:1741-1759`). Apps contain algorithmic padding, but not OOB-policy tests: convolution builds explicit zero-padded host data, Sobel zeroes image borders, BTC pads SHA blocks, and CHStone/MachSuite comments call out padding concerns (`test/spatial/tests/feature/dense/Convolutions.scala:127-135`; `test/spatial/tests/feature/dense/Sobel.scala:17-33`; `test/spatial/tests/apps/BTC.scala:168-173`; `test/spatial/tests/apps/CHStone.scala:1349-1354`).

## Policy Implication

Rust should separate direct OOB policy from the larger invalid-value bug corpus. For regression parity, provide a ScalaGen-compatible simulator mode: log OOB, return invalid on reads, discard writes, and do not abort. For synthesis/HLS, add an assertion or assumption mode rather than silently synthesizing invalid-value fallback hardware. The missing tests are clear: OOB read-to-invalid, log assertions, empty FIFO invalid reads, vector invalid generation, and padded-region reads.
