# D-16 Risk Analysis And Failure Modes

## Compatibility Risks

The Scalagen-compatible policy is the easiest way to preserve legacy behavior, but it is also the easiest way to hide real design bugs. Current emul behavior catches only `ArrayIndexOutOfBoundsException`: OOB reads log and return the caller's invalid value, while OOB writes log and disappear (`emul/src/emul/OOB.scala:19-38`; `src/spatial/codegen/scalagen/ScalaGenMemories.scala:36-50`). That matches existing evidence such as `OOBTest`, where writes to indices 13 and 14 of a 13-element SRAM are expected not to abort and not to affect the in-bounds result (`test/spatial/tests/feature/unit/OOBTest.scala:8-18`). The failure mode is obvious: a scheduling, banking, or bound-analysis bug can become a green simulator run with a line in `./logs/`.

Write discard is especially risky because it is not a hardware model. A discarded simulator write can make a test look stable, while synthesized logic may drive a wrong bank, wrong offset, or external-memory transaction. Reads are only slightly better: returning invalid makes the failure visible only if downstream code preserves validity. Logs themselves are a weak diagnostic contract. Current OOB logging uses process-level `./logs/reads.log` and `./logs/writes.log` streams opened and closed around generated execution (`src/spatial/codegen/scalagen/ScalaGenMemories.scala:14-21`; `emul/src/emul/OOB.scala:6-17`). A Rust simulator with threads, vector lanes, or replay can make global text-log ordering nondeterministic unless it records stable event IDs.

## Strict And Assertion Risks

Strict trap mode fixes bug hiding but breaks Scalagen parity. Safe Rust indexing naturally points in this direction: array and slice indexing performs a runtime check and panics if the check fails ([Rust Reference: array indexing](https://doc.rust-lang.org/reference/expressions/array-expr.html#array-and-slice-indexing-expressions)). That is useful for HLS-conformance CI, but it would fail legacy tests that intentionally rely on tolerated OOB writes. A trap policy also stops at the first failure, so it can obscure multi-lane or multi-memory failure patterns unless paired with a collect mode.

Synthesis assertions are not a free middle ground. If emitted as real hardware checks, they add address comparators, enable gating, lane reduction, and some reporting path; in heavily unrolled banked memories, this cost multiplies by lane. If emitted as ordinary C/C++ `assert`, they may be disabled under `NDEBUG`, and AMD's Vitis HLS 2021.2 documentation says the C/C++ `assert` macro is not supported and can create bad logic or block optimization ([C/C++ `assert` and `NDEBUG`](https://en.cppreference.com/w/cpp/error/assert); [AMD UG1399 assertions](https://docs.amd.com/r/2021.2-English/ug1399-vitis-hls/Assertions)). Hiding assertions behind `__SYNTHESIS__` is also dangerous: AMD warns that using that macro to change functionality can make C simulation and synthesis diverge in hard-to-debug ways ([AMD system calls / `__SYNTHESIS__`](https://docs.amd.com/r/en-US/ug1399-vitis-hls/System-Calls)).

## Undefined And Invalid Propagation

`assume_inbounds` is the highest-performance policy and the sharpest knife. In Rust host code, replacing checked access with `get_unchecked` makes an out-of-bounds index undefined behavior even if the reference is never used ([Rust slice `get_unchecked`](https://doc.rust-lang.org/std/primitive.slice.html#method.get_unchecked)). The Rust Reference also treats place projections that violate in-bounds pointer arithmetic requirements as undefined behavior ([Rust undefined behavior](https://doc.rust-lang.org/reference/behavior-considered-undefined.html)). Therefore `assume_inbounds` must never mean "run unchecked and hope"; it must mean "proved or explicitly waived, with violation outside the defined result."

Invalid propagation is a second contamination path. Scalagen invalids are not just `None`: fixed invalid is a value plus `valid=false`, float invalid is `NaN` plus `valid=false`, and bit invalid is `Bool(false,false)` (`src/spatial/codegen/scalagen/ScalaGenFixPt.scala:26-28`; `src/spatial/codegen/scalagen/ScalaGenFltPt.scala:32-34`; `src/spatial/codegen/scalagen/ScalaGenBit.scala:21-23`; `emul/src/emul/FixedPoint.scala:161`; `emul/src/emul/FloatPoint.scala:287`). Numerics that preserve validity can surface the X; numerics that accidentally unwrap, cast, compare, or pack only the payload can turn an OOB read into plausible data. Vector lanes add another edge case: each enabled lane needs its own OOB status, and `ScalaGenVec.invalid` is already flagged as malformed compatibility surface (`src/spatial/codegen/scalagen/ScalaGenVec.scala:14-16`).

## Mitigations And Diagnostics

The least risky architecture is separate named modes with shared diagnostics. Keep `compat_log_invalid` for regression parity, but mark any run with OOB as contaminated. Use `strict_trap` for tests that claim hardware legality. Use `assert_collect` to continue after failures while recording all bad accesses. Use `synth_assert` only when the emitted HLS/RTL assertion mechanism is specified. Use `assume_inbounds` only after static proof or manifest waiver.

Diagnostics should be structured before they are textual. Each event should include mode, memory id/name, source context, static op id, dynamic instance, lane id, enable, abstract indices, bank, offset, flattened address, extent, in-bounds result, returned value, and whether a write committed. Disabled lanes should record optional debug coordinates but must not assert or log OOB, matching current emul thunks that avoid touching storage when disabled (`emul/src/emul/BankedMemory.scala:28-43`; `emul/src/emul/ShiftableMemory.scala:39-64`). Finally, every synthesis artifact should record whether OOB was checked, asserted, or assumed; otherwise a clean simulator run can still be a hardware bug report waiting to happen.
