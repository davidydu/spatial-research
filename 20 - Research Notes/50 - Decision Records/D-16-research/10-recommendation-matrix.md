# D-16 Recommendation Matrix

## Frame

Q-142 is a mode policy question, not just a bounds-checking question. Current Scalagen/emul behavior is recoverable: enabled OOB reads log and return the caller's invalid value, while enabled OOB writes log and discard the write (`emul/src/emul/OOB.scala:19-38`; `src/spatial/codegen/scalagen/ScalaGenMemories.scala:36-50`). The vault already records that [[30 - Memory Simulator]] is the simulator reference, but synthesis may instead assert or assume in-bounds access ([[40 - Memory Semantics]]; [[03-spec-open-question-contract]]). The decision should therefore name both the default user experience and the HLS contract.

## Matrix

| Option | Correctness | Compatibility | HLS safety/performance | Implementation cost | Diagnostics/provenance | Testing |
|---|---|---|---|---|---|---|
| `compat_log_invalid` default | Correct for Scalagen replay; weak as hardware truth because invalid values are simulator artifacts. | Highest: matches `OOBTest` discard semantics and legacy logs ([[05-tests-apps-evidence]]). | Unsafe as synthesis default; synthesized invalid recovery would add guards, muxes, and lane logic. | Medium: requires typed invalid values and checked helpers on every access surface. | Good if structured records mirror `reads.log`/`writes.log`; provenance must say "compat". | Needs OOB read invalid, write discard, disabled lane, log, vector invalid, and transfer tests. |

## Readout

The default should not be `strict_trap` alone: it would make the Rust simulator more hardware-shaped, but it would erase the reference behavior documented in [[30 - Memory Simulator]] and directly exercised by `OOBTest` ([[05-tests-apps-evidence]]). Conversely, `compat_log_invalid` alone is too permissive for HLS: current Chisel/HLS-facing memory paths do not have logfile or typed-invalid recovery semantics, and OOB hardware access is at best implementation-dependent ([[04-hls-synthesis-assertions]]; [[07-memory-banking-addressing-surfaces]]).

The most defensible base policy is explicit separation, with `synth_assert plus sim compat` as the first concrete profile. Rust simulation keeps Scalagen parity by default or by a clearly named `compat_log_invalid` profile. HLS checked builds emit enabled-lane bounds assertions over the same bank/offset/transfer coordinates the simulator checks. Optimized release builds may use `assume_inbounds`, but only when the manifest records static proof or an explicit waiver.

## Preliminary Recommendation

Preliminarily choose **separate simulator/synthesis modes**, implemented first as **sim compat plus checked synthesis assertions**. Keep `strict_trap` as a CI/conformance simulator profile, not the legacy default. Add `assume_inbounds` only as a labelled optimized mode, and use compare/differential mode for migration triage across tests/apps. This preserves replay compatibility while making HLS safety and performance decisions auditable instead of hidden behind one overloaded "Rust behavior" switch.
