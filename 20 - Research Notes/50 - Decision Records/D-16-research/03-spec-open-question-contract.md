# D-16 Angle 3: Spec/Open-Question Contract

## Question Anchor

Q-142 asks for the Rust policy for out-of-bounds memory access across simulator and synthesis. The consolidated question says Scalagen logs OOB reads and writes, returns invalid values for reads, discards writes, and that "hardware behavior is not naturally the same" (`Spatial Research/20 - Research Notes/20 - Open Questions.md:2027-2037`). The decision queue restates the same D-16 surface: choose whether Rust offers Scalagen-compatible OOB log/invalid behavior, synthesis assertions, or separate simulator/synthesis modes (`Spatial Research/20 - Research Notes/40 - Decision Queue.md:71-73`). The HLS mapping index keeps the question open in HLS terms: preserve simulator fallbacks, synthesize guards, or leave OOB undefined (`Spatial Research/30 - HLS Mapping/40 - Open HLS Questions.md:18`).

## Written Contract

The current spec already gives the simulator side a strong answer. `Memory Semantics` says [[30 - Memory Simulator]] is normative for runtime behavior, "especially out-of-bounds access" (`Spatial Research/10 - Spec/20 - Semantics/40 - Memory Semantics.md:39-41`). Its reference implementation section says runtime memory behavior is defined by Scalagen and `emul`, then names the OOB contract directly: `OOB.readOrElse` logs and returns the caller's invalid value; `OOB.writeOrElse` logs and discards; neither aborts (`Spatial Research/10 - Spec/20 - Semantics/40 - Memory Semantics.md:61-65`). The same paragraph also records the generated-code wrapper: memory operations catch `ArrayIndexOutOfBoundsException`, print a warning, and substitute `invalid(tp)` on read (`Spatial Research/10 - Spec/20 - Semantics/40 - Memory Semantics.md:63-65`).

The Scalagen memory simulator entry is even more explicit. Its summary calls itself the reference semantics anchor and says Rust+HLS must reproduce "OOB-checked access with logging-and-fallback behavior" (`Spatial Research/10 - Spec/50 - Code Generation/20 - Scalagen/30 - Memory Simulator.md:30-32`). Its ground-truth list says OOB reads return `invalid(tp)`, log, and do not abort; OOB writes are discarded, logged, and do not abort (`Spatial Research/10 - Spec/50 - Code Generation/20 - Scalagen/30 - Memory Simulator.md:104-112`). Its HLS note then draws the current boundary: Rust simulation should match Scalagen for testing parity, while synthesis-mode should emit asserts (`Spatial Research/10 - Spec/50 - Code Generation/20 - Scalagen/30 - Memory Simulator.md:114-116`). `Memory Semantics` phrases the synthesis side a little more loosely: synthesis-mode code may emit assertions or assume in-bounds access (`Spatial Research/10 - Spec/20 - Semantics/40 - Memory Semantics.md:67-69`).

## Source Evidence

The implementation matches the simulator contract. `OOB.readOrElse` executes the read thunk, logs successful enabled reads, catches `ArrayIndexOutOfBoundsException`, logs `[OOB]`, and returns `invalid` (`/Users/david/Documents/David_code/spatial/emul/src/emul/OOB.scala:19-28`). `OOB.writeOrElse` likewise executes the write, logs success, and on OOB only logs, with no rethrow and no replacement write (`/Users/david/Documents/David_code/spatial/emul/src/emul/OOB.scala:30-37`). `BankedMemory.apply` wraps each lane in `OOB.readOrElse`; disabled lanes return invalid without touching data, and enabled lanes index bank then offset (`/Users/david/Documents/David_code/spatial/emul/src/emul/BankedMemory.scala:28-33`). `BankedMemory.update` wraps writes in `OOB.writeOrElse` (`/Users/david/Documents/David_code/spatial/emul/src/emul/BankedMemory.scala:37-42`). `ShiftableMemory` uses the same OOB wrappers for flat/vector reads and writes (`/Users/david/Documents/David_code/spatial/emul/src/emul/ShiftableMemory.scala:39-58`).

Generated Scala adds a second layer for operations such as DRAM transfers. `ScalaGenMemories.emitPreMain` and `emitPostMain` open and close OOB logs (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenMemories.scala:14-21`). Its private `oob` helper emits a `try`/`catch` around generated memory code, prints an out-of-bounds warning, and emits `invalid(tp)` only for reads (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenMemories.scala:36-49`). DRAM `SetMem` and `GetMem` compose `oobUpdate` and `oobApply` around host-memory copies (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenDRAM.scala:30-40`).

## Remaining Ambiguity

The ambiguity is not "what does current Scalagen do?" It is whether the Rust system has one reference behavior or an explicit mode split. The written spec favors a split, but uses three policies without pinning defaults or obligations: simulator parity, synthesized assertions, and assume-in-bounds/undefined synthesis behavior. Q-142 is still marked `needs-architectural-decision` and blocked by the simulator/synthesis mode split (`Spatial Research/20 - Research Notes/20 - Open Questions.md:2032-2037`), so the note should not be considered resolved just because the simulator contract is clear.

There is also a related invalid-value coverage risk. Q-135 records that `ScalaGenVec.invalid` appears to emit malformed Scala and asks when `invalid(Vec[A])` is reached, noting OOB and switch fallbacks as likely paths (`Spatial Research/20 - Research Notes/20 - Open Questions.md:1929-1947`). The source confirms the malformed `Array.fill` expression (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenVec.scala:14-16`). D-16 therefore should require the Rust contract to define invalid values for every simulatable element type, including vector elements, rather than inheriting Scalagen's codegen typo.

## Decision Surface

The clean contract would say: Rust's default simulator mode is Scalagen-compatible and non-aborting for OOB, with structured logs and invalid propagation; synthesis mode either inserts explicit assertions/guards or documents OOB as an in-bounds precondition. The decision must specify the user-visible mode flag, default behavior for tests, whether synthesis assertions are simulated-only or emitted into HLS C/C++, and whether optimized synthesis may remove guards after proof of in-bounds access. Without that split, "Rust matches Spatial" will mean different things to simulator users and hardware users.
