# D-12 — Rust Ownership For Spatial Mutable Aliases

## Question

Choose how Rust ownership handles Spatial mutable aliases.

Source: `argon/src/argon/static/Staging.scala:39-75`; `argon/src/argon/Config.scala:46-47`; [[30 - Effects and Aliasing]].

Decision criteria: User decision: Rust IR preserves the `enableMutableAliases` compatibility flag or makes mutable aliasing a hard error.

## Research findings

- Argon's base checker is narrow and precise. `checkAliases` computes immutable writes separately from mutable-alias candidates, rejects mutable aliases only when `aliases.nonEmpty && !config.enableMutableAliases`, and always rejects writes to immutable symbols. The mutable-alias candidate set is `(sym.mutableAliases diff effects.writes) diff Set(sym)`, so aliases written by the current op are intentionally exempted. See [[01-argon-current-alias-check]].

- Spatial deliberately changes Argon's default. Argon initializes `enableMutableAliases = false`, but `Spatial.settings()` sets it to true with the comment that Spatial allows mutable aliases for DRAM, SRAM, and similar memories. This is a language-level policy, not an app-local workaround. See [[02-spatial-settings-usage]].

- Memory views are first-class surface syntax and first-class IR nodes. Range views, row/column/region views, `.par` views, dense aliases, and sparse gather/scatter aliases all stage mutable alias nodes that point back to an underlying memory. `NestedAlias`, transfer tests, `HelloSpatial`, inference kernels, gather/scatter tests, and helper-passed SRAM examples show that the corpus expects these forms to compile. See [[02-spatial-settings-usage]] and [[06-tests-apps-corpus]].

- The alias model is graph metadata plus effect propagation, not Rust `&mut` ownership. Staging records effects and shallow/deep alias metadata, registers reverse aliases, runs flow rules, then checks aliases. `computeEffects` expands writes through alias closure, adds reads of mutable inputs, and builds anti-dependencies. Transformers recompute ignored effect/alias data through normal staging rather than blindly transferring it. See [[03-effects-alias-propagation]].

- HLS risk comes from losing alias facts, not from aliases existing. The current Scala pipeline can allow aliases because it expands `reads`, `writes`, and `antiDeps`, rewrites aliases through `MemoryDealiasing`, and later banks/allocates concrete memories. A Rust/HLS backend that treats multiple mutable handles as independent arrays risks false parallelism, missed dependencies, or wrong banking. See [[05-hls-scheduling-risk]].

- The written spec already leans toward preserving the graph contract. The high-level semantics note says mutable aliases may exist only when explicitly configured and warns that Spatial aliases should not be replaced by borrow-checker intuition. The lower-level Argon note says `enableAtomicWrites` and `enableMutableAliases` should be consumed as config or feature flags, but Q-137 remained open because the Rust ownership policy had not been confirmed. See [[07-spec-open-question-contract]].

- A blanket hard error is cheap in the checker but expensive in product behavior. It would reject normal Spatial views, nested aliases, DRAM slice transfers, `.par` aliases, and sparse gather/scatter patterns unless the Rust port first reclassified them as special scoped views. Allow-listing memory aliases is closer, but it requires provenance that the current `mutableAliases` set alone does not retain. See [[08-migration-compatibility-strategy]] and [[09-implementation-cost-matrix]].

- The best long-term design is an alias-capability/provenance layer, but it is too large to require up front. Capability tokens could carry base memory, range, access mode, source span, and disjointness information for HLS dependence and partitioning. The practical v1 path is to preserve compatibility while collecting enough provenance to migrate toward that model. See [[04-rust-ownership-options]], [[09-implementation-cost-matrix]], and [[10-recommendation-matrix]].

## Recommendation

Adopt **Spatial Alias Graph Compatibility With HLS-Resolved Boundaries**.

The Rust IR should preserve the `enableMutableAliases` behavior as a named alias policy, not replace it with Rust's native exclusive-`&mut` intuition and not delete it as a hard language error. A suitable policy surface is:

- `alias_policy = strict_argon`: mutable aliases are errors using the Argon candidate formula.
- `alias_policy = spatial_compat`: mutable-alias diagnostics are recorded but nonfatal, matching Spatial's current global setting.
- `alias_policy = migration_warn`: same accepted programs as `spatial_compat`, but warnings are promoted and summarized for cleanup.
- `alias_policy = capability_checked`: future mode requiring explicit alias provenance/capabilities for mutable writes.

For the Spatial frontend, default to `spatial_compat` in the first Rust implementation. That preserves existing programs and honors the fact that memory views are a core Spatial construct. The compatibility mode must still run the full alias/effect pipeline: compute the same candidate set, preserve the exception for aliases already in `effects.writes`, expand writes through alias closure, add mutable-input reads, compute anti-dependencies, and keep immutable writes as hard errors. The flag suppresses one legality error; it must not suppress alias reasoning.

Add an HLS boundary rule: mutable aliases may exist in the Rust IR only while they are represented as Spatial alias graph facts. Before HLS emission, every mutable memory alias must either be dealiased into base-memory accesses with conditions/ranges/sparse addresses and dependence information, or rejected as an unresolved mutable alias. This keeps compatibility from becoming a backend-wide permission to emit independent pointers for the same storage.

Diagnostics should be unconditional. Even in `spatial_compat`, the compiler should record the current symbol, alias definitions, base memory, alias op kind, range or sparse-address provenance, whether the alias was excluded because it was in `effects.writes`, and the policy that allowed it. This gives users a migration report and gives the later `capability_checked` mode a data model to build on.

Reject **blanket hard error** as the v1 Spatial policy. It aligns with Argon's base config, but not with Spatial's actual language configuration or corpus. It would make accepted range views, `.par` views, nested aliases, and gather/scatter patterns illegal before the Rust port has an alternate representation.

Reject **warning-only with no HLS boundary**. It preserves compatibility but leaves the backend exposed to silent dependence and banking errors if alias metadata is lost.

Reject **immediate capability-only ownership** as a launch requirement. It is the right end state, but it would require new semantics for token creation, lifetime, transfer, range proof, disjointness, and lowering before the first Rust/HLS implementation can match existing Spatial behavior.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:

1. Add `alias_policy` to Rust compiler configuration with `strict_argon`, `spatial_compat`, `migration_warn`, and reserved `capability_checked` values.
2. Implement Argon-compatible alias metadata, reverse-alias registration, write propagation, mutable-input reads, anti-dependency computation, and post-flow alias checking.
3. Preserve immutable-write errors as unconditional hard errors in every alias policy.
4. In `spatial_compat`, demote only the mutable-alias error to a recorded diagnostic; do not skip effect/alias propagation.
5. Add provenance to every mutable alias diagnostic: `SymId`, source span, base memory, alias op kind, ranges or sparse addresses, conditions, current `effects.writes`, and policy.
6. Require HLS lowering to consume dealiased base-memory accesses or reject unresolved mutable aliases before emitting HLS arrays/pointers/pragmas.
7. Add tests for the exact Argon candidate formula, co-written alias exemption, immutable-write independence, Spatial range views, nested aliases, `.par` views, sparse gather/scatter aliases, and unresolved-HLS-boundary rejection.
8. Defer `capability_checked` until alias tokens/provenance have a formal schema and migration tests.
