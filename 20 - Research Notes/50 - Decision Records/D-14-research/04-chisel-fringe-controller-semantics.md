# D-14 Angle 4: Chisel/Fringe Controller Semantics

## Bottom line

For the Rust/HLS scheduler, `ForkJoin` should be modeled as concurrent child issue plus an all-child per-iteration join, while `Fork` should be modeled as mutually exclusive selected-branch issue plus completion when the selected branch finishes. This is stronger than the enum names: Fringe only defines five hardware schedules, `Sequenced`, `Pipelined`, `Streaming`, `Fork`, and `ForkJoin` (`fringe/src/fringe/templates/Controllers.scala:62-67`). `PrimitiveBox` exists in Spatial control metadata (`src/spatial/metadata/control/ControlData.scala:7-14`) but not as a Fringe controller schedule. Any hardware-level rule below is an inference from Chisel/Fringe wiring, not a separately verified formal spec.

## Controller interface and child protocol

The reusable controller interface exposes parent enable/done/reset, counter control, parent acknowledgement, backpressure, break, child `doneIn`/`maskIn`, child `enableOut`/`childAck`, switch selects, FSM state, and streaming counter-copy done inputs (`fringe/src/fringe/templates/Controllers.scala:9-43`). Generated kernels wire these ports through `InputKernelSignals` and `OutputKernelSignals`: child done/mask flow from the concrete child body to the parent state machine, while parent child-enable and child-ack flow back down (`fringe/src/fringe/SpatialBlocks.scala:47-71`, `fringe/src/fringe/SpatialBlocks.scala:128-131`). The parent also reports each kernel's `done` and `mask` to its own parent and gates datapath enable by the controller datapath signal, mask, and counter completion (`fringe/src/fringe/SpatialBlocks.scala:155-157`).

ChiselGen instantiates each ordinary controller state machine from `lhs.level` and `lhs.rawSchedule`, with outer controllers forced to II 1 in the emitted `IICounter` path (`src/spatial/codegen/chiselgen/ChiselGenController.scala:333-358`). It separately sets the controller mask from the local counter-chain noop and parent valid enables (`src/spatial/codegen/chiselgen/ChiselGenController.scala:317-325`). Thus, for HLS, `mask` is not merely a branch predicate: it is part of the child-execution protocol and can make a child count as iteration-complete without issuing its body.

## ForkJoin

`OuterControl` dispatches schedules in one central pattern match (`fringe/src/fringe/templates/Controllers.scala:94-95`). In the `ForkJoin` case, the counter increments only when `synchronize` is true, and `synchronize` is the AND of all child `iterDone` bits (`fringe/src/fringe/templates/Controllers.scala:141-147`). Every child can become active under the same parent enable/backpressure window; a child's `iterDone` is set either by that child's `doneIn` or by a masked-out skip condition, and each child-level `done` is tied to parent `ctrDone` or break (`fringe/src/fringe/templates/Controllers.scala:148-154`).

The shared post-match logic then acknowledges each child when its `iterDone` is set and enables a child only when the parent is enabled, that child is active, not yet iteration-done, masked in, and the parent is not all-done (`fringe/src/fringe/templates/Controllers.scala:212-215`). Unverified hardware inference: this means a `ForkJoin` HLS region should launch all enabled children for an iteration, join after every non-skipped child reports done, then advance the parent counter once.

## Fork and switch selects

`Fork` is different. Its synchronization is OR over `doneIn`, its counter increments on that OR, and each child active bit is gated by `selectsIn(i)` and `!io.done` (`fringe/src/fringe/templates/Controllers.scala:173-187`). Unlike `ForkJoin`, it does not wait for all children; once any selected child reports done, the case sets all `done` SRFFs through `synchronize`, allowing the outer `allDone` reduction to complete (`fringe/src/fringe/templates/Controllers.scala:80-81`, `fringe/src/fringe/templates/Controllers.scala:186-187`, `fringe/src/fringe/templates/Controllers.scala:244-248`). Unverified hardware inference: Rust/HLS should treat `Fork` as an if/switch branch controller, not a parallel fork.

For switches, ChiselGen drives inner-switch selects directly into `sm.io.selectsIn`; for outer switches it creates a per-case register on rising enable, but the emitted assignment still drives `selectsIn` from the original select expression (`src/spatial/codegen/chiselgen/ChiselGenController.scala:424-436`). Inner controllers forward `selectsIn` to `selectsOut` under enable, and inner switch cases use the parent's `smSelectsOut(childId)` as base enable (`fringe/src/fringe/templates/Controllers.scala:271-272`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:452-456`). The intended outer-select latch semantics are therefore unverified from these lines alone.

## PrimitiveBox

Flow rules assign `PrimitiveBox` to `SpatialBlackboxImpl`, while `ParallelPipe` becomes `ForkJoin`, `Switch` and `IfThenElse` become `Fork`, and `SwitchCase` becomes `Sequenced` (`src/spatial/flows/SpatialFlowRules.scala:317-339`). But Fringe has no `PrimitiveBox` schedule object (`fringe/src/fringe/templates/Controllers.scala:62-67`). Primitive blackbox use codegen instead enables the wrapper from the parent's `smEnableOuts(idx)`, reports `bbox.io.done` as child done, and reports a parent-mask expression as child mask (`src/spatial/codegen/chiselgen/ChiselGenBlackbox.scala:147-160`). Therefore, in Rust/HLS, `PrimitiveBox` should be an opaque primitive/blackbox leaf with done/mask handshake semantics, not a sixth executable Fringe controller schedule.
