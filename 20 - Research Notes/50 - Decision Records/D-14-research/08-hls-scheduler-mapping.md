# D-14 Angle 8: HLS Scheduler Mapping Options

## Semantic boundary

The Rust/HLS scheduler should keep `ForkJoin`, `Fork`, and `PrimitiveBox` as distinct effective schedule classes, but map them to different operational mechanisms. Spatial's enum lists all three beside `Sequenced`, `Pipelined`, and `Streaming`, and the effective schedule helper preserves them even when single-control `Pipelined` collapses to `Sequenced` (`src/spatial/metadata/control/ControlData.scala:7-14`; `src/spatial/metadata/control/package.scala:193-219`). Flow assignment is structural: `ParallelPipe` becomes `ForkJoin`, `Switch` and hardware `IfThenElse` become `Fork`, `SpatialBlackboxImpl` becomes `PrimitiveBox`, while `SwitchCase` and `SpatialCtrlBlackboxImpl` are `Sequenced` (`src/spatial/flows/SpatialFlowRules.scala:317-339`). Therefore the HLS mapping must not treat the enum as a flat set of controller pragmas. Source fact: only `Fork` and `ForkJoin` have corresponding Fringe controller schedules; `PrimitiveBox` does not (`fringe/src/fringe/templates/Controllers.scala:62-68`).

## `ForkJoin` as dataflow plus barrier

Operationally, `ForkJoin(parent, children)` starts every child whose parent enable and branch/mask conditions are true for the current logical parent iteration. A masked child is considered skipped-done; an enabled child is done when its own completion fires. The parent iteration completes only at the AND barrier over all child `iterDone` bits, matching the Chisel controller's `synchronize = andR(iterDone)` and counter advance on synchronize (`fringe/src/fringe/templates/Controllers.scala:141-154`). The runtime model also charges `ForkJoin` as max-child latency per iteration, not sum-child latency (`models/src/models/RuntimeModel.scala:321`).

Unverified HLS mapping: lower legal `ForkJoin` regions to `#pragma HLS dataflow`, persistent `hls::task`, or equivalent out-of-line task calls with explicit channels. AMD documents `DATAFLOW` as task-level pipelining that overlaps functions/loops through FIFO/PIPO channels, but also lists single-producer-consumer violations, feedback, conditional task execution, and multi-exit loops as blockers ([AMD UG1399 dataflow](https://docs.amd.com/r/2021.2-English/ug1399-vitis-hls/pragma-HLS-dataflow)). Legality should require no unresolved memory/effect conflicts across children, explicit channel or storage edges for producer-consumer communication, bounded FIFO depths, and a known policy for start/done propagation. Back-pressure from any child or channel must stall that child and may delay the join; it must not allow the parent to advance past an unfinished enabled child.

## `Fork` as selected branch enable

`Fork` is not a parallel fork. Source `if`/`else` lowers through `Switch`/`SwitchCase`, with path predicates constructed by the switch transformer, and Chisel gates each child by `selectsIn(i)` then synchronizes on OR of child `doneIn` (`src/spatial/transform/SwitchTransformer.scala:52-84`; `fringe/src/fringe/templates/Controllers.scala:173-187`). The Scala executor takes the first true selector, while the hardware path can observe multiple asserted selects differently (`src/spatial/executor/scala/ControlExecutor.scala:961-972`). The Rust/HLS semantics should therefore choose one contract explicitly: either require/prove one-hot branch masks and reject multi-true dynamic selects, or offer a labelled Scalagen-compatible priority mode lowered as ordered `if`/`else if`.

Unverified HLS mapping: emit ordinary conditional control for `Fork`, not HLS dataflow. Selective child enable means nonselected children perform no effects and count as skipped for the parent. If no case is enabled, the parent completes as a no-op only when the source construct has an explicit empty/default path; otherwise diagnose. Branch masks must be carried into every memory, stream, and `PrimitiveBox` side effect so a disabled branch cannot enqueue, write, or start a blackbox. A selected branch may itself contain pipelined or dataflow structure, but the outer `Fork` completion is the selected child's done event, not an all-branch join.

## `PrimitiveBox` as opaque primitive region

`PrimitiveBox` should be an atomic scheduler node with declared input/output ports, latency or latency range, initiation interval, read/write/stream effects, and start/done/mask behavior. Source facts point this way: flow assigns `PrimitiveBox` only to `SpatialBlackboxImpl`; primitive blackbox control children are empty; Chisel blackbox use wires wrapper enable from the parent child-enable, reports `bbox.io.done`, and supplies a parent-mask expression, while implementation generation emits a module body rather than a `PrimitiveBox` controller (`src/spatial/flows/SpatialFlowRules.scala:331-335`; `src/spatial/metadata/control/package.scala:907-910`; `src/spatial/codegen/chiselgen/ChiselGenBlackbox.scala:147-160`, `166-238`).

Unverified HLS mapping: implement as an inline function, noinline function, vendor/library primitive, or RTL blackbox, depending on target support. For RTL blackboxes, AMD requires a C function signature, JSON description, and RTL files, with clock/reset/CE and protocol constraints ([AMD UG1399 RTL blackbox](https://docs.amd.com/r/2024.1-English/ug1399-vitis-hls/Adding-RTL-Blackbox-Functions?contentId=m41sRvTVaCuJZiddNmWLwA)). The scheduler may not reorder across a `PrimitiveBox` unless its declared effects prove independence. Unknown latency is legal only as a conservative estimate or report-calibrated placeholder; final scheduling/DSE should record whether latency/II came from declaration, estimation, or HLS report.

## Cross-decision implications

D-15 should probably split value compatibility from hardware scheduling: a Scalagen-serial `ParallelPipe` simulator mode is useful for old goldens, but the D-14 hardware contract for `ForkJoin` implies same-iteration overlap plus a barrier. D-22 is a blocker for realistic HLS semantics because dataflow channels are bounded; AMD notes streams act like FIFOs, reads are destructive, and insufficient FIFO depth can stall or deadlock ([AMD UG1399 task/dataflow](https://docs.amd.com/r/2024.1-English/ug1399-vitis-hls/Data-driven-Task-level-Parallelism?contentId=pB34wfx8QwrhXN~Mx6Ehdg); [AMD UG1399 channel sizing](https://docs.amd.com/r/2024.1-English/ug1399-vitis-hls/Configuring-Dataflow-Memory-Channels?contentId=0Bb7~Z7_UrafAZDyLBH4PA)). Thus `ForkJoin` legality depends on the D-22 bounded back-pressure policy, while `Fork` depends mainly on one-hot/priority branch-mask semantics. D-21/D-08 should consume backend-accepted latency and II after synthesis rather than assuming these mappings force the requested schedule.
