# D-14 Angle 6: Runtime Model and DSE Equations

## Model boundary

The generated runtime model has its own `Runtime.CtrlSchedule` enum: `Sequenced`, `Pipelined`, `Streaming`, `ForkJoin`, `Fork`, and transfer pseudo-schedules, but not `PrimitiveBox` (`models/src/models/RuntimeModel.scala:29-40`). A `ControllerModel` stores `level`, `schedule`, counter-chain data, body latency `L`, initiation interval `II`, and fixed overhead constants: `seqSync = 1`, `metaSync = 1`, `seqAdvance = 2`, `dpMask = 1`, `startup = 2`, `shutdown = 1` (`models/src/models/RuntimeModel.scala:192-212`). Tuneable schedule parameters are also narrower than full metadata: `false` resolves to `Sequenced`; anything else resolves to `Pipelined` (`models/src/models/RuntimeModel.scala:214-224`).

## Outer schedule equations

For outer controls, the runtime model distinguishes schedule shape through `cycsPerParent` (`models/src/models/RuntimeModel.scala:313-333`). Let `N = cchainIters`, `No = cchain.head.N`, `Ni = cchain.last.N`, `S = sum(child.cycsPerParent)`, and `M = max(child.cycsPerParent)`, where `sumChildren` and `maxChild` are defined directly over children (`models/src/models/RuntimeModel.scala:301-305`). `Sequenced` is serialized: for one counter chain, `startup + shutdown + S*N + seqSync*children*N + seqAdvance*N`; for two levels, `startup + shutdown + (S + Ni)*No + seqSync*children*No + seqAdvance*No`. `Pipelined` overlaps iterations but still pays fill/drain child work: one level is `startup + shutdown + M*(N-1) + S + metaSync*N*children`; two levels use `max(M, Ni)*(No-1) + S + metaSync*No*children`. `ForkJoin` is barrier parallelism: `startup + shutdown + M*N + metaSync`. `Streaming` is similar to max-child overlap but omits the per-child pipeline fill term: one level `startup + shutdown + M*N + metaSync`; two levels `startup + shutdown + max(M, Ni)*No + metaSync`.

## Fork and inner collapse

`Fork` is modeled as expected selected-branch cost, not concurrent branch execution. The model asks for duty-cycle percentages for all but the last child, assigns the remaining percentage to the last child, multiplies each child cost by its probability, sums, and truncates to `Int` (`models/src/models/RuntimeModel.scala:325-327`). There is no explicit startup/shutdown term in that `Fork` branch, unlike `ForkJoin`, `Streaming`, and `Pipelined`. For inner controls, schedule detail mostly collapses: `Sequenced` is `N*L + startup + shutdown`, while every non-sequenced schedule uses the same pipeline equation `(N-1)*II + L + startup + shutdown + dpMask` (`models/src/models/RuntimeModel.scala:334-337`). Runtime execution then multiplies each controller's per-parent cycles by its dynamic parent-derived iteration count (`models/src/models/RuntimeModel.scala:340-347`).

## Generator and DSE consumption

`RuntimeModelGenerator` emits `ControllerModel`s from raw schedules, with `L` from `bodyLatency.sum` for inner controls and `II` forced to `1` for outer controls or non-retimed cases (`src/spatial/model/RuntimeModelGenerator.scala:261-284`). `ParallelPipe` is emitted as a controller with `Left(lhs.rawSchedule)`, which the flow rules assign to `ForkJoin` (`src/spatial/model/RuntimeModelGenerator.scala:305-310`; `src/spatial/flows/SpatialFlowRules.scala:331-333`). `Switch` is emitted as an `OuterControl` with raw schedule and an added `+ 2` latency/II adjustment; `SwitchCase` is emitted separately (`src/spatial/model/RuntimeModelGenerator.scala:398-412`). The generated model prints `Total Cycles for App` after `root.execute()` for each tune map (`src/spatial/model/RuntimeModelGenerator.scala:145-153`). DSE compiles `model_dse.scala` through `scripts/assemble.sh`, runs the resulting `RuntimeModel-assembly` jar in batches, parses those `Total Cycles for App` lines, and writes the result as the `Cycles` column consumed by workers, CSV output, Sensitivity, and HyperMapper (`src/spatial/dse/DSEAnalyzer.scala:96-128`; `src/spatial/dse/LatencyAnalyzer.scala:35-45`; `src/spatial/dse/DSEThread.scala:109-134`; `src/spatial/dse/HyperMapperDSE.scala:70-88`; `models/src/models/Sensitivity.scala:68-87`).

## PrimitiveBox implication

`PrimitiveBox` is real metadata: the control enum defines it, and flow assigns it to `SpatialBlackboxImpl` (`src/spatial/metadata/control/ControlData.scala:7-14`; `src/spatial/flows/SpatialFlowRules.scala:331-335`). It does not appear in the runtime-model enum or equations, and the generator has no specific `PrimitiveBox` controller case in the schedule-emitting cases above. Chisel primitive blackbox codegen instead constructs a wrapper with `OuterControl(Sequenced, ...)` and an `IICounter(1, ...)`, while retaining the calculated latency only as wrapper metadata (`src/spatial/codegen/chiselgen/ChiselGenBlackbox.scala:255-261`). For a Rust/HLS scheduler, this suggests `PrimitiveBox` should be treated as an atomic primitive or blackbox boundary with latency/II provenance, not as another fork/join ordering policy. The open semantic issue is whether HLS DSE should preserve Spatial's legacy equations as a compatibility estimator, or replace each schedule class with backend-accepted loop/task/report facts while keeping the same batchable `Cycles` contract.
