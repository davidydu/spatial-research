# D-25 - Multi-True OneHotMux Semantics

## Question

Define multi-true `OneHotMux` semantics.

Source: `spatial/src/spatial/codegen/scalagen/ScalaGenBits.scala:30-37`; `src/spatial/node/Mux.scala:19-37`; [[60 - Counters and Primitives]]

Decision criteria: User decision: HLS `OneHotMux` with multiple true selectors matches Scalagen OR-reduce, asserts one-hotness, or defines priority semantics.

## Research findings

- The IR exposes three distinct operators: `Mux`, `OneHotMux`, and `PriorityMux`. `OneHotMux` rewrites only simple static cases; multiple literal-true selectors trigger a warning but then fold to the first true lane. Dynamic multi-true cases reach codegen. Scalagen lowers dynamic `OneHotMux` as filtered true lanes followed by OR-reduce. See [[01-source-ir-scalagen]].

- Backend behavior is split. Scalagen implements OR-reduce; Chiselgen emits Chisel `Mux1H`; Chisel/Fringe preserve `PriorityMux` separately; Cppgen and Roguegen do not provide useful `OneHotMux` precedent. HLS should therefore not inherit OR, priority, or native tool behavior without a named policy. See [[02-backend-chisel-cppgen-hls]].

- The most important call sites are compiler-generated. `MemoryDealiasing`, `SwitchOptimizer`, `FlatteningTransformer`, and `HierarchicalToStream` create `OneHotMux` values under intended exclusivity assumptions, while the direct unit test `MuxTest` intentionally exercises dynamic multi-true OR behavior. See [[03-tests-app-call-sites]].

- OR-reduce is a compatibility behavior, not clean selection semantics. For `Bool`, it is logical OR with validity conjunction; for `FixedPoint`, it is raw scaled-integer bitwise OR; for floats and structs, Scalagen's OR path is incomplete or type-problematic. One-hot assertion best matches the operator name and Chisel `Mux1H`; priority belongs to `PriorityMux`. See [[04-semantic-options-consequences]].

- HLS lowering should use named modes. `or_reduce_masked_values` matches Scalagen; `priority_if_else_chain` matches `PriorityMux`; `assume_or_assert_onehot_then_or_reduce` treats OR-reduce as an implementation of a proved one-hot select. Target capabilities for assertions and one-hot assumptions must be explicit. See [[05-hls-tool-mapping-assertions]].

- The main risk of OR-reduce default is silent data corruption, especially for fixed-point raw bit unions. The main risk of strict one-hot default is migration breakage for existing OR-golden tests. Priority default hides bugs by turning overlap into ordered arbitration. Dual mode gives the best risk profile. See [[06-risk-failure-modes]].

- Implementation cost is bounded. Add a policy enum, static rejection or warning paths, runtime `popcount(selects) == 1` checks, HLS assertion/failure plumbing, and report fields. Keep OR-reduce for compatibility readback and deny one-hot optimization assumptions in that mode. See [[07-implementation-cost-migration]].

- Validation should cover zero-true, single-true, static multi-true, dynamic multi-true, `Bool` validity, fixed raw OR, float/struct/vector payloads, priority-vs-OR contrast, and HLS assertion behavior. Diagnostics need stable D-25 labels and per-mux report fields. See [[08-validation-fixtures-diagnostics]].

- The recommendation matrix selects dual-mode strict one-hot default plus Scalagen OR legacy. It rejects OR-reduce default, one-hot-only, priority default, and invalid/poison as the primary semantics. See [[10-recommendation-matrix]].

## Recommendation

Adopt **Dual-Mode OneHotMux Semantics With Strict One-Hot Default**.

The canonical Rust/HLS policy should be:

`onehot_mux_policy = strict_onehot_v1`

In this mode, `OneHotMux` is defined only when exactly one selector is true. Static overlap or static zero-true should be rejected when provable. Dynamic simulator and HLS debug builds should check `popcount(selects) == 1` before producing a value. After proof or assertion, the backend may lower through any equivalent one-hot network: `Mux1H`, masked OR, balanced mux tree, or target-native one-hot primitive. The semantic contract is the exclusivity check, not a specific tree shape.

Keep the existing Scalagen behavior as a labelled compatibility mode:

`onehot_mux_policy = compat_scalagen_or_reduce_v1`

This mode reproduces current Scalagen readback: collect all true selectors and OR-reduce their payloads. It exists for legacy tests, especially `MuxTest`, and for comparing old Scalagen artifacts. It should label results as non-strict and should disable any optimization that assumes selector exclusivity. It must also record zero-true behavior explicitly, because Scalagen's empty `reduce` is not a typed invalid fallback.

Priority behavior should not be a `OneHotMux` interpretation:

`priority_multi_true_v1` belongs to `PriorityMux`

If a source or compiler pass wants first-true arbitration, it should emit or rewrite to `PriorityMux`. Reinterpreting `OneHotMux` as priority would erase an already clear API distinction and would make ordering semantically important in a node whose name and Chisel lowering imply exclusivity.

For debug and reporting, a violation can produce typed invalid/poison evidence, but poison is diagnostic output rather than the default value semantics. Each mux artifact should record policy, selector count, payload type, static proof source, runtime assertion mode, zero-true policy, multi-true policy, target capabilities used, and whether compatibility OR was observed.

### Rejected alternatives

Reject **OR-reduce as the default**. It preserves Scalagen parity but makes raw bit union the official answer to an exclusivity violation, which is unsafe for fixed-point and poorly defined for floats/structs.

Reject **strict one-hot only with no compatibility lane**. It is clean but needlessly breaks existing Scalagen goldens and obscures intentional legacy replay.

Reject **priority as the default**. `PriorityMux` already defines ordered arbitration. Making `OneHotMux` priority would hide overlap bugs and block one-hot optimization assumptions.

Reject **invalid/poison as primary semantics**. It is useful after an assertion failure, but it requires broader typed invalid propagation and is better treated as diagnostic evidence.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Define `onehot_mux_policy` labels: `strict_onehot_v1`, `compat_scalagen_or_reduce_v1`, `priority_multi_true_v1`, and `poison_on_violation_v1`.
2. Upgrade static multiple-literal-true and static zero-true handling to errors under `strict_onehot_v1`; keep warnings under compatibility mode.
3. Add Rust simulator checks for `popcount(selects) == 1` in strict mode, with diagnostics for zero-true and multi-true.
4. Implement `compat_scalagen_or_reduce_v1` by reproducing Scalagen's filtered OR-reduce, including current fixed/Bool raw behavior and labelled zero-true failure.
5. Keep priority behavior as `PriorityMux` lowering or an explicit rewrite, not as a `OneHotMux` fallback.
6. Add HLS lowering modes: checked one-hot with assertion/failure flag when available, masked OR after proof/check, and target-native one-hot only with capability records.
7. Add per-mux report fields: op id, source span, policy, selector count, payload type/width, proof source, assertion mode, runtime true-count range, fallback reason, and mismatch kind.
8. Add fixtures for zero-true, single-true, static multi-true, dynamic multi-true, Bool validity, fixed raw OR, float/struct/vector payloads, priority-vs-OR contrast, and HLS assertion behavior.
9. Audit compiler-generated `OneHotMux` sites in `MemoryDealiasing`, `SwitchOptimizer`, `FlatteningTransformer`, and `HierarchicalToStream` for exclusivity proof or runtime assertion coverage.
10. Update `MuxTest` into separate compatibility and strict-policy fixtures so the legacy OR behavior remains tested without defining the new default.
