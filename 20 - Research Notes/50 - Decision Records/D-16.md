# D-16 — OOB Simulator Versus Synthesis Behavior

## Question

Choose the Rust policy for out-of-bounds simulator versus synthesis behavior.

Source: `spatial/emul/src/emul/OOB.scala:19-38`; `spatial/src/spatial/codegen/scalagen/ScalaGenMemories.scala:36-50`; [[30 - Memory Simulator]].

Decision criteria: User decision: Rust offers Scalagen-compatible OOB log/invalid behavior, synthesis assertions, or separate simulator/synthesis modes.

## Research findings

- The current emulation contract is non-aborting. `OOB.readOrElse` logs enabled reads, logs `[OOB]` on caught `ArrayIndexOutOfBoundsException`, and returns the caller's invalid value. `OOB.writeOrElse` logs enabled writes, logs `[OOB]` on caught OOB writes, and discards the write. Disabled accesses avoid the storage touch in caller thunks. See [[01-emul-oob-current-behavior]].

- ScalaGen has both runtime memory wrappers and generated-code wrappers. Banked/shiftable memories route through `OOB.readOrElse`/`writeOrElse`; DRAM/transfer code can use generated `try`/`catch` wrappers that print a warning and return `invalid(tp)` for reads. Invalid values are typed runtime values, not plain host exceptions: fixed, float, and bit carry validity, while vector invalid is a known compatibility hazard. See [[02-scalagen-memory-wrapper-invalid]].

- The written spec already treats the simulator side as reference behavior, but leaves synthesis policy open. Memory Semantics and Memory Simulator say Rust simulation should match Scalagen log/invalid/discard behavior unless a deliberate divergence mode is chosen. The open question is whether synthesis emits assertions, assumes in-bounds, or exposes a mode split. See [[03-spec-open-question-contract]].

- Hardware/HLS should not synthesize Scalagen's invalid-value recovery by default. Current Chisel/Fringe memory paths wire banks, offsets, enables, and data into memory ports; they do not emit logfiles or typed invalid fallback. HLS assertions and assumptions have different meanings and costs, so final HLS must distinguish debug checks from release in-bounds contracts. See [[04-hls-synthesis-assertions]].

- Test evidence is narrow but real. `OOBTest` intentionally writes beyond a 13-element SRAM and expects legal writes to survive without aborting. Direct coverage for OOB reads, log contents, vector invalid, and synthesized behavior is weak or missing. Adjacent invalid-value behavior appears in empty FIFO/stream reads, disabled lanes, vector invalid questions, and memory padding questions. See [[05-tests-apps-evidence]].

- A Rust implementation needs explicit modes and structured diagnostics. Useful profiles include `compat_log_invalid`, `strict_trap`, `assert_collect`, `synth_assert`, and `assume_inbounds`. Logs should mirror legacy text for compatibility but use structured records as the canonical artifact. See [[06-rust-mode-design-diagnostics]].

- OOB checking must cover more than flat arrays. Banked SRAMs need bank-coordinate and offset checks; RegFile/LUT/ShiftableMemory need flattened address checks; line buffers need circular-row plus finite-column checks; DRAM dense/sparse transfers need offset/size/address checks; alias selection needs separate unresolved-alias diagnostics before OOB is applied to the selected target. Disabled lanes should not trigger OOB. See [[07-memory-banking-addressing-surfaces]].

- Implementation and risk analyses reject single-policy answers. Compatibility-only hides hardware bugs; trap-only breaks legacy behavior; assertion-only is not the simulator reference; unchecked assumptions are only safe under proof or waiver. The best migration path is explicit simulator/synthesis separation plus compare tooling. See [[08-implementation-migration-cost]], [[09-risk-failure-modes]], and [[10-recommendation-matrix]].

## Recommendation

Adopt **Separate Simulator And Synthesis OOB Modes**, implemented first as **Scalagen-compatible simulation plus checked synthesis assertions**.

Define these policies:

- `sim_compat_log_invalid`: Rust simulator matches Scalagen. Enabled in-bounds reads/writes log normally; enabled OOB reads log and return a typed invalid value; enabled OOB writes log and discard the write; disabled lanes do not log or touch storage. This is the compatibility profile for existing value goldens and OOB logs.
- `sim_strict_trap`: enabled OOB is a simulator error. It stops before returning invalid or committing later side effects. This is the HLS-conformance CI profile, not the legacy default.
- `sim_assert_collect`: record every enabled OOB violation, then continue with `sim_compat_log_invalid` data behavior. Final values from this mode must be labelled contaminated.
- `synth_checked_assert`: HLS/debug lowering emits enabled-lane bounds checks or equivalent assertion hooks for every addressable surface that cannot be statically proved in-bounds.
- `synth_assume_inbounds`: release/optimized lowering removes dynamic OOB recovery and records a proof or explicit waiver. Violating the assumption has no defined Rust simulator value and must not silently fall back to invalid.

Do not use ordinary Rust panics as the compatibility behavior. Rust panics are useful for `sim_strict_trap`, but they do not match Scalagen's log-and-continue semantics. Likewise, do not use unchecked Rust indexing or `get_unchecked` in simulator code unless the mode is explicitly `assume_inbounds` and the access is proved or waived.

Canonical diagnostics should be structured JSON Lines, with legacy text mirrors optional:

- `event`, `simulator_mode`, `synthesis_policy`
- memory name/id, source context, static access id, dynamic instance id
- access kind, lane id, enable, abstract indices
- bank coordinates, offset, flattened address, extent
- in-bounds result, returned value, write committed
- assertion id, proof/waiver id, replay step

Every simulator and HLS artifact should carry the OOB mode. A clean `sim_compat_log_invalid` run with OOB events is not a hardware-valid run; it is a compatibility run with contaminated accesses. A clean `synth_assume_inbounds` artifact is valid only as far as its proofs/waivers are trusted.

Reject **single default strict trap**. It is attractive for Rust safety and HLS legality, but it breaks the documented simulator reference and the existing OOB write-discard test.

Reject **single default log/invalid** for HLS. It preserves old behavior but makes simulator-only invalid propagation look like a hardware contract.

Reject **hidden macro divergence**. If C simulation logs invalid values while synthesis assumes in-bounds under `__SYNTHESIS__`, that split must be a named mode and manifest entry, not an invisible preprocessor accident.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:

1. Implement typed invalid values for every Rust simulator type, including vectors, so OOB reads do not rely on host exceptions or `Option<T>` alone.
2. Implement shared checked-access helpers for banked SRAM, RegFile/LUT/ShiftableMemory, LineBuffer, DRAM dense/sparse transfer, and selected alias targets.
3. Add `sim_compat_log_invalid`, `sim_strict_trap`, and `sim_assert_collect` simulator modes with mode provenance in all outputs.
4. Emit legacy-compatible `reads.log`/`writes.log` mirrors and canonical structured OOB event logs.
5. Add HLS policies `synth_checked_assert` and `synth_assume_inbounds`, with assertions enable-gated per lane and assumptions tied to proof/waiver ids.
6. Add compare/differential tooling that runs compatibility plus strict/assert modes and reports accesses where values match but hardware legality differs.
7. Add tests for OOB read invalid, OOB write discard, disabled-lane suppression, vector invalid, log contents, dense/sparse transfer OOB, alias selection failure, and proof/waiver manifest behavior.
8. Mark simulator outputs with OOB events as contaminated for HLS-conformance purposes unless the selected policy is strict/assert and no violations occurred.
