# Precedent Survey for HLS Banking/Partition Planning

## Local Baseline

Spatial's current planner is not a simple HLS partition picker. `MemoryConfigurer` first chooses candidate views, `NStrictness`, `AlphaStrictness`, and dimension-duplication controls; the defaults try both flat and hierarchical banking, then `NBestGuess`/`NRelaxed` and `AlphaBestGuess`/`AlphaRelaxed` unless user constraints are present (`src/spatial/traversal/banking/MemoryConfigurer.scala:25`, `src/spatial/traversal/banking/MemoryConfigurer.scala:34`, `src/spatial/traversal/banking/MemoryConfigurer.scala:38`, `src/spatial/traversal/banking/MemoryConfigurer.scala:42`). The metadata API already lets users constrain those families through `nConstraints` and `alphaConstraints` (`src/spatial/metadata/memory/package.scala:139`, `src/spatial/metadata/memory/package.scala:142`).

`NBestGuess` expands factors of concurrent access counts, factors of staged-dimension products, powers of two, and Mersenne-like values, while `NRelaxed` enumerates the full range (`src/spatial/metadata/memory/BankingData.scala:560`, `src/spatial/metadata/memory/BankingData.scala:562`, `src/spatial/metadata/memory/BankingData.scala:568`). `AlphaBestGuess` expands factor-, dimension-, coprime-, and power/sum-of-power-derived vectors, while `AlphaRelaxed` enumerates residues modulo `N` (`src/spatial/metadata/memory/BankingData.scala:618`, `src/spatial/metadata/memory/BankingData.scala:623`, `src/spatial/metadata/memory/BankingData.scala:631`). In `bankGroups`, these axes become a sorted cross product, then the selected strategy receives access groups, reaching writes, directives, and buffer depth (`src/spatial/traversal/banking/MemoryConfigurer.scala:610`, `src/spatial/traversal/banking/MemoryConfigurer.scala:616`, `src/spatial/traversal/banking/MemoryConfigurer.scala:619`, `src/spatial/traversal/banking/MemoryConfigurer.scala:633`).

## Partition Precedents

The common user-specified partition model in HLS tools is directive-first: users name a memory, dimension, partition kind, and factor, usually choosing complete, cyclic, or block forms (unverified). Spatial has local analogues, but they are more expressive: explicit schemes can name `Ns`, `Bs`, `Alphas`, and optional `Ps`, and forced explicit banking can bypass normal safety rewriting (`src/spatial/metadata/memory/BankingData.scala:644`, `src/spatial/metadata/memory/BankingData.scala:645`, `src/spatial/metadata/memory/BankingData.scala:662`; `src/spatial/traversal/banking/ExhaustiveBanking.scala:397`). The local spec already frames HLS banking as rework because Vitis-like partition pragmas only cover cyclic, block, and complete per dimension (unverified except for the local note) (`10 - Spec/40 - Compiler Passes/70 - Banking.md:295`, `10 - Spec/40 - Compiler Passes/70 - Banking.md:297`).

Compiler-inferred partitioning is the second precedent: downstream HLS may infer memories, ports, scalarization, or simple banking from source structure, but relying on that inference as the only planner makes results tool-version and code-shape dependent (unverified). Spatial's stronger local precedent is to infer before codegen: `findBanking` derives `Nmin`, `Ncap`, candidate `N`s, alpha vectors, and then checks cyclic or block-cyclic conflicts before constructing `ModBanking` (`src/spatial/traversal/banking/ExhaustiveBanking.scala:374`, `src/spatial/traversal/banking/ExhaustiveBanking.scala:379`, `src/spatial/traversal/banking/ExhaustiveBanking.scala:409`, `src/spatial/traversal/banking/ExhaustiveBanking.scala:417`, `src/spatial/traversal/banking/ExhaustiveBanking.scala:422`). For HLS, inference should be an optimization layer, not the semantic contract (unverified).

## Report-Guided Iteration

Report-guided HLS flows commonly iterate after synthesis reports expose achieved II, memory-port bottlenecks, and resource use (unverified). Spatial already has an internal analogue: `MemoryReporter` emits per-memory instances, resource, depth, padding, bank counts, banking details, ports, direct-banking status, and broadcast metadata (`src/spatial/report/MemoryReporter.scala:26`, `src/spatial/report/MemoryReporter.scala:52`, `src/spatial/report/MemoryReporter.scala:57`, `src/spatial/report/MemoryReporter.scala:61`, `src/spatial/report/MemoryReporter.scala:74`, `src/spatial/report/MemoryReporter.scala:81`). `bankGroups` also keeps failure visible by returning `UnbankableGroup` when no banking is found and logging read/write groups plus the result (`src/spatial/traversal/banking/MemoryConfigurer.scala:674`, `src/spatial/traversal/banking/MemoryConfigurer.scala:676`, `src/spatial/traversal/banking/MemoryConfigurer.scala:680`).

This supports a hybrid: compile-time planning should request an HLS-visible partition, while post-HLS reports should reconcile whether the tool honored it (unverified). D-06 reached the same shape for scheduling: keep Spatial-style analysis as a conservative/request signal, but record accepted HLS results separately (unverified) (`20 - Research Notes/50 - Decision Records/D-06-research/06-policy-options.md:35`).

## Manifests And Diagnostics

Explicit layout manifests are a strong fit for HLS migration: many FPGA build systems separate source code, directives, target configuration, and report artifacts (unverified). Locally, D-04 already recommends recording HLS partition modifiers as fields such as `partition_kind`, `partition_factor`, `partition_dim`, and `requested_port_model` (`20 - Research Notes/50 - Decision Records/D-04-research/05-hls-taxonomy.md:30`). D-05 similarly argues for replacing hidden Chisel/global construction with explicit target and application manifests (`20 - Research Notes/50 - Decision Records/D-05-research/03-target-fringe-coupling.md:21`).

Fallback diagnostics should be first-class. `MemoryConfigurer` throws when no banking options are allowed, returns hazards from buffer-port analysis before accepting schemes, and otherwise returns `UnbankableGroup` (`src/spatial/traversal/banking/MemoryConfigurer.scala:621`, `src/spatial/traversal/banking/MemoryConfigurer.scala:625`, `src/spatial/traversal/banking/MemoryConfigurer.scala:631`, `src/spatial/traversal/banking/MemoryConfigurer.scala:672`, `src/spatial/traversal/banking/MemoryConfigurer.scala:674`). HLS precedent argues for visible fallback when a requested partition cannot be expressed, is ignored, or fails timing/II (unverified).

## D-07 Implication

Do not reuse the full alpha/`N`/`B` search as the HLS authority. It can find valid Spatial modular schemes that source-level HLS partition directives cannot state (unverified), because Spatial constructs `ModBanking(N, B, alpha, axes, P)` after arbitrary alpha-vector conflict checks (`src/spatial/traversal/banking/ExhaustiveBanking.scala:409`, `src/spatial/traversal/banking/ExhaustiveBanking.scala:422`). The better precedent-aligned plan is a new HLS-form planner that reuses Spatial's access grouping, `NBestGuess` factor priorities, conflict tests, duplication choices, buffering metadata, reports, and diagnostics, but emits a manifest of complete/block/cyclic/reshape/duplicate/binding requests rather than arbitrary alpha-vector `ModBanking` plans (`src/spatial/traversal/banking/MemoryConfigurer.scala:616`, `20 - Research Notes/50 - Decision Records/D-04-research/05-hls-taxonomy.md:30`).
