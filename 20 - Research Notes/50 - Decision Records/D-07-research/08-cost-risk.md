# D-07 Angle 8: Implementation Cost and Risk

## Baseline Cost Surface

`MemoryConfigurer` is not just a partition-factor picker. It chooses banking views, `N` strictness, alpha strictness, and dimension duplication before search (`src/spatial/traversal/banking/MemoryConfigurer.scala:25`, `src/spatial/traversal/banking/MemoryConfigurer.scala:34`, `src/spatial/traversal/banking/MemoryConfigurer.scala:38`, `src/spatial/traversal/banking/MemoryConfigurer.scala:42`). `configure` then clears old dispatch/port/duplicate metadata, extracts affine access matrices, invokes banking, summarizes instances, finalizes metadata, and runs PIR legality checks (`src/spatial/traversal/banking/MemoryConfigurer.scala:55`, `src/spatial/traversal/banking/MemoryConfigurer.scala:71`, `src/spatial/traversal/banking/MemoryConfigurer.scala:78`, `src/spatial/traversal/banking/MemoryConfigurer.scala:81`). The output contract is a physical plan: each `Instance` carries access groups, metapipeline, `Seq[Banking]`, buffer depth, cost, ports, padding, and accumulator type (`src/spatial/metadata/memory/BankingData.scala:109`, `src/spatial/metadata/memory/BankingData.scala:114`, `src/spatial/metadata/memory/BankingData.scala:117`, `src/spatial/metadata/memory/BankingData.scala:118`). Any HLS option that replaces the search still has to produce that shape or replace every downstream consumer.

## Option Comparison

**Exact reuse of Spatial alpha/N/B search** has the lowest implementation cost because it preserves grouping, `Instance` production, `Dispatch`, `Ports`, and `Duplicates` finalization (`src/spatial/traversal/banking/MemoryConfigurer.scala:115`, `src/spatial/traversal/banking/MemoryConfigurer.scala:117`, `src/spatial/traversal/banking/MemoryConfigurer.scala:123`, `src/spatial/traversal/banking/MemoryConfigurer.scala:125`). Its risk is high. The search builds a Cartesian product of view, `N`, alpha, and regrouping directives, then reorders preferred and relaxed cases (`src/spatial/traversal/banking/MemoryConfigurer.scala:616`, `src/spatial/traversal/banking/MemoryConfigurer.scala:619`, `src/spatial/traversal/banking/MemoryConfigurer.scala:628`). Inside `findBanking`, it iterates candidate `N`s and alpha vectors, optionally checks configured `B`s, and stops only after enough schemes or the banking timeout (`src/spatial/traversal/banking/ExhaustiveBanking.scala:390`, `src/spatial/traversal/banking/ExhaustiveBanking.scala:392`, `src/spatial/traversal/banking/ExhaustiveBanking.scala:394`, `src/spatial/traversal/banking/ExhaustiveBanking.scala:417`). Because `ModBanking` is `(alpha dot addr / B) mod N`, arbitrary alpha vectors and block-cyclic `B` values do not map cleanly to ordinary HLS array partition directives (unverified) (`src/spatial/metadata/memory/BankingData.scala:49`, `src/spatial/metadata/memory/BankingData.scala:64`).

**Constraining the existing search to HLS partitionable forms** is medium cost. It can reuse access grouping, reaching-write filtering, buffer-depth computation, cost selection, and port assignment (`src/spatial/traversal/banking/MemoryConfigurer.scala:610`, `src/spatial/traversal/banking/MemoryConfigurer.scala:613`, `src/spatial/traversal/banking/MemoryConfigurer.scala:631`, `src/spatial/traversal/banking/MemoryConfigurer.scala:664`). The key work is to prune before solving: hierarchical single-dimension views, simple factors from `NBestGuess`, unit or trivial alpha, complete partition for full dimensions, and only block/cyclic forms that the target HLS accepts (unverified). Filtering after full search is riskier because Spatial may find valid modular schemes that the HLS backend cannot spell.

**A new HLS partition planner** is highest initial cost but cleaner long term. It should search `partition_kind x factor x dimension x duplicate_count x binding_preference`, then emit the same duplicate/port/padding contract. This avoids pretending HLS pragmas can express the full alpha/N/B space (unverified), but it must reimplement enough legality, diagnostics, and reporting to be trusted.

**Hybrid fallback** is medium-high cost. Use the constrained HLS planner first, then fall back either to a manually lowered array-of-banks form for non-HLS partitions or to a conservative unpartitioned/requested-II diagnostic mode. This limits v1 risk, but only if fallback is visible in reports rather than silently changing performance.

## Cross-Decision Risks

D-04 resource accounting is tightly coupled to the chosen banking plan. `MemoryAllocator` iterates memory duplicates, tests resource area and bank depth against remaining capacity and `minDepth`, then assigns a final catch-all resource to leftovers (`src/spatial/traversal/MemoryAllocator.scala:56`, `src/spatial/traversal/MemoryAllocator.scala:83`, `src/spatial/traversal/MemoryAllocator.scala:85`, `src/spatial/traversal/MemoryAllocator.scala:102`). `AreaModel` multiplies resource-per-bank by total bank count and buffer depth (`src/spatial/targets/AreaModel.scala:145`, `src/spatial/targets/AreaModel.scala:146`, `src/spatial/targets/AreaModel.scala:153`). Therefore HLS partition factors must become first-class resource/report facts, not just pragmas (unverified).

D-06 dependence/II depends on the final memory plan. `IterationDiffAnalyzer` already has TODOs around writer-leading-reader, cross-hierarchy iterators, and lane-distance assumptions (`src/spatial/traversal/IterationDiffAnalyzer.scala:36`, `src/spatial/traversal/IterationDiffAnalyzer.scala:64`, `src/spatial/traversal/IterationDiffAnalyzer.scala:71`). `InitiationAnalyzer` collapses present `iterDiff`s into `compilerII` and final `II` (`src/spatial/traversal/InitiationAnalyzer.scala:29`, `src/spatial/traversal/InitiationAnalyzer.scala:37`, `src/spatial/traversal/InitiationAnalyzer.scala:38`). If the HLS tool accepts, relaxes, or rejects partition/II directives, the accepted II must override the request for HLS performance reporting (unverified).

D-08 latency/DSE adds the feedback loop. Current DSE compiles a Scala runtime model, runs the generated jar, parses `"Total Cycles for App"`, and records area/runtime/valid in the DSE row (`src/spatial/dse/DSEAnalyzer.scala:90`, `src/spatial/dse/DSEAnalyzer.scala:127`, `src/spatial/dse/LatencyAnalyzer.scala:35`, `src/spatial/dse/LatencyAnalyzer.scala:42`, `src/spatial/dse/DSEThread.scala:119`, `src/spatial/dse/DSEThread.scala:123`). HLS needs equivalent report-backed latency and resource feedback; otherwise DSE may optimize against a partition plan the tool did not actually implement (unverified).

## Recommendation

Best cost/risk posture: **constrain the existing search for v1, with a new HLS planner as the destination and exact reuse only as an explicit fallback**. This preserves `MemoryConfigurer` outputs while cutting the N/alpha/B explosion before it produces unspellable HLS plans. The required guardrails are: record requested partition versus tool-accepted partition, record requested II versus accepted II, expose fallback kind in the memory report, and make DSE consume accepted report data when available. Spatial already reports duplicates, resources, banking, depth, padding, and ports; the HLS path should extend that report rather than hiding tool feedback (`src/spatial/report/MemoryReporter.scala:44`, `src/spatial/report/MemoryReporter.scala:57`, `src/spatial/report/MemoryReporter.scala:61`, `src/spatial/report/MemoryReporter.scala:81`).
