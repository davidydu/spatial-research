# DSE consumers and latency contract

## Evaluation path

`DSEAnalyzer.process` builds the explored parameters from tile sizes, parallelization factors, and pipeline toggles, creates integer and Boolean domains, compiles the generated latency/runtime model, then dispatches to heuristic, brute-force, experiment, or HyperMapper mode (`src/spatial/dse/DSEAnalyzer.scala:59-80`). The current non-HyperMapper path calls `threadBasedDSE`, which creates bounded queues, `spatialConfig.threads` worker threads, and one writer thread (`src/spatial/dse/DSEAnalyzer.scala:312-363`). Each worker is a `DSEThread` with its own copied `State`, target area analyzer, and target cycle analyzer (`src/spatial/dse/DSEAnalyzer.scala:320-345`, `src/spatial/dse/DSEThread.scala:56-77`).

Latency is consumed batch-first. `DSEThread.run` converts each requested `DesignPoint` into `paramRewrites`, calls `evaluateLatency(paramRewrites)` once for the whole batch, then iterates points to set domains, rerun area, compute validity, and emit CSV rows (`src/spatial/dse/DSEThread.scala:109-127`). `evaluateLatency` calls `cycleAnalyzer.test(paramRewrites)` and returns `cycleAnalyzer.totalCycles` (`src/spatial/dse/DSEThread.scala:130-134`). The concrete `LatencyAnalyzer` shells out to a compiled `RuntimeModel-assembly` jar in groups of 1000 tune commands, parses `"Total Cycles for App"`, and maps negative raw cycles to `Long.MaxValue` (`src/spatial/dse/LatencyAnalyzer.scala:29-47`). Therefore a replacement latency source must be batch-addressable by the DSE parameter values, return one non-negative `Long`-like cycle value per input point in the same order, and behave safely across multiple worker instances.

## Cycles, area, and validity

Cycles do not decide `Valid` today. A row is valid only when summarized area is within target capacity and the worker state has no errors: `valid = area <= capacity && !state.hadErrors` (`src/spatial/dse/DSEThread.scala:119-123`). The area vector is constrained to `capacity.nonZeroFields`, so the CSV area columns are target-capacity-dependent (`src/spatial/dse/DSEThread.scala:48-50`, `src/spatial/dse/DSEThread.scala:122-123`). The target supplies both analyzers through `HardwareTarget.areaAnalyzer` and `HardwareTarget.cycleAnalyzer`; the former receives both an area model and a latency model, while the latter receives a latency model (`src/spatial/targets/HardwareTarget.scala:47-48`).

That second latency dependency matters for D-08. `DSEAreaAnalyzer` uses latency information to compute delay-line area: it calls `latenciesAndCycles`, checks `latencyModel.requiresRegisters`, subtracts per-node latency from critical paths, and sums `areaOfDelayLine` for required retiming delays (`src/spatial/dse/DSEAreaAnalyzer.scala:66-108`). Reductions and memory reductions also use `latencyOfPipe(reduce)` when sizing reduction-tree delay area (`src/spatial/dse/DSEAreaAnalyzer.scala:177-180`, `src/spatial/dse/DSEAreaAnalyzer.scala:198-201`). A replacement source for top-level cycles is not sufficient unless it preserves, replaces, or deliberately disables the per-node latency/retiming interface used by area.

## CSV/report contract

For normal DSE, `DSEAnalyzer` builds the header as parameter names, worker area headings, `Cycles`, and `Valid`; `DSEWriterThread` appends `Timestamp` while writing (`src/spatial/dse/DSEAnalyzer.scala:354-361`, `src/spatial/dse/DSEWriterThread.scala:18-21`). Worker rows mirror that order: current parameter values, selected area resources, `runtimes(i)`, validity, and elapsed timestamp (`src/spatial/dse/DSEThread.scala:121-123`). After completion, the analyzer reloads the CSV, sorts rows by `Cycles` via the third-from-last column, prints samples, then tries `Sensitivity.around` (`src/spatial/dse/DSEAnalyzer.scala:420-439`).

`Sensitivity` is another hard consumer of the contract. It expects area-ish columns named `BRAM`, `DSPs`, `Regs`, `SLICEL`, `SLICEM`, `Slices`, and `Valid`, filters those from the CSV, then looks up `"Cycles"` for the current point and neighboring parameter values (`models/src/models/Sensitivity.scala:5-7`, `models/src/models/Sensitivity.scala:57-72`). The code comment says it filters timestamp too, but `Timestamp` is not in `area_cols`; this path is best treated as fragile rather than authoritative (`models/src/models/Sensitivity.scala:60-62`). Renaming `Cycles`, omitting parameter columns, or changing Boolean strings would break downstream consumers unless those consumers are updated together.

## Pruning and worker assumptions

Heuristic DSE prunes single-parameter restrictions by filtering domains, then enumerates multi-parameter legal points with `PruneWorker` before sampling up to 75,000 normal points or ten experiment trials capped at 100,000 (`src/spatial/dse/DSEAnalyzer.scala:165-186`, `src/spatial/dse/DSEAnalyzer.scala:204-231`, `src/spatial/dse/DSEAnalyzer.scala:238-278`). `PruneWorker` maps linear indices into domain settings and keeps only points whose restrictions evaluate true (`src/spatial/dse/PruneWorker.scala:21-29`). Brute force sends the full Cartesian product directly to `threadBasedDSE` (`src/spatial/dse/DSEAnalyzer.scala:442-454`). HyperMapper does not use that heuristic legal-point prepass; it emits domains into a JSON config and lets HyperMapper request points (`src/spatial/dse/HyperMapperDSE.scala:52-104`).

The replacement latency source should therefore assume it receives already-selected candidate points, not own pruning. It must tolerate many small or medium batches, bounded queues, poison-pill termination, and copied compiler states (`src/spatial/dse/DSEAnalyzer.scala:369-397`, `src/spatial/dse/DSEThread.scala:82-107`). Full HLS synthesis or simulation per point may be too slow for the existing heuristic sample counts and HyperMapper iteration loop (unverified), so report-backed or cached estimators would need an explicit batching/cache story.

## HyperMapper contract

HyperMapper mode makes `Slices` and `Cycles` optimization objectives and `Valid` the feasible output in its JSON (`src/spatial/dse/HyperMapperDSE.scala:70-78`). Its header is parameter names, area fields, `Cycles`, `Valid`, and `Timestamp` (`src/spatial/dse/HyperMapperDSE.scala:127-128`). The receiver accepts `Request n` or `FRequest n file`, maps incoming CSV headers back to domain names, parses integer/Boolean values into `Point`s, and submits blocks to the same `DSEThread` workers (`src/spatial/dse/HyperMapperReceiver.scala:27-56`, `src/spatial/dse/HyperMapperReceiver.scala:81-103`). The sender replies with the same header and row strings, or writes `.out` files and sends a `Ready` ACK for file requests (`src/spatial/dse/HyperMapperSender.scala:28-58`). Thus D-08’s latency replacement must keep `Cycles` as a numeric minimization column in both ordinary CSV and HyperMapper responses, or HyperMapper’s objective schema must change in lockstep.
