# Policy Options

## Baseline

Q-116 asks for the HLS DSE cycle source because today's DSE latency path is a Scala runtime-model loop, not an HLS feedback loop: it shells out to a compiled model jar and parses `"Total Cycles for App"` (`20 - Research Notes/20 - Open Questions.md:1565-1575`). The compiler enables this path through DSE flags that also turn on the runtime model (`src/spatial/Spatial.scala:268-272`), emits the DSE runtime model before `dsePass` (`src/spatial/Spatial.scala:178-183`), then `DSEAnalyzer` requires `gen/model/model_dse.scala` and runs `scripts/assemble.sh` (`src/spatial/dse/DSEAnalyzer.scala:90-128`). `LatencyAnalyzer` finds the `RuntimeModel-assembly` jar, invokes it with batched `tune` arguments, and maps negative totals to `Long.MaxValue` (`src/spatial/dse/LatencyAnalyzer.scala:35-46`). `DSEThread` writes the returned runtime beside modeled area and validity (`src/spatial/dse/DSEThread.scala:114-123`).

The model's useful core is a controller tree with tripcounts, body latency `L`, and initiation interval `II`; inner pipelines use `(cchainIters - 1)*II + L + startup + shutdown + dpMask` (`models/src/models/RuntimeModel.scala:192-200`, `models/src/models/RuntimeModel.scala:334-337`). But D-08's equation survey already warns that transfer/congestion equations, fixed control overheads, sparse placeholders, and invisible `+2` patches are Spatial/Chisel assumptions rather than HLS facts (`20 - Research Notes/50 - Decision Records/D-08-research/03-runtime-model-equations.md:17-27`).

## Policy Comparison

| Option | Pros | Cons / required metadata / failure modes |
|---|---|---|
| Keep Spatial runtime-model parity | Fastest v1 and keeps existing DSE mechanics: one pre-HLS latency per design point from the same jar path DSE already compiles and runs (`src/spatial/dse/DSEAnalyzer.scala:71-79`, `src/spatial/dse/LatencyAnalyzer.scala:35-46`). Requires existing controller hierarchy, tripcounts, `L`, `II`, tunables, and transfer metadata. | Fails when HLS accepted II, partitioning, operator latency, memory ports, or transfer behavior diverge from Spatial assumptions (unverified). D-06 explicitly says old `II` should not decide final HLS DSE latency after reports exist (`20 - Research Notes/50 - Decision Records/D-06.md:85-90`). |
| Use HLS reports as authority | Best post-HLS truth source if reports expose accepted loop latency/II and schedule reasons (tool-specific, unverified). It aligns with D-06's `hls_accepted_ii` and `hls_report_source` fields (`20 - Research Notes/50 - Decision Records/D-06.md:65-69`). | Too late and too expensive for inner-loop DSE search if every point must synthesize (unverified). Current checked-in report flow emits Vivado timing, utilization, hierarchical utilization, and RAM utilization reports, not an HLS schedule-latency parser (`resources/synth/zcu.hw-resources/build/getReports.tcl:19-30`). Failure mode: missing/unparseable reports silently masquerade as estimates unless status is explicit. |
| Use HLS simulation / co-simulation | Can measure emitted behavior under real test inputs and catch wrapper/back-pressure effects that static reports may miss (unverified). Existing Spatial host instrumentation already reads per-controller cycle/iteration counters from generated args (`src/spatial/codegen/cppgen/CppGenAccel.scala:71-89`, `src/spatial/codegen/cppgen/CppGenInterface.scala:148-156`). | Input-dependent and unsuitable as the only DSE objective unless vectors, seeds, start/stop policy, and coverage are recorded (unverified). Failure mode: a fast point wins because the benchmark input missed the slow path. |
| Build a new estimator | Clean long-term architecture: estimate HLS-visible loops, operator/memory latencies, partitions, dependencies, and resource-port pressure directly. It can consume D-06 dependence fields and D-07 partition plans instead of reverse-fitting Spatial runtime constants. | Highest cost and calibration risk. Requires a structured HLS controller/dependence graph, operator latency table, memory-port model, `HlsPartitionPlan`, accepted-II calibration data, and discrepancy labels. Failure mode: a polished estimator becomes a second stale oracle. |
| Hybrid pre/post-HLS workflow | Best staged fit: use a fast pre-HLS estimate for search, then replace or reconcile final numbers with report/simulation facts when available. D-06 recommends requested-II/model-report reconciliation (`20 - Research Notes/50 - Decision Records/D-06.md:45-48`), and D-07 recommends HLS-first partition plans with requested-versus-accepted report fields feeding D-08/D-21 (`20 - Research Notes/50 - Decision Records/D-07.md:83-104`, `20 - Research Notes/50 - Decision Records/D-07.md:141-148`). | Requires provenance discipline. Failure mode: consumers mix estimated, requested, and accepted latency without noticing. |

## Required Metadata

All options need a `latency_source` enum, `latency_cycles`, `latency_status`, `design_point_id`, target/tool version, and source controller mapping. The hybrid/new-estimator path also needs `spatial_compiler_ii`, `user_requested_ii`, `hls_requested_ii`, `hls_accepted_ii`, `hls_report_source`, and `ii_reconciliation_status` from D-06 (`20 - Research Notes/50 - Decision Records/D-06.md:51-72`). D-07 adds the memory side: partition kind, dimension, factor, reshape width, duplicate count, bank depth, port model, binding preference, fallback reason, source evidence, and `tool_accepted_partition` (`20 - Research Notes/50 - Decision Records/D-07.md:85-104`). For simulation authority, also record workload identity, input parameters, seed/files, counter scope, and whether the measurement is functional simulation, RTL simulation, or HLS co-simulation (tool-specific, unverified).

## Recommendation

Choose the hybrid. Keep Spatial runtime-model parity only as the v1 pre-HLS estimator and regression baseline; do not make it final authority. Prefer HLS reports for accepted II and final post-HLS performance when available, with simulation/co-simulation as a validation or workload-specific measurement lane. Build a new estimator incrementally only after D-06/D-07 metadata is stable. The DSE row should expose both the pre-HLS estimate and post-HLS accepted result instead of overwriting one with the other; D-06 already calls for report parsing hooks and runtime/DSE summaries that prefer accepted II only when available (`20 - Research Notes/50 - Decision Records/D-06.md:119-124`).
