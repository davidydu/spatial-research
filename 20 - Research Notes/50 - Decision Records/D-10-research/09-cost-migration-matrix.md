# Implementation Cost and Migration Matrix

## Baseline

D-10/Q-130 asks whether the Rust numeric runtime should match Scalagen f64 transcendentals, use per-format MPFR/exact math, or match synthesized hardware units (`20 - Research Notes/40 - Decision Queue.md:47-49`; `20 - Research Notes/20 - Open Questions.md:1818-1836`). The current reference path is not ambiguous: `Number` explicitly says both fixed and float helpers rely on `Double`, then calls `Math.*(x.toDouble)` and reclamps to the original format (`emul/src/emul/Number.scala:96-114`; `emul/src/emul/Number.scala:139-155`). The spec also makes Scalagen plus `emul` the Rust+HLS ground truth, while noting f64 as the reference unless a higher-precision path is deliberately chosen (`10 - Spec/50 - Code Generation/20 - Scalagen/20 - Numeric Reference Semantics.md:27`; `10 - Spec/50 - Code Generation/20 - Scalagen/20 - Numeric Reference Semantics.md:80-90`; `10 - Spec/20 - Semantics/50 - Data Types.md:36-39`).

## Migration Work

The migration should first port the current f64 path exactly: preserve `pow` validity copying only `x.valid`, keep sigmoid as `1/(exp(-x)+1)`, keep fixed `log2` as an internal helper, and run all results through existing fixed wrap or float clamp semantics (`emul/src/emul/Number.scala:102-114`; `emul/src/emul/Number.scala:141-155`; `emul/src/emul/FixedPoint.scala:203-209`; `emul/src/emul/FloatPoint.scala:417-433`). Then add provenance labels such as `scalagen_f64`, `mpfr_exact`, `hardfloat_sim`, `vivado_ip`, or `hls_native_unverified`, because local evidence shows BigIP is a capability boundary, not a single precision oracle (`fringe/src/fringe/templates/math/Math.scala:9-12`; `10 - Spec/80 - Runtime and Fringe/50 - BigIP and Arithmetic.md:43-49`).

## Recommendation: Provenance-Gated Scalagen Compatibility

Choose **Provenance-Gated Scalagen Compatibility**. Default Rust simulation should match Scalagen f64 plus current reclamp/wrap behavior, because that is the documented reference and the cheapest migration path. Add opt-in MPFR/exact and synthesized-hardware modes only behind explicit policy/provenance fields, with hardware mode gated by D-01 capability and custom-format audit gated by D-18. This avoids surprising users, keeps current tests useful, and leaves room to tighten math correctness without silently changing the reference oracle.
