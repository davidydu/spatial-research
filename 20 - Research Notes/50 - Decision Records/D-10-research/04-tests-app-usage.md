# Tests and Application Usage of Transcendentals

## API and Scalagen Baseline

The public math surface mostly exists for both `Fix` and `Flt`: `pow`, `exp`, `ln`, `sqrt`, `sin`, `cos`, `tan`, `sinh`, `cosh`, `tanh`, `asin`, `acos`, `atan`, and `sigmoid` are exposed through `MathAPI` and the `Num` instances (`src/spatial/lang/api/MathAPI.scala:28`, `src/spatial/lang/api/MathAPI.scala:36`, `src/spatial/lang/api/MathAPI.scala:48`; `argon/src/argon/lang/Fix.scala:134`, `argon/src/argon/lang/Fix.scala:147`; `argon/src/argon/lang/Flt.scala:65`, `argon/src/argon/lang/Flt.scala:78`). I did not find a staged `log2` API or `Flt` implementation; the only direct `log2` implementation is `Number.log2` for `FixedPoint`, plus internal rewrite uses for powers-of-two (`emul/src/emul/Number.scala:102`; `argon/src/argon/node/Fix.scala:119`).

Scalagen is the effective oracle. Both Fix and Flt codegen lower these ops to `Number.*`, or to formulas using `Number.exp`/`Number.sqrt` for sigmoid and Fix reciprocal square root (`src/spatial/codegen/scalagen/ScalaGenFixPt.scala:137`, `src/spatial/codegen/scalagen/ScalaGenFixPt.scala:152`; `src/spatial/codegen/scalagen/ScalaGenFltPt.scala:107`, `src/spatial/codegen/scalagen/ScalaGenFltPt.scala:122`). `Number` itself says these paths rely on `Double`, then calls Java/Scala `Math.*` on `toDouble` and converts back to the target format (`emul/src/emul/Number.scala:96`, `emul/src/emul/Number.scala:98`, `emul/src/emul/Number.scala:114`; `emul/src/emul/Number.scala:139`, `emul/src/emul/Number.scala:141`, `emul/src/emul/Number.scala:155`).

## Checked Unit Coverage

Direct checked coverage is narrow. `FloatBasics` asserts `sqrt(in1)` and `pow(in1,2)` against host-side DSL golds, with absolute margins of `0.5` for sqrt and `0.01` for pow (`test/spatial/tests/feature/math/FloatBasics.scala:34`, `test/spatial/tests/feature/math/FloatBasics.scala:58`, `test/spatial/tests/feature/math/FloatBasics.scala:75`). `FloatBasics2`, the only math test in both mini and nightly regression lists, comments out true `sqrt`, `exp`, `ln`, `sigmoid`, and `tanh`; it does check `1/sqrt(sram1(i))`, accepts NaNs, and uses a very loose `+-8` margin for sqrt/reciprocal-sqrt lanes versus `1e-6` elsewhere (`test/spatial/tests/feature/math/FloatBasics2.scala:42`, `test/spatial/tests/feature/math/FloatBasics2.scala:49`, `test/spatial/tests/feature/math/FloatBasics2.scala:52`, `test/spatial/tests/feature/math/FloatBasics2.scala:91`; `regressions/mini.list:1`, `regressions/nightly.list:1`).

Fixed-point tests mostly avoid the real transcendental ops. `FixBasics` checks `sqrt_approx` and `1/sqrt_approx`, not `sqrt`, with comments that `atan`, `exp`, and `ln` are disabled, and tolerates `<=3` integer error or `<=2` Q16 units because "vcs simulation of sqrt is sketchy" (`test/spatial/tests/feature/math/FixBasics.scala:14`, `test/spatial/tests/feature/math/FixBasics.scala:26`, `test/spatial/tests/feature/math/FixBasics.scala:80`). `FixBasics2` similarly uses `sqrt_approx(abs(...))`, comments out `exp`, `ln`, reciprocal, reciprocal-sqrt, `sigmoid`, and `tanh`, and gives sqrt lanes `+-8` (`test/spatial/tests/feature/math/FixBasics2.scala:42`, `test/spatial/tests/feature/math/FixBasics2.scala:49`, `test/spatial/tests/feature/math/FixBasics2.scala:91`). `HalfBasics` is a stronger consistency test for `sqrt`, `1/sqrt`, and `exp_taylor`, but still compares accelerator output to the same host DSL expressions with `1e-6` tolerance (`test/spatial/tests/feature/math/HalfBasics.scala:42`, `test/spatial/tests/feature/math/HalfBasics.scala:49`, `test/spatial/tests/feature/math/HalfBasics.scala:89`).

## Approximation Tests

Several files exercise approximation helpers, not the core ops. `MathAPI` defines `exp_taylor`, `log_taylor`, `sin_taylor`, `cos_taylor`, and `sqrt_approx` with hand-coded Taylor/linear regions (`src/spatial/lang/api/MathAPI.scala:80`, `src/spatial/lang/api/MathAPI.scala:104`). `FunctionApproximations` and `FixBasics3` generate arrays for exp, sin, cos, and sqrt approximations, print host golds, then assert only the unrelated multiply result; the transcendental checks are commented out (`test/spatial/tests/feature/math/FunctionApproximations.scala:81`, `test/spatial/tests/feature/math/FunctionApproximations.scala:103`; `test/spatial/tests/feature/math/FixBasics3.scala:81`, `test/spatial/tests/feature/math/FixBasics3.scala:103`). `FloatBasics_buggy_boolean` computes `exp`, `ln`, `sigmoid`, and `tanh`, but only prints them; the gold/cksum block is commented (`test/spatial/tests/feature/math/FloatBasics_buggy_boolean.scala:82`, `test/spatial/tests/feature/math/FloatBasics_buggy_boolean.scala:184`).

## Apps and Libraries

Application coverage is mostly algorithm-level. `BlackScholes` uses `exp_taylor`, `log_taylor`, and `sqrt_approx` and checks option prices within `0.5`, so it constrains application drift but not exact primitive behavior (`test/spatial/tests/feature/dense/BlackScholes.scala:16`, `test/spatial/tests/feature/dense/BlackScholes.scala:51`, `test/spatial/tests/feature/dense/BlackScholes.scala:151`; `regressions/pir.list:4`). `LogReg` uses real `sigmoid` in accelerator and host gold, but accepts per-element error within `5` (`test/spatial/tests/feature/dense/LogReg.scala:55`, `test/spatial/tests/feature/dense/LogReg.scala:90`, `test/spatial/tests/feature/dense/LogReg.scala:107`; `regressions/pir.list:6`). SGD-style apps use `pow(...,2)` for costs and convergence thresholds, not primitive accuracy (`test/spatial/tests/apps/LP_SGD.scala:225`, `test/spatial/tests/apps/LP_SGD.scala:247`). Library demand is real in batch norm and kernels: `LinearAlgebra.batchNorm2d` uses `pow` and `sqrt`, and `ML.polynomial_kernel` uses `pow` (`src/spatial/lib/LinearAlgebra.scala:386`, `src/spatial/lib/LinearAlgebra.scala:389`; `src/spatial/lib/ML.scala:300`).

## D-10 Implications

Compatibility risk is highest for anything that must match Scalagen f64 behavior exactly. `sqrt` and `pow` have some checked coverage, but with broad or self-referential host-gold tolerances. `recipSqrt` is indirect through `1/sqrt` or `pow(_, -0.5)` rewrites (`argon/src/argon/node/Fix.scala:260`, `argon/src/argon/node/Flt.scala:45`). `exp` and `sigmoid` are covered mainly through coarse app outcomes. `ln`, `log2`, `sin`, `cos`, `tan`, `sinh`, `cosh`, `tanh`, `asin`, `acos`, and `atan` have little to no asserted Fix/Flt behavioral coverage. D-10 should assume tests protect broad compatibility smoke, not bit-level or tight numerical compatibility.
