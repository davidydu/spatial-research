# HLS/DSL Memory Binding Precedent

## Precedent Pattern

Mainstream HLS practice separates *what the program declares* from *where the compiler binds it*: arrays can be partitioned or reshaped, storage can be pushed toward BRAM/URAM/LUTRAM/registers with resource or `bind_storage`-style directives, streams/FIFOs get their own depth-bearing surface, and the result is checked in synthesis reports (unverified). That is a useful precedent for D-04, but the local anchor is Spatial's own DSL. Spatial exposes distinct storage forms at construction time: `SRAM.apply` stages dimensional `SRAMNew`, `RegFile.apply` stages dimensional `RegFileNew`, `FIFO.apply` stages `FIFONew(depth)`, and `DRAM.apply` stages host DRAM allocations (`src/spatial/lang/SRAM.scala:139`, `src/spatial/lang/RegFile.scala:75`, `src/spatial/lang/FIFO.scala:86`, `src/spatial/lang/DRAM.scala:40`). This argues for an HLS taxonomy that records source storage class separately from backend binding.

## Binding And Partitioning

Spatial's closest local precedent for HLS `array_partition` is explicit banking metadata. The SRAM API exposes `forcebank`, `fullybankdim`, and `fullybanked`, and `fullybanked` lowers to a bank scheme based on constant dimensions (`src/spatial/lang/SRAM.scala:120`, `src/spatial/lang/SRAM.scala:122`, `src/spatial/lang/SRAM.scala:126`). The metadata layer records user-specified `BankingScheme` and `ExplicitBanking`, rather than folding this into the memory's type (`src/spatial/metadata/memory/BankingData.scala:637`, `src/spatial/metadata/memory/BankingData.scala:644`). Therefore D-04 should treat HLS partitioning as allocation modifiers: `partition_kind`, `partition_factor`, `partition_dim`, `bank_count`, and `port_model`, not as separate resource classes. Vendor names such as Vitis `array_partition`, Intel memory attributes, and `bind_storage` pragmas remain external precedent (unverified), but their conceptual role matches Spatial's split between memory kind and banking/binding directives.

## Resource Binding Taxonomy

Spatial already has a resource-binding taxonomy below the user-facing memory API. `HardwareTarget` requires each target to publish area fields, capacity, `memoryResources`, and `defaultResource` (`src/spatial/targets/HardwareTarget.scala:14`, `src/spatial/targets/HardwareTarget.scala:29`, `src/spatial/targets/HardwareTarget.scala:44`). A `MemoryResource` has a resource `name`, area model, summary function, and optional `minDepth` (`src/spatial/targets/MemoryResource.scala:6`, `src/spatial/targets/MemoryResource.scala:9`). Xilinx-style targets bind through an ordered list: URAM, BRAM, URAM overflow, then LUTs, with BRAM as the default metadata fallback (`src/spatial/targets/xilinx/XilinxDevice.scala:13`, `src/spatial/targets/xilinx/XilinxDevice.scala:18`, `src/spatial/targets/xilinx/XilinxDevice.scala:23`, `src/spatial/targets/xilinx/XilinxDevice.scala:28`, `src/spatial/targets/xilinx/XilinxDevice.scala:33`, `src/spatial/targets/xilinx/XilinxDevice.scala:39`). HLS should mirror this with explicit classes such as `register_scalar`, `lutram_or_distributed`, `bram`, `uram`, `fifo_stream`, and `external_dram`, plus a vendor binding string when the backend has a more precise spelling (unverified).

## Depth, Capacity, And Reports

HLS reports normally become the feedback loop for whether the binding was honored (unverified). Spatial's local loop is capacity-field based. The allocator starts with `target.capacity`, computes candidate resource area, and assigns only if area fits and bank depth meets the resource minimum (`src/spatial/traversal/MemoryAllocator.scala:46`, `src/spatial/traversal/MemoryAllocator.scala:63`, `src/spatial/traversal/MemoryAllocator.scala:85`). The depth being tested is physical depth per bank: `memoryBankDepth` delegates to `Memory.bankDepth`, while raw memory area multiplies resource-per-bank by total banks and buffer depth (`src/spatial/targets/AreaModel.scala:115`, `src/spatial/metadata/memory/BankingData.scala:195`, `src/spatial/targets/AreaModel.scala:145`, `src/spatial/targets/AreaModel.scala:153`). Capacity fields need to line up with reports: Zynq publishes `SLICEL`, `SLICEM`, `Slices`, `Regs`, `BRAM`, and `DSPs`, while AWS also publishes `URAM` (`src/spatial/targets/xilinx/Zynq.scala:11`, `src/spatial/targets/xilinx/AWS_F1.scala:13`). `MemoryReporter` emits per-instance resource names, and DSE validates `area <= capacity` while emitting only capacity-backed headings (`src/spatial/report/MemoryReporter.scala:57`, `src/spatial/dse/DSEThread.scala:120`, `src/spatial/dse/DSEThread.scala:122`). D-04 should therefore make `physical_depth_per_bank`, `bank_count`, `buffer_depth`, `total_bits`, and report field names first-class.

## Catch-All Implication

The most important precedent is the fallback rule. Spatial tries every target resource except the last, then assigns every remaining SRAM/FIFO/LIFO-capable instance and every non-SRAM duplicate to `resources.last` without another capacity or depth check (`src/spatial/traversal/MemoryAllocator.scala:56`, `src/spatial/traversal/MemoryAllocator.scala:59`, `src/spatial/traversal/MemoryAllocator.scala:102`). For Xilinx this means `LUTs`; for ASIC and Plasticine, their singleton `SRAM_RESOURCE` is both default and catch-all (`src/spatial/targets/generic/ASIC.scala:14`, `src/spatial/targets/generic/ASIC.scala:20`, `src/spatial/targets/plasticine/Plasticine.scala:20`, `src/spatial/targets/plasticine/Plasticine.scala:26`). D-04 should preserve that behavior deliberately: define a final `fallback_soft_local` allocation class, document that it is last in priority order, and require report validation to surface overuse rather than silently pretending the fallback is BRAM.
