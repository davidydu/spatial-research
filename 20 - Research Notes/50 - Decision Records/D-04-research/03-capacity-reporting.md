## Capacity Shape

`HardwareTarget` makes area accounting field-driven: each target declares `AFIELDS`, derives `AreaFields` from it, and exposes `capacity: Area` as the device maximum in those fields (`src/spatial/targets/HardwareTarget.scala:14`, `src/spatial/targets/HardwareTarget.scala:22`, `src/spatial/targets/HardwareTarget.scala:29`). `Area` is a `Model[Double, AreaFields]`; missing entries read as the field default, and `<=` checks every configured field, not only fields present in the left-hand map (`models/src/models/package.scala:6`, `models/src/models/Model.scala:26`, `models/src/models/Model.scala:48`). This means capacity omissions are semantically zero for validation.

For FPGA-like targets, `AFIELDS` contains primitive memory fields (`RAM18`, `RAM36`, distributed RAM fields), aggregate fields (`BRAM`, `URAM`), and non-memory fields (`DSPs`, `Regs`, slices) (`src/spatial/targets/xilinx/XilinxDevice.scala:10`). Concrete capacities normally publish aggregate limits such as `BRAM`, `DSPs`, `Regs`, and slices; AWS also publishes `URAM` (`src/spatial/targets/xilinx/Zynq.scala:11`, `src/spatial/targets/xilinx/AWS_F1.scala:13`). Therefore HLS capacity import must feed the aggregate fields consumed by reporting and validity, especially `BRAM`, `URAM` where present, `DSPs`, `Regs`, `SLICEL`, `SLICEM`, and `Slices`.

## Resource Taxonomy And Depth

`MemoryResource` is the allocation taxonomy: each resource has a `name`, an area function, a summary function, and an optional `minDepth` defaulting to zero (`src/spatial/targets/MemoryResource.scala:6`). Xilinx-style targets define `URAM_RESOURCE` with `minDepth = 512`, `BRAM_RESOURCE` with `minDepth = 32`, an overflow URAM resource also named `URAM` with `minDepth = 32`, and a final `LUTs_RESOURCE` with no depth override (`src/spatial/targets/xilinx/XilinxDevice.scala:13`, `src/spatial/targets/xilinx/XilinxDevice.scala:18`, `src/spatial/targets/xilinx/XilinxDevice.scala:23`, `src/spatial/targets/xilinx/XilinxDevice.scala:28`). The ordered list is `URAM`, `BRAM`, overflow `URAM`, then `LUTs`, while the default metadata fallback is `BRAM_RESOURCE` (`src/spatial/targets/xilinx/XilinxDevice.scala:33`, `src/spatial/targets/xilinx/XilinxDevice.scala:39`).

Area estimation for allocation is bank-aware. `memoryBankDepth` delegates to the `Memory` instance's `bankDepth`, and `bankDepth` divides each banked axis size by its number of banks before multiplying across banking levels (`src/spatial/targets/AreaModel.scala:115`, `src/spatial/metadata/memory/BankingData.scala:195`). `rawMemoryBankArea` asks the candidate resource for area at that bank depth, and `rawMemoryArea` multiplies per-bank area by total banks and buffer depth (`src/spatial/targets/AreaModel.scala:124`, `src/spatial/targets/AreaModel.scala:145`). The min-depth rule therefore applies to per-bank depth, not total logical memory size.

## Allocation Behavior

`MemoryAllocator` starts with `target.capacity`, then iterates all memory resources except the last (`src/spatial/traversal/MemoryAllocator.scala:46`, `src/spatial/traversal/MemoryAllocator.scala:59`). For each candidate resource it computes a raw area through `rawMemoryArea`, reduces that through `resource.summary`, and wraps the result as `Area(resource.name -> count)` for capacity comparison (`src/spatial/traversal/MemoryAllocator.scala:31`, `src/spatial/traversal/MemoryAllocator.scala:64`). Assignment requires both `area <= capacity` and `depth >= resource.minDepth`; accepted assignments store `dup.resourceType = Some(resource)` and subtract the area from remaining capacity (`src/spatial/traversal/MemoryAllocator.scala:80`, `src/spatial/traversal/MemoryAllocator.scala:85`, `src/spatial/traversal/MemoryAllocator.scala:87`).

The final resource is the catch-all: after the capacity-checked loop, every still-unassigned SRAM-capable instance and every non-SRAM duplicate is assigned `resources.last` without another capacity or depth check (`src/spatial/traversal/MemoryAllocator.scala:102`). For current Xilinx-style targets, that means `LUTs_RESOURCE`; for ASIC and Plasticine, the singleton list makes `SRAM_RESOURCE` both default and catch-all (`src/spatial/targets/generic/ASIC.scala:14`, `src/spatial/targets/plasticine/Plasticine.scala:20`).

## Metadata, Reports, And DSE

Allocated choices live in `BankingData.Memory.resourceType`; reads of `inst.resource` fall back to the target default when no allocation has set a resource (`src/spatial/metadata/memory/BankingData.scala:186`, `src/spatial/metadata/memory/BankingData.scala:187`). `MemoryReporter` uses `areaModel.areaOf` to print memory area totals, but its per-instance summary prints `inst.resource.name`, so the report shows allocation metadata even though its area totals do not call `rawMemoryArea` with that resource (`src/spatial/report/MemoryReporter.scala:21`, `src/spatial/report/MemoryReporter.scala:57`).

The DSE path is stricter about capacity fields but currently looser about resource metadata. `DSEAreaAnalyzer.areaOf` delegates to `areaModel.areaOf`, and `AreaModel.areaOf` handles `MemAlloc` through `areaOfMem`, not `Memory.resource` (`src/spatial/dse/DSEAreaAnalyzer.scala:62`, `src/spatial/targets/AreaModel.scala:171`). Its postprocess summarizes raw fields into aggregate fields, for example Xilinx converts `RAM18/RAM36` into `BRAM`, distributed RAM into memory slices, and emits totals for `SLICEM`, `SLICEL`, `Slices`, `Regs`, `DSPs`, `BRAM`, and `URAM` (`src/spatial/dse/DSEAreaAnalyzer.scala:50`, `src/spatial/targets/xilinx/XilinxAreaModel.scala:18`, `src/spatial/targets/xilinx/XilinxAreaModel.scala:97`).

`DSEThread` then derives CSV area headings from `capacity.nonZeroFields`, validates each design point with `area <= capacity`, and emits only those capacity-backed fields (`src/spatial/dse/DSEThread.scala:49`, `src/spatial/dse/DSEThread.scala:50`, `src/spatial/dse/DSEThread.scala:120`, `src/spatial/dse/DSEThread.scala:123`). HLS capacity fields must therefore feed both allocator resource names (`BRAM`, `URAM`, and any non-final named resource) and DSE/report aggregate fields (`Slices`, `SLICEL`, `SLICEM`, `Regs`, `DSPs`, `BRAM`, `URAM`). If HLS adds a new named memory resource that is not the last catch-all, its `resource.name` must also be an `AFIELDS`/capacity field, or allocation sees zero available capacity by default.
