# Min-Depth And Threshold Policy

## Current Spatial Policy

`MemoryResource` has two required behaviors, `area(width, depth)` and `summary(area)`, plus a concrete default `minDepth = 0`; any resource that does not override it is eligible at every bank depth (`src/spatial/targets/MemoryResource.scala:6-10`). Xilinx, Generic, Altera, and Euresys all use the same ordered taxonomy: primary `URAM_RESOURCE`, `BRAM_RESOURCE`, `URAM_RESOURCE_OVERFLOW`, then `LUTs_RESOURCE` (`src/spatial/targets/xilinx/XilinxDevice.scala:33-39`, `src/spatial/targets/generic/GenericDevice.scala:38-44`, `src/spatial/targets/altera/AlteraDevice.scala:33-39`, `src/spatial/targets/euresys/EuresysDevice.scala:33-39`). In all four, primary URAM requires `minDepth = 512`, BRAM requires `minDepth = 32`, overflow URAM requires `minDepth = 32`, and LUTs inherit the zero default (`src/spatial/targets/xilinx/XilinxDevice.scala:13-31`, `src/spatial/targets/generic/GenericDevice.scala:13-31`, `src/spatial/targets/altera/AlteraDevice.scala:13-31`, `src/spatial/targets/euresys/EuresysDevice.scala:13-31`).

## Depth And Capacity Mechanics

The allocator treats `target.memoryResources` as an ordered priority list but iterates only `dropRight(1)`, so the final element is not capacity-tested during greedy packing (`src/spatial/traversal/MemoryAllocator.scala:56-60`). For each non-final resource, it computes raw memory area, summarizes it, wraps the result as `Area(resource.name -> count)`, sorts remaining candidates by descending raw count, and assigns a duplicate only if `area <= capacity && depth >= resource.minDepth` (`src/spatial/traversal/MemoryAllocator.scala:62-88`). The tested `depth` is bank depth, not logical element count or n-buffer depth: `memoryBankDepth` calls `instance.bankDepth(dims)`, and `Memory.bankDepth` multiplies per-banking-axis `ceil(size / nBanks)` values (`src/spatial/targets/AreaModel.scala:109-117`, `src/spatial/metadata/memory/BankingData.scala:190-200`). Capacity accounting uses the target's `Area` schema; lookup of a missing field returns the `AreaFields` default, so an omitted capacity behaves as zero for that resource name (`src/spatial/targets/HardwareTarget.scala:22`, `models/src/models/Fields.scala:11-12`, `models/src/models/Model.scala:25-26`).

## Consequences

The current thresholds are hard eligibility gates, not tuning hints: a bank at depth 511 cannot receive primary URAM automatically, and a bank at depth 31 cannot receive BRAM or overflow URAM, even if the area model says it would consume one block and the target has capacity (`src/spatial/traversal/MemoryAllocator.scala:80-96`). The overflow URAM entry is intentionally a second pass over the same `URAM` resource with a lower depth threshold, after BRAM has had a chance to consume medium memories (`src/spatial/targets/xilinx/XilinxDevice.scala:23-27`, `src/spatial/traversal/MemoryAllocator.scala:49-56`). Because depth is bank depth, aggressive banking can push a large logical memory below the BRAM/URAM threshold, while buffering multiplies resource area later through `resourcesPerBuffer * bufferDepth` without changing eligibility (`src/spatial/targets/AreaModel.scala:145-153`). Finally, all still-unassigned SRAM-able duplicates and all non-SRAM-able duplicates are assigned to `resources.last` unconditionally (`src/spatial/traversal/MemoryAllocator.scala:102-103`). On the Xilinx-like targets this means LUT memory; on ASIC and Plasticine it means the sole `SRAM_RESOURCE` catch-all (`src/spatial/targets/xilinx/XilinxDevice.scala:33-39`, `src/spatial/targets/generic/ASIC.scala:14-20`, `src/spatial/targets/plasticine/Plasticine.scala:20-26`).

## HLS Recommendation

For HLS, preserve the allocator shape but make the taxonomy explicit: capacity-accounted resources should be ordered as `URAM`, `BRAM`, optional `URAM_OVERFLOW`, optional `LUTRAM`, and a final `AUTO` catch-all. `AUTO` should emit no hard storage binding and should not be treated as proof that area is free; it is the place where HLS inference, downstream synthesis reports, or later diagnostics resolve storage (unverified). This is safer than making LUTRAM the universal final resource, because HLS tools may infer registers, BRAM, distributed RAM, or fail based on access pattern and pragma support (unverified).

The default HLS thresholds should start from Spatial's proven policy: `URAM.minDepth = 512`, `BRAM.minDepth = 32`, `URAM_OVERFLOW.minDepth = 32`, `LUTRAM.minDepth = 0`, and `AUTO.minDepth = 0`. However, they should be target-configurable fields, not global constants, because HLS memory inference and available report fields vary by vendor, device, and pragma family (unverified). Treat them as hard eligibility for automatic allocation to named physical resources, since the allocator's capacity accounting assumes that an accepted duplicate really consumes that resource. Explicit user binding can be allowed as an override, but should be diagnostic/advisory: warn when the requested resource violates the configured min-depth and let the backend either emit the pragma or reject it if the HLS target cannot honor it (unverified).

Capacity fields should match `resource.name`, not internal primitive details. If HLS reports `BRAM_18K`, `URAM`, or `LUTRAM_LUTS`, the HLS target should either name resources in those terms or provide a summary function that reduces primitive fields into the capacity key being decremented, mirroring Spatial's BRAM summary over `RAM18` and `RAM36` (`src/spatial/targets/xilinx/XilinxDevice.scala:18-22`, `src/spatial/traversal/MemoryAllocator.scala:62-88`).
