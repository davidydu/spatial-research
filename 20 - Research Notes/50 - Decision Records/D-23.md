# D-23 - Rust/HLS Host ABI Manifest

## Question

Define the Rust/HLS host ABI manifest.

Source: `/Users/david/Documents/David_code/spatial/src/spatial/codegen/cppgen/CppGenInterface.scala:138-164`; `src/spatial/codegen/chiselgen/ChiselGenInterface.scala:156-187`; [[10 - Cppgen]]

Decision criteria: User decision: define the Rust/HLS host ABI manifest for scalar args, DRAM pointers, ArgIO/ArgOuts, counters, exits, and streams.

## Research findings

- Cppgen already has a host ABI, but it is distributed across mutable backend buffers and generated constants. Its order is ArgIns, DRAM pointer args, ArgIOs, ArgOuts, instrumentation counters, then early exits; launch code separately sets the corresponding runtime counts. Streams mostly do not get a host transport in legacy Cppgen. See [[01-cppgen-host-interface]].

- Chisel/Fringe repeats the ordinal scalar ABI and adds command/status registers, wrapper fields, memory stream metadata, AXI stream descriptors, instrumentation counters, exits, and target globals. VCS and AWS host contexts lower the same logical ABI differently, so target-specific mechanics need to sit behind one logical manifest. See [[02-chisel-fringe-host-interface]].

- The language/API surface is broader than a top-level HLS function signature. `setArg`/`getArg`, `setMem`/`getMem`, `setFrame`/`getFrame`, tensor flattening, DRAM ranks/dimensions, FileIO, StreamIn/Out buses, Frame linkage, and host tensors all imply manifest fields. See [[03-language-host-data-model]].

- Fixed-point and scalar conversion must be represented without deciding D-24. Cppgen uses fractional fixed values as host `double` in some places and shifted raw integers in transport paths; vectors, structs, bit views, sub-byte packing, floats, and bools each need explicit numeric and packing records. See [[04-fixed-point-d24-overlap]].

- Streams and files must reuse D-22 labels. FileBus/FileEOFBus are simulator/testbench resources, AXI streams need packet sidebands, and stream endpoints need mode labels such as `sim_compat_file_elastic` versus `hls_ready_valid_bounded`. See [[05-stream-file-hostio-d22-overlap]].

- A concrete `abi_manifest_v1` schema should include kernel identity, ABI version, target/backend facts, host register entries, memory entries, transfer records, stream descriptors, host-file records, numeric format registry, packing policy, lifecycle, diagnostics, and compatibility modes. See [[06-abi-manifest-schema]].

- Tests and harnesses depend on argv order, run wrappers, environment, success parsing, ArgAPI offsets, runtimeArgs, stream prompts, frame transfers, counters, and early exits. Missing fixtures include noninteractive FileBus replay, multiple AXI streams, frame transport in Cppgen/Scalagen/HLS, manifest readback, and stale layout detection. See [[07-tests-apps-run-harness]].

- The main risk of no manifest is silent drift. Ad hoc compatibility can hide scalar conversion, pointer-order, shape/stride, stream sideband, counter/exit, endian/packing, and target runtime differences until they appear as value mismatches. See [[08-risk-failure-modes]].

- Migration cost is highest for manifest-first but lowest long-term. A staged rollout can first emit the manifest sidecar while still generating `ArgAPI.hpp`/`ArgAPI.scala`, then derive legacy APIs and Rust host bindings from it, then require version/hash checks. See [[09-implementation-cost-migration]].

- The recommendation matrix selects an explicit versioned manifest-first ABI. Cppgen and Chisel/Fringe should become projections; target-native HLS signatures should be generated from the manifest rather than treated as the portable source contract. See [[10-recommendation-matrix]].

## Recommendation

Adopt **Explicit Versioned Manifest-First Host ABI**.

D-23 should define `abi_manifest_v1` as the source of truth for the Rust/HLS host boundary. Cppgen-compatible constants, Chisel/Fringe constants, Rust host bindings, HLS top signatures, simulator adapters, and reports should all be generated from or checked against this manifest. Legacy layouts remain important, but they should be named projections:

- `legacy_cppgen_ordinal_v0`
- `legacy_fringe_register_v0`
- `target_native_hls_<backend>_v1`

The canonical manifest should include these root groups:

1. `kernel`: name, entry symbol, source module, source hash, IR hash, producer versions.
2. `abi`: schema version, ABI version, manifest hash, minimum runtime version.
3. `target`: backend, tool/vendor, part/board/shell, clock, address width, data width, stream width, memory channel count, capabilities.
4. `host_abi`: ordered scalar and pointer slots for `ArgIn`, DRAM pointers, `ArgIO`, `ArgOut`, counters, and exits, including logical counts versus padded hardware counts.
5. `memories`: DRAM/LockDRAM/Frame records with rank, shape, row-major flat layout, element type, allocation owner, pointer slot, transfer directions, and packing policy.
6. `streams`: internal memory-stream descriptors and external StreamIn/StreamOut endpoint descriptors, including bus kind, AXI sidebands, file metadata, capacity/back-pressure policy, and D-22 queue-policy labels.
7. `host_files`: file path, mode, parser/serializer, encoding, EOF policy, and attached stream or DRAM.
8. `numeric_formats`: bool, integer, fixed, float, struct, vector, storage bits, signedness, fractional bits, exponent/mantissa bits, field offsets, endian/packing, and `conversion_policy`.
9. `lifecycle`: bind args, allocate/copy memories, bind streams/files, start, poll/wait, read outputs/counters/exits, copy back, close/free.
10. `diagnostics`: stable source locations, controller/channel/memory ids, conversion mismatches, OOB, stream stalls, early exits, stale reports, unsupported target features, and manifest hash mismatch.

D-23 must deliberately stop short of deciding D-24. Numeric entries should carry `conversion_policy` values such as `cppgen_fractional_double_shift_v1`, `bit_exact_scaled_integer_v1`, `raw_memcpy_v1`, `packed_subbyte_lsb0_v1`, and `unsupported_fractional_subbyte_v1`; D-24 can choose the default fixed-point policy. Likewise, D-23 should reuse D-22 stream labels rather than invent a parallel stream model.

The manifest should make compatibility explicit. During migration, continue emitting `ArgAPI.hpp` and `ArgAPI.scala`, but test that they match `abi_manifest_v1`. Once Rust/HLS host bindings consume the manifest directly, stale or missing manifests should be rejected rather than guessed from traversal order.

### Rejected alternatives

Reject **no manifest/ad hoc ABI**. The ABI exists already; leaving it implicit preserves hidden coupling and makes failures hard to localize.

Reject **Cppgen-compatible as the canonical ABI**. It is a necessary legacy projection, but it cannot fully describe streams, fixed-point ambiguity, target-specific register mechanics, or HLS endpoint policies.

Reject **Chisel/Fringe-compatible as the sole ABI**. It is the right RTL projection, but it overfits 64-bit register files, command/status slots, generated globals, and current target shells.

Reject **target-native HLS ABI as source contract**. Vendor-native signatures are backend projections. They should be generated from the Spatial manifest, not define Spatial's host semantics.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Define and document `abi_manifest_v1` with JSON as canonical serialization and an optional human-readable rendering.
2. Add a shared ABI collector before backend-specific codegen emits Cppgen, Chisel/Fringe, Rust, or HLS artifacts.
3. Emit `abi_manifest_v1.json` alongside legacy `ArgAPI.hpp`/`ArgAPI.scala` and generated run wrappers.
4. Add manifest readback tests that compare ArgAPI constants, runtime count setters, DRAM pointer slots, counter slots, early exits, stream descriptors, and target widths to the manifest.
5. Add Rust host bindings that consume the manifest for scalar writes, DRAM allocation/copy, ArgIO/ArgOut readback, counters, exits, streams, files, and lifecycle.
6. Add D-22 stream policy fields to every stream endpoint and D-24 `conversion_policy` fields to every numeric payload.
7. Add target adapters that lower the manifest to VCS/AWS/other legacy register mechanics and to target-native HLS top signatures.
8. Add compatibility modes for `legacy_cppgen_ordinal_v0` and `legacy_fringe_register_v0`, with warnings for fields that cannot be represented losslessly.
9. Add stale/missing manifest diagnostics and include the manifest hash in reports, simulations, and HLS artifacts.
10. Add missing ABI fixtures for FileBus/FileEOFBus, multiple AXI streams, payload-only versus full AXI packets, frame transfers, ArgIO loopback, fixed-point conversion labels, and early-exit/counter readback.
