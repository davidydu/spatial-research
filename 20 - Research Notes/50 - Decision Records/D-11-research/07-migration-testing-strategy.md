# Migration/testing strategy for `breakWhen` semantics

## Migration posture and simulator modes

D-11/Q-132 is a real backend split, not a documentation nit: the decision queue asks whether Rust simulation should match Scalagen end-of-iteration `breakWhen` or synthesized immediate-break behavior (`20 - Research Notes/40 - Decision Queue.md:51-53`), and the open question says the two produce different memory/register effects (`20 - Research Notes/20 - Open Questions.md:1868-1877`). Treat this as a named simulator policy with three modes. `compat_scalagen_eoi` matches the current Scala oracle: Scalagen emits `while(hasItems_$lhs && !stopWhen.value)` or `cchain.takeWhile(!stopWhen.value)`, with the warning that Scala breaks at loop end while `--synth` breaks immediately (`src/spatial/codegen/scalagen/ScalaGenController.scala:75-93`); `Counter.takeWhile` also checks the predicate before `func` and then commits the body (`emul/src/emul/Counter.scala:24-31`). `hls_immediate` is the architectural mode: the API promises immediate break/reset (`src/spatial/lang/control/Control.scala:41-56`), Chisel drives `sm.io.break` from the stop register and resets it on controller done (`src/spatial/codegen/chiselgen/ChiselGenController.scala:312-315`), and memory enables are gated by `~break` (`src/spatial/codegen/chiselgen/ChiselGenMem.scala:40-50`). `dual_diff` runs both policies and emits divergence diagnostics.

## Golden tests that expose the split

Create small, named golden tests before migrating broad suites. First, a bounded `Sequential` where `stop := i == 3` precedes an SRAM write. End-of-iteration commits index 3; immediate mode leaves it zero. This mirrors `Break`, where `stop2` is assigned before `s2(i) = src(i)` but `gold2` excludes the break index (`test/spatial/tests/feature/control/Breakpoint.scala:62-68`, `test/spatial/tests/feature/control/Breakpoint.scala:98-111`). Second, a nested row-copy test matching `stop4`: load a row, set the breaker from the row header, then store the row; immediate mode keeps the previous row, while Scalagen mode stores the triggering row (`test/spatial/tests/feature/control/Breakpoint.scala:71-80`, `test/spatial/tests/feature/control/Breakpoint.scala:99-110`). Third, keep a non-divergent drain case where all useful stores happen before the break write; `StreamPipeFlush` fills output SRAM and only then writes `break := true`, so both modes should preserve the eight asserted outputs (`test/spatial/tests/feature/dynamic/StreamPipeFlush.scala:41-47`, `test/spatial/tests/feature/dynamic/StreamPipeFlush.scala:51-56`). Fourth, add a priority-dequeue sentinel variant that enqueues/dequeues after `kill := true` so queue effects distinguish the modes; existing `PriorityDeq` cases already use `Stream(breakWhen = kill)` and sentinel kill paths, but some only assert count or non-hang (`test/spatial/tests/feature/dynamic/PriorityDeq.scala:48-85`, `test/spatial/tests/feature/dynamic/PriorityDeq.scala:195-197`).

## Updating existing tests and diagnostics

Classify every `breakWhen` test as `mode_independent`, `immediate_required`, `scalagen_legacy`, or `expect_divergence`. The `Breakpoint.scala` `stop2`/`stop4` checks are `immediate_required`; its `stop5` stream case is mode-independent because the store precedes the break write (`test/spatial/tests/feature/control/Breakpoint.scala:85-100`). `StreamPipeFlush` should run in both modes as a drain invariant. `PriorityDeq` should split liveness/order checks from semantic checks: preserve "does not hang" and payload-order tests, but add mode-specific assertions near the kill FIFO patterns (`test/spatial/tests/feature/dynamic/PriorityDeq.scala:284-312`, `test/spatial/tests/feature/dynamic/PriorityDeq.scala:329-358`).

The `dual_diff` diagnostic should record the first breaker write and every later same-controller side effect that commits in Scalagen mode but is killed in HLS mode: SRAM/DRAM writes, RegWrites, FIFO/LIFO enqueue/dequeue, StreamIn/Out handshakes, and ArgOut updates. This is supported by existing metadata contracts: `UseAnalyzer` marks stop registers as `isBreaker` (`src/spatial/traversal/UseAnalyzer.scala:57-62`), `breaker`/`stopWhen` lookup identifies the owning controller (`src/spatial/metadata/control/package.scala:423-430`, `src/spatial/metadata/control/package.scala:952-958`), and binding already treats break memories as ordering boundaries (`src/spatial/transform/BindingTransformer.scala:40-42`).

## Metadata, reports, and Rust-vs-HLS comparison

Every simulator artifact and golden record should include `break_when_semantics`, `break_controller`, `breaker_sym`, source context, trigger iteration, committed output hash, post-break side-effect count, and whether divergence was expected. Extend the existing report pattern rather than inventing a side channel: Chisel already emits `numArgOuts_breakpts`, breakpoint contexts, and app characteristics (`src/spatial/codegen/chiselgen/ChiselGenController.scala:537-557`), with `HasBreakpoint` as a first-class app property (`src/spatial/codegen/chiselgen/AppProperties.scala:3-13`). For Rust-vs-HLS, compare `hls_immediate` against HLS C-sim/RTL co-sim outputs and, where possible, a compact side-effect trace. Exact cycle equality is optional until schedules stabilize; equality of committed memories, streams, ArgOuts, and killed post-break effects is mandatory. If `compat_scalagen_eoi` differs but the divergence report names only post-break effects, mark it expected.

## Recommendation implication

Adopt `hls_immediate` as the Rust default, keep `compat_scalagen_eoi` as an explicit legacy/debug mode, and require all break-sensitive tests to declare their mode. This is the narrow exception to the general Scalagen-as-reference rule in the codegen spec (`10 - Spec/50 - Code Generation/00 - Codegen Index.md:13-14`), already called out by the Scalagen controller spec as a policy choice for Rust (`10 - Spec/50 - Code Generation/20 - Scalagen/50 - Controller Emission.md:98-108`).
