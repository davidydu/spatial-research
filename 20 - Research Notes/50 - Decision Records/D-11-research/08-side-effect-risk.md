# Side-effect compatibility risk for `breakWhen`

## 1. Boundary of the divergence

The compatibility risk is not merely “one extra loop trip.” ScalaGen tests `breakWhen` only at controller entry: forever loops become `while(hasItems && !stop.value)` and bounded loops become `cchain.takeWhile(!stop.value)` before the body is emitted (`src/spatial/codegen/scalagen/ScalaGenController.scala:75`, `src/spatial/codegen/scalagen/ScalaGenController.scala:90`, `src/spatial/codegen/scalagen/ScalaGenController.scala:101`). The warning says the Scala break is end-of-loop while synth is immediate (`src/spatial/codegen/scalagen/ScalaGenController.scala:76`). Chisel instead reads `stopWhen`, connects reset to controller done, and drives `sm.io.break` (`src/spatial/codegen/chiselgen/ChiselGenController.scala:314`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:315`). Thus the simulator-visible delta is: after a body writes the break register, do later same-iteration effects commit, and do nested children keep running?

## 2. Scalar, SRAM, DRAM, and ArgOut writes

In end-of-iteration mode, ordinary Scala side effects after the breaker write commit. Reg and ArgOut writes are just `reg.set(v)` (`src/spatial/codegen/scalagen/ScalaGenReg.scala:36`), SRAM writes call `mem.update(...)` (`src/spatial/codegen/scalagen/ScalaGenMemories.scala:179`, `src/spatial/codegen/scalagen/ScalaGenMemories.scala:185`), and dense DRAM stores drain every command/data token into the backing array before clearing the command stream (`src/spatial/codegen/scalagen/ScalaGenDRAM.scala:57`, `src/spatial/codegen/scalagen/ScalaGenDRAM.scala:66`). Immediate hardware suppresses many such writes through break/controller gating: local memory writes use `~break && backpressure` (`src/spatial/codegen/chiselgen/ChiselGenMem.scala:45`, `src/spatial/codegen/chiselgen/ChiselGenMem.scala:50`), local Reg writes lower through the same path (`src/spatial/codegen/chiselgen/ChiselGenMem.scala:277`, `src/spatial/codegen/chiselgen/ChiselGenMem.scala:278`), and ArgOut writes are visible only when the controller datapath enable reaches the write port (`src/spatial/codegen/chiselgen/ChiselGenInterface.scala:89`, `src/spatial/codegen/chiselgen/ChiselGenInterface.scala:99`). The exact user symptom is a different final `getMem`/`getArg`: in `Break`, `stop2` is assigned before `s2(i) = src(i)`, while the gold excludes the break element (`test/spatial/tests/feature/control/Breakpoint.scala:64`, `test/spatial/tests/feature/control/Breakpoint.scala:67`, `test/spatial/tests/feature/control/Breakpoint.scala:98`). Likewise `stop4` is set before copying/storing the loaded row, and gold expects the previous row (`test/spatial/tests/feature/control/Breakpoint.scala:73`, `test/spatial/tests/feature/control/Breakpoint.scala:79`, `test/spatial/tests/feature/control/Breakpoint.scala:99`).

## 3. FIFO and stream enqueue/dequeue

ScalaGen FIFO and stream operations mutate queues directly: FIFO deq/enq use `dequeue()`/`enqueue(...)` (`src/spatial/codegen/scalagen/ScalaGenFIFO.scala:33`, `src/spatial/codegen/scalagen/ScalaGenFIFO.scala:43`), StreamIn dequeues when nonempty, and StreamOut enqueues when enabled (`src/spatial/codegen/scalagen/ScalaGenStream.scala:84`, `src/spatial/codegen/scalagen/ScalaGenStream.scala:95`). Immediate Chisel FIFO accesses use read/write ports whose implicit enables include `~break`, and priority dequeue adds a nonempty guard at codegen (`src/spatial/codegen/chiselgen/ChiselGenMem.scala:40`, `src/spatial/codegen/chiselgen/ChiselGenMem.scala:46`, `src/spatial/codegen/chiselgen/ChiselGenMem.scala:324`). Struct/stream field dequeue readiness is also gated by `~break` (`src/spatial/codegen/chiselgen/ChiselGenStruct.scala:32`, `src/spatial/codegen/chiselgen/ChiselGenStruct.scala:35`). The user sees different queue depths and therefore different store lengths or packet counts. Dynamic `PriorityDeq` stores `result(0::outQueue.numel) store outQueue`, so one extra enqueue or a suppressed dequeue changes the host array length/content (`test/spatial/tests/feature/dynamic/PriorityDeq.scala:70`, `test/spatial/tests/feature/dynamic/PriorityDeq.scala:77`). `StreamPipeFlush` is a teardown-sensitive case: it drains eight `outFIFO` values to SRAM, then sets `break := true`; killing before those dequeues/stores complete would make `outputDRAM` fail its deterministic gold (`test/spatial/tests/feature/dynamic/StreamPipeFlush.scala:41`, `test/spatial/tests/feature/dynamic/StreamPipeFlush.scala:46`, `test/spatial/tests/feature/dynamic/StreamPipeFlush.scala:54`).

## 4. Nested controllers, reductions, and stores

Nested controllers are another divergence surface. ScalaGen’s outer stream lowering can run readable child controls to exhaustion with `while(hasItems_child) { visit(child) }`, independent of a breaker written earlier in the entered parent body (`src/spatial/codegen/scalagen/ScalaGenController.scala:31`, `src/spatial/codegen/scalagen/ScalaGenController.scala:35`). Hardware break participates in active/reset/done and child-ack logic, so children can be acknowledged/reset rather than allowed to keep producing side effects (`fringe/src/fringe/templates/Controllers.scala:106`, `fringe/src/fringe/templates/Controllers.scala:212`, `fringe/src/fringe/templates/Controllers.scala:265`). Reductions are in scope: `OpReduce` and `OpMemReduce` carry `stopWhen` in the IR (`src/spatial/node/Control.scala:57`, `src/spatial/node/Control.scala:92`), and unrolling preserves it into generated `UnitPipe`/`UnrolledReduce` controllers (`src/spatial/transform/unrolling/ReduceUnrolling.scala:23`, `src/spatial/transform/unrolling/ReduceUnrolling.scala:123`, `src/spatial/transform/unrolling/MemReduceUnrolling.scala:139`). A simulator user would see off-by-one accumulator values or stores fed by reductions if the break iteration’s reduce-store commits in one mode but is gated in the other.

## 5. D-11 implication

D-11 should model side-effect visibility explicitly, not only loop termination. A Scala-parity simulator must commit all source-later same-iteration effects once the body has entered. A hardware-faithful simulator must suppress effects whose controller/port enable is killed by `break`, while preserving already-committed drain/final-store work: `FilterStream1D` sets `done` only after storing final packed results (`test/spatial/tests/feature/streaming/FilterStream1D.scala:386`, `test/spatial/tests/feature/streaming/FilterStream1D.scala:389`), and Rosetta drains internal FIFOs before `done := true` and then stores the full frame (`test/spatial/tests/apps/Rosetta.scala:592`, `test/spatial/tests/apps/Rosetta.scala:595`, `test/spatial/tests/apps/Rosetta.scala:601`). The safest D-11 policy is two named modes, with the default chosen by whether Rust simulation is an RTL predictor or a ScalaGen compatibility oracle.
