# Pass Scheduling Interactions for `breakWhen`

## 1. Semantic Split Enters Before Passes

The user API says `Stream(breakWhen=...)` and `Sequential(breakWhen=...)` break immediately and reset the break register when the controller finishes (`src/spatial/lang/control/Control.scala:41-46`, `src/spatial/lang/control/Control.scala:53-56`). That value is not loose metadata; it is stored as `stopWhen` in the IR constructors for `UnitPipe`, `OpForeach`, `OpReduce`, `OpMemReduce`, `UnrolledForeach`, and `UnrolledReduce` (`src/spatial/node/Control.scala:33-50`, `src/spatial/node/Control.scala:57-68`, `src/spatial/node/Control.scala:78-92`, `src/spatial/node/Control.scala:121-139`). The backend split is explicit: ScalaGen warns that Scala breaks at loop end while synthesis breaks immediately (`src/spatial/codegen/scalagen/ScalaGenController.scala:75-93`), whereas Chisel wires `stopWhen` to `sm.io.break` and resets the register on controller done (`src/spatial/codegen/chiselgen/ChiselGenController.scala:312-315`). This means pass behavior is already oriented around hardware immediacy, even when the Scala model is top-tested.

## 2. Pipeline Order and Unrolling

The main pipeline runs DCE before streamify, optional streamify, banking analysis, then unrolling; after unrolling it reruns reg CSE and DCE before rewrite, memory cleanup, flattening, and binding (`src/spatial/Spatial.scala:190-220`). Unrolling treats the breaker as a duplicated memory: foreach, reduce, memreduce, and unitpipe mirroring remap `stopWhen` through the `memories(...,0)` duplicate map before staging the transformed controller (`src/spatial/transform/unrolling/ForeachUnrolling.scala:13-16`, `src/spatial/transform/unrolling/ReduceUnrolling.scala:17-28`, `src/spatial/transform/unrolling/MemReduceUnrolling.scala:14-30`, `src/spatial/transform/unrolling/UnrollingBase.scala:202-204`). Fully unrolled foreach/reduce can become `UnitPipe(..., stopWhen)`, while partial unrolling keeps `UnrolledForeach` or `UnrolledReduce` with the same stop field (`src/spatial/transform/unrolling/ForeachUnrolling.scala:33-45`, `src/spatial/transform/unrolling/ForeachUnrolling.scala:67-80`, `src/spatial/transform/unrolling/ReduceUnrolling.scala:55-79`, `src/spatial/transform/unrolling/ReduceUnrolling.scala:113-123`). The unroller therefore preserves the dependency but does not insert per-lane kill enables; same-iteration cancellation is left to controller scheduling/codegen.

## 3. UnitPipe/Foreach and Streamify Rewrites

The UnitPipe/Foreach conversion passes preserve `stopWhen` structurally. `UnitPipeToForeachTransformer` converts a hardware `UnitPipe` to a one-iteration `OpForeach(..., stopWhen)` and sets the new controller schedule to pipelined (`src/spatial/transform/UnitPipeToForeachTransformer.scala:18-32`). `ForeachToUnitpipeTransformer` converts static single-iteration foreach controllers back to `UnitPipe(..., f(stopWhen))` (`src/spatial/transform/ForeachToUnitpipeTransformer.scala:21-28`), and `UnitIterationElimination` follows the same preservation rule for all-unit or partially eliminated counters (`src/spatial/transform/UnitIterationElimination.scala:15-23`, `src/spatial/transform/UnitIterationElimination.scala:34-43`). Schedule rules then reclassify single controls: UnitPipe/Accel default to sequenced, and single controls cannot remain pipelined (`src/spatial/flows/SpatialFlowRules.scala:321-327`, `src/spatial/flows/SpatialFlowRules.scala:346-365`). A D-11 simulator should therefore not infer semantics solely from the current node shape; a one-iteration foreach and a unitpipe may be pass-equivalent carriers for the same break register.

Stream-specific rewrites are more conservative when a user break exists. `StreamTransformer` folds outer stream foreach counters into children only when `stopWhen.isEmpty`; with a defined break it takes the generic path and preserves the foreach (`src/spatial/transform/StreamTransformer.scala:35-49`, `src/spatial/transform/StreamTransformer.scala:77-82`). Streamify also synthesizes internal breaker registers for generated stream wrappers: `HierarchicalToStream` creates `${lhs}_stop`, passes it as `Some(stopWhen)` to a streaming unitpipe, and `releaseGen` writes it on last-token release or missing time map (`src/spatial/transform/streamify/HierarchicalToStream.scala:601-669`, `src/spatial/transform/streamify/HierarchicalToStream.scala:782-810`). These generated breakers are termination protocol state, so D-11 should separate user-visible break semantics from compiler-inserted stream termination.

## 4. Cleanup, Binding, and Memory Ordering

Use analysis marks every controller stop register as `isBreaker` across pre- and post-unroll forms (`src/spatial/traversal/UseAnalyzer.scala:57-62`). That flag prevents unused-memory cleanup from treating an unread break register as dead (`src/spatial/traversal/UseAnalyzer.scala:25-34`), which matters because `TransientCleanup` removes unused registers and register writes when `isUnusedMemory` is set (`src/spatial/transform/TransientCleanup.scala:113-129`). Binding also treats the break register as an ordering dependency: it refuses to bundle a child if its active memories include the breaker of the parent or an ancestor (`src/spatial/transform/BindingTransformer.scala:30-42`, `src/spatial/transform/BindingTransformer.scala:93-103`). The latency model has the same assumption: when a write targets a breaker register belonging to an ancestor stop condition, later nodes in the parent scope are pushed after that write (`src/spatial/util/modeling.scala:458-467`, `src/spatial/util/modeling.scala:529-532`). Thus immediate semantics are not just a final codegen detail; they shape cleanup reachability, binding legality, and the modeled order of memory effects.

## 5. D-11 Implications

Tests encode the ordering question directly. In `Break`, the body sets `stop2` and then writes `s2(i)`, while the gold excludes the break-at index (`test/spatial/tests/feature/control/Breakpoint.scala:62-68`, `test/spatial/tests/feature/control/Breakpoint.scala:98-115`). End-of-iteration semantics would commit that post-break write; immediate semantics suppress it. D-11 should therefore model `breakWhen` as a control dependency over subsequent same-iteration effects, not only as a next-iteration loop guard, when running in HLS-faithful mode. If Rust also offers ScalaGen-compatible mode, it should name the difference and flag programs where writes, FIFO actions, or memory stores occur after a break-register write in the same controller body. The compiler passes already preserve, protect, and order `stopWhen`; the simulator decision is whether those ordered post-break effects commit or are killed.
