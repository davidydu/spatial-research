# Tests and Application Usage of `breakWhen`

## Coverage Map

`breakWhen` is not a casual API in the tests; it appears in targeted dynamic-control workloads and a few larger apps. The complete test/app surface I found is `StreamPipeFlush`, `PriorityDeq`, `PseudoCoprocTest`, `DummyWorkload`, the `Break` test in `Breakpoint.scala`, `FilterStream1D`, and `Rosetta` (`test/spatial/tests/feature/dynamic/StreamPipeFlush.scala:13`, `test/spatial/tests/feature/dynamic/PriorityDeq.scala:49`, `test/spatial/tests/feature/dynamic/PseudoCoprocTest.scala:24`, `test/spatial/tests/feature/dynamic/DummyWorkload.scala:131`, `test/spatial/tests/feature/control/Breakpoint.scala:64`, `test/spatial/tests/feature/streaming/FilterStream1D.scala:347`, `test/spatial/tests/apps/Rosetta.scala:560`). The API docs describe both `Stream` and `Sequential` as breaking "immediately" when the register is true and resetting the register when the controller finishes (`src/spatial/lang/control/Control.scala:41`, `src/spatial/lang/control/Control.scala:53`). Scalagen, however, warns that scala breaks at the end of the loop while synth breaks immediately (`src/spatial/codegen/scalagen/ScalaGenController.scala:75`).

## Exact Timing Checks

The strongest checked timing dependency is `Break` inside `Breakpoint.scala`. In the first `Sequential(breakWhen = stop2)`, the body sets `stop2` when `src(i) == iBreakAt`, then writes `s2(i)` afterward (`test/spatial/tests/feature/control/Breakpoint.scala:64`, `test/spatial/tests/feature/control/Breakpoint.scala:66`). The gold array excludes the break iteration itself, `i < args(0)`, so a pure end-of-iteration model would store `s2(9) = 9` and fail the default runtime (`test/spatial/tests/feature/control/Breakpoint.scala:98`, `test/spatial/tests/feature/control/Breakpoint.scala:109`). The second `Sequential` is similar: `stop4` is set from the newly loaded row before copying/storing it, while the gold expects the previous row (`test/spatial/tests/feature/control/Breakpoint.scala:73`, `test/spatial/tests/feature/control/Breakpoint.scala:75`, `test/spatial/tests/feature/control/Breakpoint.scala:99`). These checks assume immediate synthesized behavior that suppresses later work in the break-triggering iteration.

## End-of-Body and Same-Iteration Assumptions

Other tests set the break register after the useful side effects. The stream case in `Break` stores `x` across `d5` and only then assigns `stop5 := x == oBreakAt`; the gold expects the break value itself to be present (`test/spatial/tests/feature/control/Breakpoint.scala:85`, `test/spatial/tests/feature/control/Breakpoint.scala:90`, `test/spatial/tests/feature/control/Breakpoint.scala:91`, `test/spatial/tests/feature/control/Breakpoint.scala:100`). `DummyWorkload` is subtler: each worker sets `breakN := true` on a DONE packet and then enqueues `doneN` for the finisher (`test/spatial/tests/feature/dynamic/DummyWorkload.scala:131`, `test/spatial/tests/feature/dynamic/DummyWorkload.scala:139`, `test/spatial/tests/feature/dynamic/DummyWorkload.scala:140`). The final checks require all work to be counted and `status == 1`, so D-11 cannot interpret "immediate" as "abort all remaining same-iteration statements after the break register write" (`test/spatial/tests/feature/dynamic/DummyWorkload.scala:216`, `test/spatial/tests/feature/dynamic/DummyWorkload.scala:220`, `test/spatial/tests/feature/dynamic/DummyWorkload.scala:221`).

## Eventual-Termination Workloads

Most dynamic apps use `breakWhen` as a completion latch rather than an exact timing oracle. `StreamPipeFlush` drains eight results, writes the break at the end of a `Sequential`, and checks deterministic arrays; either immediate synth or end-of-iteration scalagen is acceptable as long as the internal forever pipes flush and no extra outer iteration commits (`test/spatial/tests/feature/dynamic/StreamPipeFlush.scala:41`, `test/spatial/tests/feature/dynamic/StreamPipeFlush.scala:46`, `test/spatial/tests/feature/dynamic/StreamPipeFlush.scala:54`). `PseudoCoprocTest` writes all `TASKS` outputs before `kill := true`, then asserts the whole output sequence (`test/spatial/tests/feature/dynamic/PseudoCoprocTest.scala:86`, `test/spatial/tests/feature/dynamic/PseudoCoprocTest.scala:97`, `test/spatial/tests/feature/dynamic/PseudoCoprocTest.scala:103`). The dynamic `PriorityDeq` variants mostly check packet counts or just non-hang behavior after a sentinel kill (`test/spatial/tests/feature/dynamic/PriorityDeq.scala:64`, `test/spatial/tests/feature/dynamic/PriorityDeq.scala:72`, `test/spatial/tests/feature/dynamic/PriorityDeq.scala:85`, `test/spatial/tests/feature/dynamic/PriorityDeq.scala:197`). The dependency-control variants assert ordered enqueue/dequeue behavior, not a precise final iteration count (`test/spatial/tests/feature/dynamic/PriorityDeq.scala:284`, `test/spatial/tests/feature/dynamic/PriorityDeq.scala:312`, `test/spatial/tests/feature/dynamic/PriorityDeq.scala:329`, `test/spatial/tests/feature/dynamic/PriorityDeq.scala:358`).

## D-11 Implications

D-11 has to preserve two observable contracts, not one. First, existing checked tests require synthesized immediacy when the break condition is raised before later stores in the same controller body. Second, they also rely on already-issued or source-later completion signals when the break write is part of normal teardown. The larger apps reinforce the second point: `FilterStream1D` sets `done` only after the final store and compares all requested rows to CSV golds (`test/spatial/tests/feature/streaming/FilterStream1D.scala:386`, `test/spatial/tests/feature/streaming/FilterStream1D.scala:389`, `test/spatial/tests/feature/streaming/FilterStream1D.scala:452`), while `Rosetta` explicitly drains FIFOs before setting `done` and then image-compares the full frame (`test/spatial/tests/apps/Rosetta.scala:592`, `test/spatial/tests/apps/Rosetta.scala:594`, `test/spatial/tests/apps/Rosetta.scala:638`). A replacement semantics should therefore specify the cut point: it may be immediate with respect to future controller work, but it must not invalidate committed work, drain protocol, or same-iteration completion handshakes that current tests already encode.
