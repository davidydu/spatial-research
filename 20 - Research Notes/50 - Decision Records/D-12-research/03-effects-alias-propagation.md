# Effects/Alias Metadata Propagation Through Argon and Spatial

## Creation at Staging

Argon creates ownership-relevant facts in `register`: it derives shallow and deep aliases from the op, computes effects, then either reuses a CSE hit with identical effects or creates a fresh symbol (`argon/src/argon/static/Staging.scala:106-118`). For a fresh symbol, staging appends it to the scope, caches it only when `effects.mayCSE`, records impure effects in `state.impure`, writes `Effects`, `DeepAliases`, and `ShallowAliases` metadata, and registers reverse shallow aliases on every alias in the closure (`argon/src/argon/static/Staging.scala:120-129`). It also records forward consumers for all op inputs before immediate flow hooks and compiler flow rules run, so consumer-derived metadata sees the new staged node (`argon/src/argon/static/Staging.scala:131-140`). The alias lattice comes from `Op`: `aliases`, `contains`, `extracts`, and `copies` are declarative hints, while `shallowAliases`, `deepAliases`, `allAliases`, and `mutableAliases` close those hints through already-stored symbol metadata (`argon/src/argon/Op.scala:48-67`, `argon/src/argon/Op.scala:80-104`).

## Spatial Alias Producers

Spatial makes memory view aliases first-class mutable allocations: `MemAlias` returns `Effects.Mutable` when `mutable` is true, and both dense and sparse aliases override `aliases` to point at the underlying memory symbols (`src/spatial/node/HierarchyMemory.scala:28-38`, `src/spatial/node/HierarchyMemory.scala:48-63`, `src/spatial/node/HierarchyMemory.scala:81-98`). Ordinary memory allocations are also mutable effects, while reads/writes and transfers express mutation through `Effects.Writes(mem)` or `Effects.Writes(local/dram)` (`src/spatial/node/HierarchyMemory.scala:8-20`, `src/spatial/node/HierarchyAccess.scala:64-81`, `src/spatial/node/HierarchyAccess.scala:115-122`, `src/spatial/node/DenseTransfer.scala:34-48`, `src/spatial/node/SparseTransfer.scala:27-39`). Spatial explicitly flips Argon’s default ban by setting `config.enableMutableAliases = true` because DRAM/SRAM/etc. aliases are expected in the language (`argon/src/argon/Config.scala:45-48`, `src/spatial/Spatial.scala:583-586`). Flow rules also create memory-consumer facts: accessors append the staged access to `mem.readers` or `mem.writers`, and controllers accumulate nested read/write memory sets (`src/spatial/flows/SpatialFlowRules.scala:21-40`, `src/spatial/flows/SpatialFlowRules.scala:245-277`).

## Storage, Transfer, and Invalidation

Metadata is stored inside each symbol’s `_data` map, while `metadata.add`, `metadata.remove`, and `metadata.all` are thin accessors over that map (`argon/src/argon/Ref.scala:123-129`, `argon/src/argon/Data.scala:87-123`). `Effects`, `ShallowAliases`, and `DeepAliases` are `Data` with `Transfer.Ignore`, so transformers do not explicitly copy or delete these facts; new symbols get fresh staging-time effects and aliases instead (`argon/src/argon/Effects.scala:5-30`, `argon/src/argon/Aliases.scala:1-5`). The transfer contract is split three ways: `Transfer.Mirror` is copied through `Data.mirror`, `Transfer.Remove` is removed, and `Transfer.Ignore` is left alone by explicit transfer (`argon/src/argon/Data.scala:36-60`, `argon/src/argon/transform/Transformer.scala:94-109`). Transformers preserve selected metadata with `transferData`, `transferDataIfNew`, and `transferDataToAllNew`, while `mirror` stages a new mirrored node with `transferDataIfNew` before normal flow rules run (`argon/src/argon/transform/Transformer.scala:111-150`). Global metadata with `Transfer.Remove` is invalidated before transforms, and specific passes can explicitly strip metadata such as banking data (`argon/src/argon/GlobalMetadata.scala:19-31`, `argon/src/argon/transform/Transformer.scala:228-232`, `src/spatial/transform/MetadataStripper.scala:6-31`).

## Consumption and Dealiasing

Argon consumes alias facts immediately in `checkAliases`: it errors on multiple mutable aliases unless the compatibility flag is enabled, and separately errors when a write targets a non-mutable symbol (`argon/src/argon/static/Staging.scala:34-75`). Effects also consume alias closure: `propagateWrites` expands every written symbol to all aliases, `computeEffects` adds reads of mutable inputs, and effect dependencies impose WAR/RAW/WAW/simple/global ordering through `state.impure` (`argon/src/argon/static/Staging.scala:202-233`, `argon/src/argon/static/Staging.scala:252-260`). The scheduler summarizes scope effects while excluding locally allocated mutables, DCE keeps non-idempotent effects, and register-read CSE compares recomputed effects including anti-dependencies (`argon/src/argon/schedule/Scheduler.scala:10-26`, `argon/src/argon/schedule/SimpleScheduler.scala:18-30`, `src/spatial/transform/RegReadCSE.scala:20-35`). Spatial later consumes view aliases structurally: `MemoryDealiasing` removes single-source dense/sparse aliases by substituting the base memory, expands dense-alias reads into guarded reads and writes into guarded demuxed writes, and handles alias-derived dimensions and DRAM addresses (`src/spatial/transform/MemoryDealiasing.scala:84-162`). Banking then consumes `mem.readers` and `mem.writers`, so any ownership model must keep those consumer facts coherent with alias rewrites (unverified; `src/spatial/traversal/MemoryAnalyzer.scala:141-150`, `src/spatial/traversal/banking/MemoryConfigurer.scala:71-86`).

## Hard Error or Compatibility Flag Constraints

A Rust ownership layer cannot treat every Spatial mutable alias as an independent `&mut` without also preserving Argon’s write-closure rule, because writes are deliberately expanded across all aliases before scheduling and hazard checks (unverified; `argon/src/argon/static/Staging.scala:252-260`). A hard error mode would need to reject only alias patterns that cannot be represented as scoped, non-overlapping, or compiler-dealiased views, because the current Spatial pipeline intentionally creates mutable `MemDenseAlias` and `MemSparseAlias` nodes before `MemoryDealiasing` removes or lowers them (unverified; `src/spatial/node/HierarchyMemory.scala:48-63`, `src/spatial/node/HierarchyMemory.scala:81-98`, `src/spatial/transform/MemoryDealiasing.scala:84-162`). A compatibility flag would need to preserve current staging behavior: allow `checkAliases` to pass, keep alias metadata recomputed on restaging, keep flow-derived readers/writers and controller read/write sets valid after transforms, and keep `Transfer.Ignore` effects/aliases from being blindly mirrored as stale facts (unverified; `argon/src/argon/static/Staging.scala:52-60`, `argon/src/argon/static/Staging.scala:124-140`, `argon/src/argon/transform/Transformer.scala:104-109`, `src/spatial/flows/SpatialFlowRules.scala:21-40`). It would also need to preserve diagnostic behavior for illegal immutable writes, since that check is independent of the mutable-alias compatibility switch (`argon/src/argon/static/Staging.scala:39-75`).
