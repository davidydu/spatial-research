# HLS Scheduling Risk From Mutable Aliases

## Recommendation

For Rust/HLS, do not silently preserve `enableMutableAliases` as a backend-wide permission. Preserve it only as a Spatial-compatibility front-end mode whose aliases must be lowered to concrete base-memory accesses, or else reject the unresolved mutable alias before HLS emission. Current Spatial can tolerate the flag because the Scala pipeline expands writes through alias closure, builds anti-dependencies, de-aliases memory views, then banks and emits concrete ports. An HLS backend that accepts multiple mutable aliases without reproducing that chain risks emitting an apparently independent memory schedule for storage that may be identical (unverified).

## Alias and Effect Facts

Alias facts start in Argon. `Op.aliases` is explicitly an alias hint, and `allAliases` plus `mutableAliases` are derived from shallow/deep alias closures (`argon/src/argon/Op.scala:48-52`, `argon/src/argon/Op.scala:85-98`). `mutableInputs` turns reads of mutable aliases into a set of mutable resources read by the operation (`argon/src/argon/Op.scala:100-104`). During staging, `computeEffects` calls `propagateWrites`, adds `Effects.Reads(d.mutableInputs)`, computes effect dependencies, and stores `antiDeps` (`argon/src/argon/static/Staging.scala:230-234`). Write propagation is alias-sensitive: it replaces each write by `s.allAliases`, or by `extractAtomicWrite(s).allAliases` when atomic writes are enabled (`argon/src/argon/static/Staging.scala:252-259`). The normal checker then rejects `(sym.mutableAliases diff effects.writes diff Set(sym))` only when the flag is disabled (`argon/src/argon/static/Staging.scala:39-53`). Spatial explicitly enables the flag because it allows mutable DRAM/SRAM aliases (`src/spatial/Spatial.scala:583-585`).

## Dependence and Scheduling Consumers

The dependence model depends on those expanded effect sets. `effectDependencies` treats previous readers as WAR hazards for current writes, previous writers as RAW/WAW hazards for reads or writes, and allocations as dependencies of accesses (`argon/src/argon/static/Staging.scala:195-223`). Symbol scheduling exposes those effect edges through `antiDeps` and `allDeps` (`argon/src/argon/static/Implicits.scala:127-131`). Scope scheduling summarizes impure nodes by unioning their reads and writes and preserving the impure sequence as anti-dependencies (`argon/src/argon/schedule/Scheduler.scala:14-25`). Spatial also consumes written/read memory sets when binding adjacent controller stages: `BindingTransformer` refuses to bundle stages when active memories intersect prior read/write or write sets (`src/spatial/transform/BindingTransformer.scala:21-42`). `PipeInserter` has a smaller but telling dependency hook: transient nodes are placed after a stage if their inputs intersect that stage's nodes or `sym.effects.writes` (`src/spatial/transform/PipeInserter.scala:117-124`).

## Banking, Allocation, and Codegen Consumers

Spatial's memory backend mostly stops consulting alias metadata directly after staging. Flow rules attach each concrete accessor to `mem.writers` or `mem.readers` (`src/spatial/flows/SpatialFlowRules.scala:21-29`). `MemoryDealiasing` rewrites dense-alias reads into reads of every possible underlying memory followed by `oneHotMux`, and rewrites dense-alias writes into demuxed writes guarded by alias conditions (`src/spatial/transform/MemoryDealiasing.scala:22-49`, `src/spatial/transform/MemoryDealiasing.scala:135-143`). Access analysis then attaches affine access patterns and unrolled matrices to each reader/writer (`src/spatial/traversal/AccessAnalyzer.scala:215-225`, `src/spatial/traversal/AccessExpansion.scala:66-126`). `MemoryConfigurer` consumes `mem.readers`, `mem.writers`, and their affine matrices, groups conflicts by possible concurrency and address overlap, computes ports/buffer depths, searches banking options, and finalizes dispatch/port metadata (`src/spatial/traversal/banking/MemoryConfigurer.scala:71-82`, `src/spatial/traversal/banking/MemoryConfigurer.scala:250-289`, `src/spatial/traversal/banking/MemoryConfigurer.scala:610-665`). `MemoryAllocator` then assigns each analyzed duplicate to a target resource, not to an alias handle (`src/spatial/traversal/MemoryAllocator.scala:35-40`, `src/spatial/traversal/MemoryAllocator.scala:56-103`). Chisel codegen finally instantiates memories from the computed bank counts, ports, readers, and writers (`src/spatial/codegen/chiselgen/ChiselGenMem.scala:162-199`).

## HLS Risk

The risk is not "aliases exist"; Spatial's existing pipeline is built around aliases. The risk is allowing the Rust/HLS path to see only Rust-level mutable handles after losing Argon's alias-expanded `writes`, mutable-input `reads`, `antiDeps`, and de-aliased base-memory accesses. HLS scheduling, memory partitioning, and dependence pragmas are usually sensitive to whether two accesses may hit the same storage, whether they are concurrent, and whether they require the same physical port (unverified). If multiple mutable aliases are silently treated as independent arrays/pointers, the HLS tool may pipeline across a true RAW/WAW dependence or bank/partition the wrong object (unverified). If the tool is conservative, it may instead serialize accesses and miss Spatial's intended banking/II choices (unverified). Therefore Rust/HLS should reject unresolved mutable aliases at the HLS boundary unless it can prove or emit the same base-memory, access-matrix, banking, port, and dependence contract Spatial already derives.
