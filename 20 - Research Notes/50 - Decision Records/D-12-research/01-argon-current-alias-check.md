# Current Argon Mutable-Alias Check

## Scope and Timing

Argon’s current mutable-alias behavior is concentrated in `checkAliases(sym, effects)`, whose local comment names two correctness checks: “No mutable aliasing” and “No mutation of immutable symbols” (`Staging.scala:34-39`). In the normal staging path, `register` computes operation alias metadata and effects before calling the check: it captures `op.shallowAliases` and `op.deepAliases` (`Staging.scala:106-107`), computes `effects = computeEffects(op)` (`Staging.scala:109`), stores `sym.effects`, `sym.deepAliases`, and `sym.shallowAliases` (`Staging.scala:124-128`), registers reverse shallow aliases through `sym.allAliases.foreach` (`Staging.scala:129`), runs immediate flow and registered flow passes (`Staging.scala:134-140`), then calls `checkAliases(sym, effects)` (`Staging.scala:147-148`). Therefore the check observes the post-registration symbol, its alias metadata, and the computed effect set rather than raw `Op` fields (`Staging.scala:124-148`).

## Sets Compared

Inside `checkAliases`, immutable writes and mutable aliases are derived as two separate sets (`Staging.scala:39-42`). Immutable-write candidates are exactly `effects.writes.filterNot(_.isMutable)`, so the source classifies illegal immutable mutation by asking whether each written symbol reports `isMutable` (`Staging.scala:40`). Mutable-alias candidates are `((sym.mutableAliases diff effects.writes) diff Set(sym))` (`Staging.scala:41`). The comparison excludes aliases already present in the current effect write set, then excludes the current `sym` itself (`Staging.scala:41`). As a direct consequence of that expression, `checkAliases` reports only mutable aliases of the staged symbol that are not among this operation’s writes and are not the symbol being checked (`Staging.scala:41-42`). The code does not compare immutable writes against `sym.mutableAliases`; immutable writes come solely from `effects.writes`, while alias violations come from `sym.mutableAliases` minus writes and `sym` (`Staging.scala:40-41`).

## Mutable Alias Behavior

The mutable-alias error path is guarded by both set non-emptiness and the configuration flag: `if (aliases.nonEmpty && !config.enableMutableAliases)` (`Staging.scala:52`). With the default configuration, `enableMutableAliases` is `false` (`Config.scala:45-47`), so nonempty alias candidates enter the error block. That block reports an error at `sym.ctx` saying the value has multiple mutable aliases and mutable aliasing is disallowed (`Staging.scala:52-53`). It then iterates over `(aliases + sym)`, emits “defined here” diagnostics at each alias’s context with `noError = true`, emits an additional context error, and logs debug provenance for each alias/symbol (`Staging.scala:54-60`). If `enableMutableAliases` is set to `true`, the guard is false and this alias-specific error/debug block is skipped (`Staging.scala:52-60`). The source exposes the flag but does not explain why it exists; calling it a compatibility flag is a decision-record framing, not a source comment (unverified).

## Immutable Writes

Immutable-write checking is independent of `enableMutableAliases`: after the alias block, `checkAliases` separately tests `if (immutables.nonEmpty)` with no configuration guard (`Staging.scala:61`). For each immutable symbol in `effects.writes.filterNot(_.isMutable)`, it emits an error at the current symbol context for illegal mutation, emits another current-context error, reports where the immutable value was originally defined, and emits another definition-context error (`Staging.scala:61-67`). It also logs debug details for each immutable write, including the statement and metadata (`Staging.scala:69-73`). Current Argon therefore distinguishes “mutable aliasing allowed by flag” from “immutable write always diagnosed”: the alias path is conditional on `!config.enableMutableAliases`, while the immutable write path is unconditional once `immutables.nonEmpty` (`Staging.scala:52-67`).

## Atomic Write Relationship

`enableAtomicWrites` is adjacent to the mutable-alias flag in `Config`, defaulting to `true` while `enableMutableAliases` defaults to `false` (`Config.scala:45-47`). The relationship in `Staging` is indirect through the `effects` argument passed into `checkAliases`. `computeEffects` first calls `propagateWrites(d.effects)` and combines that with `Effects.Reads(d.mutableInputs)`, then attaches scheduling anti-dependencies (`Staging.scala:230-234`). `propagateWrites` uses `enableAtomicWrites` to decide how writes are expanded: when atomic writes are disabled, it replaces each written symbol with `s.allAliases`; when enabled, it first maps through `extractAtomicWrite(s)` and then takes `.allAliases` (`Staging.scala:252-259`). The nearby comment says this is for nested “atomic” writes, where writes through an intermediate atomic read should be reflected on the top mutable object rather than the intermediate value (`Staging.scala:236-244`). Thus `checkAliases` itself has no atomic-write branch, but its `effects.writes` input has already been shaped by the atomic-write setting (`Staging.scala:230-259`, `Staging.scala:39-42`).
