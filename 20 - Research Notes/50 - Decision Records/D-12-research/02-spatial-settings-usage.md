# Spatial settings and usage expectations

## Finding

Preserve Spatial's `enableMutableAliases` behavior for Rust ownership. A Rust front end should not make Spatial mutable aliasing a blanket hard error, because current Spatial deliberately opts out of Argon's default alias prohibition and exposes memory views as normal user syntax. Argon's base configuration defaults `enableMutableAliases` to `false` (`argon/src/argon/Config.scala:45`, `argon/src/argon/Config.scala:47`), and staging emits the user-facing diagnostic "has multiple mutable aliases. Mutable aliasing is disallowed" when aliases exist and the setting is not enabled (`argon/src/argon/static/Staging.scala:52`, `argon/src/argon/static/Staging.scala:53`). Spatial then overrides that default in `settings()` with the explicit comment "Spatial allows mutable aliases for DRAM, SRAM, etc." (`src/spatial/Spatial.scala:583`, `src/spatial/Spatial.scala:584`, `src/spatial/Spatial.scala:585`). I found no other `enableMutableAliases` enablement sites with `rg`, so this looks like a language-level policy decision, not a per-app workaround.

## User-visible syntax

The public memory APIs create aliases for ordinary slice/view operations. One-dimensional memories expose `apply(range)` as "a view of this memory" and stage `MemDenseAlias` (`src/spatial/lang/types/Mem.scala:128`, `src/spatial/lang/types/Mem.scala:129`). Multi-dimensional memories similarly expose row, column, and region views that all stage dense aliases (`src/spatial/lang/types/Mem.scala:138`, `src/spatial/lang/types/Mem.scala:139`, `src/spatial/lang/types/Mem.scala:142`, `src/spatial/lang/types/Mem.scala:145`). This is not only DRAM syntax: `SRAM.par` says it "Creates an alias of this SRAM with parallel access" and stages a dense alias (`src/spatial/lang/SRAM.scala:21`, `src/spatial/lang/SRAM.scala:22`, `src/spatial/lang/SRAM.scala:26`); `RegFile.par` does the same for register files (`src/spatial/lang/RegFile.scala:19`, `src/spatial/lang/RegFile.scala:20`, `src/spatial/lang/RegFile.scala:24`); `LockSRAM.par` does the same for locked SRAMs (`src/spatial/lang/LockMem.scala:84`, `src/spatial/lang/LockMem.scala:85`, `src/spatial/lang/LockMem.scala:89`). At the IR level, both dense and sparse memory aliases are marked mutable and report the underlying memory as an alias target (`src/spatial/node/HierarchyMemory.scala:58`, `src/spatial/node/HierarchyMemory.scala:60`, `src/spatial/node/HierarchyMemory.scala:62`, `src/spatial/node/HierarchyMemory.scala:93`, `src/spatial/node/HierarchyMemory.scala:95`, `src/spatial/node/HierarchyMemory.scala:97`).

## Existing tests and apps

Tests and examples imply users expect aliases to compile. `NestedAlias` constructs `y = x(2::2::16)` and then `z = y(4::2::8)`, with the comment "technically allowed," then asserts the nested view reads the expected value (`test/spatial/tests/feature/memories/Aliasing.scala:11`, `test/spatial/tests/feature/memories/Aliasing.scala:12`, `test/spatial/tests/feature/memories/Aliasing.scala:21`, `test/spatial/tests/feature/memories/Aliasing.scala:22`). Transfer tests use DRAM views as load/store endpoints for 3D-to-2D rotations (`test/spatial/tests/feature/transfers/Aliases3Dto2DTransfers.scala:62`, `test/spatial/tests/feature/transfers/Aliases3Dto2DTransfers.scala:63`, `test/spatial/tests/feature/transfers/Aliases3Dto2DTransfers.scala:64`, `test/spatial/tests/feature/transfers/Aliases3Dto2DTransfers.scala:67`, `test/spatial/tests/feature/transfers/Aliases3Dto2DTransfers.scala:68`). The sample app `HelloSpatial` loads a 2D DRAM quadrant through a ranged/parallel view and stores another ranged/parallel view (`apps/src/HelloSpatial.scala:42`, `apps/src/HelloSpatial.scala:43`, `apps/src/HelloSpatial.scala:46`). It also uses `sram1D par 1` as gather addresses (`apps/src/HelloSpatial.scala:48`, `apps/src/HelloSpatial.scala:49`). The exact blast radius across all apps is unverified, but the pattern is clearly first-class.

## Compiler expectation

Spatial has dedicated support for aliases after staging. `MemoryDealiasing` says dense and sparse aliases are removed after unrolling, forwards single-base aliases, and rewrites reads and writes through dense aliases into mux/demux operations (`src/spatial/transform/MemoryDealiasing.scala:85`, `src/spatial/transform/MemoryDealiasing.scala:86`, `src/spatial/transform/MemoryDealiasing.scala:87`, `src/spatial/transform/MemoryDealiasing.scala:135`, `src/spatial/transform/MemoryDealiasing.scala:140`, `src/spatial/transform/MemoryDealiasing.scala:143`). `AliasRewrites` also combines nested dense aliases rather than rejecting them (`src/spatial/rewrites/AliasRewrites.scala:49`, `src/spatial/rewrites/AliasRewrites.scala:50`, `src/spatial/rewrites/AliasRewrites.scala:54`, `src/spatial/rewrites/AliasRewrites.scala:63`, `src/spatial/rewrites/AliasRewrites.scala:68`, `src/spatial/rewrites/AliasRewrites.scala:69`). There are limits: accumulation over multiple aliased memories still throws (`src/spatial/transform/unrolling/ReduceUnrolling.scala:133`, `src/spatial/transform/unrolling/ReduceUnrolling.scala:134`). That argues for preserving aliases while retaining targeted errors for unsupported cases.

## Rust ownership implication

Rust lowering should model Spatial memory views as Spatial handles/views with compiler-managed alias semantics, not as ordinary Rust `&mut` borrows that must be exclusive. Making every mutable alias a hard error would turn common slices, parallel views, sparse gather/scatter regions, and nested aliases into rejected programs. The safer user-visible choice is: preserve `enableMutableAliases` for Spatial, reject only cases Spatial already cannot lower, and document that alias safety is discharged by Spatial's memory analysis/dealiasing rather than by Rust's native mutable-reference rule.
