# Migration and Compatibility Strategy

## Baseline Contract

Q-137 is explicitly a migration decision: Argon can disable mutable-alias errors with `enableMutableAliases`, and Rust IR must choose whether to preserve that compatibility flag or make mutable aliasing a hard error (`20 - Open Questions.md:1968-1973`). The source split is real. Argon defaults `enableAtomicWrites = true` and `enableMutableAliases = false` (`Config.scala:45-47`), while `checkAliases` computes immutable writes and mutable aliases separately, then errors on mutable aliases only when the alias set is nonempty and the flag is false (`Staging.scala:39-60`). Spatial overrides the base default in `settings()` with the comment "Spatial allows mutable aliases for DRAM, SRAM, etc." (`Spatial.scala:583-585`). The vault spec already captures the compatibility boundary: multiple mutable aliases may exist only when explicitly configured, and immutable writes remain illegal (`10 - Effects and Aliasing.md:55`; `30 - Effects and Aliasing.md:129-133`).

## Option A: Hard Error By Default

One migration shape is to make Rust IR strict by default: run the Argon candidate calculation, treat any `(sym.mutableAliases diff effects.writes diff Set(sym))` as fatal, and omit or ignore the compatibility escape hatch (unverified). This aligns with Argon's base config rather than Spatial's configured behavior, so it must be described as a language break, not a mechanical port. The breakage is broad. Public memory APIs intentionally stage aliases: one-dimensional `apply(range)` returns a memory view by staging `MemDenseAlias` (`Mem.scala:128-129`), and two-dimensional row, column, and region views do the same (`Mem.scala:138-145`). Dense and sparse memory aliases are mutable and report the base memory as their alias target (`HierarchyMemory.scala:28-38`, `HierarchyMemory.scala:48-63`, `HierarchyMemory.scala:81-98`). Disallowing aliases outright would also reject existing examples and tests: `NestedAlias` forms nested SRAM views and asserts the expected value (`Aliasing.scala:9-22`), transfer tests load and store DRAM slices (`Aliases3Dto2DTransfers.scala:62-68`), and `HelloSpatial` uses ranged/parallel DRAM views for load, store, gather, and scatter (`HelloSpatial.scala:42-51`).

## Option B: Preserve A Legacy Flag

A second shape is to preserve the flag, possibly as a clearer `alias_policy = strict | legacy_spatial | warn` compiler option (unverified). In `strict`, Rust emits the current Argon mutable-alias diagnostic as an error. In `legacy_spatial`, Rust still computes the same alias candidate set and effect `writes`, but demotes only the mutable-alias failure; immutable writes remain hard errors because Argon has no configuration guard for them (`Staging.scala:61-74`; `30 - Effects and Aliasing.md:133`). This option keeps Rust's data structures under normal Rust ownership while modeling Spatial values as `SymId` handles in an effects/alias graph (unverified). It also respects the HLS-oriented vault note that alias closures should not be replaced by borrow-checker intuition, and that `reads`, `writes`, and `antiDeps` need to match Argon before HLS simplification (`10 - Effects and Aliasing.md:61-63`).

## Option C: Warning Mode And Allow-list

Warning/lint mode can be layered over either default (unverified). The compiler would always run the check, always record a finding, and choose severity by policy: error in strict, warning in legacy or migration mode, and perhaps note-only for aliases proven to be compiler-generated memory views (unverified). An allow-list can be narrow: `MemDenseAlias` and `MemSparseAlias` over known Spatial memories, plus established `.par`/view constructors cited in the D-12 usage note (`02-spatial-settings-usage.md:7-13`), while non-memory mutable aliases still surface as errors or warnings (unverified). This cannot skip effect propagation. `computeEffects` first propagates writes, adds reads of mutable inputs, and then computes anti-dependencies (`Staging.scala:230-234`); `propagateWrites` expands each write through alias closure, with `enableAtomicWrites` controlling atomic-read extraction (`Staging.scala:252-259`). Any allow-list is therefore a legality policy, not permission to remove alias edges (unverified).

## Diagnostics And Tests

Rust diagnostics should preserve Argon's provenance shape (unverified). The current error reports the current symbol, every alias definition, and debug provenance for `(aliases + sym)` (`Staging.scala:52-60`); the Rust version should add stable `SymId`, source span, originating op, base memory, range/condition, and whether the alias was excluded because it was already in `effects.writes` (unverified). Tests need four layers (unverified): exact unit tests for the alias candidate formula and immutable-write independence; mode tests for strict, legacy, and warning severities; pass tests showing `AliasRewrites` still combines nested dense aliases (`AliasRewrites.scala:49-69`) and `MemoryDealiasing` still rewrites alias reads/writes into mux/demux operations (`MemoryDealiasing.scala:84-163`); and negative tests for unsupported cases such as accumulation across multiple aliased memories, which already throws (`ReduceUnrolling.scala:133-135`). If aliases are disallowed outright, the expected failures are not cosmetic: normal views, nested aliases, DRAM slice transfers, `.par` aliases, and downstream dealiasing behavior all stop being accepted or stop being exercised.
