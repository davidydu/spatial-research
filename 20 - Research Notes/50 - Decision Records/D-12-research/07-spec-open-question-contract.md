# D-12 Angle 7: Spec/Open-Question Contract

## Current question

Q-137 frames the ownership question narrowly: Argon can disable mutable-alias errors with `enableMutableAliases`, and the Rust port must decide whether to preserve that compatibility flag or make mutable aliasing a hard error. The open question cites `argon/src/argon/static/Staging.scala:39-75` and `argon/src/argon/Config.scala:46-47` as the source surface, and marks the issue blocked on Rust IR ownership design (`20 - Open Questions.md:1963-1974`). So the decision is not whether Rust's borrow checker can discover Spatial aliasing automatically. It is whether the Rust implementation preserves Spatial's staged alias graph and configurable legality check, or strengthens the language contract during the port.

## Contract already written

The high-level semantics spec says Spatial inherits Argon's effect and alias semantics as one contract: staged symbols carry `Effects`, staging expands those effects through mutable inputs and alias-write propagation, and the scheduler/CSE read the resulting `Impure` records (`10 - Effects and Aliasing.md:35-37`). It defines effects as an explicit tuple with boolean flags plus `reads`, `writes`, and `antiDeps`, citing `argon/src/argon/Effects.scala:20-96` (`10 - Effects and Aliasing.md:41-47`). The same note is explicit about the Rust port: alias closures must not be replaced by "borrow-checker intuition"; Spatial aliases are graph metadata, and atomic-write propagation, mutable-alias errors, and allocation-local effect subtraction should be preserved before HLS simplification (`10 - Effects and Aliasing.md:61-63`).

## Mutable-alias rule as specified

The lower-level Argon spec makes mutability a property of the symbol's effects, not a separate ownership bit: `sym.isMutable = sym.effects.isMutable`, and `sym.mutableAliases` filters all aliases by that predicate (`30 - Effects and Aliasing.md:116-123`). `checkAliases` has two legality paths. First, mutable aliasing is rejected when `(sym.mutableAliases diff effects.writes diff Set(sym))` is nonempty and `enableMutableAliases` is false; aliases being written by the current op are intentionally subtracted (`30 - Effects and Aliasing.md:127-132`). Second, writes to immutable symbols are always illegal, with no config escape hatch (`30 - Effects and Aliasing.md:127-133`). The config section records the defaults: `enableAtomicWrites = true` and `enableMutableAliases = false`, both read at registration time (`30 - Effects and Aliasing.md:196-203`; source `Config.scala:46-47`).

## Contradictions and coverage drift

There is no direct contradiction in the Argon behavior model: default Spatial rejects multiple mutable aliases, but the legacy config can permit them. The contradiction is in decision status. The high-level spec says "multiple mutable aliases may exist only when explicitly configured" and calls that the mutable-alias contract (`10 - Effects and Aliasing.md:55`). The lower-level HLS notes go further, saying the `enableAtomicWrites` and `enableMutableAliases` toggles "should be consumed" as compile-time feature flags or runtime compiler config in Rust (`30 - Effects and Aliasing.md:216-218`). But Q-137 still says the user must decide whether Rust preserves that flag or makes mutable aliasing a hard error (`20 - Open Questions.md:1968-1973`). There is also bookkeeping drift: the high-level note links both atomic recursion and mutable-alias ownership as open questions (`10 - Effects and Aliasing.md:65-68`), while the lower-level Argon note's open-question section only lists the atomic-write question (`30 - Effects and Aliasing.md:222-224`).

## Decision surface

The decision must specify the Rust IR ownership contract in terms of Spatial semantics, not Rust implementation mechanics. If compatibility wins, Rust should model syms as IDs in an Argon-style effects/alias graph, default `enableMutableAliases` to false, and retain a deliberate escape hatch for legacy programs; Rust ownership then protects compiler data structures, while Spatial legality is enforced by `checkAliases`-equivalent validation. If hard-error wins, the config flag should disappear or be ignored, and the spec must say that programs relying on configurable mutable aliasing are no longer accepted. Either way, the decision should preserve the always-illegal immutable-write rule, define the diagnostic point during staging/register, and state whether the current "aliases being written are not flagged" exception remains part of the Rust contract (`30 - Effects and Aliasing.md:129-133`).
