# Implementation Cost Matrix: Alias Policy Options

## Baseline Surfaces

Alias policy is concentrated but cross-cutting. `register` computes `op.shallowAliases` and `op.deepAliases`, computes effects, installs `sym.effects`/alias metadata, registers reverse aliases, runs flow rules, then calls `checkAliases` (`argon/src/argon/static/Staging.scala:106-148`). `checkAliases` has two separable rules: mutable aliases are `((sym.mutableAliases diff effects.writes) diff Set(sym))` and are rejected only when `!config.enableMutableAliases`, while immutable writes are always rejected (`argon/src/argon/static/Staging.scala:39-75`). The alias closure comes from `Op.aliases`, `contains`, `extracts`, and `copies`, then derives `shallowAliases`, `deepAliases`, `allAliases`, `mutableAliases`, and `mutableInputs` (`argon/src/argon/Op.scala:48-104`). Spatial deliberately flips the default in `settings()` with "Spatial allows mutable aliases for DRAM, SRAM, etc." (`src/spatial/Spatial.scala:583-585`). Dense and sparse memory views are mutable allocations whose alias source is the underlying memory (`src/spatial/node/HierarchyMemory.scala:28-38`, `src/spatial/node/HierarchyMemory.scala:48-63`, `src/spatial/node/HierarchyMemory.scala:81-98`).

## Cost Matrix

| Policy | Implementation cost | Code surface cost | Spec/test cost |
|---|---:|---|---|
| Exact Argon flag parity | Medium | Reimplement `Effects`, alias metadata, write propagation, and `checkAliases` ordering exactly. The hard part is matching `register` timing around CSE, flows, reverse aliases, and post-flow checking, not the boolean flag itself (`argon/src/argon/static/Staging.scala:106-148`). `Op` alias closures and `mutableInputs` must remain graph metadata, not Rust borrow facts (`argon/src/argon/Op.scala:85-104`). | Low-medium. Current specs already say alias closures should not be replaced by borrow-checker intuition and that `enableMutableAliases` belongs in compiler config (`10 - Spec/20 - Semantics/10 - Effects and Aliasing.md:63`; `10 - Spec/30 - IR/00 - Argon Framework/30 - Effects and Aliasing.md:216-218`). Tests mainly assert parity on diagnostics, write-set expansion, and existing memory-view programs. |
| Hard error default | Low in checker, high in product | The local checker change is small: make the mutable-alias branch unconditional or ignore the flag (`argon/src/argon/static/Staging.scala:52-60`). The expensive part is fallout: `Spatial.settings()` would stop enabling the escape hatch, and ordinary `MemDenseAlias`/`MemSparseAlias` nodes are currently mutable aliases by construction (`src/spatial/Spatial.scala:583-585`; `src/spatial/node/HierarchyMemory.scala:60-63`, `src/spatial/node/HierarchyMemory.scala:95-98`). | High. Specs must override the current Rust-port guidance and remove or sharply narrow the config-toggle contract (`10 - Spec/20 - Semantics/10 - Effects and Aliasing.md:55-68`). Many memory-view and transfer tests become rejection tests or need rewritten non-alias lowering. |
| Allow memory-view aliases only | High | Requires preserving provenance that `sym.mutableAliases` currently erases into sets. The checker would need to distinguish aliases introduced by `MemDenseAlias`/`MemSparseAlias` from generic `Op.aliases`, `contains`, `extracts`, or `copies` (`argon/src/argon/Op.scala:48-67`, `argon/src/argon/Op.scala:85-98`). Nested view flattening and post-unroll dealiasing already assume memory aliases survive as normal IR nodes (`src/spatial/rewrites/AliasRewrites.scala:37-70`; `src/spatial/transform/MemoryDealiasing.scala:84-143`). | Medium-high. Specs need a new whitelisted-alias category. Tests must cover dense, sparse, nested, and non-memory mutable aliases, including the existing exception where aliases in `effects.writes` are not flagged (`10 - Spec/30 - IR/00 - Argon Framework/30 - Effects and Aliasing.md:129-133`). |
| Explicit `AliasToken`/capability model | Very high | Adds a new permission object across staging, memory APIs, effect computation, diagnostics, and lowering. `MemDenseAlias` and `MemSparseAlias` would produce both a view and a capability describing base memory, conditions, ranges or sparse addresses, and write permission (`src/spatial/node/HierarchyMemory.scala:48-63`, `src/spatial/node/HierarchyMemory.scala:81-98`). Reads/writes, `MemoryDealiasing`, and HLS dependence derivation would consume that token instead of relying only on alias closure. | Very high. Requires new formal semantics for token creation, transfer, lifetime, disjointness, and diagnostics. Existing spec text that treats aliases as graph metadata would need an ownership/capability layer rather than a direct port (`10 - Spec/20 - Semantics/10 - Effects and Aliasing.md:49-63`). |
| Phased lint-to-error | Medium | Starts from parity, then adds severity modes around the existing alias candidate set. The first pass can reuse `checkAliases` provenance, including reporting the current symbol and alias definitions (`argon/src/argon/static/Staging.scala:52-60`). Later phases can raise memory-view, non-memory, or capability-missing findings from warn to error without changing the initial alias graph. | Medium. Specs need a policy schedule and named modes, but the current open question already frames the choice as preserving the flag versus hard error (`20 - Research Notes/20 - Open Questions.md:1963-1974`). Tests need dual expectations: compatibility suites, lint snapshots, and future strict-mode rejection cases. |

## Cross-Cutting Costs

All options must keep immutable-write errors separate from mutable-alias policy because that path is unconditional today (`argon/src/argon/static/Staging.scala:40`, `argon/src/argon/static/Staging.scala:61-74`). All options also interact with write propagation: `computeEffects` expands writes through aliases before dependency analysis, optionally via `extractAtomicWrite`, so changing alias legality without preserving write-set closure risks changing scheduling and HLS dependence facts (`argon/src/argon/static/Staging.scala:230-261`). The cheapest paths reuse the existing alias graph; the expensive paths introduce provenance, capabilities, or migration states above it.
