# Tests/Apps Corpus Evidence for Mutable Aliases

## Proven Source Facts

Argon defaults to rejecting mutable aliases: `enableMutableAliases` is initialized to `false` (`argon/src/argon/Config.scala:47`). Spatial then globally overrides that default in `settings`, with the comment "Spatial allows mutable aliases for DRAM, SRAM, etc." immediately before `config.enableMutableAliases = true` (`src/spatial/Spatial.scala:583`, `src/spatial/Spatial.scala:584`, `src/spatial/Spatial.scala:585`). I found no app/test-local setting of this flag; tests appear to inherit the Spatial-wide setting.

The surface constructs used by tests/apps really do create mutable alias IR, not just parser sugar. Dense memory range views stage `MemDenseAlias`: `Mem1.apply(range)` returns "a view" and stages `MemDenseAlias` (`src/spatial/lang/types/Mem.scala:128`, `src/spatial/lang/types/Mem.scala:129`), and 2D row/column/region views do the same (`src/spatial/lang/types/Mem.scala:138`, `src/spatial/lang/types/Mem.scala:139`, `src/spatial/lang/types/Mem.scala:142`, `src/spatial/lang/types/Mem.scala:145`). `SRAM.par` is also documented as creating an alias and stages `MemDenseAlias` (`src/spatial/lang/SRAM.scala:21`, `src/spatial/lang/SRAM.scala:26`). Sparse DRAM gather/scatter views stage `MemSparseAlias` (`src/spatial/lang/DRAM.scala:65`, `src/spatial/lang/DRAM.scala:70`, `src/spatial/lang/DRAM.scala:71`). Both alias node families are mutable and report the underlying memory as an alias: `MemDenseAlias` sets `mutable = true` and `aliases = syms(mem)` (`src/spatial/node/HierarchyMemory.scala:60`, `src/spatial/node/HierarchyMemory.scala:62`); `MemSparseAlias` does the same (`src/spatial/node/HierarchyMemory.scala:95`, `src/spatial/node/HierarchyMemory.scala:97`).

The corpus contains a direct user-level alias test. `NestedAlias` allocates one SRAM, assigns `val y = x(2::2::16)`, then assigns `val z = y(4::2::8)`; the comment says this nested alias is "massively confusing but technically allowed" (`test/spatial/tests/feature/memories/Aliasing.scala:9`, `test/spatial/tests/feature/memories/Aliasing.scala:11`, `test/spatial/tests/feature/memories/Aliasing.scala:12`). The test asserts the nested view resolves to element 10 (`test/spatial/tests/feature/memories/Aliasing.scala:21`, `test/spatial/tests/feature/memories/Aliasing.scala:22`). This is the strongest checked-in evidence that aliases can be first-class `val`s, not only transient transfer operands.

Dense DRAM/SRAM views are routine in apps. `HelloSpatial` loads a 2D DRAM quadrant into SRAM and stores an SRAM back through a DRAM region view (`apps/src/HelloSpatial.scala:43`, `apps/src/HelloSpatial.scala:46`). `Aliases3Dto2DTransfers` repeatedly loads 2D slices from a 3D DRAM and stores them into rotated 3D positions (`test/spatial/tests/feature/transfers/Aliases3Dto2DTransfers.scala:63`, `test/spatial/tests/feature/transfers/Aliases3Dto2DTransfers.scala:64`, `test/spatial/tests/feature/transfers/Aliases3Dto2DTransfers.scala:67`, `test/spatial/tests/feature/transfers/Aliases3Dto2DTransfers.scala:68`). `Inference` aliases both source and destination regions in convolution kernels, including a local SRAM sub-view loaded from an input DRAM sub-view (`test/spatial/tests/apps/Inference.scala:101`, `test/spatial/tests/apps/Inference.scala:133`, `test/spatial/tests/apps/Inference.scala:465`, `test/spatial/tests/apps/Inference.scala:503`).

Sparse aliases are also checked behavior. `HelloSpatial` gathers from `dram1D_longer(sram1D par 1, 64)` and scatters through `dram_scatter1D(sram1D par 1, 64)` (`apps/src/HelloSpatial.scala:49`, `apps/src/HelloSpatial.scala:51`). Transfer tests use the same shape with explicit address SRAMs: `sram gather gatherData(addrs par P, tileSize)` and `scatterDRAM(addrs par P, tileSize) scatter data` (`test/spatial/tests/feature/transfers/GatherStore.scala:149`, `test/spatial/tests/feature/transfers/GatherStore.scala:150`, `test/spatial/tests/feature/transfers/GatherStore.scala:201`, `test/spatial/tests/feature/transfers/GatherStore.scala:202`). `ScatterGatherSRAM` even leaves a TODO asking what to do about parallel scatter to the same burst, which is alias-adjacent behavioral pressure rather than a proven alias-legality assertion (`test/spatial/tests/feature/transfers/ScatterGatherSRAM.scala:24`, `test/spatial/tests/feature/transfers/ScatterGatherSRAM.scala:25`).

Mutable memories are passed through helpers in nontrivial apps. `Bank4DHard` defines `fill`, `read`, `slidingWindowRead`, and `run` over `SRAM3[Int]`, then calls `run(hier)` and `run(flat)` (`test/spatial/tests/feature/banking/Bank4DHard.scala:53`, `test/spatial/tests/feature/banking/Bank4DHard.scala:56`, `test/spatial/tests/feature/banking/Bank4DHard.scala:65`, `test/spatial/tests/feature/banking/Bank4DHard.scala:77`, `test/spatial/tests/feature/banking/Bank4DHard.scala:88`, `test/spatial/tests/feature/banking/Bank4DHard.scala:92`). MachSuite radix sort passes SRAMs into `hist` and `update`, then swaps `a_sram` and `b_sram` roles across calls (`test/spatial/tests/apps/MachSuite.scala:1893`, `test/spatial/tests/apps/MachSuite.scala:1938`, `test/spatial/tests/apps/MachSuite.scala:1960`, `test/spatial/tests/apps/MachSuite.scala:1971`, `test/spatial/tests/apps/MachSuite.scala:1975`). Rosetta similarly passes SRAM/FIFO buffers through staged rendering helpers (`test/spatial/tests/apps/Rosetta.scala:34`, `test/spatial/tests/apps/Rosetta.scala:54`, `test/spatial/tests/apps/Rosetta.scala:278`, `test/spatial/tests/apps/Rosetta.scala:280`, `test/spatial/tests/apps/Rosetta.scala:354`, `test/spatial/tests/apps/Rosetta.scala:362`, `test/spatial/tests/apps/Rosetta.scala:587`, `test/spatial/tests/apps/Rosetta.scala:588`).

## Inferred Risk

Making mutable aliasing a hard Rust IR error would likely reject real corpus programs unless the Rust port reclassifies Spatial memory views as scoped borrows before legality checking. The risk is not just exotic nested aliases: common transfer syntax, `par` views, gather/scatter address views, and helper-passed mutable memories all stage mutable alias nodes today. However, many proven aliases are transfer-scoped views rather than long-lived competing writers. A stricter Rust design could still be viable if it preserves these as affine, operation-scoped views and reserves hard errors for persistent overlapping mutable `val`s. The corpus evidence argues against deleting the compatibility behavior wholesale; it argues for modeling Spatial's intentional view aliases explicitly, then tightening only the cases that remain genuinely ambiguous.
