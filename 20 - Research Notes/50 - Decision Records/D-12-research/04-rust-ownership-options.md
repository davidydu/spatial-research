# Rust Ownership Options for Mutable Aliases

## Source Baseline

Argon does not model aliasing as Rust ownership. It stages each symbol with an `Effects` record containing boolean flags plus `reads`, `writes`, and `antiDeps` (`argon/src/argon/Effects.scala:20-30`). The semantics note summarizes the same contract: every staged symbol has local effects, staging expands them through mutable inputs and alias-write propagation, and scheduling/CSE preserve observable behavior through the resulting impure records (`10 - Effects and Aliasing.md:37-47`). Alias facts are graph metadata. `Op.aliases`, `contains`, `extracts`, and `copies` feed `shallowAliases` and `deepAliases`; `allAliases` is their union, and `mutableAliases` filters that set by `isMutable` (`argon/src/argon/Op.scala:48-99`; `argon/src/argon/static/Implicits.scala:153-170`).

The current legality switch is narrow. Argon defaults `enableMutableAliases` to `false` (`argon/src/argon/Config.scala:45-47`). `checkAliases` rejects `((sym.mutableAliases diff effects.writes) diff Set(sym))` when that set is nonempty and the flag is not enabled, while immutable writes are always rejected (`argon/src/argon/static/Staging.scala:39-75`). Spatial then deliberately enables the flag with the comment "Spatial allows mutable aliases for DRAM, SRAM, etc." (`src/spatial/Spatial.scala:583-585`). Spatial memory alias nodes make that concrete: `MemAlias` allocates an alias with `Effects.Mutable` when mutable, and dense/sparse aliases set `mutable = true` and report the underlying memory as their alias source (`src/spatial/node/HierarchyMemory.scala:28-38`, `src/spatial/node/HierarchyMemory.scala:48-63`, `src/spatial/node/HierarchyMemory.scala:81-98`).

Hardware effects depend on preserving those sets before HLS-specific simplification. `computeEffects` propagates writes, adds reads of mutable inputs, and attaches anti-dependencies (`argon/src/argon/static/Staging.scala:230-234`). Write propagation expands writes through alias closure, optionally after atomic-write extraction (`argon/src/argon/static/Staging.scala:236-260`). The semantics note explicitly warns that alias closures should not be replaced by borrow-checker intuition and says HLS memory-dependence pragmas can be derived later from `reads`, `writes`, and `antiDeps`, once those match Argon (`10 - Effects and Aliasing.md:61-63`).

## Option 1: Strict Borrow-Like IR

The Rust IR could make multiple mutable aliases a hard error everywhere, treating each mutable memory as if it had one active mutable owner (unverified). That gives a clean Rust-facing story and makes hardware memory effects easier to reason about: a write set names one unique mutable resource, while views are either immutable reads or scoped exclusive borrows (unverified). The cost is semantic drift. Current Spatial intentionally creates mutable DRAM/SRAM/SRAM-view aliases and globally enables the escape hatch, so a hard error would reject programs and transformations that the Scala implementation accepts today (`src/spatial/Spatial.scala:583-585`; `src/spatial/node/HierarchyMemory.scala:48-63`). It would also move correctness out of the existing alias/write closure and into a new proof discipline, which the sources do not define (unverified).

## Option 2: Explicit Alias Capability Tokens

A middle path is to represent mutable aliasing with explicit capability tokens: creating `MemDenseAlias` or `MemSparseAlias` would produce an alias value plus an `AliasCap` tying it to a base memory, range expression, access mode, and possibly a disjointness predicate (unverified). The IR could require the token for writes through the alias, while still recording Argon-compatible `reads`, `writes`, alias closure, and anti-dependencies (unverified). This maps well to hardware because the token can become the bridge between type-level permission and HLS memory facts: same storage means conservative dependence; proven disjoint ranges or banks can justify parallel ports, partitioning, or dependence pragmas (unverified). The risk is complexity: Spatial's current source facts define alias metadata and effects, not capability construction, token lifetime, or proof checking.

## Option 3: Legacy Compatibility Mode

The Rust IR can preserve `enableMutableAliases` as a compatibility mode, ideally renamed to make the policy explicit, such as `alias_policy = LegacySpatial | Strict | CapabilityChecked` (unverified). In legacy mode, the Rust staging/checking pass would reproduce Argon's alias candidate set, immutable-write rule, write propagation, and anti-dependency computation. Mutable alias findings would be nonfatal when the mode permits them, matching Spatial's current behavior; immutable writes would remain hard errors because Argon has no flag for them (`argon/src/argon/static/Staging.scala:52-67`). Hardware lowering would then inherit the current conservative contract: aliases may denote the same storage, so scheduling and memory-dependence decisions must respect expanded `writes`, mutable-input `reads`, and `antiDeps` (unverified, source-grounded by `argon/src/argon/static/Staging.scala:202-260`).

## Diagnostics And Lints

Diagnostics should exist in all modes (unverified). Argon already reports the current symbol and every alias definition when it rejects mutable aliasing (`argon/src/argon/static/Staging.scala:52-60`). The Rust implementation should preserve that provenance, then add mode-aware lints: warn in legacy mode, error in strict mode, and suggest capability annotations when the capability path exists (unverified). A useful lint distinction is whether the alias is already in `effects.writes`: Argon excludes co-written aliases from the error set, so the diagnostic should not imply that every alias relation is illegal (`argon/src/argon/static/Staging.scala:39-42`) (unverified).

## Recommendation

Do not make mutable aliasing a hard IR error in the first Rust implementation. Preserve a compatibility flag/mode, make diagnostics unconditional, and design explicit alias capability tokens as the migration path to stricter checking (unverified). This keeps Rust+HLS aligned with the existing Spatial hardware model while leaving room for a more Rust-like surface once alias capabilities and range proofs are specified.
