# Hardware, Chisel, and Fringe Rounding

## Generator Surface

The Chisel generator treats `Unbiased` as an explicit fixed-point rounding mode, not as a generic "accurate" or deterministic rounding name. `ChiselGenMath` lowers `UnbMul`, `UnbDiv`, `UnbSatMul`, and `UnbSatDiv` into `Math.mul` or `Math.div` calls with `Unbiased` and either `Wrapping` or `Saturating` overflow (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenMath.scala:33`, `/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenMath.scala:40`). Casts make the same distinction: normal fixed-to-fixed casts pass `Truncate`, while `FixToFixUnb` and `FixToFixUnbSat` pass `Unbiased` into `Math.fix2fix` (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenMath.scala:75`, `/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenMath.scala:78`). The interface therefore already separates truncation from stochastic-unbiased intent at codegen time.

## Fringe Hardware Policy

Fringe names only two fixed-point rounding modes: `Truncate` and `Unbiased`; the source comment defines `Unbiased` as "Use a LFSR for stochastic (unbiased) rounding" (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/RoundingMode.scala:3`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/RoundingMode.scala:8`). The concrete `fix2fixBox` implements that policy only when shrinking fractional width. For `Truncate`, it slices away the lower `shave_f` bits; for `Unbiased`, it instantiates `PRNG`, adds `prng.io.output(shave_f - 1, 0)` to the full input word, and then takes the same shifted slice (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Converter.scala:36`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Converter.scala:44`). [Inference] This is stochastic round-by-random-offset: the discarded residue creates a carry into the kept field with probability approximately residue / `2^shave_f`, modulo LFSR quality and edge cases.

## Seed and Replay

The hardware seed is elaboration-time ambient Scala randomness, not a stable semantic input. `fix2fixBox` seeds `PRNG` with `scala.math.abs(scala.util.Random.nextInt)` and permanently enables it with `prng.io.en := true.B` (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Converter.scala:39`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Converter.scala:42`). The PRNG itself is a Chisel register initialized from that seed and advanced by XOR/shift feedback when `io.en` is true (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/PRNG.scala:6`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/PRNG.scala:16`). [Inference] Once Verilog is generated, replay is deterministic for a fixed emitted seed and cycle trace, but two elaborations can choose different seeds unless the host Scala RNG is controlled. Because `fix2fixBox` enables the PRNG every cycle rather than tying it to `flow`, stalls or retiming can also affect which random word aligns with which value.

## BigIPSim Literal Path

`BigIPSim` uses the hardware box for non-literal `fix2fix` inputs, but literals bypass the PRNG module entirely (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/BigIPSim.scala:237`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/BigIPSim.scala:245`). The literal path computes `salt` once at elaboration with `scala.math.random` when `rounding == Unbiased` and fractional width shrinks, then folds a single `newlit` constant and retimes its sliced bits (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/BigIPSim.scala:247`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/BigIPSim.scala:270`). This is not runtime stochastic rounding; it is randomized constant folding. The nearby source warning, "Likely that there are mistakes here," is attached directly to this literal block (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/BigIPSim.scala:247`).

## Signed Behavior and ScalaGen Divergence

Visible signed handling in Fringe is structural rather than a separate signed stochastic rule. Decimal expansion sign-extends from `io.a.msb` when `s1` is true, while saturating shrink uses `io.a.msb`, `expect_pos`, `expect_neg`, and `s1/s2` to choose max/min-style outputs (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Converter.scala:65`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Converter.scala:88`). Arithmetic paths such as multiply and divide provide expected sign flags before downcasting (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Math.scala:251`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Math.scala:317`), but direct `BigIPSim.fix2fix` clears both flags (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/BigIPSim.scala:241`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/BigIPSim.scala:244`). [Inference] The canonical hardware policy is stochastic rounding on fractional shrink, but signed exactness is under-specified in the visible code.

ScalaGen diverges in both probability granularity and RNG contract. It emits `*&`, `/&`, or `FixedPoint.unbiased` for unbiased operations and casts (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenFixPt.scala:96`, `/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenFixPt.scala:114`). `FixedPoint.unbiased` always uses four guard bits, draws `scala.util.Random.nextFloat()`, and on a stochastic carry increments nonnegative biased values but decrements negative ones (`/Users/david/Documents/David_code/spatial/emul/src/emul/FixedPoint.scala:232`, `/Users/david/Documents/David_code/spatial/emul/src/emul/FixedPoint.scala:240`). D-17 should therefore treat existing hardware evidence as favoring stochastic `Unbiased`, while seedability, literal-fold replay, and signed parity remain open canonical-policy requirements.
