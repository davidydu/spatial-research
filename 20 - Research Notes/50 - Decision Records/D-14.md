# D-14 — Operational Semantics For `Fork`, `ForkJoin`, And `PrimitiveBox`

## Question

Define operational semantics for `Fork`, `ForkJoin`, and `PrimitiveBox`.

Source: `src/spatial/metadata/control/ControlData.scala:7-14`; `src/spatial/metadata/control/package.scala:193-219`; [[10 - Control]].

Decision criteria: User decision: define operational semantics for `Fork`, `ForkJoin`, and `PrimitiveBox` schedules in the Rust/HLS scheduler.

## Research findings

- The schedule enum is a metadata vocabulary, not a complete execution model. `Fork`, `ForkJoin`, and `PrimitiveBox` are preserved by effective-schedule queries even when single-control `Pipelined` collapses to `Sequenced`, but their meaning comes from structural origins and consumers. `ForkJoin` is a parallel barrier region, `Fork` is a branch region, and `PrimitiveBox` is an opaque primitive/blackbox boundary. See [[01-control-metadata-effective-schedule]].

- Flow rules assign these schedules categorically. `ParallelPipe` becomes `ForkJoin`; `Switch` and `IfThenElse` become `Fork`; `SpatialBlackboxImpl` becomes `PrimitiveBox`; `SpatialCtrlBlackboxImpl` and `SwitchCase` are `Sequenced`; transfers are `Pipelined`. Generic fallback scheduling is separate and should not be used to reinterpret these structural tags. See [[02-flow-rules-schedule-assignment]].

- User/API origins reinforce the split. `Parallel { ... }` stages `ParallelPipe`, and unrolling/binding can synthesize additional `ParallelPipe`s for sibling child controllers. Conditional `if`/`else` lowers through `IfThenElse`, `Switch`, and `SwitchCase`. `Blackbox.SpatialPrimitive` stages a primitive blackbox implementation, while controller blackboxes are a different sequenced control form. See [[03-api-ir-origins]].

- Chisel/Fringe gives the hardware reference for `ForkJoin` and `Fork`. `ForkJoin` launches enabled children for the same parent iteration and advances only after all non-skipped child `iterDone` bits are true. `Fork` gates children by select bits and completes when a selected child reports done. Fringe has no `PrimitiveBox` schedule, so primitive boxes must be handled as atomic leaves rather than sixth controller state machines. See [[04-chisel-fringe-controller-semantics]].

- ScalaGen is a labelled compatibility behavior, not the synthesis contract. It emits `ParallelPipe` textually and emits `Switch` as an ordered `if`/`else if` chain. That is useful for deterministic regression, but it does not model `ForkJoin` child overlap, join timing, or stream/back-pressure interactions. See [[05-scalagen-codegen-behavior]].

- The runtime model already prices these classes differently. `ForkJoin` uses max-child cost per iteration plus synchronization; `Fork` uses branch-duty-cycle expected cost; `PrimitiveBox` does not appear in the model enum and needs primitive latency/II provenance. DSE currently consumes cycle totals through this model, so HLS-native semantics must feed calibrated or labelled estimates rather than silently reusing stale equations. See [[06-runtime-model-dse]].

- Tests/apps support the structural interpretation. `Parallel` tests expect all sibling side effects to be visible after the region, branch tests expect only selected effects, and blackbox tests expect primitive input/output replacement with latency/effect assumptions. They do not prove every cycle detail, but they rule out treating the tags as opaque decoration. See [[07-tests-apps-evidence]].

- HLS mapping should be semantic rather than waveform-literal. `ForkJoin` can map to dataflow/task parallelism with explicit channels, memory dependences, and a join barrier. `Fork` should map to ordinary conditional control with branch-mask propagation. `PrimitiveBox` should map to an inline/noinline/vendor/IP primitive with declared effects, latency, II, and reference model or explicit rejection. See [[08-hls-scheduler-mapping]].

- Risk analysis rejects both extremes. Exact Chisel handshake parity is expensive and brittle; ScalaGen serial semantics are cheap but wrong as HLS schedule semantics; opaque metadata invites inconsistent lowering. The preferred path is HLS-native semantics plus explicit compatibility modes and diagnostics. See [[09-risk-implementation-cost]] and [[10-recommendation-matrix]].

## Recommendation

Adopt **Hybrid Modes With HLS-Native Synthesis Semantics**.

The canonical Rust/HLS scheduler semantics should be:

- `ForkJoin`: For each logical parent iteration, start every enabled, non-masked child region as a peer task. A skipped/masked child counts as done for that iteration. The parent iteration completes only when every non-skipped child completes. Memory, stream, and scalar communication between children must be represented by explicit dependences, channels, or conservative scheduling barriers. Latency estimation is max-child plus synchronization until backend reports replace the estimate.

- `Fork`: Evaluate the select predicates for the logical parent iteration, enable only the selected case region(s), propagate the branch mask to all writes, enqueues, primitive starts, and nested controllers, and complete the parent when the selected case completes. The default HLS mode should require/prove one-hot selectors for `Switch`/branch controllers, because Scala executor priority and Chisel OR-done behavior can diverge when multiple selectors are true. A labelled `compat_scalagen_priority` mode may use ordered first-true `if`/`else if` behavior for reference simulation.

- `PrimitiveBox`: Treat as an atomic primitive/blackbox leaf, not a controller schedule with children. Each primitive box must declare ports, read/write/stream effects, latency or latency range, initiation interval, reset/start/done behavior, and a Rust reference implementation or explicit "synthesis-only" marker. HLS lowering may use an inline function, noinline function, vendor/library call, or RTL blackbox wrapper. The scheduler may reorder across a primitive box only when declared effects prove independence.

Use two named execution modes:

- `hls_semantic`: the default for Rust/HLS synthesis, HLS-aware simulation, DSE, and report reconciliation. It uses the three semantics above and records HLS mapping/provenance.
- `compat_scalagen_serial`: deterministic compatibility mode for old ScalaGen value regression. It serializes `ParallelPipe` textually and uses ordered branch priority, and it must be labelled as not cycle/back-pressure faithful.

Reserve future `cycle_backpressure` or `bounded_dataflow` mode for after D-15 and D-22 settle ParallelPipe simulation and FIFO/stream boundedness. D-14 should not bake unbounded or serial stream behavior into the schedule definitions.

Reject **opaque metadata**. A scheduler that preserves the enum but leaves meaning to later passes will eventually serialize a `ForkJoin` in one place, price it as max-child in another, and emit HLS dataflow in a third.

Reject **Scalagen serial as canonical HLS semantics**. It is a useful compatibility oracle for values, but it contradicts Chisel/Fringe `ForkJoin` and the runtime model's max-child pricing.

Reject **exact Chisel/Fringe handshake parity as the required Rust/HLS implementation**. The HLS rewrite should preserve the semantic obligations: selected branch execution, all-child join, masks, effects, back-pressure hooks, and done provenance. It does not need to clone the Chisel state-machine wiring when HLS-native task/channel constructs provide an equivalent contract.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:

1. Add an explicit schedule semantics layer with `ForkJoin`, `Fork`, and `PrimitiveBox` operational rules separate from raw/effective metadata.
2. Implement `hls_semantic` mode with `ForkJoin` peer-task launch plus all-child join, `Fork` selected-branch execution, and `PrimitiveBox` atomic effectful leaf semantics.
3. Implement `compat_scalagen_serial` mode for deterministic Rust reference simulation and label all outputs from that mode.
4. Add selector policy diagnostics: prove one-hot selectors in HLS mode, reject or warn on dynamic multi-true selectors, and optionally route through `compat_scalagen_priority`.
5. Require a primitive-box manifest: ports, latency/II provenance, effects, reference-model availability, synthesis lowering kind, and start/done/reset behavior.
6. Update DSE/HLS reporting to record whether schedule latency came from Spatial runtime estimate, declared primitive latency, HLS report, or compatibility fallback.
7. Add tests for all-child `ForkJoin` join, selected-only `Fork` effects, multi-true selector diagnostics, masked children, primitive-box latency/effect manifests, and labelled ScalaGen-serial compatibility.
8. Cross-reference D-15 for `ParallelPipe` simulator defaults and D-22 for bounded FIFO/stream back-pressure in dataflow children.
