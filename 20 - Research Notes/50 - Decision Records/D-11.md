# D-11 — Rust Simulator Semantics For `breakWhen`

## Question

Choose the Rust simulator semantics for `breakWhen`.

Source: `spatial/src/spatial/codegen/scalagen/ScalaGenController.scala:74-93`.

Decision criteria: User decision: Rust simulation matches Scalagen end-of-iteration `breakWhen` or synthesized immediate-break behavior.

## Research findings

- The current ScalaGen behavior is a documented end-of-iteration compatibility oracle, not a clean architectural contract. `ScalaGenController.scala` emits a warning that Scala break occurs at the end of the loop while synthesis breaks immediately, then lowers forever and bounded controllers through top-tested `while`/`takeWhile` guards. A breaker update inside the body is therefore observed only before the next iteration, so same-iteration side effects after the breaker write can still commit. See [[01-scalagen-current-semantics]].

- The hardware-oriented implementation points the other way. API comments describe `Stream(breakWhen)` and `Sequential(breakWhen)` as immediate controller breaks with reset when the controller finishes. Chisel connects the breaker register to the controller `break` input, Fringe folds that input into controller active/done behavior, and Chisel memory/stream lowerings gate many effects with `~break`. That makes immediate behavior the better Rust default for HLS correlation. See [[02-chisel-rtl-hls-semantics]].

- `breakWhen` is an explicit IR operand, not merely codegen metadata. It appears on `UnitPipe`, `OpForeach`, `OpReduce`, `OpMemReduce`, and unrolled controller forms; unrolling and IR copying preserve it. Related metadata marks breaker registers and controller ownership, while `UseAnalyzer` classifies break registers as special dependencies. This gives the Rust rewrite enough structure to model `breakWhen` directly instead of inheriting ScalaGen loop syntax. See [[03-api-ir-metadata-contract]].

- Existing passes already treat breaker state as a scheduling and dependency boundary. Binding and modeling logic use breaker writes to constrain binding order and push dependent nodes later in latency estimates. That is consistent with a hardware-visible control cut point, and it means a simulator that blindly commits all post-break same-iteration effects would disagree with assumptions elsewhere in the compiler. See [[04-pass-scheduling-interactions]].

- The test and app corpus shows three distinct usage classes. Some tests, especially `Breakpoint` cases where the breaker is assigned before a store, require immediate suppression of later same-iteration effects. Others intentionally do useful work before setting the breaker and should still commit those earlier effects. A larger group uses `breakWhen` for eventual stream or workload termination and mainly needs stable drain behavior. See [[05-tests-app-usage]].

- The viable policy space is not just "Scala" versus "hardware." A pure ScalaGen default preserves legacy emitted-code behavior but bakes in a known mismatch. A pure immediate simulator gives the right architectural answer but makes legacy differential testing harder. The most useful migration path is a hardware-default mode plus an explicit ScalaGen compatibility mode and a compare/diagnostic mode. See [[06-policy-options]] and [[07-migration-testing-strategy]].

- Side effects are the core risk. Register, ArgOut, SRAM/DRAM, FIFO, stream, nested-controller, and reduction effects can diverge depending on whether they occur before or after the breaker write. Immediate semantics must not retroactively erase work that already occurred before the breaker was produced, but it should suppress future `~break`-guarded work once the controller-visible break is active. See [[08-side-effect-risk]].

- External HLS and control-DSL precedent generally favors explicit immediate exit as the architectural model, with compatibility modes reserved for legacy simulators. The useful lesson is to make the commit point, breaker reset, and diagnostic provenance visible rather than relying on host-language loop semantics to define hardware behavior. See [[09-precedent-hls-patterns]].

- The recommendation matrix converges on an **HLS-Default Dual Provenance** policy: default to `hls_immediate`, retain `scalagen_end_of_iteration` as an opt-in compatibility mode, and add `dual_compare`/diagnostic reporting for post-break effects that differ between the two. See [[10-recommendation-matrix]].

## Recommendation

Adopt **HLS-Default Dual Provenance**.

The Rust simulator should default to `hls_immediate`: `breakWhen` is an architectural controller break whose visible effect is to stop later controller work and `~break`-guarded side effects once the breaker becomes active. The default reset policy should be `on_controller_done`, matching the existing API and Chisel/Fringe shape. This makes the Rust reference simulator useful for the Rust+HLS rewrite rather than reproducing a ScalaGen warning as a silent contract.

The policy must still define the cut point precisely. Immediate break is not "time travel." Effects that occur before the breaker assignment in the controller protocol remain committed. Effects after the breaker assignment, or effects whose hardware lowering is guarded by the active break signal, are suppressed. That distinction preserves drain/completion patterns such as stores before a final breaker update while fixing cases like `Breakpoint` where a breaker is set before later writes that hardware should kill.

Keep `scalagen_end_of_iteration` as an explicit compatibility mode. That mode should match the current ScalaGen lowering: a top-tested loop whose breaker value is observed at the next iteration boundary. It is useful for old Scala-generated goldens and for demonstrating historical divergence, but it should never be the unlabelled Rust default.

Add `dual_compare` as a migration/debug mode. In this mode, the simulator should evaluate or trace both policies and report divergent side effects: register writes, ArgOut commits, memory stores, stream/FIFO enqueues and dequeues, reductions, nested-controller entry, and controller done/reset timing. Reports should include controller id, breaker register id, source span, lowering stage, and whether the breaker is user-authored or compiler-inserted.

Use these manifest fields, or close equivalents:

- `break_when_mode`: `hls_immediate | scalagen_end_of_iteration | dual_compare | diagnostic_only`
- `break_when_reset`: `on_controller_done | manual_legacy | compare_modes`
- `compatibility_label`: `hls_default | scalagen_compat | migration_compare | strict_conformance`
- `divergent_side_effect_policy`: `kill_hls_guarded_effects | commit_scalagen_iteration | warn_on_divergence | error_on_divergence`
- `break_when_provenance`: controller id, breaker reg id, source span, lowering stage, and user-authored/compiler-inserted origin

Reject pure ScalaGen end-of-iteration as the architectural default. It contradicts the API comments, the Chisel/Fringe controller protocol, the effect gating shape, and tests that already encode immediate suppression. It would make the Rust rewrite optimize for parity with a known simulator artifact.

Reject pure immediate-only semantics without compatibility or provenance. It would be simpler, but it would make legacy comparisons opaque and would turn real migration differences into unexplained golden failures.

Reject a diagnostic-only non-decision. D-03 already places early exit on structured controller mechanisms rather than raw `while`/`return`; `breakWhen` is the central mechanism that needs a real default.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:

1. Add the `break_when_mode`, reset, compatibility label, divergent-side-effect policy, and provenance fields to the Rust simulator configuration/report schema.
2. Implement `hls_immediate` with a clear side-effect cut point and `on_controller_done` reset.
3. Implement `scalagen_end_of_iteration` as a legacy compatibility mode matching ScalaGen's top-tested loop behavior.
4. Implement `dual_compare` diagnostics for post-break divergence across register, memory, stream, FIFO, reduction, and nested-controller effects.
5. Classify existing tests as `hls_default`, `scalagen_compat`, `mode_independent`, or `expect_divergence`.
6. Add distinguishing goldens based on `Breakpoint` before-store breaks, after-store drain breaks, nested controllers, and stream termination.
7. Preserve `stopWhen` as an explicit IR field through Rust IR copying, unrolling, and metadata/provenance generation.
8. Audit HLS lowerings for breaker reset and drain semantics before marking the decision applied to specs.
