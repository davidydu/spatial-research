# D-18 Research: Spec and Open-Question Context

## Decision Frame

D-18 is the architectural decision for whether Rust float packing must reproduce the current `FloatPoint.clamp` heuristic bit-for-bit. The queue ties the decision directly to Q-145 and names the source as `spatial/emul/src/emul/FloatPoint.scala:335-339`, with the choice framed as exact reproduction of the `x > 1.9` guard versus a cleaner custom-float algorithm that accepts divergence (`20 - Research Notes/40 - Decision Queue.md:79-81`). Q-145 says the same thing in open-question form: the guard is unexplained, and the decision is blocked by numeric conformance tests (`20 - Research Notes/20 - Open Questions.md:2065-2075`). In other words, [[20 - Numeric Reference Semantics]] specifies what Spatial does today; D-18 asks whether the Rust/HLS rewrite treats that behavior as compatibility law.

## Specified Behavior

The relevant specs make Scalagen the reference. [[10 - Overview]] says Scalagen plus `emul` is the de facto dynamic-semantics ground truth, including the bit layout of packed `FloatPoint` values (`10 - Spec/50 - Code Generation/20 - Scalagen/10 - Overview.md:24-26`), and says Rust test-vector compatibility requires bit-identical outputs for every op family (`10 - Spec/50 - Code Generation/20 - Scalagen/10 - Overview.md:146-150`). [[20 - Numeric Reference Semantics]] repeats that numeric IR node meaning is the corresponding `emul` operator meaning (`10 - Spec/50 - Code Generation/20 - Scalagen/20 - Numeric Reference Semantics.md:25-29`). [[50 - Data Types]] states that when Spatial differs from native Scala, JVM, or hardware expectations, Scalagen wins for simulator/reference semantics (`10 - Spec/20 - Semantics/50 - Data Types.md:36-38`).

For floats, `FltFormat` defines `sbits`, `ebits`, `bias`, `MIN_E`, `MAX_E`, and `SUB_E = MIN_E - sbits` (`spatial/emul/src/emul/FltFormat.scala:3-8`). `FloatPoint` stores `NaN`, signed infinity, signed zero, or `Value(BigDecimal)` rather than native float payloads (`10 - Spec/50 - Code Generation/20 - Scalagen/20 - Numeric Reference Semantics.md:29`). `Value.bits` calls `FloatPoint.clamp`, then emits mantissa bits, exponent bits, and sign (`spatial/emul/src/emul/FloatPoint.scala:120-126`). Arithmetic operators also feed results through `FloatPoint.clamped` (`spatial/emul/src/emul/FloatPoint.scala:153-165`), so clamp is not just serialization; it is the runtime rounding/packing boundary.

## Clamp Details

`FloatPoint.clamp(value, fmt)` first maps exact zero to positive zero, then decomposes nonzero magnitudes into `y = floor(log2(abs(value)))` and `x = abs(value) / 2^y` (`spatial/emul/src/emul/FloatPoint.scala:318-325`). The D-18 branch fires when `y < fmt.SUB_E && x > 1.9`, increments `y`, and forces `x = 1` (`spatial/emul/src/emul/FloatPoint.scala:335-339`). If that branch does not fire, a separate `x >= 2` guard adjusts exponent/mantissa, and a cutoff rule can decrement `y` for small mantissas (`spatial/emul/src/emul/FloatPoint.scala:340-351`).

Range dispatch is then explicit: overflow becomes signed infinity; normal values build a mantissa from `(x - 1) * 2^(sbits + 1)` and round with the low bit; subnormals build `x * 2^(sbits + 1)`, shift according to `MIN_E - y + 1`, and round by the discarded bit; too-small values become signed zero (`spatial/emul/src/emul/FloatPoint.scala:358-395`). `convertBackToValue` then reconstructs the canonical stored value, and `clamped` performs this round trip for `Value(_)` while bypassing `NaN`, `Inf`, and `Zero` (`spatial/emul/src/emul/FloatPoint.scala:399-433`). [[20 - Numeric Reference Semantics]] summarizes this as custom clamp with subnormal flushing, not native `f32`/`f64` behavior (`10 - Spec/50 - Code Generation/20 - Scalagen/20 - Numeric Reference Semantics.md:55-72`).

## Unresolved and Blocked

The unresolved part is not the code path; it is the design status of the magic threshold. Q-120 records the same snippet, notes that `1.9` is undocumented, and offers only a conjecture that it may guard a near-subnormal rounding edge where the mantissa would escape to 2.0 (`20 - Research Notes/20 - Open Questions.md:1617-1633`). That older entry is marked resolved by spec via [[20 - Numeric Reference Semantics]] (`20 - Research Notes/20 - Open Questions.md:1635-1637`), but Q-145 reopens the architectural choice for Rust/HLS: preserve bit compatibility or replace the heuristic with a cleaner custom-float algorithm (`20 - Research Notes/20 - Open Questions.md:2070-2075`).

For Rust, bit compatibility means implementing the Scalagen algorithm, including the `Math.log`-based `log2BigDecimal` path, `BigDecimal`-like decomposition, the `x > 1.9` adjustment, subnormal rounding, signed-zero underflow, `FloatValue` variants, and valid-bit propagation. [[50 - Data Types]] warns that custom floating formats cannot be delegated to native `f32`/`f64` if bit parity is required (`10 - Spec/20 - Semantics/50 - Data Types.md:64-66`). For HLS, the hardware can use fixed-width target arithmetic, but the simulator-side replica remains `rework`: it needs arbitrary-precision equivalents and bit-for-bit Scalagen conformance unless D-18 explicitly accepts divergence (`10 - Spec/50 - Code Generation/20 - Scalagen/20 - Numeric Reference Semantics.md:88-94`). The immediate blocker is therefore numeric conformance tests around the subnormal boundary, especially cases that distinguish the `x > 1.9` path from principled IEEE-style packing.
