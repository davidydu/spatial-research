## Search scope

D-18 asks whether the Rust port should preserve `FloatPoint.clamp` exactly, especially the unexplained subnormal guard that bumps `y` when `y < fmt.SUB_E && x > 1.9` (`emul/src/emul/FloatPoint.scala:318`, `emul/src/emul/FloatPoint.scala:335`). I searched the source tests, app examples, regression lists, and vault specs for `FloatPoint`, `Flt`, custom `FltPt`, `Half`, `clamp`, saturation, subnormal/overflow vocabulary, and float/fixed casts. The relevant semantic baseline is already stated in [[50 - Data Types]]: Scalagen/emul wins over native Scala/JVM behavior, and `FloatPoint.clamp` packing is one of the important bit-exact data-type facts (`10 - Spec/20 - Semantics/50 - Data Types.md:38`). The decision queue frames D-18 as choosing bit-for-bit preservation of the `x > 1.9` heuristic versus a cleaner custom-float algorithm with accepted divergence (`20 - Research Notes/40 - Decision Queue.md:79`).

## Direct float tests found

There is no test that calls `FloatPoint.clamp`, `FloatPoint.bits`, `FloatPoint.fromBits`, or `isSubnormal` directly. The strongest ordinary `Float` regression is `FloatBasics2`, which is listed in both mini and nightly (`regressions/mini.list:1`, `regressions/nightly.list:1`). It uses `type T = Float`, random values bounded by 255, and checks add, multiply, subtract, comparisons, reciprocal, reciprocal sqrt, clipping-to-5, and multiply-add against host-side Spatial expressions (`test/spatial/tests/feature/math/FloatBasics2.scala:5`, `test/spatial/tests/feature/math/FloatBasics2.scala:12`, `test/spatial/tests/feature/math/FloatBasics2.scala:39`, `test/spatial/tests/feature/math/FloatBasics2.scala:91`). `FloatBasics` covers scalar `Float` add/mul/div/sqrt/pow/max/comparison with runtime args `3.2752 -283.70`, but its tolerances are numerical, not bit-level (`test/spatial/tests/feature/math/FloatBasics.scala:7`, `test/spatial/tests/feature/math/FloatBasics.scala:75`). `FloatTransfer` only moves quarter-step `Float` values through DRAM/SRAM (`test/spatial/tests/feature/transfers/FloatTransfer.scala:16`, `test/spatial/tests/feature/transfers/FloatTransfer.scala:46`).

## Custom formats and casts

`HalfBasics` is the only substantial `Half` math test. It randomly samples values up to 1024, performs arithmetic, sqrt, abs, exp approximation, reciprocal, reciprocal sqrt, and FMA-like multiply-add, then compares against the same Spatial/emul reference path (`test/spatial/tests/feature/math/HalfBasics.scala:5`, `test/spatial/tests/feature/math/HalfBasics.scala:12`, `test/spatial/tests/feature/math/HalfBasics.scala:39`, `test/spatial/tests/feature/math/HalfBasics.scala:89`). This can incidentally hit half overflow in multiply, but because the expected value is computed through the same semantics and the inputs are random, it is not a stable D-18 fixture. `HalfTest` only checks `0.to[Half]` and `1.to[Half]` through a mux (`test/spatial/tests/feature/math/HalfTest.scala:16`, `test/spatial/tests/feature/math/HalfTest.scala:22`).

The only custom `FltPt` use in syntax tests is `FltPt[_32,_32]` in lifting checks, and it only lifts the exact literal `32.0` from Scala float/double into `F32`, `F64`, fixed, and the custom float (`test/spatial/tests/syntax/Lifting.scala:167`, `test/spatial/tests/syntax/Lifting.scala:178`, `test/spatial/tests/syntax/Lifting.scala:181`, `test/spatial/tests/syntax/Lifting.scala:199`). `SpecialMath` uses `FltPt[_24,_8]` as an analysis type for fixed-point unbiased-rounding means, not as a clamp-boundary target (`test/spatial/tests/feature/math/SpecialMath.scala:75`, `test/spatial/tests/feature/math/SpecialMath.scala:86`). `Float32ToInt8` and `LowPrecisionRep` are clamp-sensitive only through float-to-byte casts after scaling, and both compute golden values with the same cast pipeline (`test/spatial/tests/feature/math/Float32ToInt8.scala:34`, `test/spatial/tests/feature/math/Float32ToInt8.scala:40`, `test/spatial/tests/feature/math/LowPrecisionRep.scala:29`, `test/spatial/tests/feature/math/LowPrecisionRep.scala:31`).

## Gaps and D-18 fixtures

The gap is sharp: current tests protect user-visible float arithmetic enough to catch gross breakage, but not `FloatPoint.clamp` packing. The codegen surface makes that risky because Scalagen lowers `FltToFlt` to `toFloatPoint(FltFormat(...))`, text constants to `FloatPoint(..., FltFormat(...))`, random floats to `FloatPoint.random`, and bit reinterpretation to `.bits` / `FloatPoint.fromBits` (`src/spatial/codegen/scalagen/ScalaGenFltPt.scala:84`, `src/spatial/codegen/scalagen/ScalaGenFltPt.scala:90`, `src/spatial/codegen/scalagen/ScalaGenFltPt.scala:96`, `src/spatial/codegen/scalagen/ScalaGenBits.scala:47`, `src/spatial/codegen/scalagen/ScalaGenBits.scala:53`). D-18 should add deterministic fixtures that assert exact bit arrays and round-trip values for `Half`, `Float`, `Double`, and at least one nonstandard `FltPt[M,E]` around: max normal to infinity, min normal to subnormal, smallest subnormal to signed zero, the `x > 1.9` branch boundary, negative zero, NaN, and +/- infinity. Add staged tests for `to[Half]`, `to[Float]`, `to[FltPt[_32,_32]]`, `bits.as[...]` round trips, and float-to-fixed casts near saturation. Without these fixtures, either D-18 choice can silently pass the existing test/app suite.
