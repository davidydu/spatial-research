# D-24 - Fixed-Point Host Conversion Policy

## Question

Choose bit-exact or Cppgen-compatible fixed-point host conversion.

Source: `/Users/david/Documents/David_code/spatial/src/spatial/codegen/cppgen/CppGenCommon.scala:75-100`; `/Users/david/Documents/David_code/spatial/src/spatial/codegen/cppgen/CppGenInterface.scala:42-80`; [[10 - Cppgen]]

Decision criteria: User decision: Rust host-side fixed-point conversions match Cppgen double approximations or enforce bit-exact shifted-integer values.

## Research findings

- Cppgen uses two fixed-point representations. Fractional `FixPtType` is a host `double` in ordinary C++ expressions, but register and memory transport use shifted raw integers through `toTrueFix`/`toApproxFix`; structs, bit views, binary files, shifts, and casts each expose related quirks. See [[01-cppgen-conversion-source]].

- Scalagen/emul's reference is raw scaled state, not native floating point. `FixedPoint` stores `value: BigInt`, `valid`, and `FixFormat`; constructors, arithmetic, casts, `bits`, `fromBits`, invalid values, saturation, and unbiased helpers operate in that scaled-integer domain. See [[02-scalagen-emul-reference]].

- Fixed conversion appears at many ABI surfaces: `ArgIn`, `HostIO`/ArgIO, `ArgOut`, DRAM `SetMem`/`GetMem`, binary files, CSV/text, frames, streams, structs, vectors, bit views, CLI arguments, reports, and tests. Raw integer transport should be the canonical evidence; decimal/text views should be labelled adapters. See [[03-host-transfer-abi-surfaces]].

- Existing tests cover the surface but are not sharp raw-bit oracles. `FixPtArgInOut`, `SmallTypeTransfers`, BLAS/SGD-like apps, struct transfers, CSV/Binary IO, and sub-byte transfers mostly use representable values or tolerant comparisons. Missing fixtures need raw goldens, lossy decimals, signed readback, sub-byte fractional rejection, and struct/bitcast payloads. See [[04-tests-apps-affected]].

- D-24 composes with D-17, D-18, D-19, and D-23 but should not take them over. It decides host-boundary fixed conversion only. Stochastic unbiased rounding, custom float packing, fused/unfused FMA, and ABI ordering remain owned by their respective decisions. See [[05-overlap-numeric-abi]].

- The highest risk is silent corruption from preserving Cppgen's approximate host model as the default. A `double` can lose raw fixed precision and still look plausible after readback. Bit-exact conversion is more reproducible but less ergonomic unless Rust provides explicit typed wrappers. See [[06-risk-failure-modes]].

- Migration cost favors dual mode. Cppgen-compatible default is cheap but extends semantic debt; bit-exact-only is clean but sharp for legacy users; dual mode needs more schema/tests but allows staged migration and clear reports. See [[07-implementation-cost-migration]].

- The policy contract should use versioned labels. `cppgen_fractional_double_shift_v1` claims legacy Cppgen host-source compatibility only. `bit_exact_scaled_integer_v1` claims raw scaled-integer parity. `target_native_fixed_v1` is backend/co-sim evidence only. See [[08-policy-options-contract]].

- The recommendation matrix selects dual mode with `bit_exact_scaled_integer_v1` as the Rust default and `cppgen_fractional_double_shift_v1` as explicit legacy compatibility. This gives D-23 a concrete manifest policy and keeps old behavior available without defining correctness around it. See [[10-recommendation-matrix]].

## Recommendation

Adopt **Dual-Mode Fixed-Point Host Conversion With Bit-Exact Default**.

The canonical Rust/HLS host conversion policy should be:

`conversion_policy = bit_exact_scaled_integer_v1`

In this mode, the host boundary treats fixed-point payloads as raw scaled integers with declared format metadata: signedness, integer bits, fractional bits, total storage bits, validity policy, endian/packing, and sign-extension rule. The raw scaled integer is the source of truth for scalar registers, DRAM payloads, binary files, stream/frame payloads, structs, vectors, and bit views. Host APIs may provide ergonomic decimal constructors and formatters, but those adapters must parse and round through a specified exact path and report truncation, wrap, or rejection. Raw APIs such as `from_raw_bits`, `from_scaled_int`, `to_raw_bits`, and `to_scaled_int` should be first-class.

Keep legacy Cppgen behavior as an explicit compatibility lane:

`conversion_policy = cppgen_fractional_double_shift_v1`

This mode preserves the historical C++ host view: fractional fixed values appear as `double`, text parsing uses double-like decimal parsing, scalar writes multiply by `2^f` and cast to an integer transport value, reads sign-extend and divide by `2^f`, DRAM and binary paths rawify where Cppgen does, and Cppgen's unsupported fractional sub-byte cases remain unsupported. This lane may be used for legacy Cppgen replay, old host-source comparison, and migration tests, but it should not be the default correctness oracle for new Rust/HLS artifacts.

D-23 manifests must carry the policy per ABI entry, not as a hidden global. Every scalar, memory element, stream/frame payload, file column, struct field, vector element, and diagnostic should record the conversion policy and numeric format. Reports and tests should include raw hex or raw scaled integer beside logical decimal when possible. If a result is compared under `cppgen_fractional_double_shift_v1`, the report should say so; if it is compared under `bit_exact_scaled_integer_v1`, approximate host decimals should not be accepted as proof unless their exact raw interpretation is recorded.

Target-native/vendor-native conversion is allowed only as a labelled projection:

`conversion_policy = target_native_fixed_<backend>_v1`

It may support co-simulation or generated wrapper integration, but it is not a portable Spatial host contract unless a backend-specific proof maps it to the bit-exact policy.

### Rejected alternatives

Reject **Cppgen-compatible default**. It preserves legacy behavior but makes an approximate double/shift hybrid the canonical semantics, which is exactly the ambiguity D-24 is meant to remove.

Reject **bit-exact-only with no compatibility lane**. It is semantically clean, but it would make migration needlessly abrupt and obscure whether a mismatch is a real bug or legacy Cppgen conversion drift.

Reject **target-native/vendor-native default**. Vendor assignment/co-sim behavior is a backend projection, not a cross-backend Rust/HLS host contract.

Reject **unlabelled decimal convenience**. Decimal host values are fine as adapters, but a test or report must state how they became raw fixed bits.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Define `conversion_policy` labels: `bit_exact_scaled_integer_v1`, `cppgen_fractional_double_shift_v1`, `raw_memcpy_v1`, `packed_subbyte_lsb0_v1`, `unsupported_fractional_subbyte_v1`, and `target_native_fixed_<backend>_v1`.
2. Implement Rust fixed-point host types with raw scaled integer storage, signedness, width, fractional bits, validity, sign extension, wrapping, saturation hooks, and LSB-first bit views.
3. Implement exact decimal/text adapters for bit-exact mode with diagnostics for truncation, out-of-range raw values, and rejected lossy strict parses.
4. Implement a faithful Cppgen compatibility adapter for host `double`, `toTrueFix`, `toApproxFix`, scalar set/get, DRAM rawification, binary file conversion, struct pretty-printing, and unsupported fractional sub-byte cases.
5. Add D-23 manifest fields for conversion policy on scalar args, ArgIO/ArgOuts, DRAMs, streams/frames, host files, structs, vectors, and bitcasts.
6. Add raw golden fixtures for scalar set/get, DRAM set/get, binary fixed payloads, CSV/text fixed parsing, struct `toRaw`, `DataAsBits`, signed negative readback, overflow/wrap, and fractional sub-byte rejection.
7. Preserve legacy tests under `cppgen_fractional_double_shift_v1` while adding canonical Rust/HLS tests under `bit_exact_scaled_integer_v1`.
8. Add diagnostics that report type, format, raw hex, scaled integer, logical decimal, input surface, policy label, truncation/wrap/reject decision, and comparison mode.
9. Mark target-native fixed conversion incomparable with bit-exact goldens unless an adapter proves equivalence.
10. Cross-reference D-17, D-18, D-19, and D-23 in numeric/ABI docs so host conversion is not confused with internal arithmetic rounding, float packing, FMA precision, or ABI ordering.
