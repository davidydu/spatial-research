# D-15 — Rust Simulation Semantics For `ParallelPipe`

## Question

Choose simulation semantics for `ParallelPipe`.

Source: `spatial/src/spatial/codegen/scalagen/ScalaGenController.scala:187-191`; [[50 - Controller Emission]].

Decision criteria: User decision: Rust simulation follows Scalagen sequential `ParallelPipe` execution or models parallel interleaving/back-pressure.

## Research findings

- ScalaGen is a deterministic serial value oracle. `ParallelPipe` is emitted by wrapping the body, calling `gen(func)`, and then invoking a no-op done hook. The block generator visits statements in source order; no threads, child scheduler, join protocol, or back-pressure model is introduced. See [[01-scalagen-parallelpipe-semantics]].

- Hardware-facing semantics are different and were settled at the scheduler level by D-14. `ParallelPipe` is assigned `ForkJoin`; Fringe peer-issues enabled children, treats masked children as skipped-done, and advances the parent only after all non-skipped child `iterDone` bits synchronize under back-pressure. This is not textual serial execution. See [[02-hardware-forkjoin-contrast]] and [[D-14]].

- The Scala executor is a third point in the design space. It routes `ParallelPipe` to `StreamUnitPipeExecutor`, constructs child executors, ticks them in deterministic source order on each parent tick, and completes only when all children are done/disabled/indeterminate. It is cooperative interleaving, not ScalaGen text emission and not full Chisel/ForkJoin handshaking. See [[03-scala-executor-simulator]].

- The tests/apps corpus mostly validates final values, not interleavings. Many parallel/unrolled/FIFO/stream tests compare arrays, reductions, or checksums. Several schedule-sensitive tests have disabled or weak assertions. That supports retaining ScalaGen serial compatibility for legacy goldens, but it is not evidence that serial execution is the correct HLS simulator. See [[04-tests-apps-evidence]].

- Back-pressure cannot be honestly solved inside D-15 alone. ScalaGen queues are elastic, while Chisel/Fringe FIFOs and streams have bounded full/empty/ready/valid pressure. A `ParallelPipe` simulator can model peer issue and join structure now, but bounded stalls, overflow, and deadlock diagnostics depend on D-22's FIFO/LIFO/stream decision. See [[05-stream-backpressure-overlap]].

- Runtime/DSE already assume `ForkJoin` is closer to max-child parallelism than serial sum-child execution. Serial simulation remains useful for value compatibility, but simulator-derived cycle numbers need provenance. HLS reports and accepted II remain the authority for post-tool latency under D-08. See [[06-runtime-model-latency]], [[D-08]], and [[D-14]].

- The practical mode design is explicit and reproducible. `compat_scalagen_serial` answers "what would ScalaGen do?" `hls_semantic_step` answers "what does the D-14 ForkJoin semantics permit?" `randomized_interleaving` is a seeded bug-finder. `bounded_cycle/backpressure` should remain experimental until D-22. See [[07-simulator-mode-design]].

- Implementation and risk analyses agree that serial-only is too weak and bounded-cycle-first is premature. The migration-safe path is dual-mode/compare: preserve serial goldens while making the HLS-facing simulator deterministic peer/barrier by default. See [[08-implementation-migration-cost]], [[09-risk-failure-modes]], and [[10-recommendation-matrix]].

## Recommendation

Adopt **Dual-Mode Simulation With HLS-Semantic Peer/Barrier Default**.

The Rust simulator should support at least two stable modes:

- `hls_semantic`: default for Rust/HLS development, HLS-facing tests, DSE validation, and schedule diagnostics. A `ParallelPipe` starts every enabled, masked-in child for the current logical parent activation as peer work; masked-out children count as skipped-done; the parent completes only when all non-skipped children complete. Tie-breaks and child polling must be deterministic by stable controller/symbol identity. This mode models D-14's `ForkJoin` structure, but until D-22 it must label queue behavior as `elastic` or `bounded_experimental`.

- `compat_scalagen_serial`: explicit compatibility mode for old ScalaGen value goldens. A `ParallelPipe` executes block statements in stable source/IR order using ScalaGen-style elastic FIFO/LIFO/stream behavior. It is deterministic and useful for regression, but every result must be labelled as not cycle-, join-, or back-pressure-faithful.

Add optional validation modes:

- `compare_serial_hls`: run both stable modes and report first divergent side effect, final memory/output difference, and whether the difference is expected under mode semantics.
- `randomized_interleaving_seeded`: preserve D-14 legality but randomly choose among enabled child steps using a required seed and replay trace. This is a fuzzing mode, not a reference semantics.
- `bounded_dataflow` or `cycle_backpressure`: reserve as experimental until D-22 defines finite FIFO/LIFO/stream semantics; after D-22, promote it to the liveness/back-pressure validation mode.

Do not make `compat_scalagen_serial` the unlabelled default for the Rust/HLS simulator. It preserves existing value behavior, but it hides child races, producer/consumer phasing bugs, queue overflow, deadlocks, missing join barriers, and memory-port conflicts. It also encourages users to read serial order as a hardware guarantee.

Do not make bounded-cycle/back-pressure the v1 default before D-22. A simulator that claims back-pressure while still using elastic queues is worse than honest serial compatibility: producers never stall, overflow disappears, empty reads become invalid values rather than stalls, and deadlocks can look like normal completion.

For every mode, persist provenance:

- `simulator_mode`
- queue policy: `elastic | bounded | bounded_experimental`
- scheduler policy and tie-break rule
- seed and replay trace for randomized modes
- max-cycle bound and timeout policy
- latency source/provenance when cycles are reported
- first divergent side effect in compare mode

Diagnostics for `hls_semantic` and future bounded modes should include per-step enabled children, chosen child, blocked reason, masks, done state, child acknowledgements, queue occupancy/high-water marks, memory conflicts, and parent join status.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:

1. Implement `compat_scalagen_serial` with source/IR-order `ParallelPipe` execution and ScalaGen-compatible elastic queue behavior.
2. Implement `hls_semantic` with deterministic peer child issue, skipped-mask handling, all-child join, and stable tie-breaks.
3. Emit mode provenance in all simulator outputs, test artifacts, and any DSE/latency sidecars.
4. Add warnings when `hls_semantic` runs with elastic queues: "ForkJoin peer/barrier semantics enabled; bounded queue back-pressure disabled pending D-22."
5. Implement `compare_serial_hls` with final-state diff and first-side-effect divergence reporting.
6. Add seeded randomized interleaving as an opt-in fuzzing/test mode with replay traces.
7. Defer `bounded_dataflow` promotion until D-22; keep any early bounded simulator labelled experimental.
8. Add tests for serial order compatibility, peer/barrier completion, masked children, deterministic tie-breaks, divergent serial-vs-HLS side effects, and explicit D-22-blocked back-pressure cases.
