# D-18 — FloatPoint Clamp Compatibility Policy

## Question

Choose whether to reproduce `FloatPoint.clamp` heuristics bit-for-bit.

Source: `spatial/emul/src/emul/FloatPoint.scala:335-339`; [[20 - Numeric Reference Semantics]]

Decision criteria: User decision: Rust float packing reproduces the `x > 1.9` clamp heuristic bit-for-bit or adopts a cleaner custom-float algorithm with accepted divergence.

## Research findings

- The `FloatPoint.clamp` surface is broad. It is not just a text or bit-packing helper: `FloatPoint.clamped` calls it for every finite value, ordinary float arithmetic reclamps finite results, and `Value.bits` uses it for `DataAsBits`/`BitsAsData` compatibility. The magic `x > 1.9` branch appears before normal/subnormal range dispatch and can rescue near-underflow values into a minimum-subnormal result. See [[01-source-algorithm-call-surface]].

- The specs already place `FloatPoint.clamp` in the Scalagen reference path. [[20 - Numeric Reference Semantics]] treats the routine as the formal packing function, and [[50 - Data Types]] warns that custom floating formats cannot be delegated to native `f32`/`f64` if bit parity is required. Q-120 records the magic constant as an old Scalagen issue; Q-145 reopens it as an architectural Rust/HLS decision. See [[02-spec-open-question-context]].

- Existing tests do not protect the dangerous boundary directly. The float tests exercise ordinary `Float`, `Half`, transfer, cast, and math behavior, but they do not call `FloatPoint.clamp`, `.bits`, `fromBits`, or `isSubnormal` directly. They also lack deterministic fixtures around the `x > 1.9` band, min subnormal, max normal, signed zero, or NaN/infinity encodings. See [[03-tests-apps-usage]].

- Format edge cases show why native-float intuition is insufficient. `Flt[M,E]` lowers to `FltFormat(mbits - 1, ebits)`, packs mantissa bits first, exponent bits next, sign last, canonicalizes NaNs, preserves signed zeros, and handles normals/subnormals with one discarded rounding bit rather than a fully specified IEEE rounding mode. See [[04-format-edge-cases]].

- Backend behavior is already split. Scalagen uses `emul.FloatPoint` and therefore the clamp path. Cppgen maps custom floats to native `float` and plain casts, and Chisel/Fringe keep custom widths but delegate runtime arithmetic/conversions to `Math.*`, BigIP, and HardFloat-style paths. That divergence is evidence for a backend-capability story, not permission to weaken the Rust reference. See [[05-backend-codegen-variation]].

- Cleaner algorithms are technically attractive. Exact normalization with integer/decimal math, explicit guard/round/sticky policy, IEEE-like/MPFR or HLS-native conversions would be easier to explain and test as a new semantic center. But any such algorithm is a semantic migration unless it is proven bit-identical against Scalagen. See [[06-cleaner-algorithm-candidates]].

- HLS implications argue for a capability split. Rust reference simulation can clone the Scala path with arbitrary-precision support. HLS hardware may prepack constants, implement a custom packer, reject exact runtime conversions, or use native/vendor float behavior under a declared fallback. Native HLS behavior should not be labelled Scalagen parity. See [[07-hls-toolchain-implications]].

- The main risk is silent bit divergence: ordinary numeric tests may pass while packed mantissa/exponent/sign bits differ. Dual-mode support is useful, but only if every run, golden, mismatch report, and HLS artifact records which float-packing policy was used. See [[08-risk-failure-modes]].

- The recommendation matrix favors a dual-mode structure with legacy clamp as the reference/default: `scalagen_legacy_clamp_v1` for parity and a future `clean_custom_float_vNext` or HLS-native mode only behind explicit provenance. See [[10-recommendation-matrix]].

## Recommendation

Adopt **Dual-Mode Float Packing With Legacy Clamp As Reference**.

The canonical Rust reference mode should be:

`float_pack_policy = scalagen_legacy_clamp_v1`

In this mode, Rust reproduces the current `FloatPoint.clamp` path bit-for-bit enough for Scalagen parity, including:

1. `log2BigDecimal`-style exponent estimation.
2. `x > 1.9` subnormal rescue.
3. `x >= 2` exponent repair.
4. The cutoff repair for small mantissas.
5. Overflow to signed infinity.
6. Normal and subnormal one-discarded-bit rounding.
7. Underflow to signed zero.
8. Canonical NaN, infinity, zero, mantissa, exponent, and sign bit layouts.
9. `convertBackToValue` round-trip canonicalization after packing.

This is compatibility-first, not an endorsement of the magic constant as good numeric design. The point is to make Scalagen parity explicit and testable instead of letting a Rust rewrite accidentally clean up a behavior that the vault currently treats as reference semantics.

Add a second, non-default policy for future cleanup:

`float_pack_policy = clean_custom_float_vNext`

That mode may use exact normalization, a principled rounding rule, MPFR-style or IEEE-like reference behavior, or a better custom-float algorithm. It must be documented as a semantic migration and must not be used to generate Scalagen-parity goldens unless the user confirms the divergence.

For HLS, use backend capability labels. Exact parity may be implemented by prepacking constants, by a local custom packer, or by rejecting runtime conversions that cannot match `scalagen_legacy_clamp_v1`. Native/vendor float behavior is allowed only as a named fallback such as `hls_native_float`, with provenance showing that it is not the Rust reference.

### Rejected alternatives

Reject **cleaner custom float as the immediate default**. It is likely the right long-term numeric cleanup, but adopting it silently would turn D-18 into an unannounced semantic migration.

Reject **native `f32`/`f64` or vendor HLS float as reference parity**. Those may be useful backend implementations, but the source and specs show that `FloatPoint.clamp` defines the current reference behavior for custom formats and packed bits.

Reject **single-mode legacy without provenance**. Copying the magic constant is acceptable only if the policy name makes it visible. Future tests, mismatch reports, and generated artifacts need to say which float-packing oracle they used.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Define `float_pack_policy` with at least `scalagen_legacy_clamp_v1`, `clean_custom_float_vNext`, and `hls_native_float`/fallback labels.
2. Implement `scalagen_legacy_clamp_v1` in Rust with arbitrary-precision decimal/integer support and a clear audit trail back to `FloatPoint.clamp`.
3. Add exact fixtures for the `x > 1.9` band, min subnormal, underflow to signed zero, max normal to infinity, NaN, +/- infinity, negative zero, and ordinary powers of two.
4. Add staged tests for `to[Half]`, `to[Float]`, nonstandard `FltPt[M,E]`, `.bits`, `FloatPoint.fromBits`, `DataAsBits`, `BitsAsData`, and float/fixed casts near boundaries.
5. Record `float_pack_policy`, algorithm version, backend capability, and fallback in Rust simulator runs, golden files, mismatch reports, and HLS artifacts.
6. For HLS, either implement parity packing for runtime conversions, prepack constants with Rust, or reject/label unsupported exact conversions.
7. Keep cleaner algorithms behind explicit non-default modes until the user confirms a semantic migration.
8. Cross-reference D-19, because fused versus unfused float FMA decides how often finite results are reclamped.
