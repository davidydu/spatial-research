# D-20 — DelayLine Simulation Semantics

## Question

Choose value-only or cycle-aware `DelayLine` semantics.

Source: `spatial/src/spatial/codegen/scalagen/ScalaGenDelays.scala:7-15`; `src/spatial/transform/RetimingTransformer.scala:205-220`; [[C0 - Retiming]]

Decision criteria: User decision: Rust simulation is value-only and elides `DelayLine`, or is cycle-aware and models retiming registers.

## Research findings

- The source surface is deliberately split. `retime(delay, payload)` stages a `DelayLine` and marks it `userInjectedDelay`, while compiler retiming also inserts `DelayLine` through `RetimingTransformer`. Scalagen erases both as value aliases; ChiselGen lowers them to real retimed wires. See [[01-source-ir-scalagen-surface]].

- Retiming positions are scheduling facts, not ordinary value facts. `RetimingAnalyzer` computes `fullDelay` and cycle membership; `RetimingTransformer` materializes `ValueDelay`s and substitutes delayed inputs for the consumers that need them. User-injected retimes are source-visible timing directives, while automatic delay lines are schedule-alignment state. See [[02-retiming-scheduling-role]].

- Tests currently check IR shape and final values more than cycle traces. `SimpleRetimePipe` counts delay-line nodes; explicit `retime(10, data)` appears in priority-dequeue tests; streaming and line-buffer apps have realistic cycle pressure but mostly final-value or checksum assertions. D-20 needs dedicated trace fixtures. See [[03-tests-apps-usage]].

- The RTL path is truly cycle-aware. ChiselGen emits `DL(...)`, tracks `maxretime`, and Fringe implements flow-gated shift registers that advance only when the flow predicate permits. Scalagen, Cppgen, Roguegen, and PIR do not preserve delay time. See [[04-chisel-hls-backend-behavior]].

- Simulator options favor dual mode. Value-only elision is the right compatibility oracle; cycle-aware shift-register simulation is useful only as an explicit timing/debug mode with a real tick and flow model. HLS-only timing is needed as final authority, but it is too late for pre-HLS debugging. See [[05-simulator-semantics-options]].

- The key risk of value-only mode is blindness to retiming bugs; the key risk of cycle-aware mode is false precision. A partial cycle model that delays payloads but not enables, masks, queues, stream readiness, or controller progress can invent or hide deadlocks. See [[06-risk-failure-modes]].

- Migration cost favors value-only as the default and cycle mode as staged diagnostic work. The Rust baseline is cheap; cycle mode needs traces, flow predicates, reset/init behavior, report fields, and D-15/D-21/D-22 coordination. See [[08-implementation-cost-migration]].

- D-20 overlaps D-15, D-21, and D-22. A cycle-aware delay line only makes sense under a simulator mode that defines parent/child ticks, accepted II assumptions, and queue/backpressure behavior. See [[09-overlap-d15-d21-d22]].

- The recommendation matrix selects dual mode: `compat_value_elided` as the Rust reference/default, `cycle_shift_register_debug` as opt-in, and HLS reports/co-simulation as final timing authority. See [[10-recommendation-matrix]].

## Recommendation

Adopt **Dual-Mode DelayLine Semantics With Value-Elided Reference Default**.

The canonical Rust functional reference should be:

`delayline_policy = compat_value_elided`

In this mode, `DelayLine(size, data)` evaluates to `data` for value comparison, matching Scalagen's behavior. This is the default for legacy functional goldens, ordinary Rust simulation, and any comparison that claims Scalagen parity. It should still record the presence of delay lines as provenance, especially `origin = user_retime` versus `origin = compiler_retime`, but it should not claim timing correctness.

Add an explicit opt-in diagnostic mode:

`delayline_policy = cycle_shift_register_debug`

This mode models delay lines as temporal state only when the simulator also declares:

1. A tick policy and controller-step model.
2. Flow/backpressure/forward-pressure gating.
3. Reset and initialization behavior.
4. Queue/stream elasticity or bounded backpressure policy.
5. The assumed II source: compiler II, user-requested II, estimated HLS II, or parsed HLS accepted II.
6. Whether the line came from user `retime` or automatic retiming.

Cycle mode should be treated as a diagnostic and HLS-adjacent validation aid, not as the default semantic oracle. It can report "value matched, timing diverged" or "incomparable cycle policy" rather than turning every timing disagreement into a functional failure.

For HLS, automatic compiler `DelayLine`s should usually lower as schedule obligations rather than user-visible memory. User-injected `retime` may need explicit static shift-register lowering or report-backed proof that the HLS schedule preserves the requested latency. D-21 should own accepted-II and latency reconciliation.

### Rejected alternatives

Reject **cycle-aware as the default Rust simulator**. It would break Scalagen parity and silently decide D-15/D-21/D-22 subquestions that are not settled by D-20 alone.

Reject **value-only as the only mode**. It is too blind for retiming bugs, explicit `retime`, and Chisel parity debugging.

Reject **HLS-report-only for all timing questions**. HLS reports are final timing authority, but developers still need pre-HLS diagnostics for source-visible retiming directives.

Reject **unlabelled delay behavior**. Every simulator run, golden, mismatch, and HLS artifact should declare whether delay lines were value-elided, cycle-modeled, explicitly lowered, or delegated to the tool schedule.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Define `delayline_policy` values: `compat_value_elided`, `cycle_shift_register_debug`, `hls_schedule_authority`, and `rejected_unsupported`.
2. Record delay-line provenance: origin, size, trace/source producer, consumers, `fullDelay`, `userInjectedDelay`, creation pass, and comparison mode.
3. Implement Rust value-mode semantics by aliasing `DelayLine` to its payload while retaining provenance.
4. Add opt-in cycle-mode support only after defining tick, flow, reset/init, and queue/backpressure policies.
5. Add trace fixtures for explicit `retime` delays 0/1/3/10, automatic retiming from `SimpleRetimePipe`, `retimeGate` FIFO ordering, streaming starvation, and line-buffer window boundaries.
6. Add mismatch output that distinguishes value divergence, timing divergence, and incomparable delay-line policy.
7. For HLS lowering, elide automatic delay lines into scheduling by default; explicitly lower or verify user retimes.
8. Record requested II, compiler II, accepted HLS II, latency source, and delay-line reconciliation status in HLS artifacts.
9. Cross-reference D-15, D-21, and D-22 wherever cycle-mode `DelayLine` behavior is documented.
