# D-22 Angle 2: Chisel/RTL Bounded Back-Pressure Behavior

## ScalaGen Baseline

ScalaGen is capacity-aware only as a query surface. `FIFO` lowers to `mutable.Queue`, `FIFONew` creates an unbounded queue object, `FIFOIsFull` compares `size >= stagedSize`, and enqueue just calls `enqueue` when enabled; it does not test full or stall the producer (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenFIFO.scala:12-20`, `/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenFIFO.scala:41-44`). `LIFO` is the same pattern with `mutable.Stack`: `isFull` reads `size >= stagedSize`, while `LIFOBankedPush` always calls `push` under its enable (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenLIFO.scala:12-20`, `/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenLIFO.scala:39-44`). Streams also lower to mutable queues, file-backed stream inputs are initialized before execution, stream reads return invalid on empty, and stream writes enqueue unconditionally when enabled (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenStream.scala:12-15`, `/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenStream.scala:60-97`). The spec names this surprise directly: silent elastic enqueue can pass Scala simulation while overflowing synthesized hardware (`10 - Spec/50 - Code Generation/20 - Scalagen/40 - FIFO LIFO Stream Simulation.md:140-149`).

## ChiselGen FIFO and LIFO

ChiselGen emits real memory templates for FIFO/LIFO rather than Scala collections. `FIFONew` and `LIFONew` call `emitMem`, while status nodes become direct hardware fields: `empty`, `full`, `almostEmpty`, `almostFull`, and `numel` (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenMem.scala:310-343`). Both FIFO and LIFO use `FIFOInterface`, so LIFO has the same status-field shape even though its template implements stack addressing (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenCommon.scala:111-122`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/memory/MemPrimitives.scala:447-541`). Reads and writes route through `connectRPort` and `connectWPort`; reads pass the current `backpressure`, writes are enabled only through `implicitEnableWrite`, whose flow term is `~break && backpressure` and optionally `forwardpressure` for halt-if-starved controllers (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenMem.scala:45-50`, `/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenMem.scala:89-103`). A nuance: activity maps and pressure collection visibly cover FIFO/FIFOReg, not LIFO, so LIFO is bounded state but not obviously included in controller pressure filters (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenMem.scala:202-205`, `/Users/david/Documents/David_code/spatial/src/spatial/metadata/control/package.scala:1360-1379`).

## Streams and Controller Pressure

Chisel streams use ready/valid, not elastic enqueue. `StreamOutBankedWrite` drives `stream.valid` from delayed datapath issue, lane enable, and `backpressure`; `StreamInBankedRead` drives `stream.ready` from read enables and datapath enable, with AXI stream reads further filtering by `TID` and `TDEST` (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenStream.scala:34-43`, `/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenStream.scala:139-167`). The controller-level predicate is centralized: forward pressure requires stream-in valid, FIFO non-empty or not-active-read, merge-buffer non-empty, and priority-dequeue availability; back pressure requires stream-out ready, FIFO not-full or not-active-write, and merge-buffer capacity (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenCommon.scala:247-279`). `instantiateKernel` wires those expressions into each kernel's `backpressure` and `forwardpressure`, ORed with the done latch (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenController.scala:308-309`). Fringe then consumes them as control flow: child enable is `baseEn & forwardpressure`, counter chains increment with `forwardpressure`, and state machines receive `backpressure` (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/SpatialBlocks.scala:132-158`). Fork-join and streaming controllers also gate activity and iteration progress with `io.backpressure` (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/Controllers.scala:141-169`).

## Fringe Queue Templates

The queue templates are bounded by construction. `MemParams.volume` is derived from dimensions, banks, blocks, and neighborhoods, and FIFO instantiates head/tail counters over `p.volume`, a numel counter with `p.volume`, and physical `Mem1D` banks totaling that volume (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/memory/MemParams.scala:7-38`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/memory/MemPrimitives.scala:380-399`). FIFO exports `empty`, `full`, `almostEmpty`, `almostFull`, and `numel` from the element counter; LIFO exports the same from its inc/decrement counter (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/memory/MemPrimitives.scala:436-442`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/memory/MemPrimitives.scala:535-541`). The interface makes those fields first-class, plus per-access active loopbacks (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/memory/MemInterfaceType.scala:137-148`). Non-memory streams are Chisel `Decoupled`: `StreamOut` is `Decoupled(StreamIO)`, `StreamIn` is flipped `Decoupled(StreamIO)` (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/FringeBundles.scala:193-206`). DRAM arbiter FIFOs show the canonical ready/valid contract: write fires on `in.valid & in.ready`, read fires on `out.valid & out.ready`, `out.valid := !empty`, and `in.ready := !full` (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/dramarbiter/FIFO.scala:7-31`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/dramarbiter/FIFO.scala:83-89`).

## Decision Implications

D-22 should not call ScalaGen "bounded with advisory overflow"; it is elastic simulation with status queries. RTL is bounded back-pressure semantics for FIFO/stream paths: full and empty are physical state, ready/valid controls external streams, and controller progress is gated by pressure. LIFO needs an explicit policy because the template is bounded but the pressure filters do not name `isLIFO`. For [[D-15]], any interleaved `ParallelPipe` simulator that claims hardware realism must share this bounded queue model; otherwise producers never stall and fork/join progress is fiction. For [[D-20]], delay-line advancement must use the same pressure predicate, because Chisel `DL` advances under `backpressure` and sometimes `forwardpressure` (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenCommon.scala:288-293`). For [[D-21]], those predicates are also II-sensitive through `.D(II-1)` on FIFO pressure (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenCommon.scala:249-277`). The clean policy is two labelled modes: `compat_elastic_scalagen` for historical value goldens, and `bounded_backpressure` for HLS/RTL validation with overflow, underflow, stall, LIFO inclusion, and deadlock diagnostics.
