# D-10 â€” Rust Reference Precision For Transcendental Functions

## Question

Choose the Rust reference precision for transcendental functions.

Source: `spatial/emul/src/emul/Number.scala:97-156`.

Decision criteria: User decision: Rust numeric runtime matches Scalagen f64 transcendentals, uses per-format MPFR/exact math, or matches synthesized hardware units.

## Research findings

- The current reference path is explicitly f64-based. `Number.scala` sends fixed and floating transcendental helpers through `x.toDouble`, JVM `Math.*`, then reconstructs `FixedPoint` or `FloatPoint` in the original format and copies validity. This includes fixed `sqrt`, `exp`, `ln`, `log2`, `pow`, trig, hyperbolic, inverse trig, and composed `sigmoid`; the float surface is parallel but lacks `log2`. See [[01-current-number-implementation]].

- The staged API and Scalagen lowering reinforce that `Number.*` is the simulator oracle. Public fixed and float math expose `pow`, `exp`, `ln`, `sqrt`, trig, hyperbolic, inverse trig, and `sigmoid`; Scalagen lowers them to `Number.*` or to expressions using `Number.sqrt`/`Number.exp`. There is no public staged float `log2`, and fixed reciprocal-square-root differs from float reciprocal-square-root in Scalagen emission shape. See [[02-api-scalagen-surface]].

- The specs currently make Scalagen plus `emul` normative, but Q-130 explicitly asks whether Rust should keep that rule or upgrade it. Numeric Reference Semantics says `Number` transcendentals route through `Double` and that f64 is the current reference unless a higher-precision path is preferred. D-10 overlaps D-01 on operation availability, D-18 on final float packing/clamp, and D-19 on adjacent fused/unfused precision policy. See [[03-spec-open-question-survey]].

- Test coverage is broad enough for smoke but weak for bit-level precision. `FloatBasics`, `FloatBasics2`, `FixBasics`, and `HalfBasics` cover selected `sqrt`, `pow`, `recipSqrt`, or approximations with loose or self-referential tolerances. `exp`, `sigmoid`, and `ln` are often commented out or checked only through coarse application-level outcomes. Many operations have little asserted coverage. See [[04-tests-app-usage]].

- Format analysis shows why this is a real policy decision. Current f64 mode double-rounds half/f32-like values, collapses custom formats wider than binary64 at `toDouble`, and then re-enters `FixedPoint` or `FloatPoint` through the existing wrap/clamp rules. It also preserves current special-value and validity quirks, including fixed `pow` and float `pow` copying only base validity. See [[05-format-precision-effects]].

- Hardware/BigIP behavior is not a single precision oracle. `BigIP` makes many transcendental-adjacent operations optional; some defaults throw, `BigIPSim` has identity TODOs for fixed trig/hyperbolics, and Zynq/CXP map some float ops to Vivado IP with half/single/custom distinctions. Matching synthesized hardware therefore means matching a target, operation, IP family, and version, not choosing one global numeric reference. See [[06-hls-hardware-bigip-implications]].

- MPFR/per-format exact math is the clean mathematical upgrade, but it is not Scalagen parity. It would need correct rounding for every supported `FixFormat`/`FltFormat`, explicit final packing rules, slower runtime behavior, native dependency management, and new conformance tests. It also depends on D-18: exact real evaluation still needs a final `FloatPoint.clamp` policy. See [[07-mpfr-exact-option]].

- f64 compatibility must be treated as a named mode, not hand-waved as "Rust f64." To match Scalagen, Rust must reproduce BigInt/BigDecimal storage boundaries, `toDouble`, JVM `Math` behavior or a tested compatibility shim, reclamping, specials, validity quirks, fixed-only `log2`, and composed sigmoid/reciprocal-square-root shapes. Rust libm bit-identity with Java `Math` is unverified and should be tested or labelled approximate. See [[08-f64-compatibility-option]].

- Cost analysis favors starting with Scalagen compatibility and gating other modes by provenance. f64 compatibility is cheapest and preserves current tests. MPFR is an opt-in audit oracle. Hardware matching is target-specific and tied to D-01 capability tiers. Hybrid/provenance modes isolate these meanings instead of hiding them behind one word like "correct." See [[09-cost-migration-matrix]].

- The recommendation matrix names the preferred architecture: **Scalagen-Compatible Default with Declared Precision Modes**. Default Rust reference behavior should be `scalagen_f64_reclamp`, while `mpfr_per_format` and `hardware_matched` remain explicit, labelled modes with supported operation sets and mismatch tolerances. See [[10-recommendation-matrix]].

## Recommendation

**Adopt Scalagen-Compatible Default with Declared Precision Modes.**

The Rust numeric runtime should default to matching current Scalagen/emul behavior:

`trans_precision_mode = scalagen_f64_reclamp`

This means: convert the source fixed/float value to f64 in the same sense as current emul, call the compatibility implementation of the corresponding JVM `Math.*` operation, then reclamp or wrap back to the source `FixFormat` or `FltFormat`, preserving current special-value and validity behavior. This is the v1 reference mode for simulator parity and golden-test migration.

MPFR/per-format exact math and synthesized-hardware matching should be supported only as explicit non-default modes. They are valuable, but they answer different questions:

- MPFR answers "what is a better format-aware mathematical oracle?"
- Hardware matching answers "what did this target/tool/IP unit do?"
- Scalagen f64 answers "what does the current reference simulator mean?"

Those answers should not silently overwrite each other.

### Precision-mode contract

Record precision mode at run level and, where needed, per operation. Per-operation metadata overrides the run default.

```yaml
trans_precision_mode: scalagen_f64_reclamp | mpfr_per_format | hardware_matched | disabled
trans_backend: jvm-math-compatible | rust-libm | mpfr | hls-native | vivado-ip | hardfloat | custom
trans_algorithm: jvm_math_to_double_reclamp | rust_libm_to_double_reclamp | correctly_rounded_mpfr | vendor_ip_v7_1 | newton_raphson_n | composed_exp_recip
compatibility_label: scalagen-compatible | exact-format | hardware-correlated | approximate | unsupported
mismatch_tolerance:
  metric: exact_bits | ulp | absolute | relative | application
  value: 0
supported_operation_set:
  - sqrt
  - recip_sqrt
  - exp
  - ln
  - pow
  - sin
  - cos
  - tan
  - sinh
  - cosh
  - tanh
  - asin
  - acos
  - atan
  - sigmoid
notes: "Fixed log2 is an internal helper; sigmoid and some reciprocal-sqrt paths match Scalagen composition."
```

For `scalagen_f64_reclamp`, include fixed-only `log2` as an internal helper and preserve Scalagen expression shapes:

- fixed `recipSqrt` as `one / Number.sqrt(x)`,
- float `recipSqrt` through `Number.recipSqrt(x)`,
- fixed and float `sigmoid` as `one / (Number.exp(-x) + one)`,
- `pow` validity copied from the base operand as current emul does.

### Rejected alternatives

Reject **MPFR/per-format exact as the default v1 reference**. It is cleaner mathematically and should be available for audit, but it intentionally changes the oracle, adds dependency/build risk, slows simulation, and depends on D-18's packing policy.

Reject **synthesized-hardware matching as the default reference**. It is target- and tool-specific. BigIP support is uneven, some current paths are TODO or throwing, and HLS-native math bit-identity with old vendor IP is unverified. Hardware mode should be a correlation mode gated by D-01 capability, not the default Rust simulator meaning.

Reject **unlabelled Rust libm f64 as "Scalagen compatible"**. Rust f64 calls may agree with Java `Math` for many ordinary inputs, but bit identity across operations, platforms, NaNs, signed zero, and edge cases is unverified. Either implement/test a `jvm-math-compatible` shim or label the backend `rust_libm_approx_scalagen`.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Implement `scalagen_f64_reclamp` as the Rust default for fixed and float transcendentals.
2. Add a tested `jvm-math-compatible` boundary or explicitly label Rust libm results as approximate until proven bit-identical.
3. Preserve current `Number.scala` quirks: fixed-only `log2`, composed sigmoid, fixed vs float reciprocal-square-root emission shape, and `pow` base-validity behavior.
4. Route results through the same fixed wrap/truncation and float clamp semantics used by current emul.
5. Add precision-mode metadata fields to the simulator/config/manifest and per-op reports.
6. Create exact-bit regression tests for representative `scalagen_f64_reclamp` inputs, including NaN, infinities, signed zero, invalid values, f32-like, half-like, and wide custom formats.
7. Add opt-in `mpfr_per_format` mode only after defining final packing/rounding behavior and D-18 clamp policy.
8. Add opt-in `hardware_matched` mode only for operations and targets that D-01 capability policy says are supported.
9. Record HLS/native/vendor-IP backend, algorithm, target/tool version, and tolerance for hardware-correlated runs.
10. Cross-reference D-18 and D-19 so float packing and fused/unfused arithmetic do not silently diverge from the selected transcendental precision mode.
