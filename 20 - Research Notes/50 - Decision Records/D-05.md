# D-05 — Rust/HLS Replacement For Chisel Instantiation Globals

## Question

Design the Rust/HLS replacement for Chisel instantiation globals.

Source: `src/spatial/codegen/chiselgen/ChiselCodegen.scala:129-302`; `src/spatial/codegen/chiselgen/ChiselGenInterface.scala:121-152`; `src/spatial/codegen/chiselgen/ChiselGenController.scala:550-581`; `fringe/src/fringe/SpatialIP.scala:18-40`; `fringe/src/fringe/globals.scala:7-79`.

Decision criteria: User decision: define the Rust/HLS top-level manifest that replaces mutable Chisel/Fringe instantiation globals.

## Research findings

- The current "manifest" is real but implicit. `ChiselCodegen` opens `AccelWrapper.scala`, `Instantiator.scala`, `ArgInterface.scala`, and `AccelUnit.scala`; then generated code and `SpatialIP` cooperate through constructor arguments plus mutable `fringe.globals`. The global surface includes target identity, derived widths, flags, scalar counts, stream lists, AXI stream descriptors, allocator count, loopback metadata, and side-effecting build state such as `bigIP.tcl`. See [[01-source-global-surface]].

- Host ABI, accelerator IO, and Fringe routing are currently mixed together. Scalar ArgIns, DRAM pointer slots, ArgIOs, ArgOuts, instrumentation counters, and breakpoints are host ABI. Padded arg counts, stream descriptors, AXI stream descriptors, and allocator count are accelerator IO shape. Loopbacks and lane maps are routing implementation metadata. A replacement should keep these as separate named sections rather than a single undifferentiated config blob. See [[02-abi-stream-metadata]].

- Target configuration and app shape need different ownership. `globals.target` exposes target-derived widths, stream geometry, channel count, BigIP/shell binding, and IO policy, while generated wrapper code writes app-specific argument and stream facts into the same singleton. Controller code also mutates target latency/SRAM assumptions and global flags. See [[03-target-fringe-coupling]].

- The largest risk is lifecycle ordering. Today `SpatialIP` sets `globals.target` before calling the delayed accelerator generator, while `AccelWrapper` writes globals immediately before constructing `CustomAccelInterface`. Empty stream lists trigger non-neutral target-derived defaults, and string target checks are duplicated across codegen and `SpatialIP`. The Rust/HLS flow should make manifest validation precede top-signature construction and shell integration. See [[04-lifecycle-hazards]].

- A concrete schema is feasible. The proposed Rust API is a `TopManifest` containing `TargetSpec`, `HostAbi`, `InterfaceLayout`, `AllocatorInfo`, `Instrumentation`, `ControlFlags`, `AssumptionSet`, and `ArtifactPaths`. The typed Rust value should be authoritative; JSON and optional TOML should be emitted from it, not maintained as separate schemas. See [[05-manifest-schema]].

- Consumers need different views of the same data. HLS kernel generation needs compile-time-visible target/profile and IO-shape constants. Host wrappers need stable symbolic ABI names and offsets. Runtime/simulator code needs name-to-slot metadata and compatibility checks. DSE mostly needs target/model fields and report reconciliation, not Chisel-style globals. See [[06-consumer-matrix]].

- Existing spec entries already point away from Chisel globals. [[60 - Instantiation]] and [[10 - Fringe Architecture]] mark `SpatialIP`, `CommonMain`, and `globals` as Chisel-specific mechanics, while [[90 - Host-Accel Boundary]] makes the host register map and `ArgAPI` agreement the stable semantic boundary. Q-084 is already resolved by not treating the empty `ArgInterface.scala` as the ABI owner. See [[07-spec-open-question-survey]].

- Cost/risk analysis rejects three tempting shortcuts. A mutable singleton is cheapest but preserves the ordering hazard. Constants-only generation is HLS-friendly but loses provenance and schema validation. Serialized-only manifests are reviewable but push type checking into every consumer. The best center is a typed builder plus derived generated artifacts. See [[08-cost-risk-alternatives]].

- External HLS/SoC precedent supports declarative top contracts rather than hidden process state. Top-function signatures, interface pragmas, kernel metadata, and IP packaging all separate the machine-readable contract from implementation side effects, though the external details here are general precedent and marked unverified in the research. See [[09-precedent]].

- The final matrix favors "typed manifest plus generated artifacts", shaped so it can split into a target profile and an application manifest. That option scores best on determinism, host ABI safety, HLS friendliness, and migration path, despite medium-high initial implementation cost. See [[10-recommendation-matrix]].

## Recommendation

**Adopt a versioned typed `TopManifest` as the single source of truth, with generated artifacts derived from it.**

The Rust/HLS port should not recreate `fringe.globals`, even as a nicer Rust singleton. It should build one immutable manifest value before emitting any HLS top signature, host wrapper, simulator binding, sidecar file, or shell integration. All consumers receive either `&TopManifest`, a validated deserialized copy, or a generated artifact produced from the same manifest.

### Canonical structure

Define `TopManifest` with these required sections:

1. `schema_version`
2. `app_name`
3. `target_profile`
4. `application_top`
5. `host_abi`
6. `interface_layout`
7. `allocator_info`
8. `instrumentation`
9. `build_options`
10. `assumptions`
11. `artifact_paths`

`target_profile` replaces `globals.target` as a mutable handle. It owns canonical target id, target kind, device/board part when relevant, shell or fringe provider, BigIP-equivalent binding, address/data widths, stream word geometry, external and target widths, top-level IO width/vectorization, channel count, and target latency/SRAM defaults.

`application_top` owns per-app top shape: root name, logical scalar counts, padded hardware counts, generated wrapper/kernel names, reset/control conventions, and whether dummy/padded interfaces were introduced. This must explicitly distinguish logical zero resources from the current Chisel habit of padding argument vectors to at least one.

`host_abi` owns ordered symbolic register slots: ArgIns, host DRAM pointer args, ArgIOs, ArgOuts, instrumentation counters, and early-exit/breakpoint outputs. Each entry should carry at least name, kind, direction, stable index, value type, bit width when known, and provenance. This is the D-05 link to Q-152/D-23: the instantiation manifest and host ABI manifest must be generated from the same object.

`interface_layout` owns load/store/gather/scatter `StreamParInfo`, AXI stream specs, stream lane assignments, buffer lane assignments, and any target-derived default policy. Empty resources should be represented explicitly as `none`, `dummy_for_tool`, or `target_default`, never inferred by an accessor hidden behind an empty list.

`allocator_info` owns heap/accelerator DRAM allocator count and, when available, per-allocator descriptors. `instrumentation` owns controller count, counter output count, named counters, breakpoint count, and named exits. `build_options` owns retiming, perpetual IP, modular generation, verbose/debug flags, channel assignment, and codegen switches. `assumptions` owns fixed-point and memory latency values, mux latency, `cheapSRAMs`, `SramThreshold`, max latency, and any target/model assumptions used to emit HLS pragmas or schedule estimates.

### Derived artifacts

From the typed manifest, emit:

- HLS/Rust constants or headers for top-function generation.
- Rust simulator metadata for name-to-slot and stream mapping.
- Host ABI bindings replacing `ArgAPI.scala`/`ArgAPI.hpp` style duplication.
- A JSON sidecar for build, compatibility, and tooling interchange.
- Optional TOML or markdown-style review output for human inspection.
- A manifest hash, schema version, and ABI version embedded into host/runtime launch checks.

The JSON/TOML sidecars are not the source of truth; they are serialized views. The source of truth is the typed manifest builder plus validation pass.

### Rejected alternatives

Reject a mutable global context. It keeps the exact risk D-05 is meant to remove: construction order, stale singleton state, duplicated target strings, and hidden defaulting.

Reject constants-only generation. Constants are useful derived artifacts for HLS, but they cannot validate target/app/ABI consistency, explain default policies, or feed non-HLS consumers.

Reject serialized-only authority. A JSON-only or TOML-only schema is useful at the artifact boundary, but a stringly manifest would move validation into every consumer and make schema drift likely.

Reject runtime topology discovery as the primary handshake. A launch-time manifest/version check is good; discovering argument or stream topology dynamically is not appropriate for HLS top signatures, which must be fixed before synthesis.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Define Rust `TopManifest`, `TargetProfile`, `ApplicationTop`, `HostAbi`, `InterfaceLayout`, `AllocatorInfo`, `Instrumentation`, `BuildOptions`, `AssumptionSet`, and `ArtifactPaths` structs with schema versioning.
2. Add a manifest builder that gathers the current Chisel-global facts from the corresponding Rust/HLS IR/codegen phases before any top signature is emitted.
3. Add validation for canonical target id, target/profile consistency, logical versus padded counts, stream-default policies, AXI stream limits, allocator count, instrumentation count, and breakpoint count.
4. Generate host ABI bindings, HLS/Rust constants, simulator metadata, and JSON sidecar from the typed manifest.
5. Add a manifest hash/schema/ABI-version compatibility check to host launch and simulator launch paths.
6. Replace mutable global-style target/build overrides with explicit `BuildOptions` and `AssumptionSet` values.
7. Add tests that compare generated ABI offsets against the existing `ArgAPI` order: ArgIns, DRAM pointers, ArgIOs, ArgOuts, instrumentation, exits.
8. Add tests for empty stream/default policies, dummy padded arg vectors, target canonicalization, and repeated builds in one process.
9. Cross-reference this decision when processing D-23 and D-24, because host ABI and fixed-point host conversion should consume the same manifest rather than define a second contract.
