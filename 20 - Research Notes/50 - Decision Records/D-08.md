# D-08 — HLS DSE Latency Source

## Question

Choose the latency source for HLS DSE.

Source: `src/spatial/dse/LatencyAnalyzer.scala:29-47`; `src/spatial/dse/DSEAnalyzer.scala:90-147`.

Decision criteria: User decision: HLS DSE latency comes from Spatial runtime-model parity, HLS reports/simulation, or a new estimator.

## Research findings

- The current DSE latency path is a generated Scala runtime model, not HLS feedback. DSE enables runtime-model generation, compiles `model_dse.scala` through SBT assembly, invokes a `RuntimeModel-assembly` jar in batches of 1000 `tune` maps, parses `"Total Cycles for App"`, and writes the result as `Cycles`. That makes runtime-model parity the legacy baseline and HLS reports/simulation a real replacement, not a small parser swap. See [[01-current-latency-pipeline]].

- Existing DSE consumers impose a narrow contract: replacement latency must be batch-addressable by DSE design point, return one cycle value per point in order, preserve a numeric `Cycles` minimization column, and work with worker threads, CSV reload/sort, Sensitivity, and HyperMapper. Area validity is independent of cycles today, but area analysis also consumes latency metadata for delay-line sizing, so D-08 cannot replace only the top-level scalar without preserving lower-level latency interfaces. See [[02-dse-consumers-contract]].

- The runtime equations are useful but Spatial-specific. The controller-tree shape, tripcount math, and inner-pipeline equation `latency + (N - 1) * II` can inform an HLS estimator. However, the current model serializes Spatial `bodyLatency`, `lhs.II`, fixed controller overheads, fitted dense-transfer congestion, sparse-transfer placeholders, and several TODO-like assumptions. These are not backend-accepted HLS schedule facts. See [[03-runtime-model-equations]].

- HLS reports are the best post-tool authority, but not a hot-loop oracle. Schedule/synthesis reports can carry accepted II, per-loop latency, top latency, and tool status once the backend has run; this aligns directly with D-06's requested-versus-accepted II split. Simulation/co-simulation is better treated as validation and failure triage because it is workload-dependent and too expensive to require for every candidate. See [[04-hls-reports-simulation]].

- A new estimator should be HLS-aware and report-calibrated. It should consume loop identity, tripcounts, requested/accepted II, operator latency class, memory-port pressure, D-07 partition-plan features, and report provenance. Its output should include cycles plus source, confidence, report/calibration IDs, and fallback reason rather than only a number. See [[05-new-estimator-calibration]].

- Policy comparison rejects all single-source extremes. Spatial parity is fast but semantically stale; reports-only is authoritative but too slow for broad DSE; simulation-only is concrete but workload-bound; estimator-only risks becoming another uncalibrated oracle. The hybrid policy uses the right source at each stage. See [[06-policy-options]].

- The spec and open questions already point away from the old jar as the final HLS answer. Q-116 names reports, simulation, and estimation as alternatives, and the DSE spec says the queue/writer architecture can carry over while HLS cycle estimation should not depend on the generated Scala runtime jar. The timing spec and D-06 also require compiler-requested and backend-accepted II to remain separate. See [[07-spec-open-question-survey]].

- Cost/risk analysis favors a fast estimator plus selective tool runs. Current heuristic and experiment DSE can consider tens of thousands of points; running HLS reports or simulation for every point would collapse throughput. Keeping the old Scala model avoids cost but imports Spatial schedule constants into an HLS backend. A report-calibrated estimator localizes maintenance while preserving search speed. See [[08-cost-risk]].

- Precedent from DSL-to-HLS and HLS DSE flows favors hybrid calibration: analytic/surrogate models for search, report-guided feedback for selected points, cache reuse, and simulation for validation. External precedent claims are unverified in the note, but the pattern maps cleanly onto Spatial's existing CSV/HyperMapper architecture. See [[09-precedent]].

- The recommendation matrix names the best policy: **Hybrid Report-Calibrated Estimator**. It scores best across throughput, semantic fidelity, D-06/D-21 accepted-II reconciliation, D-07 partition-plan interaction, and provenance/debuggability. See [[10-recommendation-matrix]].

## Recommendation

**Adopt the Hybrid Report-Calibrated Estimator policy.**

HLS DSE latency should come from a fast HLS-aware estimator during broad search, calibrated and overridden by HLS reports when tool results exist, with simulation/co-simulation reserved for validation and mismatch triage. Keep Spatial runtime-model parity only as a labelled compatibility fallback and regression baseline. Do not make the old Scala runtime-model jar the final HLS latency authority.

### Source tiers

1. `spatial_parity_fallback`

   Use the existing runtime-model controller shape only as a pre-HLS fallback, legacy-comparison baseline, or diagnostic path. This source can preserve DSE throughput while the HLS estimator is immature, but every value must be labelled as Spatial parity rather than HLS-grounded.

2. `hls_estimate`

   Use a new HLS estimator for broad DSE. Its first version may reuse the controller-tree composition idea, but its inputs must be HLS-facing: loop/controller identity, tripcounts, requested pipeline pragmas, D-06 `hls_requested_ii`, known `hls_accepted_ii` when available, operator latency tables, memory-port assumptions, D-07 partition plans, D-04 storage/resource binding, and target/tool profile.

3. `hls_report`

   Treat parsed HLS schedule/synthesis reports as the post-tool authority for accepted II and latency when available. Report latency should override or calibrate estimates, not silently overwrite the provenance of earlier search values.

4. `simulation_validation`

   Use C simulation, RTL simulation, or co-simulation for sampled validation, final-candidate confidence, deadlock/protocol triage, and report/parser discrepancy investigation. Do not use simulation as the default DSE objective for every point.

### Required latency result contract

Preserve the existing numeric `Cycles` column for CSV, Sensitivity, and HyperMapper compatibility, but add explicit provenance beside it. Each evaluated design point should have:

1. `cycles`
2. `latency_source`
3. `latency_status`
4. `confidence`
5. `estimator_version`
6. `calibration_id`
7. `hls_report_source`
8. `hls_report_status`
9. `simulation_status`
10. `fallback_reason`
11. `spatial_compiler_ii`
12. `hls_requested_ii`
13. `hls_accepted_ii`
14. `ii_reconciliation_status`
15. `partition_plan_id`
16. `tool_accepted_partition`

The old one-number contract can remain as an adapter: `Cycles` is the ranking value selected by policy, while the provenance fields explain whether that value is a Spatial fallback, an HLS estimate, a report result, or a simulation-backed measurement.

### Rejected alternatives

Reject **Spatial runtime-model parity as authority**. It is the cheapest path and should remain useful for regression, but it encodes Spatial/Chisel-era schedule and transfer assumptions rather than backend-accepted HLS latency.

Reject **reports-only DSE**. Reports are authoritative after selected HLS runs, but requiring synthesis for every explored point breaks the throughput assumptions of the current worker/CSV/HyperMapper architecture.

Reject **simulation-only DSE**. Simulation can provide high-value validation for concrete workloads, but it is input-dependent, expensive, and does not replace per-loop accepted-II/report information.

Reject **new-estimator-only authority**. A clean estimator is the right search-loop destination, but without report calibration it risks becoming a second stale oracle. Its authority should increase with calibration confidence and fall back explicitly when evidence is missing.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Define a `LatencyResult` or equivalent schema with `cycles`, source, status, confidence, report, simulation, estimator, II, and partition-plan provenance.
2. Preserve `Cycles` as the numeric DSE/HyperMapper minimization column while adding provenance columns or sidecar records.
3. Implement an HLS estimator interface that can batch design points and return ordered results.
4. Seed the initial estimator from controller hierarchy, tripcounts, D-06 requested/accepted-II fields, operator latency tables, and D-07 partition-plan fields.
5. Add HLS report parsers for top latency, per-loop latency, accepted II, report source, missing-field states, and loop/controller mapping failures.
6. Add a cache keyed by design point, backend, tool version, target profile, partition plan, estimator version, and report/calibration ID.
7. Use simulation/co-simulation only for sampled validation, final-candidate confirmation, and mismatch triage; record workload identity and status.
8. Update DSE CSV readers, Sensitivity, HyperMapper, and summaries to tolerate the added provenance while preserving existing `Cycles` behavior.
9. Add reconciliation diagnostics for estimate-versus-report and report-versus-simulation mismatches.
10. Keep the old Scala runtime model available as `spatial_parity_fallback` for legacy comparison until the HLS estimator/report path is stable.
