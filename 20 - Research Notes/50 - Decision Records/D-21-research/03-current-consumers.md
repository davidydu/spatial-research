# D-21 Angle 3: Current Consumers of II Metadata

[[D-21]] should treat current Spatial as a selected-II flow, not an accepted-II flow. Control metadata defines four related fields: `bodyLatency` for analysis-owned body timing, `II` as the selected initiation interval, `userII` as the optional DSL request, and `compilerII` as the compiler-owned estimate (`/Users/david/Documents/David_code/spatial/src/spatial/metadata/control/ControlData.scala:144-177`). The package accessors preserve those names directly (`/Users/david/Documents/David_code/spatial/src/spatial/metadata/control/package.scala:826-839`). The surface `.II(n)` path writes only `userII`, via `CtrlOpt.ii` into `x.userII`; it does not represent a backend-accepted schedule (`/Users/david/Documents/David_code/spatial/src/spatial/lang/control/CtrlOpt.scala:8-21`; [[10 - Controllers]]).

The active final producer is `InitiationAnalyzer`. For outer controls, it computes `compilerII` from the max child `II` and then applies `userII` to selected `II` (`/Users/david/Documents/David_code/spatial/src/spatial/traversal/InitiationAnalyzer.scala:14-20`). For inner controls, it computes block latency and interval, includes blackbox `bboxII`, collects `iterDiff`, writes `bodyLatency`, derives `compilerII`, and finally chooses selected `II` from sequenced override, `userII`, or `min(compilerII, latency)` (`/Users/david/Documents/David_code/spatial/src/spatial/traversal/InitiationAnalyzer.scala:23-40`; [[70 - Timing Model]]). `PerformanceAnalyzer` can also write `lhs.II`, but it appears as an older/rerunnable estimator; the main pass chain runs final `initiationAnalyzer` immediately before runtime model generation and reports (`/Users/david/Documents/David_code/spatial/src/spatial/traversal/PerformanceAnalyzer.scala:24-27`, `/Users/david/Documents/David_code/spatial/src/spatial/Spatial.scala:232-237`; [[60 - Use and Access Analysis]]).

Codegen has the strongest hard consumers. `ChiselGenController.createSMObject` converts `bodyLatency.sum` and selected `lhs.II` into state-machine latency and an emitted `IICounter`, forcing II 1 for outer controls or disabled retiming (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenController.scala:333-358`). Stream pressure generation also delays FIFO, merge-buffer, and blackbox pressure by `sym.s.get.II - 1`; therefore selected `II` affects handshake timing, not only printed estimates (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenCommon.scala:249-277`). The blackbox wrapper computes the same `lat` and `ii`, but emits an outer-control `IICounter(1)` and leaves the calculated value in a comment (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenBlackbox.scala:255-261`). The Fringe template confirms the counter semantics: when `ii > 1`, `issue` and `done` are gated by a wrapping counter (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/counters/Counter.scala:121-150`).

Report-like consumers are mostly diagnostic. `TreeGen` prints controller latency, selected `II`, and bold `CompilerII` only when `compilerII != II`; this is the clearest existing mismatch surface (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/treegen/TreeGen.scala:116-131`). Direct `src/spatial/report` search found no `II`, `compilerII`, `bodyLatency`, or `userII` reads. `MemoryReporter` prints a control tree, but that helper includes level, loop, and schedule rather than II (`/Users/david/Documents/David_code/spatial/src/spatial/report/MemoryReporter.scala:99-100`, `/Users/david/Documents/David_code/spatial/src/spatial/metadata/control/package.scala:1331-1340`). `RetimeReporter` reports retiming cycles, latency-model data, and delay lines, not controller II metadata (`/Users/david/Documents/David_code/spatial/src/spatial/report/RetimeReporter.scala:35-63`).

Model and scheduler consumers also read selected `II`, not `compilerII`. `RuntimeModelGenerator` repeatedly emits `ControllerModel(L, II)` from `bodyLatency.sum` and `lhs.II`, again forcing outer controls to II 1 (`/Users/david/Documents/David_code/spatial/src/spatial/model/RuntimeModelGenerator.scala:261-284`, `/Users/david/Documents/David_code/spatial/src/spatial/model/RuntimeModelGenerator.scala:288-344`, `/Users/david/Documents/David_code/spatial/src/spatial/model/RuntimeModelGenerator.scala:373-411`). The generated model stores these as `L` and `II`; inner controls use `(N - 1) * II + L + overhead` (`/Users/david/Documents/David_code/spatial/models/src/models/RuntimeModel.scala:192-200`, `/Users/david/Documents/David_code/spatial/models/src/models/RuntimeModel.scala:334-336`). DSE compiles and runs that jar, parses `Total Cycles for App`, and records the result as the `Cycles` objective (`/Users/david/Documents/David_code/spatial/src/spatial/dse/DSEAnalyzer.scala:90-127`, `/Users/david/Documents/David_code/spatial/src/spatial/dse/LatencyAnalyzer.scala:35-45`, `/Users/david/Documents/David_code/spatial/src/spatial/dse/DSEThread.scala:114-123`, `/Users/david/Documents/David_code/spatial/src/spatial/dse/HyperMapperDSE.scala:70`). The Scala executor scheduler likewise throttles inner foreach, reduce, and mem-reduce issue using only `ctrl.II.toInt` modulo streams (`/Users/david/Documents/David_code/spatial/src/spatial/executor/scala/ControlExecutor.scala:307-330`, `/Users/david/Documents/David_code/spatial/src/spatial/executor/scala/ControlExecutor.scala:519-544`, `/Users/david/Documents/David_code/spatial/src/spatial/executor/scala/ControlExecutor.scala:795-819`).

Implication for [[D-21]]: current consumers conflate selected requested schedule with implementation truth. A Rust+HLS flow should keep `spatial_compiler_ii`, user/requested II, emitted HLS pragma II, parsed accepted II, and cycle source separate. TreeGen/model/DSE-style consumers can then prefer accepted II only when report provenance exists, matching [[D-06]] and [[D-08]] without erasing mismatch diagnostics.
