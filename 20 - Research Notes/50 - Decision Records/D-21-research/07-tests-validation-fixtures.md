# Tests, Report Fixtures, and Validation

## Existing Compiler-Side Tests

The present tests validate Spatial's selected `II`, not HLS acceptance. `FlatAffineAccess` scans nested `UnrolledForeach` controllers named `LOOP1D`, `LOOP2D`, and `LOOP3D`, collects `x.II`, and asserts every value is `1.0` (`/Users/david/Documents/David_code/spatial/test/spatial/tests/compiler/InitiationIntervals.scala:81-84`). The larger `IterationDiffs` test encodes expected interval behavior as comments beside named pipes, including `II = 1`, `II = body latency`, `II = body latency / 2`, and segmentation cases (`/Users/david/Documents/David_code/spatial/test/spatial/tests/compiler/InitiationIntervals.scala:147-237`). It checks functional results, but its exact-II assertions are mostly implicit. User syntax is also tested functionally: `ExplicitII` demonstrates `Pipe.II(1).Foreach` but only asserts the final reduction value (`/Users/david/Documents/David_code/spatial/test/spatial/tests/feature/control/ExplicitII.scala:5-25`). The Rust+HLS rewrite should turn these into explicit fixtures for `user_requested_ii`, `compiler_ii`, selected/requested HLS II, and post-report accepted II.

## Current Metadata and Display Surfaces

Spatial already has the compiler fields that D-21 must preserve. `InitiationInterval`, `UserII`, and `CompilerII` are distinct metadata records (`/Users/david/Documents/David_code/spatial/src/spatial/metadata/control/ControlData.scala:153-177`), and the accessors default `II` and `compilerII` to `1.0` while leaving `userII` optional (`/Users/david/Documents/David_code/spatial/src/spatial/metadata/control/package.scala:832-839`). `CtrlOpt.set` is the user-entry point: it assigns names, schedule, and `x.userII = ii.map(_.toDouble)` (`/Users/david/Documents/David_code/spatial/src/spatial/lang/control/CtrlOpt.scala:18-22`). `InitiationAnalyzer` selects outer `II` from max child II or user override and stores `compilerII = interval` (`/Users/david/Documents/David_code/spatial/src/spatial/traversal/InitiationAnalyzer.scala:14-20`); for inner controls it derives `compilerII`, then chooses selected `II` from sequenced latency, user override, or `min(compilerII, latency)` (`/Users/david/Documents/David_code/spatial/src/spatial/traversal/InitiationAnalyzer.scala:23-40`). TreeGen is the model UI precedent: it prints `Latency`, `II`, and bold `CompilerII` only when `compilerII != II` (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/treegen/TreeGen.scala:126-131`).

## Report Parsing and Fixture Gaps

The only local report-like II parser I found is DSE/resource scraping, not an HLS schedule-report parser. It walks generated `controller_tree.html`, creates nodes from HTML ids like `xNNN_appname` for `UnrolledForeach`, `UnitPipe`, `UnrolledReduce`, and `StateMachine`, then scrapes `>InitiationInterval<` into `current.ctrl_ii` (`/Users/david/Documents/David_code/spatial/resources/dse/scrape_node_area.py:418-484`). That field feeds model-training sessions for `ParallelPipe`, `StateMachine`, `UnitPipe`, `UnrolledForeach`, and `UnrolledReduce` (`/Users/david/Documents/David_code/spatial/resources/dse/fit_area.py:445-550`). Missing report behavior is lossy: `requirerpt = True` filters apps with no synth report (`/Users/david/Documents/David_code/spatial/resources/dse/scrape_node_area.py:14`, `/Users/david/Documents/David_code/spatial/resources/dse/scrape_node_area.py:418-420`), `scrapeFor` prints "No results" or "Faulty results" and returns false (`/Users/david/Documents/David_code/spatial/resources/dse/scrape_node_area.py:142-169`), and callers delete failed nodes (`/Users/david/Documents/David_code/spatial/resources/dse/scrape_node_area.py:501-506`). D-21 should reject this deletion pattern for HLS: missing report, parse failure, and loop-mapping failure must remain first-class statuses.

## Fixtures to Add for Rust+HLS

Add golden compiler fixtures that cover: no user II, explicit `Pipe.II(1)`, explicit II lower than Spatial `compilerII`, explicit II higher than compiler estimate, sequenced controls where selected `II` becomes latency, outer controls whose `II` composes child intervals, and FSMs where current code sets selected `II` but not `compilerII` in the special branches (`/Users/david/Documents/David_code/spatial/src/spatial/traversal/InitiationAnalyzer.scala:52-91`). Add report fixtures independent of vendor tools: matched requested/accepted II, accepted higher, accepted lower, tool rejected, report missing, unknown format, duplicate loop labels, and unmapped loop labels. General HLS behavior here is unverified, but [[C0 - Retiming]] already frames the intended HLS path as skipping Chisel `DelayLine` retiming and emitting `#pragma HLS pipeline II=<N>` from initiation metadata (`10 - Spec/40 - Compiler Passes/C0 - Retiming.md:307-311`).

Every fixture should include a mapping manifest: Spatial controller symbol, user name, source line/text, parent path, emitted HLS loop label, parsed report loop id, and reconciliation status. Existing identity surfaces are too fragile alone: TreeGen displays `$lhs` plus `lhs.ctx` (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/treegen/TreeGen.scala:123-131`), runtime models use `lhs.hashCode` with `Ctx` (`/Users/david/Documents/David_code/spatial/src/spatial/model/RuntimeModelGenerator.scala:261-284`), and DSE scraping keys on generated `xNNN` ids (`/Users/david/Documents/David_code/spatial/resources/dse/scrape_node_area.py:463-477`). The rewrite should test stable controller-to-HLS-loop identity before trusting any accepted-II mismatch diagnostic.
