# D-21 Angle 4: Overlap with D-08 and D-20

[[D-21]] should treat II reconciliation as the shared timing ledger between [[D-08]] latency sourcing and [[D-20]] `DelayLine` simulation. The compiler can compute a requested schedule, the HLS backend can accept or reject a different schedule, and simulators can run under either value-only or cycle-debug assumptions. Those are three different claims. Collapsing them into one `II` field would make DSE, delay-line debugging, and HLS report comparison look cleaner while hiding the exact mismatch that this decision is supposed to expose.

The compiler-side baseline is clear in the current timing stack. `InitiationAnalyzer` sets outer `compilerII` from the maximum child II, and inner `compilerII` from the local latency/interval model, `iterDiff`, `bboxII`, user II, and sequenced-schedule overrides (`/Users/david/Documents/David_code/spatial/src/spatial/traversal/InitiationAnalyzer.scala:14-41`). [[70 - Timing Model]] therefore classifies II as timing semantics, not only metadata, and explicitly says Rust/HLS should keep compiler-requested and backend-accepted values separate. [[C0 - Retiming]] adds the HLS boundary: `IterationDiffAnalyzer` and `InitiationAnalyzer` are reusable to emit pipeline requests, but the Chisel retiming trio and its `DelayLine(size, data)` insertion should not be ported as the HLS timing authority.

For [[D-08]], this means a latency source is not just a cycle number. The recommended hybrid report-calibrated estimator preserves `Cycles` for search compatibility, but adds `latency_source`, `hls_requested_ii`, `hls_accepted_ii`, `ii_reconciliation_status`, report status, confidence, and fallback reason. D-21 is the piece that decides how those II fields are reconciled before D-08 ranks or calibrates a point. Before HLS runs, an estimator may use `spatial_compiler_ii` or emitted `hls_requested_ii` to compute `latency + (N - 1) * II`; after reports exist, parsed accepted II and per-loop/top latency become post-tool authority. If accepted II is higher than requested, D-08 should not silently patch the old estimate; it should produce a report-backed latency with a mismatch status. If reports are missing or unmappable, D-08 can still rank with an estimate, but the point remains pre-report or fallback provenance, not accepted HLS truth.

For [[D-20]], II reconciliation determines what a cycle-aware `DelayLine` trace is allowed to mean. The default `delayline_policy = compat_value_elided` aliases `DelayLine` to its payload and can compare functional values without an II model. The opt-in `delayline_policy = cycle_shift_register_debug` is different: a shift register only advances relative to controller ticks, flow/backpressure, reset policy, and an assumed issue interval. D-20 already requires the assumed II source to be named: compiler II, user-requested II, estimated HLS II, or parsed HLS accepted II. D-21 should make that a hard dependency. A trace produced under `spatial_compiler_ii = 1` is useful diagnostic evidence, but if the HLS report accepts `II = 2`, that trace is not the accepted backend schedule. It should be labelled "cycle-debug under requested II" or "timing diverged after report", not treated as a failed functional golden.

Retiming is the reason this boundary matters. [[40 - Retiming]] records `FullDelay`, `Cycle`, `UserInjectedDelay`, `ForcedLatency`, and `ValueDelay`; [[C0 - Retiming]] explains that `RetimingTransformer` materializes delay lines from `fullDelay` and substitutes delayed producers for consumers. The IR node itself rewrites size zero to the payload, while `RetimeGate` has simple effects (`/Users/david/Documents/David_code/spatial/src/spatial/node/DelayLine.scala:8-14`). These facts are valid for Spatial's compiler/RTL timing model. HLS schedulers, however, insert registers from pragmas, dependence analysis, operator latencies, and resource binding. Therefore D-21 should reconcile II and reports at the controller/loop level, while D-20 records whether delay lines were value-elided, debug-shifted, explicitly lowered for user `retime`, or delegated to HLS schedule authority.

The reporting implication is a shared schema, not shared authority. D-21 should own fields such as `spatial_compiler_ii`, `user_requested_ii`, `hls_requested_ii`, `hls_accepted_ii`, `ii_reconciliation_status`, `hls_report_source`, and loop/controller mapping status, matching [[D-06]] and [[D-08]]. D-08 consumes those fields to select or calibrate `Cycles`. D-20 consumes them to qualify cycle-debug traces and delay-line provenance. HLS reports and co-simulation remain final timing evidence when available, but estimators and cycle simulators remain useful precisely because their assumptions are named instead of merged.
