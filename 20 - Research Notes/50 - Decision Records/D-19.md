# D-19 — Fused Versus Unfused FMA Precision

## Question

Choose fused or unfused FMA precision for Rust+HLS.

Source: `spatial/src/spatial/codegen/scalagen/ScalaGenFixPt.scala:150`; `spatial/src/spatial/codegen/scalagen/ScalaGenReg.scala:57-64`; [[60 - Counters and Primitives]]

Decision criteria: User decision: HLS port matches Scalagen unfused multiply-add precision or Chisel/HLS fused FMA precision.

## Research findings

- The source/IR surface is split by design. Users generally write `a*b+c`; the hardware rewrite pipeline turns eligible fixed and floating add-of-mul patterns into `FixFMA`/`FltFMA`, and accumulator specialization produces `RegAccumFMA`. But Scalagen emits all of these as ordinary multiply then add. See [[01-source-ir-scalagen-surface]].

- Existing tests exercise FMA-shaped programs but do not pin fused-versus-unfused bit patterns. `FMAInLoop` and `ReduceSpecialization` check optimization legality or exact integer/fixed values; float, GEMM, and ML tests often use tolerances, checksums, or host expressions that will not expose one-round versus two-round differences. See [[02-tests-apps-usage]].

- Chisel/Fringe provide real fused hardware precedent for floating FMA. ChiselGen lowers `FltFMA` to `Math.fma`; Fringe routes floating `Math.fma` through `BigIP.ffma`; Zynq/CXP-style targets instantiate an `FFma` block configured as FMA. Fixed `Math.fma` is weaker evidence, because the visible implementation still uses fixed multiply plus add templates and a final cast. See [[03-chisel-fringe-hardware-path]].

- Precision interacts directly with [[D-17]] and [[D-18]]. Current FMA fusion does not rewrite `UnbMul`, so it does not remove D-17 stochastic events today. But if an unbiased FMA is later introduced, the stochastic event would need a single final event identity. For floats, unfused FMA performs two D-18 reclamps, while fused FMA performs one final pack. See [[04-precision-overlap-d17-d18]].

- Backend behavior is inconsistent enough to require capability labels. Cppgen emits plain `a*b+c`; PIR preserves some FMA nodes but expands `RegAccumFMA`; the Scala executor has a likely bug where `FixFMA` resolves as `(m0*m1)*add`, and no `FltFMA` resolver. Resource reporting covers fixed FMA more than float FMA. See [[05-backend-variation]].

- HLS support favors fused as an implementation option, but not as a silent reference. Existing target data and vendor documentation show fused FMA can be a useful latency/area/II path. Exact unfused parity requires contraction controls and report/RTL evidence; exact fused custom behavior still needs a Rust oracle for Spatial custom floats/fixed formats. See [[06-hls-toolchain-support-cost]].

- FMA policy needs provenance beyond `CanFuseFMA`. A schema should record `fma_policy`, node kind, fusion origin, eligibility state, rounding/reclamp events, backend capability, HLS contraction controls, fallback, and test comparison mode. See [[07-fma-policy-provenance-schema]].

- Implementation cost favors a dual-mode transition. Unfused reference is cheapest for Rust and parity, but can be expensive for HLS if tools try to contract. Fused canonical is attractive for hardware correlation but expensive for custom formats and under-tested. Dual mode has the most metadata work but lowest migration risk. See [[09-implementation-cost-migration]].

- The recommendation matrix converges on **dual mode with unfused reference and fused opt-in**. It rejects a single-mode unfused mandate, rejects fused-HLS as canonical reference, and rejects tool-default contraction. See [[10-recommendation-matrix]].

## Recommendation

Adopt **Dual-Mode FMA Precision With Scalagen-Unfused As Reference**.

The canonical Rust reference mode should be:

`fma_policy = scalagen_unfused_two_round_v1`

In this mode, Rust treats `FixFMA`, `FltFMA`, and `RegAccumFMA` as the current Scalagen path does:

1. Compute the multiply in the ordinary operand/result format.
2. Apply the normal multiply rounding, clamp, wrap, valid-bit, and float-packing behavior.
3. Add the already-rounded product to the addend.
4. Apply the normal add behavior.

For fixed point, this preserves the current `FixedPoint.*` shift/clamp followed by `FixedPoint.+` clamp. For floats, this preserves two `FloatPoint.clamped` events, so the D-18 `float_pack_policy` is part of FMA provenance. For `RegAccumFMA`, the first iteration stores `m0*m1` and later iterations store `m0*m1 + reg.value`, matching `ScalaGenReg`.

Add explicit non-default modes:

`fma_policy = fused_single_round_v1`

This mode means exact product-plus-add followed by one final cast or pack. It is suitable for a future fused reference oracle, for Rust `mul_add` on native IEEE formats where that is the declared reference, and for custom fixed/float implementations that can prove one-round behavior.

`fma_policy = backend_native_fma`

This mode labels vendor/HLS/RTL fused behavior that has not been proven bit-identical to `fused_single_round_v1`. It is acceptable for HLS performance exploration or target-native builds, but it must not be compared as Scalagen parity.

This policy keeps the reference stable while acknowledging that the compiler already seeks FMA hardware and that HLS tools often reward fused multiply-add. The key is not to let the optimization silently change numeric meaning.

### Rejected alternatives

Reject **fused-HLS canonical as the default**. It would redefine current Scalagen arithmetic, reduce D-18 reclamp events, and rely on tests that do not currently distinguish the two answers.

Reject **unfused-only as the HLS mandate**. It is the right Rust reference, but forcing every HLS backend to avoid FMA would discard a hardware path the existing compiler intentionally exposes.

Reject **tool-default contraction**. Letting each HLS compiler decide whether `a*b+c` contracts makes results non-portable and hard to debug. If the backend cannot prove the requested policy, it should emit a diagnostic or declare a named fallback.

Reject **unlabelled Cppgen/PIR/executor behavior as evidence of semantics**. These surfaces disagree enough that D-19 must define the Rust/HLS contract explicitly.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Define `fma_policy` values: `scalagen_unfused_two_round_v1`, `fused_single_round_v1`, `backend_native_fma`, `backend_native_mul_add`, and `rejected_unsupported`.
2. Record FMA provenance: node kind, fusion origin, eligibility state, source add/mul symbols, reduction cycle id, first-iteration signal, backend capability, contraction controls, fallback, and report evidence.
3. Implement Rust `FixFMA`, `FltFMA`, and `RegAccumFMA` with `scalagen_unfused_two_round_v1` as the default.
4. Add optional Rust fused implementations for native IEEE and Spatial custom formats, but keep them non-default until explicitly selected.
5. Update HLS lowering to request split multiply/add for reference parity or declared fused/native paths for performance modes.
6. Require HLS reports/RTL checks or explicit fallback labels before claiming exact unfused or exact fused behavior.
7. Fix or quarantine the Scala executor `FixFMA` resolver bug and add a `FltFMA` resolver if executor parity matters.
8. Add deterministic fixtures with both fused and unfused expected bits for `Float`, `Half`, custom `FltPt`, fixed point, `RegAccumFMA`, cancellation/subnormal cases, and dot/GEMM elementwise checks.
9. Store selected `fma_policy`, `float_pack_policy`, rounding events, and backend capability in simulator runs, golden files, mismatch reports, and generated HLS artifacts.
10. Cross-reference D-17 and D-18 in numeric documentation so stochastic rounding events and float reclamp counts remain visible.
