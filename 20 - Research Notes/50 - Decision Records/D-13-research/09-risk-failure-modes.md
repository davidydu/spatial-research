# D-13 Research, Angle 9: Risk Analysis And Failure Modes

## Boundary Loss

The first failure mode is erasing a cycle/control boundary that `PipeInserter` made explicit. For each escaping inner-stage value, the pass emits a `Pipe`, visits the producer nodes inside it, writes the escape to a holder inside the pipe body, then reads the holder afterward and registers the original symbol to that read (`src/spatial/transform/PipeInserter.scala:189-201`). This is not just a local temporary: stage binding is conditional on parallel or stream control parents, and an inner stage may be bound with the immediately following outer stage when the outer stage consumes the escape (`PipeInserter.scala:135-164`). If HLS lowering converts the write/read pair to SSA too early, later code sees a pure value edge where Spatial had a staged boundary. That can also mask violations of the postcondition that mixed primitive/control content is no longer allowed in outer controls (`PipeInserter.scala:317-320`; `src/spatial/traversal/CompilerSanityChecks.scala:115-123`).

## Stream Pressure And Initialization Drift

`FIFOReg` removal is the hard back-pressure case. `resFrom` chooses `FIFOReg` for bit values under stream control, `Reg` for other bit values, and `Var` for non-bit values (`PipeInserter.scala:256-260`). The generated accesses are not interchangeable: `FIFOReg` uses `enq`/`deq`, while `Reg` uses write/value and `Var` uses assign/read (`PipeInserter.scala:277-314`). Chisel later treats `FIFOReg` as active FIFO-like storage by registering access lanes and wiring read/write access actives (`src/spatial/codegen/chiselgen/ChiselGenMem.scala:202-224`). It also includes `FIFORegNew` in forward and backward pressure equations using empty/full tests delayed by `II - 1` (`src/spatial/codegen/chiselgen/ChiselGenCommon.scala:249-271`). Replacing that with SSA can remove stall evidence or create a design that never sees the original full/empty condition.

Reset and initialization are another drift point. `PipeInserter` allocates register-family holders with `s.zero` (`PipeInserter.scala:271-275`), Scalagen initializes `Reg` through `initMem` (`src/spatial/codegen/scalagen/ScalaGenReg.scala:15-18`), and streamify infers initialization edges from `RegNew(init)` (`src/spatial/transform/streamify/HierarchicalToStream.scala:187-190`). An SSA rewrite must prove the read is always dominated by a write under the same enable. Otherwise disabled writes, switch cases, or reset cycles can turn a defined holder value into an uninitialized C++ value.

## Effect Ordering And Rewrite Surprises

Holder reads and writes are scheduling anchors. The memory-access spec records `RegWrite`, `FIFORegEnq`, and destructive dequeues as write effects; `DequeuerLike` mutates queue state, not just dataflow (`Spatial Research/10 - Spec/30 - IR/10 - Spatial Nodes/30 - Memory Accesses.md:96-100`, `:209-221`, `:260-265`). Modeling code then adds preceding writes to read dependencies through `cur.readMem`, preserving effect ordering even when no data symbol flows (`src/spatial/util/modeling.scala:257-269`). `AccessAnalyzer` also special-cases `RegRead` invariance by asking whether register writers are outside the loop (`src/spatial/traversal/AccessAnalyzer.scala:45-55`).

`Var` adds a separate rewrite hazard. Argon marks `VarNew` mutable and non-acceleratable, and `VarAssign` has `Effects.Writes(v)` (`argon/src/argon/node/Var.scala:7-14`, `:38-44`). `VarRead` tries to rewrite to the most recent impure `VarAssign` in the current scope, with a source comment assuming all statements in a block execute (`Var.scala:23-34`). If HLS lowering erases a pipe-created `Var` before branch, enable, or switch structure is represented, it can accidentally validate that assumption in places where the holder was the only explicit sequencing object.

## HLS Tool Boundary Risk

Documented AMD behavior confirms that HLS streams are stateful, sequential, and destructive: `hls::stream<>` behaves like FIFO storage in C, is read/written sequentially, top-level streams map to `ap_fifo` or AXIS, and internal streams default to FIFO depth 2 unless directed otherwise ([AMD UG1399 HLS Stream Library](https://docs.amd.com/r/en-US/ug1399-vitis-hls/HLS-Stream-Library)). AMD also documents that scalar dataflow channels are inferred as FIFOs, FIFO depth can cause stalls or deadlock, and co-simulation may be the first place a bad depth appears ([AMD UG1399 Configuring Dataflow Memory Channels](https://docs.amd.com/r/en-US/ug1399-vitis-hls/Configuring-Dataflow-Memory-Channels); [AMD UG1399 Compiler-Created FIFO Depth](https://docs.amd.com/r/2024.1-English/ug1399-vitis-hls/Specifying-Compiler-Created-FIFO-Depth)). General HLS claim, unverified for this project: if a holder becomes an ordinary scalar, the tool may schedule, merge, or move operations across the intended pipe boundary unless the generated C++ includes an equivalent dependence, protocol, stream, reset, or dataflow channel. FIFO interface misuse is a known documented risk when access order is not sequential ([AMD UG1399 FIFO Interfaces](https://docs.amd.com/r/2024.1-English/ug1399-vitis-hls/FIFO-Interfaces)).

## Over-Preservation Costs

Erasure also loses debug/provenance unless it records the eliminated holder: holder allocations and accesses receive distinct generated names such as `rd_`, `wr_`, `deq_`, and `enq_` (`src/spatial/codegen/naming/NamedCodegen.scala:41-56`). Preserving every holder literally also has costs. `RegNew` and `FIFORegNew` are emitted and counted as register resources (`src/spatial/codegen/resourcegen/ResourceCountReporter.scala:96-99`), so unnecessary holders can inflate resource reports and obscure which state was semantically required. Extra read/write effects may inhibit CSE, retiming, loop pipelining, and HLS scalar forwarding even when a single-assignment proof would be valid; this optimization-cost claim is general HLS knowledge and unverified here. Over-preservation can also overconstrain FIFO depths, add avoidable full/empty handshakes, or keep `Var` objects that cannot accelerate. The safe failure-mode framing is therefore not "always materialize every holder forever," but "do not erase holder identity until ordering, enable, reset/init, back-pressure, and provenance have an equivalent representation."
