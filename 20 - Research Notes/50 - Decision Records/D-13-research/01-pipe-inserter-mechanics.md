# PipeInserter Source Mechanics

## Choice for this angle

For angle 1, choose: **HLS lowering must preserve pipe holders as observable state by default**. `PipeInserter` does not leave an inner-stage result as an ordinary SSA edge. It emits a holder write inside a generated `Pipe`, emits a holder read afterward, then rewrites downstream users to consume that read (`src/spatial/transform/PipeInserter.scala:189-201`). The existing vault note says an HLS rewrite “can emit raw SSA and let the HLS tool inject registers as needed” (`[[50 - Pipe Insertion]]:181-188`), but the source mechanics are stricter: the transformed IR explicitly contains `Reg`, `FIFOReg`, or `Var` allocation plus read/write effects (`PipeInserter.scala:256-305`). Treat SSA replacement only as a later proven optimization, not the lowering contract (unverified).

## Escaping values and binding

The escaping computation is duplicated in binding and wrapping. For an inner stage, `calculated = stg.nodes.toSet`; a node escapes when it is non-void and either equals `block.result` or has consumers outside the calculated set (`PipeInserter.scala:141-145`, `PipeInserter.scala:174-182`). This is stage-local, not syntactic “used by next statement” logic: any consumer outside the inner-stage node set makes the value escape. Binding is conditional and only enabled when the parent is parallel or stream control (`PipeInserter.scala:135-137`). It checks whether escaping consumers intersect the immediately following stage, using `allStms` to include nested statements (`PipeInserter.scala:108-110`, `PipeInserter.scala:152`). If the next stage is outer and consumes an escape, the inner and following outer stage are placed in one bound bucket (`PipeInserter.scala:153-157`); otherwise, the inner stage becomes its own bucket (`PipeInserter.scala:157`). Outer stages are emitted separately unless already captured by a previous binding (`PipeInserter.scala:158-163`), while non-parallel/non-stream parents use one bucket per stage (`PipeInserter.scala:165-167`).

## Inner-stage wrapping

`wrapInner` handles each bucket. Outer stages are not wrapped by this helper; they are visited directly (`PipeInserter.scala:205-208`). Inner stages are lowered into a new `Pipe { isolateSubst() { ... } }`, where the original stage nodes are visited and each escaping value is written to its holder before leaving the pipe body (`PipeInserter.scala:189-193`). The result holder is special: if the escaping symbol is `block.result` and an external `res` exists, that existing holder is reused; otherwise the pass allocates a fresh holder through `resFrom(s, parent)` (`PipeInserter.scala:184-187`). After the pipe body, `wrapInner` immediately reads each holder and registers `s -> rd`, so later transformed consumers no longer see the original inner-stage symbol (`PipeInserter.scala:195-201`). Bucket emission preserves the binding shape: a multi-stage bucket is wrapped in an outer `Pipe` before `wrapInner` runs, a single inner bucket just calls `wrapInner`, and a single outer bucket visits nodes directly (`PipeInserter.scala:232-240`).

## Holder selection and operations

`resFrom` chooses by value type and parent schedule. Bit-valued escapes under a stream controller allocate a `FIFOReg`; other bit-valued escapes allocate a `Reg`; non-bit escapes allocate a `Var` (`PipeInserter.scala:256-260`). `memFrom` makes that concrete: `FIFOReg.alloc(s.zero)` for stream escapes, `Reg.alloc(s.zero)` otherwise, with pass-adjusted source context and the bit type recovered from the symbol (`PipeInserter.scala:271-276`). Reads and writes dispatch on the actual holder kind: `FIFOReg` uses `deq()` / `enq(data.unbox)`, while `Reg` uses `.value` / `.write(data.unbox)` (`PipeInserter.scala:277-293`). `Var` holders are allocated with `Var.alloc(None)`, read with `Var.read`, and written with `Var.assign(x, data.unbox)` (`PipeInserter.scala:298-305`, `PipeInserter.scala:306-314`). This is explicit state/effect construction, not merely naming a value.

## HLS consequence

Lowering should preserve the holder boundary because the pass uses it to encode communication from an inner primitive stage to later outer/control context. `FIFOReg` is the clearest hard case: replacing `enq`/`deq` with SSA would erase stream-order and buffering semantics (`PipeInserter.scala:274`, `PipeInserter.scala:281`, `PipeInserter.scala:291`). `Reg` also carries a write/read effect boundary around the generated `Pipe` (`PipeInserter.scala:189-201`, `PipeInserter.scala:282-292`). `Var` may be less hardware-visible than `Reg` (unverified), but the IR still observes it through mutable assign/read and substitution (`PipeInserter.scala:263-268`, `PipeInserter.scala:300-314`). A legal SSA peephole would need to prove no stream parent, no cross-case/block-result holder reuse, no ordering dependence, and equivalence of the generated pipe boundary (unverified). That proof is narrower than the pass’s general contract and should not be assumed by D-13 from this source slice alone.
