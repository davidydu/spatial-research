# D-13 Research, Angle 8: Implementation / Migration Cost

## 1. Baseline Surface Area

The lowest-risk reading of the current compiler is that pipe holders are already part of the post-`PipeInserter` IR contract. `PipeInserter` classifies block statements into outer and inner stages, finds non-void inner-stage values that escape, writes each escape into a holder inside a generated `Pipe`, then reads the holder after the pipe and substitutes that read for downstream users (`src/spatial/transform/PipeInserter.scala:170-201`). Holder choice is explicit: stream-controlled bit values become `FIFOReg`, other bit values become `Reg`, and non-bit values become `Var`; reads and writes dispatch to `deq`/`enq`, `.value`/`.write`, or `Var.read`/`Var.assign` (`src/spatial/transform/PipeInserter.scala:256-314`). The written spec matches this mechanism, but its HLS note also says raw SSA may be acceptable because HLS tools can insert registers, marking the area as rework rather than settled policy (`Spatial Research/10 - Spec/40 - Compiler Passes/50 - Pipe Insertion.md:179-188`).

## 2. Preserve All vs. Optimize All

Preserving all holders is medium implementation cost and lowest semantic risk. A Rust/HLS lowering can translate the existing post-pass shape directly: `Reg` to a scalar register-like object, `FIFOReg` to a one-element or small channel with destructive read/write semantics, and `Var` to mutable local state or a guarded temporary. This avoids changing pass ordering, because many consumers already observe these nodes. `AccessAnalyzer` has `RegRead`-specific invariance and reaching-write logic; `MemoryAnalyzer` configures `Reg` and `FIFOReg`; unrolling restages `RegRead`, `FIFORegDeq`, `RegWrite`, and `FIFORegEnq`; resource counting bins `RegNew` and `FIFORegNew` as register resources (`src/spatial/traversal/AccessAnalyzer.scala:45-55`, `src/spatial/traversal/AccessAnalyzer.scala:303-319`, `src/spatial/traversal/MemoryAnalyzer.scala:125-131`, `src/spatial/transform/unrolling/MemoryUnrolling.scala:550-553`, `src/spatial/codegen/resourcegen/ResourceCountReporter.scala:96-99`).

Optimizing all holders to SSA is high implementation cost disguised as cleanup. It would have to replace those memory identities with equivalent scheduling, effect, reporting, and debug facts. It is especially risky for `FIFOReg`: the node surface models enqueue/dequeue, marks dequeue as a transient unique operation, treats FIFOReg reads as destructive, and classifies FIFOReg access as stream-stage holder/enabler with pressure participation (`src/spatial/node/Reg.scala:23-44`, `src/spatial/metadata/memory/package.scala:412-424`, `src/spatial/metadata/access/package.scala:59-75`). Existing codegen also treats the holders as real state: Scalagen emits `Reg` as `Ptr`, `FIFOReg` as a mutable queue, and `Var` as `Ptr` (`src/spatial/codegen/scalagen/ScalaGenReg.scala:15-36`, `src/spatial/codegen/scalagen/ScalaGenFIFO.scala:46-52`, `src/spatial/codegen/scalagen/ScalaGenVar.scala:15-18`).

## 3. Hybrid Preservation Options

Preserving `FIFOReg`/`Reg` but converting `Var` to SSA has attractive migration cost. It keeps bit-valued hardware state aligned with existing memory and stream passes while reducing the awkwardness of lowering Argon `Var`, whose allocation and reads cannot accelerate and whose Chisel codegen is not a meaningful hardware primitive (`argon/src/argon/node/Var.scala:7-21`; prior note: `D-13-research/03-downstream-consumers.md`). The risk is that `Var` is still mutable IR state: assignment has write effects, and `PipeInserter` uses it as the non-bit escape bridge. This option therefore needs a narrow rule: SSA `Var` only when the holder is generated by `PipeInserter`, has one dominating write before all reads, and no consumer needs mutation identity.

Legality-proven holder elimination is the cleanest long-term optimization but the most expensive compiler work. It requires provenance metadata for generated holders, dominance/single-use proof, enable-equivalence proof, stream-control exclusion or replacement, and tests that compare emitted HLS scheduling against holder-preserving lowering. This option also needs negative tests, because the current test surface mostly protects related behavior indirectly: tests count `RegWrite`s, exercise explicit `FIFOReg`, and inspect register coalescing, but do not appear to pin HLS pipe-holder erasure directly (`test/spatial/tests/compiler/AccumReplacements.scala:355-359`, `test/spatial/tests/feature/memories/fifo/FIFOVector.scala:29`, `test/spatial/tests/feature/banking/RegCoalescing.scala:27`).

## 4. HLS-Native Mapping

HLS-native channel/register mapping has higher first-pass implementation cost than literal preservation, but lower migration risk than SSA-everything if it preserves the IR-level contract. `FIFOReg` should map to a channel/FIFO abstraction whenever stream pressure or destructive reads matter, because Chiselgen wires FIFOReg into FIFO interfaces, activity lanes, forward pressure, and back pressure (`src/spatial/codegen/chiselgen/ChiselGenMem.scala:40-50`, `src/spatial/codegen/chiselgen/ChiselGenMem.scala:202-224`, `src/spatial/codegen/chiselgen/ChiselGenCommon.scala:248-271`). `Reg` can map to an HLS scalar/register so long as read/write ordering remains explicit. `Var` can map to a local mutable temporary, with SSA as a proven peephole.

## 5. Preliminary Recommendation

Preliminary recommendation, not final D-13: migrate with HLS-native preservation of `FIFOReg` and `Reg`, preserve `Var` initially as mutable state, and add a later legality-proven elimination pass. Do not choose optimize-all-to-SSA as the base lowering contract. It is smaller generated C++, but it pushes correctness obligations into analysis, tests, and schedule reconstruction before the HLS backend has earned that trust.
