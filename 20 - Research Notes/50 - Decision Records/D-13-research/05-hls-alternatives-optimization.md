# D-13 Angle 5: HLS-Native Alternatives And Optimization Legality

## Recommendation

The HLS port should preserve pipe holders as semantic state in the intermediate representation, but it should not require literal `Reg`/`FIFOReg`/`Var` objects in emitted HLS. Lowering may optimize a holder to SSA when it proves the holder is only a private pipe escape, has one write before one read per dynamic firing, carries no back-pressure or status semantics, and does not persist across loop iterations or top-level calls. Otherwise, lower the holder to an HLS-native register or channel. The rule is: preserve the source-visible scheduling edge first; choose the physical HLS form only after a legality proof.

## Source Semantics To Preserve

`PipeInserter` is not a generic cleanup pass. It classifies block statements into `Transient`, `Alloc`, `Primitive`, `Control`, and `FringeNode`, attaches transient nodes to the last stage whose nodes or effect writes feed them, and then wraps inner primitive stages in `Pipe` nodes (`src/spatial/transform/PipeInserter.scala:117-134`, `src/spatial/transform/PipeInserter.scala:170-209`). For escaping values, `resFrom` chooses the holder: bit-valued values under stream control use `FIFOReg`, other bit-valued values use `Reg`, and non-bit values use `Var` (`src/spatial/transform/PipeInserter.scala:256-305`; [[50 - Pipe Insertion]]). The write occurs inside the inserted `Pipe`; the read is emitted after the pipe and substituted for downstream consumers (`src/spatial/transform/PipeInserter.scala:189-201`).

Those holders are observable in existing backends. Scalagen emits `Reg` as mutable `Ptr` state with `.value` reads and `.set` writes (`src/spatial/codegen/scalagen/ScalaGenReg.scala:15-36`), `Var` as `Ptr` read/assign state (`src/spatial/codegen/scalagen/ScalaGenVar.scala:15-18`), and `FIFOReg` as a mutable queue with enqueue/dequeue behavior (`src/spatial/codegen/scalagen/ScalaGenFIFO.scala:46-52`). Chiselgen also treats `FIFOReg` as a real FIFO-like memory: it goes through `emitMem`, gets activity-lane registration, and reads in outer controllers use the special `~break && done` enable rather than the normal datapath enable (`src/spatial/codegen/chiselgen/ChiselGenMem.scala:40-43`, `src/spatial/codegen/chiselgen/ChiselGenMem.scala:202-224`). So HLS cannot erase holders blindly.

## HLS-Native Replacements

`Local SSA temporary` (HLS knowledge, unverified): behavior-preserving when the holder is compiler-created, private, single-assignment, consumed in the same HLS process after the producer, and all original effect dependencies are still enforced. It is risky when the holder crosses a loop iteration, controller firing, stream boundary, or reset/initialization point.

`Static/local register` (HLS knowledge, unverified): behavior-preserving for a `Reg` holder that must remember a scalar across a scheduled boundary but has no FIFO status or elasticity. Use an explicit local variable/register when the HLS scheduler can place a flip-flop without changing lifetime. A C++ `static` is risky because its lifetime spans top-level invocations; it should be used only when Spatial semantics also require persistent state or the generated wrapper resets it deliberately.

`hls::stream` or `ap_fifo` (HLS knowledge, unverified): behavior-preserving for `FIFOReg`-like stream escape values with sequential one-producer/one-consumer access and back-pressure. AMD documents `hls::stream<>` as FIFO-like, sequential, destructive-read storage, implemented internally as FIFOs, with top-level streams mapping to `ap_fifo` in Vivado IP flow or AXIS in Vitis kernel flow ([AMD HLS Stream Library](https://docs.amd.com/r/2024.1-English/ug1399-vitis-hls/HLS-Stream-Library)). AMD also warns `ap_fifo` should be used only for sequential access and recommends `hls::stream` for FIFO communication ([AMD Interfaces for Vivado IP Flow](https://docs.amd.com/r/2024.1-English/ug1399-vitis-hls/Interfaces-for-Vivado-IP-Flow)). Risk: insufficient FIFO depth can stall or deadlock, and `ap_fifo` is not valid for random or bidirectional read/write access.

`Dataflow channels and scalar FIFOs` (HLS knowledge, unverified): behavior-preserving when PipeInserter's boundary corresponds to HLS `dataflow` tasks. AMD says dataflow channels between scalar, pointer, and reference parameters are implemented as FIFOs, while arrays become FIFO only when sequential access is detected; otherwise they become ping-pong buffers ([AMD Configuring Dataflow Memory Channels](https://docs.amd.com/r/2022.1-English/ug1399-vitis-hls/Configuring-Dataflow-Memory-Channels)). This is a good match for private scalar pipe escapes, but risky if the original Spatial holder expects exactly one-token storage and the HLS channel depth changes timing or deadlock behavior.

`ap_shift_reg` (HLS knowledge, unverified): behavior-preserving for fixed-latency delay-line behavior, not for generic `FIFOReg`. AMD describes `ap_shift_reg` as a class that maps directly to SRL resources and supports shift/read operations ([AMD Mapping Directly into SRL Resources](https://docs.amd.com/r/en-US/ug1399-vitis-hls/Mapping-Directly-into-SRL-Resources)). It is risky for elastic stream handshakes because a shift register has deterministic shifting, not empty/full back-pressure.

## SSA Legality Test

Optimize to SSA only if all are true: the holder was introduced by `PipeInserter`; no user memory/status operation can name it; there is exactly one dynamic producer and consumer; initial value is overwritten before read; the parent is not stream control or the FIFOReg behavior is otherwise represented; no downstream pass needs the holder for banking, activity, retiming, or debug; and the emitted HLS preserves Argon/Spatial anti-dependencies. If any condition fails, emit an explicit HLS state object or channel. This keeps Spatial compatibility while still allowing HLS to do the optimization it is good at: turning private, acyclic pipe escape values into scheduled wires/registers.
