# D-13 Research, Angle 6: Pipe Holder Observability

## Question Anchor

Q-139 asks whether HLS lowering must preserve `PipeInserter`'s explicit `Reg` / `FIFOReg` / `Var` holders, or may erase them into SSA when the HLS scheduler can legally carry the value without storage (`Spatial Research/20 - Research Notes/20 - Open Questions.md:1988-1998`). The open question is not about whether pipe insertion exists: the spec already says the pass is the structural repair that prevents an outer control block from mixing primitive statements and controllers, and that it wraps primitive runs in `Pipe` blocks while routing escaping values through holders (`Spatial Research/10 - Spec/40 - Compiler Passes/50 - Pipe Insertion.md:21-23`; source: `src/spatial/transform/PipeInserter.scala:170-209`, `:256-305`). The question is whether those inserted holders are part of observable semantics for Rust+HLS, or only one implementation tactic used by the Scala/Chisel pipeline.

## Written Contract

The current written contract has three pieces. First, pipe insertion classifies statements into `Transient`, `Alloc`, `Primitive`, `Control`, and `FringeNode`; primitives become inner stages, alloc/control/fringe nodes become outer stages, and transients such as `RegRead` are attached to the last stage whose producers or effect-writers feed them (`Spatial Research/10 - Spec/40 - Compiler Passes/50 - Pipe Insertion.md:29-47`). Second, inner stages compute their escaping non-void values, allocate one holder per escapee, write the holder inside the emitted `Pipe`, then read the holder in the parent scope and substitute that read for downstream consumers (`Spatial Research/10 - Spec/40 - Compiler Passes/50 - Pipe Insertion.md:78-92`). Third, the holder type is deliberate: bit values under stream control use `FIFOReg`, other bit values use `Reg`, and non-bit values use `Var`; writes and reads dispatch as enqueue/dequeue, register write/value, or var assign/read respectively (`Spatial Research/10 - Spec/40 - Compiler Passes/50 - Pipe Insertion.md:150-168`).

## Holder Semantics

The referenced holder types are not semantically inert in the spec. Register-family allocations are mutable local memories, with scalar `RegNew` and `FIFORegNew` entries and reset/init values (`Spatial Research/10 - Spec/30 - IR/10 - Spatial Nodes/20 - Memories.md:132-146`). Register and FIFOReg accessors also differ operationally: `RegRead` is transient/unique, `RegWrite` writes the memory, `FIFORegDeq` is a destructive dequeue, and `FIFORegEnq` writes/enqueues (`Spatial Research/10 - Spec/30 - IR/10 - Spatial Nodes/30 - Memory Accesses.md:94-100`, `:207-221`). The generic memory semantics say writers, dequeuers, and resetters carry `Effects.Writes(mem)`, so scheduling treats them as mutations rather than pure value movement (`Spatial Research/10 - Spec/20 - Semantics/40 - Memory Semantics.md:45-48`; `Spatial Research/10 - Spec/30 - IR/10 - Spatial Nodes/30 - Memory Accesses.md:260-265`).

`Var` is similarly mutable: the Argon base-type note says `Var[A]` has staged allocation, read, and assignment, with allocation marked mutable and assignment marked as a write to the variable (`Spatial Research/10 - Spec/30 - IR/00 - Argon Framework/D0 - DSL Base Types.md:62-72`). Scalagen gives `Reg` an actual runtime cell: `Ptr[T]` is a single-cell mutable container with `set`, `value`, `reset`, and `initMem` (`Spatial Research/10 - Spec/50 - Code Generation/20 - Scalagen/30 - Memory Simulator.md:90-96`). Scalagen gives `FIFOReg` queue behavior and explicitly does not distinguish it from a generic FIFO at runtime (`Spatial Research/10 - Spec/50 - Code Generation/20 - Scalagen/40 - FIFO LIFO Stream Simulation.md:80-82`).

## Ambiguity

The ambiguity is introduced by the HLS note inside the same Pipe Insertion spec. It says the Rust+HLS implementation must route escaping bit values through register-like holders, but immediately adds that HLS tools often do this automatically via SSA and that Rust could emit raw SSA and let HLS inject registers (`Spatial Research/10 - Spec/40 - Compiler Passes/50 - Pipe Insertion.md:179-188`). The scheduling-model note sharpens the conflict: HLS should preserve source-ordered staging and effect anti-dependencies, while `PipeInserter` is marked rework specifically because HLS can infer many registers, yet the explicit holder semantics are still observable in Scalagen and later passes (`Spatial Research/10 - Spec/20 - Semantics/20 - Scheduling Model.md:37-52`, `:59-66`). Retiming is one such later consumer of inserted structure: the Pipe Insertion entry says `RetimingTransformer` reads Pipe-emitted structure to compute delay-line placement (`Spatial Research/10 - Spec/40 - Compiler Passes/50 - Pipe Insertion.md:170-178`).

## Decision Required

The decision must define the observation boundary. If pipe holders are observable state, Rust+HLS must preserve holder allocation identity, read/write/dequeue effects, enable behavior, and FIFOReg stream ordering even when generated C++ could be shorter as SSA. If they are optimization artifacts, the spec must say exactly when holder erasure is legal: probably only when no downstream analysis, simulator behavior, user-visible state, enable condition, queue/back-pressure behavior, or retiming/scheduling dependency can distinguish the SSA value from the original holder path. The decision should also separate `Reg`, `FIFOReg`, and `Var`; `FIFOReg` is the riskiest to erase because dequeue/enqueue is queue state, while `Reg` and `Var` may be reducible to SSA only under a proof that their writes and reads are single-assignment within the relevant pipe boundary.
