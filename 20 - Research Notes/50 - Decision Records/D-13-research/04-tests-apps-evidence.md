# Tests and Apps Evidence for Pipe Holders

## Recommendation From Corpus Evidence

The tests/apps corpus does not prove that every `PipeInserter`-created holder must survive to final HLS, but it does argue for preserving `Reg`/`FIFOReg`/`Var` holder semantics as the default lowering contract. The strongest source-proven facts are that tests use register/FIFOReg-like state as inter-stage timing tokens, that stream tests assert observable FIFO ordering, and that open questions explicitly identify pipe-holder observability as an architectural decision (`Spatial Research/20 - Research Notes/20 - Open Questions.md:1988-1998`). The safe position for D-13 is: HLS may optimize generated holders to SSA only after proving no stream/control timing, buffering, or read/write effect is being erased.

## Inner Values Escaping to Later Control

`ParallelPipeInsertion` is the direct corpus smell test for this topic. Inside a `Foreach`, it writes `x := i`, enters a `Parallel`, then uses `x.value` both to choose between two loads and to compute `start`, which feeds a following DRAM slice load (`test/spatial/tests/feature/control/ParallelPipeInsertion.scala:22-30`). The file computes `goldA`, `notGoldA`, and `goldC`, but the final assertion is commented out (`test/spatial/tests/feature/control/ParallelPipeInsertion.scala:36-43`). Source-proven fact: the corpus contains a named pipe-insertion case with an inner primitive value consumed by outer-ish memory/control structure. Inferred risk: because the oracle is disabled, it is evidence of intended compiler coverage, not a passing behavioral golden.

`PropUses` adds a compiler-regression variant. A comment says `RegRead -> FixToFix -> CounterNew -> Controller` "used to crash TransientCleanup" (`test/spatial/tests/compiler/PropUses.scala:19-21`). The same test includes a `Pipe` containing `Reg`, struct `Reg`, FIFO, nested `if`, nested `Foreach`, and a final expression using both `fragments.deq().z` and `frag_reg` (`test/spatial/tests/compiler/PropUses.scala:24-45`), then asserts the independent reduce golden (`test/spatial/tests/compiler/PropUses.scala:48-51`). Source-proven fact: values derived from mutable reads are intentionally allowed to feed controller construction and nested control. Inferred risk: SSA collapse that loses the transient/read boundary could revive the old class of scheduling bugs.

## FIFOReg as Observable Timing Token

User-level `FIFOReg` tests are the clearest behavioral proxy for `PipeInserter`'s stream-holder choice. `FIFOVector` creates a `FIFOReg[Bit]` named `comm` between two `Pipe` blocks: the first block drains four values from `f1`, enqueues transformed values into `f2`, then `comm.enq(true)`; the following pipe performs `dummy := comm.deq()` before vector-dequeueing `f2` and enqueuing the packed result (`test/spatial/tests/feature/memories/fifo/FIFOVector.scala:21-43`). The test builds an exact packed-vector golden and asserts equality (`test/spatial/tests/feature/memories/fifo/FIFOVector.scala:54-60`). Source-proven fact: a one-element FIFOReg is used as an ordering/data-valid token, not just as a replaceable scalar name.

`DummyWorkload` makes that intent explicit. It declares `done1`, `done2`, and `done3` as FIFORegs, with the comment "FIFOs of depth 1, used for controlling streams" (`test/spatial/tests/feature/dynamic/DummyWorkload.scala:76-79`). The scheduler section also says an inserted `Pipe` guarantees earlier enqueues finish before done enqueues begin, because otherwise retiming "doesn't recognize the dependency" (`test/spatial/tests/feature/dynamic/DummyWorkload.scala:117-120`). The finisher then blocks on all three FIFOReg deqs and writes status only if all are true (`test/spatial/tests/feature/dynamic/DummyWorkload.scala:174-179`). This is source-proven timing/state evidence.

## Stream and FIFO Timing Pressure

Other tests/apps confirm that queue timing is user-visible. `Bank1D.NoMerge1D` uses two FIFORegs after populating an SRAM, then deqs them in two later pipes before reads; its comment says two reads may happen in the same cycle and should not be merged, and its `checkIR` requires exactly two SRAM duplicates (`test/spatial/tests/feature/banking/Bank1D.scala:124-145`, `test/spatial/tests/feature/banking/Bank1D.scala:150-163`). `SW` has an app-level FSM comment warning that an enqueue must be wrapped or masked because the FIFO may be full before the FSM observes full (`test/spatial/tests/apps/SW.scala:126-148`). `PriorityDeq` asserts the payload enqueue must happen before a later dequeue, with comments describing a prior timing mismatch between FIFO emptiness and activeIn (`test/spatial/tests/feature/dynamic/PriorityDeq.scala:288-299`, `test/spatial/tests/feature/dynamic/PriorityDeq.scala:310-312`, `test/spatial/tests/feature/dynamic/PriorityDeq.scala:333-358`). These are not generated pipe-holder goldens, but they prove stream/FIFO timing is a tested semantic surface.

## Open-Question Boundaries

Open questions sharpen the limit of what this angle proves. Q-017 records an empirical Chisel rule: outer-controller FIFOReg reads use `done` instead of the usual `datapathEn & iiIssue`, with the quoted comment "this is what works" (`Spatial Research/20 - Research Notes/20 - Open Questions.md:243-257`). Q-150 says FIFO/LIFO/Stream simulator elasticity versus bounded back-pressure is undecided for HLS (`Spatial Research/20 - Research Notes/20 - Open Questions.md:2129-2139`). Q-105 asks whether streamify helper passes were retired or accidentally omitted, blocked by streamify test cases (`Spatial Research/20 - Research Notes/20 - Open Questions.md:1428-1439`). Q-140 separately leaves `PrimitiveBox` operational semantics unresolved (`Spatial Research/20 - Research Notes/20 - Open Questions.md:2001-2011`). Therefore, preserving holders is source-supported; broad SSA optimization is only an inferred future optimization after these stream/control semantics are nailed down.
