# Spec/Open-Question Survey for D-05

## Decision Surface

D-05 is anchored by Q-083: replace the current Chisel path's generated Scala, `SpatialIP`, `CommonMain`, and mutable Fringe globals with a Rust/HLS top-level manifest that carries target selection, scalar counts, stream info, allocator count, retiming, and channel assignment explicitly (`20 - Research Notes/20 - Open Questions.md:1136-1147`). [[40 - Open HLS Questions]] ranks this as one of the top Rust+HLS questions and phrases the missing representation as target, stream widths, memory channels, and file dependencies without mutable Chisel globals (`30 - HLS Mapping/40 - Open HLS Questions.md:16-18`). The survey implication is that D-05 is not just a serialization decision; it is the replacement for the current elaboration-time dataflow among [[60 - Instantiation]], [[10 - Fringe Architecture]], and [[90 - Host-Accel Boundary]].

## Instantiation Constraints

[[60 - Instantiation]] shows the Chisel generator splitting top-level construction across `Main`, `AccelWrapper.scala`, `Instantiator.scala`, `ArgInterface.scala`, and `AccelUnit.scala` (`10 - Spec/80 - Runtime and Fringe/60 - Instantiation.md:34-42`). Source confirms that `emitEntry` calls `emitPreMain()`, emits the accelerator block, and calls `emitPostMain()` inside `Main.main(accelUnit)` (`src/spatial/codegen/chiselgen/ChiselCodegen.scala:293-302`). The values D-05 must preserve are computed before IO creation: combined arg counts, stream metadata lists, AXI stream metadata, and allocator count are assigned into `globals` before `CustomAccelInterface` is constructed (`src/spatial/codegen/chiselgen/ChiselCodegen.scala:196-217`). Controller codegen then writes build-specific target latencies, `perpetual`, `channelAssignment`, `retime`, and `enableModular` into the same global surface (`src/spatial/codegen/chiselgen/ChiselGenController.scala:563-581`). Therefore the Rust/HLS manifest needs separate sections for host ABI counts, interface layout, allocator layout, target assumptions/latencies, and control flags, rather than one undifferentiated "config" blob.

## Fringe/Target Constraints

[[10 - Fringe Architecture]] makes the target, not the generated accelerator body, responsible for the external pin shape (`10 - Spec/80 - Runtime and Fringe/10 - Fringe Architecture.md:28-36`). In source, `SpatialIP` maps a target string to a concrete target, installs the target IO hook, constructs the delayed accelerator, and delegates top-level shell creation through `addFringeAndCreateIP` (`fringe/src/fringe/SpatialIP.scala:18-39`). `DeviceTarget` carries the dimensions that `globals` exposes as architectural constants: address width, data width, words per stream, external width/vectorization, target width, and channel count (`fringe/src/fringe/targets/DeviceTarget.scala:33-66`; `fringe/src/fringe/globals.scala:18-24`). `fringe.globals` also supplies defaults for empty load/store/gather/scatter stream lists and AXI stream lists (`fringe/src/fringe/globals.scala:53-68`). D-05 should decide whether those defaults are represented as explicit validated manifest entries or as builder-time derivations; silently inheriting singleton fallback behavior would recreate the bug surface in another form.

## Host ABI And Scalar Layout

Q-084 is already resolved by [[90 - Host-Accel Boundary]]: the boundary should not be named after `ArgInterface.scala`, because source only opens and closes an empty `object Args`; scalar layout lives in `ArgAPI.scala`, Fringe's register file, and target bridges (`20 - Research Notes/20 - Open Questions.md:1149-1159`; `src/spatial/codegen/chiselgen/ChiselCodegen.scala:175-190`; `src/spatial/codegen/chiselgen/ChiselCodegen.scala:283-285`). Chisel `ArgAPI.scala` orders ArgIns, DRAM pointers, ArgIOs, ArgOuts, instrumentation counters, and early exits (`src/spatial/codegen/chiselgen/ChiselGenInterface.scala:156-187`), while Cppgen emits the matching host-side `ArgAPI.hpp` offsets (`src/spatial/codegen/cppgen/CppGenInterface.scala:138-164`). [[90 - Host-Accel Boundary]] states the invariant directly: Cppgen and generated hardware must agree on the scalar register map, pointer slots, stream metadata, and launch/readback behavior (`10 - Spec/20 - Semantics/90 - Host-Accel Boundary.md:48-54`). D-05 should therefore link to Q-152, the host ABI manifest question, rather than trying to solve instantiation globals independently (`20 - Research Notes/20 - Open Questions.md:2154-2165`).

## Open Questions And Decision Links

The load-bearing unresolved item is Q-083 itself, blocked by HLS top-level generation design (`20 - Research Notes/20 - Open Questions.md:1141-1146`). Adjacent constraints include Q-152 for scalar args, DRAM pointers, ArgIO/ArgOuts, counters, exits, and streams (`20 - Research Notes/20 - Open Questions.md:2154-2165`), plus [[10 - Fringe Architecture]] questions about channel assignment and arg-out loopback ownership (`10 - Spec/80 - Runtime and Fringe/10 - Fringe Architecture.md:60-64`). The manifest also has to preserve observable host/control details from Fringe: command/status registers, debug-register sizing, done/reset/enable behavior, and the target-specific read-data timing hack (`10 - Spec/80 - Runtime and Fringe/10 - Fringe Architecture.md:40-44`; `fringe/src/fringe/Fringe.scala:112-135`). Recommended D-05 wording: define a versioned Rust `TopManifest` as the sole input to HLS top generation, Rust host ABI generation, and sidecar emission; include target spec, host ABI, stream/AXI/DRAM layout, allocator info, instrumentation, control flags, latency/retiming assumptions, and artifact paths. JSON or TOML sidecars are plausible review/interchange forms (unverified), but the decision should make the typed Rust manifest authoritative.
