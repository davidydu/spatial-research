# D-05 Research Angle 3: Target/Fringe Coupling

## Coupling Shape

The current Chisel path makes target selection a side effect of constructing `SpatialIP`: the constructor pattern-matches a target string and writes a concrete `DeviceTarget` into `globals.target` before the accelerator is generated (`fringe/src/fringe/SpatialIP.scala:18`, `fringe/src/fringe/SpatialIP.scala:19`, `fringe/src/fringe/SpatialIP.scala:33`). That same constructor also installs `makeIO` as `IO(x)`, instantiates the delayed `AccelUnit`, clears its IO with `DontCare`, and asks the selected target to add Fringe and create the final IP (`fringe/src/fringe/SpatialIP.scala:36`, `fringe/src/fringe/SpatialIP.scala:37`, `fringe/src/fringe/SpatialIP.scala:39`). The Rust/HLS replacement should avoid reproducing this hidden construction order. The manifest should name the target explicitly, and the builder should consume that value as immutable input rather than mutating a global during module construction.

## Target Configuration

`globals` exposes target properties through computed accessors, so many apparently global constants are really target-specific fields: `bigIP`, `magPipelineDepth`, `ADDR_WIDTH`, `DATA_WIDTH`, `WORDS_PER_STREAM`, external/target widths, and `NUM_CHANNELS` all delegate to `target` (`fringe/src/fringe/globals.scala:12`, `fringe/src/fringe/globals.scala:16`, `fringe/src/fringe/globals.scala:18`, `fringe/src/fringe/globals.scala:24`). `ChiselCodegen` also bakes target-name checks into generated Scala for IO width/vector decisions and top-level width selection (`src/spatial/codegen/chiselgen/ChiselCodegen.scala:143`, `src/spatial/codegen/chiselgen/ChiselCodegen.scala:145`, `src/spatial/codegen/chiselgen/ChiselCodegen.scala:239`). These belong in a target/platform manifest section: target id, device family, host shell/fringe provider, BigIP or equivalent integration handle, address/data/stream geometry, external IO geometry, number of memory channels, and pipeline defaults.

## Application Manifest

The same `globals` object also stores per-application facts: argument counts, loopback maps, stream descriptors, AXI stream descriptors, and allocator count (`fringe/src/fringe/globals.scala:45`, `fringe/src/fringe/globals.scala:52`, `fringe/src/fringe/globals.scala:57`, `fringe/src/fringe/globals.scala:60`). `ChiselCodegen.emitPostMain` copies generated app-local values into those globals, then immediately creates `CustomAccelInterface` from the global stream and argument views (`src/spatial/codegen/chiselgen/ChiselCodegen.scala:197`, `src/spatial/codegen/chiselgen/ChiselCodegen.scala:203`, `src/spatial/codegen/chiselgen/ChiselCodegen.scala:213`, `src/spatial/codegen/chiselgen/ChiselCodegen.scala:215`). It also passes the same counts and stream descriptors into `AccelUnit` through constructor arguments (`src/spatial/codegen/chiselgen/ChiselCodegen.scala:245`, `src/spatial/codegen/chiselgen/ChiselCodegen.scala:263`, `src/spatial/codegen/chiselgen/ChiselCodegen.scala:276`). The replacement manifest should keep these as app instance fields: scalar register counts, instruction/breakpoint outputs, load/store/gather/scatter stream shapes, AXI stream shapes, allocator count, and any loopback map.

## Behavioral Overrides

Several properties are neither pure target defaults nor structural app shape. `globals` initializes retiming, perpetual mode, modular generation, verbosity/debug registers, and channel assignment as mutable process state (`fringe/src/fringe/globals.scala:26`, `fringe/src/fringe/globals.scala:31`). `ChiselGenController` later emits assignments for arithmetic and memory latency constants, perpetual IP mode, channel assignment, SRAM cost model knobs, SRAM threshold, retiming, and modular generation (`src/spatial/codegen/chiselgen/ChiselGenController.scala:563`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:566`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:573`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:577`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:581`). These should be explicit manifest overlays with provenance: target default, compiler option, or app-specific override. Latency constants and SRAM thresholds are especially risky because they mutate `globals.target`, which makes the target object both a platform description and a compile-result container (`src/spatial/codegen/chiselgen/ChiselGenController.scala:575`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:579`).

## D-05 Implication

For D-05, the Rust/HLS top-level manifest should be split into at least two records. `TargetConfig` should replace `globals.target` and include immutable device/fringe selection, derived bus widths, channel count, stream word geometry, BigIP/shell binding, and IO creation policy. `ApplicationTop` should replace the generated-instantiator mutations: argument surfaces, stream descriptors, allocator count, generated wrapper module name, and reset/control conventions. A third `BuildOptions` or `ImplementationOverrides` record should carry retiming, modular/perpetual flags, channel assignment, latencies, cheap-SRAM mode, and SRAM threshold. This separation prevents app codegen from silently changing platform constants while preserving the facts currently threaded through `SpatialIP`, `globals`, `AccelWrapper`, `Instantiator`, and `ChiselGenController` (`fringe/src/fringe/SpatialIP.scala:39`, `fringe/src/fringe/globals.scala:62`, `src/spatial/codegen/chiselgen/ChiselCodegen.scala:216`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:580`).
