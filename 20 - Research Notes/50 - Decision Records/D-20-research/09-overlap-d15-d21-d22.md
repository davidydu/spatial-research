# D-20 Angle 9: Overlap with D-15, D-21, and D-22

[[D-20]] cannot pick "cycle-aware `DelayLine` simulation" as a local feature. A real delay line is a per-cycle state element whose advance condition must match the enclosing controller and channel semantics. The source split is sharp: ScalaGen erases delay time by emitting no code for constant `DelayLine` and `val lhs = data` for non-constant lines (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenDelays.scala:9-13`), while ChiselGen emits a wire assigned through `DL(..., delay)` (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenDelay.scala:16-30`). `DL` gates retiming on `backpressure` and, for `haltIfStarved`, `forwardpressure` (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenCommon.scala:288-293`). The RTL template then shifts only when `flow` is high and otherwise holds state (`/Users/david/Documents/David_code/spatial/resources/synth/aws.hw-resources/RetimeShiftRegister.sv:22-39`). That means a cycle-aware Rust simulator needs the same logical "does this cycle advance?" answer as the rest of the simulated hardware.

The first dependency is [[D-15]]. If Rust runs `ParallelPipe` in ScalaGen compatibility mode, there is no honest peer-cycle for a delay line inside that region. ScalaGen emits `ParallelPipe` by wrapping the body, calling `gen(func)`, and marking done (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenController.scala:187-191`; [[01-scalagen-parallelpipe-semantics]]). Hardware metadata instead assigns every `ParallelPipe` to `ForkJoin` (`/Users/david/Documents/David_code/spatial/src/spatial/flows/SpatialFlowRules.scala:317-333`), and Fringe starts enabled children under one parent window, joins on all `iterDone`, and increments only on synchronized backpressure-qualified progress (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/Controllers.scala:141-154`). Therefore a `cycle_shift_register` mode should be legal only under a D-15 mode that defines parent ticks, child issue, child completion, masks, and deterministic tie breaks. Under `compat_scalagen_serial`, `DelayLine` should remain value-transparent except for explicitly labelled debug traces; otherwise we would be attaching hardware-looking registers to a source-order interpreter.

The second dependency is [[D-21]]. `DelayLine` size and placement come from the compiler's timing model, but HLS will have its own accepted schedule. `InitiationAnalyzer` computes outer `II` from child `II`, inner `compilerII` from interval and `iterDiff`, and selected `II` from user overrides or `min(compilerII, latency)` (`/Users/david/Documents/David_code/spatial/src/spatial/traversal/InitiationAnalyzer.scala:14-41`; [[C0 - Retiming]]). The timing spec already says Rust should store compiler-requested and backend-accepted values separately (`10 - Spec/20 - Semantics/70 - Timing Model.md:57-65`), and [[D-08]] argues HLS reports are the post-tool authority for accepted II and latency (`20 - Research Notes/50 - Decision Records/D-08.md:31-37`). So D-20's cycle mode must report which II it assumed: `compilerII`, user-requested II, estimated HLS II, or parsed HLS accepted II. It can validate phasing under that assumption, but it must not silently overrule report-backed accepted II.

The third dependency is [[D-22]]. Backpressure is meaningless if queues are unbounded. ScalaGen maps FIFO and stream writes to `enqueue` and LIFO writes to `push` without enforcing staged capacity (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenFIFO.scala:41-44`; `/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenLIFO.scala:39-44`; `/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenStream.scala:92-97`). The Scalagen spec calls this "silent elastic enqueue" and flags overflow as a simulator/synthesis divergence (`10 - Spec/50 - Code Generation/20 - Scalagen/40 - FIFO LIFO Stream Simulation.md:140-149`). RTL FIFOs, by contrast, allocate counters and physical memories of `p.volume`, then expose `empty`, `full`, and `numel` (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/memory/MemPrimitives.scala:380-399`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/memory/MemPrimitives.scala:436-442`). If D-22 chooses elastic compatibility, delay lines may shift through conditions that hardware would stall. If D-22 chooses bounded backpressure, delay advancement, queue occupancy, invalid-on-empty behavior, overflow, and deadlock diagnostics must be one shared cycle policy.

The clean boundary is dual provenance. Keep D-20 `compat_value_elided` for ScalaGen value goldens. Add `cycle_shift_register` only as a D-15-aware and D-22-aware timing/debug mode, with fields for simulator mode, queue policy, assumed II, accepted-II source, flow predicate, and first blocked reason. That keeps [[D-20]] useful without letting it quietly decide [[D-15]], [[D-21]], or [[D-22]].
