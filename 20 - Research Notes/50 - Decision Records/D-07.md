# D-07 — HLS Banking-Search Pruning And Partition Planning

## Question

Choose the HLS banking-search pruning and partition-planning strategy.

Source: `src/spatial/metadata/memory/BankingData.scala:547-563`; `src/spatial/traversal/banking/ExhaustiveBanking.scala:392-431`; `src/spatial/traversal/banking/MemoryConfigurer.scala:610-633`.

Decision criteria: User decision: HLS banking search reuses Spatial alpha/N/B search, constrains it to HLS partition forms, or replaces it with a new planner.

## Research findings

- Spatial's banking metadata is broader than ordinary HLS partition directives. The core `ModBanking(N, B, alpha, axes, P)` represents `(alpha dot addr / B) mod N`, with flat and hierarchical views, alpha vectors, block-cyclic `B`, padding/period `P`, regrouping, explicit banking, forced banking, and full-bank dimensions. The natural HLS subset is much smaller. See [[01-banking-metadata-surface]].

- `ExhaustiveBanking` searches a large `(view, N, alpha, regroup, B)` space. It expands candidate bank counts from factors, powers, Mersenne-like values, and relaxed ranges; expands alpha vectors from factors, dimensions, coprimes, and relaxed residues; checks cyclic and block-cyclic conflicts; computes `P`; and stops on effort/timeout gates. Reusing it exactly would carry useful machinery, but also its scale and HLS-unspellable wins. See [[02-exhaustive-search-mechanics]].

- `MemoryConfigurer` defines the real contract. It clears old metadata, groups accesses by concurrency/conflict/broadcast rules, computes buffer depth and hazards, invokes a banking strategy, scores returned schemes, and finalizes physical `Memory` duplicates plus `Dispatch`, `Ports`, and `GroupId`. Any replacement planner must still produce this shape or replace all downstream consumers. See [[03-memory-configurer-contract]].

- HLS partition mapping is a narrower language. Complete, cyclic, and block partitions per dimension, plus reshape, explicit duplication, and storage binding, are plausible HLS-lowerable forms; arbitrary flattened alpha vectors and multidimensional modular bank functions are not portable HLS partition pragmas without manual array-of-banks lowering. See [[04-hls-partition-mapping]].

- Downstream consumers need a physical layout, not just `factor = N`. `MemoryUnrolling`, `EarlyUnroller`, `BufferRecompute`, area/DSE models, `MemoryAllocator`, reports, Scalagen, PIR, and Chisel all consume duplicate identity, bank/offset mapping, padding, dispatch, ports, buffer depth, cast/broadcast groups, and resource/depth facts. Chisel modules are backend-specific; the semantic layout contract is not. See [[05-downstream-consumers]].

- Tests and apps require user banking directives, conflict diagnostics, and enough bandwidth planning for real kernels. Checked `.bank`, `.forcebank`, `.conflictable`, `.fullybanked`, block-cyclic controls, hierarchical/flat choices, and `effort` appear in tests and apps. Exact `SRAMNew` counts or mux-port assertions are legacy Spatial diagnostics; value correctness and user directives are real semantic/user-facing contracts. See [[06-tests-app-usage]].

- Existing specs split the answer cleanly. [[30 - Banking Math]] is HLS-clean: keep access algebra, ISL emptiness checks, `computeP`, and direct-vs-crossbar reasoning. [[70 - Banking]] is HLS-rework: the current pass owns physical bank network construction, port assignment, resource binding, and Chisel-era codegen assumptions. See [[07-spec-open-question-survey]].

- Cost/risk analysis favors pruning before solving, not filtering after full Spatial search. Exact reuse is cheap but can find schemes HLS cannot express. A new HLS planner is cleaner but higher cost. The pragmatic v1 answer is to keep `MemoryConfigurer`'s boundary and evidence, but search an HLS-lowerable partition-plan space first. See [[08-cost-risk]].

- Precedent favors explicit partition/layout manifests and report feedback. User-directed partitioning, compiler-inferred partitioning, report-guided iteration, and visible fallback diagnostics all point away from hidden full alpha/N/B reuse. Existing D-04 and D-05 records already push HLS memory/layout choices into explicit manifests. See [[09-precedent]].

- The recommendation matrix names the preferred architecture: an **HLS-first partition planner with Spatial evidence and escape hatch**. It keeps Spatial grouping/conflict analysis, reuses useful factor priorities such as `NBestGuess`, emits only HLS-lowerable plans on the normal path, and reserves arbitrary alpha/N/B banking for explicit custom lowering. See [[10-recommendation-matrix]].

## Recommendation

**Adopt an HLS-first partition planner with Spatial evidence and an explicit custom-banking escape hatch.**

Do not reuse the full Spatial alpha/N/B search as the HLS authority. Also do not merely run full Spatial search and filter results afterward. Instead, define a new HLS partition-plan search space, seeded by Spatial's access evidence and factor heuristics, and require the planner to produce the same semantic memory-layout contract that downstream passes expect.

### Normal HLS planner

The normal HLS planner searches only HLS-lowerable forms:

- `complete` partition on a dimension,
- `cyclic` partition on a dimension with factor,
- `block` partition on a dimension with factor,
- optional `reshape`/widening where the backend supports it,
- explicit duplicate count/fission where needed,
- storage binding preference from D-04 (`BRAM`, `URAM`, `LUTRAM`, `AUTO_LOCAL`, etc.),
- requested port model and estimated bank depth.

Use Spatial evidence to drive this planner:

- affine access matrices,
- access-group/concurrency/conflict/broadcast evidence,
- `segmentMapping` and recurrence hints from D-06,
- staged dimensions and unroll lanes,
- user directives such as hierarchical/flat, full-bank dimensions, effort, explicit banking, and conflict waivers,
- `NBestGuess`-style factor ordering from access counts and dimension products.

The search space should be:

`partition_kind x dimension x factor x duplicate_count x reshape_width x binding_preference`

not:

`view x NStrictness x AlphaStrictness x regroup x arbitrary alpha vector x B`.

This cuts the common path down to plans the HLS backend can actually emit as source/directives, while still preserving the conflict and layout reasoning that made Spatial banking powerful.

### Required planner output

Every HLS memory plan should emit a first-class `HlsPartitionPlan` or equivalent manifest record with:

1. `memory`
2. `logical_dims`
3. `physical_duplicates`
4. `partition_kind`
5. `partition_dimension`
6. `partition_factor`
7. `reshape_width`
8. `duplicate_count`
9. `bank_depth`
10. `padding`
11. `port_model`
12. `binding_preference`
13. `requested_resource_class`
14. `fallback_kind`
15. `fallback_reason`
16. `source_evidence`
17. `tool_accepted_partition` when report feedback exists

It must also preserve the compiler-facing layout metadata or its Rust/HLS equivalent:

- physical duplicates,
- buffer depth,
- dispatch identity,
- port identity,
- mux/cast/broadcast grouping where applicable,
- padding,
- accumulator/buffer-accumulator flags,
- unused-access diagnostics,
- unbankable/conflict diagnostics.

### Escape hatch

Support arbitrary Spatial-style `ModBanking(N, B, alpha, axes, P)` only as an explicit escape hatch:

- `custom_banking_escape = true`
- opt-in by user directive or backend capability,
- reportable as non-portable or custom-lowered,
- excluded from claims of ordinary HLS partition portability,
- either lowered manually as array-of-banks/index logic or rejected with diagnostics.

Do not silently translate arbitrary alpha vectors into approximate HLS pragmas. If the planner cannot express a layout with complete/block/cyclic/reshape/duplication, it must say so.

### Rejected alternatives

Reject exact Spatial alpha/N/B reuse as the default HLS path. It preserves the old planner but produces layouts that may be valid Spatial hardware and invalid or unspellable HLS partition directives.

Reject post-filtering full Spatial search. Filtering after exhaustive search wastes DSE time and can accidentally bias toward unlowerable schemes before rejecting them. Prune before solving.

Reject pure HLS-tool inference. Letting the HLS tool infer all partitions gives up Spatial's user directives, conflict evidence, and reproducible DSE search surface.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Define `HlsPartitionPlan` and attach it to the D-05 top/application manifest and memory reports.
2. Preserve the `MemoryConfigurer` output contract or define a Rust/HLS equivalent for duplicates, dispatch, ports, depth, padding, and group identity.
3. Implement an HLS-first planner over complete/block/cyclic/reshape/duplicate/binding choices.
4. Seed candidate factors from `NBestGuess`-style access-count and dimension-product factors, but skip relaxed alpha-vector enumeration by default.
5. Map D-04 storage classes into `binding_preference` and `requested_resource_class`.
6. Consume D-06 `segmentMapping`/dependence evidence as planner input, not final schedule authority.
7. Add report fields for requested partition versus tool-accepted partition, and connect them later to D-08/D-21 latency and accepted-II reconciliation.
8. Implement explicit diagnostics for unbankable HLS partition plans, unlowerable arbitrary alpha/B schemes, ignored user directives, and custom-banking escape-hatch use.
9. Add tests for complete, cyclic, block, reshape, duplicate/fission, explicit banking, forcebank escape hatch, conflictable waiver, HLS fallback, and report reconciliation.
10. Keep the old alpha/N/B search available only for legacy Spatial/Chisel comparison or explicit custom-lowered experiments.
