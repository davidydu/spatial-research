# D-15 Angle 2: Hardware/ForkJoin Contrast

## Prior Constraint From D-14

D-14 already separates two questions that D-15 can easily collapse: value-compatible simulation and schedule-faithful simulation. Its recommendation says canonical Rust/HLS `ForkJoin` should start every enabled, non-masked child as a peer task, count masked children as done, complete only after every non-skipped child completes, and use max-child-plus-sync latency until backend reports replace estimates (`D-14.md:44-47`). The same record labels `compat_scalagen_serial` as deterministic reference behavior, not a back-pressure-faithful model (`D-14.md:52-57`). The angle-4 hardware memo is sharper: Chisel/Fringe means concurrent child issue plus an all-child per-iteration join, while ScalaGen does not model overlap or join-time effects (`D-14-research/04-chisel-fringe-controller-semantics.md:13-18`; `D-14-research/05-scalagen-codegen-behavior.md:7-11`).

The structural origin is not optional. Spatial flow assigns `ParallelPipe` directly to `ForkJoin` (`src/spatial/flows/SpatialFlowRules.scala:317-333`), and ChiselGen instantiates a state-machine module from `lhs.level` and `lhs.rawSchedule` (`src/spatial/codegen/chiselgen/ChiselGenController.scala:333-358`). So the hardware path consumes the schedule tag as a controller protocol, not just an annotation on textually ordered Scala statements.

## ForkJoin Hardware Shape

Fringe's controller interface exposes the signals needed for a real child protocol: parent `enable`, `done`, `ctrInc`, `parentAck`, `backpressure`, `break`; child `doneIn`/`maskIn`; and child `enableOut`/`childAck` (`fringe/src/fringe/templates/Controllers.scala:9-43`). `ForkJoin` then makes all children peers within one parent iteration. In the `ForkJoin` case, `synchronize` is the reduction AND of all child `iterDone` latches, and the parent counter increment is `synchronize & io.backpressure` (`fringe/src/fringe/templates/Controllers.scala:141-147`). Each child's active latch can be set under the same parent window: not already iteration-done, not already done, parent not counter-done, parent enabled, and back-pressure allowing issue (`fringe/src/fringe/templates/Controllers.scala:148-151`).

That is the core contrast: hardware does not execute child 0 to completion, then child 1, then child 2. It attempts to start all currently enabled peers for the same logical iteration. Completion is an all-child barrier over per-child done/skipped state, not textual fall-through. The runtime model is consistent with that reading: `ForkJoin` costs max child latency per parent iteration plus metadata synchronization, rather than summing children as a sequential controller would (`models/src/models/RuntimeModel.scala:313-322`).

## Masks, Back-Pressure, And Counter Advance

Masks participate in completion, not only predicate side effects. In Fringe `ForkJoin`, a child's `iterDone` is set by either `doneIn(i)` or `!maskIn(i) & enable & backpressure`; a masked-out child therefore becomes skipped-done without issuing its body (`fringe/src/fringe/templates/Controllers.scala:150-153`). Shared post-schedule logic acknowledges a child when its `iterDone` is high and drives `enableOut(i)` only if the parent is enabled, the child is active, not yet iteration-done, masked in, not all done, and, for child 0, not counter-done (`fringe/src/fringe/templates/Controllers.scala:212-215`).

The Chisel block wrapper preserves those ports through the kernel boundary: parent state-machine enable outs and child acknowledgements are copied into `InputKernelSignals`, while child `done` and `mask` flow back to the parent state machine (`fringe/src/fringe/SpatialBlocks.scala:47-71`, `128-131`, `155-158`). The actual counter-chain enable is further gated by `sm.io.ctrInc & iiDone & forwardpressure`, so synchronization is necessary but not the only advance condition (`fringe/src/fringe/SpatialBlocks.scala:156-158`). ChiselGen also computes controller `mask` from local counter-chain non-noop and parent enables (`src/spatial/codegen/chiselgen/ChiselGenController.scala:317-325`), and wires controller back-pressure from `getBackPressure(lhs.toCtrl)` with the done latch included (`src/spatial/codegen/chiselgen/ChiselGenController.scala:304-310`). Therefore a Rust parallel simulator that ignores mask/back-pressure can get both liveness and ordering wrong: it may advance a parent iteration before an enabled child finishes, or fail to model the stall that prevents the `ForkJoin` counter from advancing.

## ScalaGen Serial Contrast

ScalaGen takes a much simpler route. Its block generator walks statements in order through `javaStyleChunk(...)(visit _)` (`src/spatial/codegen/scalagen/ScalaCodegen.scala:37-53`). The `ParallelPipe` case emits a kernel wrapper, calls `gen(func)`, then emits control-done (`src/spatial/codegen/scalagen/ScalaGenController.scala:187-191`). D-14 correctly characterizes this as deterministic serialization of the IR block, not a fork/join execution model with concurrent children, arbitrary interleavings, or join-time races (`D-14-research/05-scalagen-codegen-behavior.md:7-11`).

The one nearby ScalaGen stream concession is also serial: outer `Streaming` children that read external FIFOs/streams can be wrapped in `while (hasItems_child)` before moving to the next child (`src/spatial/codegen/scalagen/ScalaGenController.scala:22-49`). That drains one child before the next and is explicitly separate from `ForkJoin`.

## Simulator Consequence For D-15

For D-15, the crisp split is: `compat_scalagen_serial` can remain a deterministic golden-value mode for old ScalaGen parity, but it should be named as such. A hardware-oriented Rust simulator for `ParallelPipe` needs a peer-start, all-child-join model with masks and back-pressure in the step relation. It need not be waveform-identical to Chisel, but its observable rules should include: issue every enabled, masked-in child for the current parent iteration; treat masked-out children as skipped-done; stall counter advance until every non-skipped child is done and back-pressure allows synchronization; propagate child acknowledgements; and expose possible dataflow stalls rather than hiding them behind textual order.
