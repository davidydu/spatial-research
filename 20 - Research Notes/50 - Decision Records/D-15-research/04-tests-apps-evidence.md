# D-15 Angle 4: Tests/Apps Evidence for ParallelPipe Simulation

## Claim

The tests/apps corpus is mostly value-oriented evidence, not cycle-semantics evidence. It contains many explicit `Parallel { ... }`, `par`, unrolling, FIFO, and stream examples, but the executable checks usually compare final arrays, scalar reductions, or checksums. That pattern favors a Rust default that matches ScalaGen's deterministic sequential `ParallelPipe` simulation for regression parity. The same files do contain pressure toward a hardware-style mode, but that pressure is inferential: comments about fullness, lockstep, stream conversion, II assumptions, and disabled assertions show where true interleaving/back-pressure could matter without pinning down a required simulator schedule.

## Value-Level Goldens

The strongest tests simply say that parallel-looking programs must compute the same values as their sequential mathematical goldens. `ParallelLCA` is explicit about a compiler-created parallel LCA from outer-loop unrolling and says memory analysis treats the unrolled bodies as "lockstep"; the actual check is only `got == gold` for `i + getcol` (`test/spatial/tests/feature/buffering/ParallelLCA.scala:5-10`, `test/spatial/tests/feature/buffering/ParallelLCA.scala:24-40`). Outer unrolling similarly runs `Foreach(32 by 16 par 2)` and compares the whole DRAM result to a tabulated matrix, with POM/MOP variants only proving the same values (`test/spatial/tests/feature/unrolling/OuterUnrolling.scala:15-28`, `test/spatial/tests/feature/unrolling/OuterUnrolling.scala:42-64`). `ReduceTreeUnrolling` checks parallel reductions against host `data.reduce`, including uneven `par 3`, but observes no reduction-tree timing (`test/spatial/tests/feature/unrolling/ReduceTreeUnrolling.scala:13-21`, `test/spatial/tests/feature/unrolling/ReduceTreeUnrolling.scala:33-41`). Explicit `Parallel` load cases such as `BasicParams` and `FSMDotProduct` also end in scalar goldens, not child-order or pressure assertions (`test/spatial/tests/dse/BasicParams.scala:47-56`, `test/spatial/tests/dse/BasicParams.scala:80-88`; `test/spatial/tests/feature/control/fsm/FSMDotProduct.scala:24-35`).

## Disabled and Weak Asserts

The suite has several hints that schedule-faithful behavior was hard to lock down. `ParallelPipeInsertion` is the most direct D-15 smoke test: it creates a `Parallel` block with a conditional DRAM load and another load, prints `Wanted`, `GoldC`, and `GotC`, then comments out the real assertion (`test/spatial/tests/feature/control/ParallelPipeInsertion.scala:22-43`). `MatMult_systolic` uses FIFOs, `Parallel` shifting, an `II(1)` compute loop, and a producer/consumer `computeDone` FIFO, but its per-element assertion is disabled and the active check is only a checksum (`test/spatial/tests/feature/dense/MatMult_systolic.scala:44-68`, `test/spatial/tests/feature/dense/MatMult_systolic.scala:93-108`). `FIFOBanking2` performs parallel enqueues and a parallel reduce-dequeue from a depth-5 FIFO, then ends with `assert(true)`, so it does not constrain overflow, ordering, or back-pressure (`test/spatial/tests/feature/banking/FIFOBanking.scala:33-46`). The compiler `PipeMergerTest` even has an IR-shape requirement for number of unit pipes, but the collection and `require` are commented out (`test/spatial/tests/compiler/PipeMergerTest.scala:77-82`).

## IR and Compiler Checks

IR-level tests in this neighborhood are sparse and mostly absent for `ParallelPipe` itself. `PipeMergerTest` overrides `checkIR`, but after commenting out the `UnitPipe` count it delegates to `super.checkIR` only (`test/spatial/tests/compiler/PipeMergerTest.scala:77-83`). Other compiler checks nearby assert banking, counters, or initiation metadata, not fork/join execution order. `IllegalFIFOParallelization` is the important negative compiler-level signal: it expects one compile error for a `Foreach(16 par 2)` body that enqueues then dequeues the same FIFO in each parallel lane (`test/spatial/tests/compiler/IllegalBanking.scala:5-18`). That supports rejecting some unsafe FIFO parallelization statically, but it still does not define a runtime interleaving semantics for legal `ParallelPipe`.

## Inferred Cycle and Back-Pressure Pressure

The real hardware pressure shows up in apps with streams/FIFOs. `SW` notes that an FSM padding body may need wrapping or masking because "we are full before FSM sees full", then drains two FIFOs in a `Parallel` store based on `numel` (`test/spatial/tests/apps/SW.scala:147-157`). `MatMult_systolic` admits an `II=1` assumption and coordinates compute/store through `computeDone.deq`, suggesting schedule sensitivity beyond final checksum (`test/spatial/tests/feature/dense/MatMult_systolic.scala:55-68`). `StreamFIFO` and `StreamMultiFIFO` assert value preservation through stream FIFO pipelines, including conditional enqueue/dequeue, but do not expose stalls or queue capacity as observable behavior (`test/spatial/tests/feature/control/StreamFIFO.scala:23-39`, `test/spatial/tests/feature/control/StreamFIFO.scala:60-83`). `ParFIFOLoad` is especially telling: it forces Scala execution/access instrumentation, loads two FIFOs in `Parallel`, and only checks the final dot-product tile (`test/spatial/tests/feature/banking/ParFIFOLoad.scala:12-38`, `test/spatial/tests/feature/banking/ParFIFOLoad.scala:53-63`). So tests/apps justify serial ScalaGen compatibility as the regression baseline, while a parallel interleaving/back-pressure Rust mode needs additional, cycle-observing tests before it can claim conformance.
