# D-15 Angle 6: Runtime Model, DSE, And Latency Implications

## Two Latency Oracles

The D-15 simulator choice should not be confused with the existing DSE latency source. ScalaGen `ParallelPipe` simulation is textual: the case emits a kernel wrapper, calls `gen(func)`, then marks control done, with no child scheduler or done-token protocol (`spatial/src/spatial/codegen/scalagen/ScalaGenController.scala:187-191`; [[01-scalagen-parallelpipe-semantics]]). By contrast, current DSE does not measure ScalaGen execution time. It compiles a generated runtime-model jar, runs batched `tune` commands, parses `Total Cycles for App`, and writes those values into `Cycles` (`spatial/src/spatial/dse/LatencyAnalyzer.scala:35-45`; `spatial/src/spatial/model/RuntimeModelGenerator.scala:149-153`; [[D-08]]). So a Rust serial simulator can preserve value parity without automatically preserving DSE latency parity.

## Max-Child Versus Sum-Child

The runtime model already encodes `ParallelPipe`'s hardware-facing meaning more than ScalaGen's textual order. It defines `maxChild` and `sumChildren` separately (`models/src/models/RuntimeModel.scala:301-305`). Outer `Sequenced` uses `sumChildren * cchainIters` plus synchronization and advance overheads, while outer `ForkJoin` uses `startup + shutdown + maxChild * cchainIters + metaSync` (`models/src/models/RuntimeModel.scala:315-321`; [[06-runtime-model-dse]]). That difference is the core latency implication: a serial simulator that treats child regions as one-after-another suggests sum-child latency; the runtime and D-14 `ForkJoin` semantics suggest max-child latency plus join/sync overhead (`D-14.md:44-47`; [[D-14]]).

There is a catch. Max-child is not the same as "free overlap." The Chisel/Fringe controller sets `synchronize` as the AND of child `iterDone` bits and advances the counter only under `synchronize & io.backpressure` (`fringe/src/fringe/templates/Controllers.scala:141-147`). It also treats masked children as skipped-done and drives child enable from active, mask, done, and all-done state (`fringe/src/fringe/templates/Controllers.scala:148-154`, `212-215`; [[02-hardware-forkjoin-contrast]]). Therefore a realistic parallel/back-pressure simulator may produce `max(children) + stalls`, not just `max(children)`. The serial mode can overestimate pure parallel regions but can also hide bounded-channel stalls, deadlocks, or producer/consumer phasing.

## DSE Contract Pressure

DSE currently expects latency to be batchable and ordered: each worker turns design points into parameter rewrites, calls `evaluateLatency` once for the batch, then emits rows with area, `runtimes(i)`, validity, and timestamp (`spatial/src/spatial/dse/DSEThread.scala:109-134`; [[02-dse-consumers-contract]]). The CSV contract names `Cycles` and `Valid`, and downstream reload/sort/Sensitivity/HyperMapper paths depend on that shape (`spatial/src/spatial/dse/DSEAnalyzer.scala:354-361`, `420-439`; `D-08.md:69-90`).

If Rust simulation becomes a latency source, the mode label matters. `compat_scalagen_serial` cycles would be a compatibility measurement of textual execution, not an HLS latency estimate. `hls_semantic` or future `cycle_backpressure` cycles would be closer to D-14's peer-start/all-child-join contract, but still workload- and channel-policy-dependent. Either way, the old one-number `Cycles` adapter can remain only if sidecar provenance records whether the value is serial compatibility, analytic runtime estimate, bounded-dataflow simulation, HLS estimate, or report-backed latency (`D-08.md:71-90`; [[05-new-estimator-calibration]]).

## Validation Role

Serial simulation is still useful validation, but mostly for values, side effects, and regression against existing ScalaGen goldens. It should not validate max-child latency because it never exercises concurrent child issue or join timing (`D-14-research/05-scalagen-codegen-behavior.md:7-11`; [[05-scalagen-codegen-behavior]]). Parallel/back-pressure simulation validates a different surface: skipped masked children, blocked channels, child acknowledgement, parent counter advance, and liveness. That makes it better for D-22-style bounded FIFO/stream questions, but more expensive and more sensitive to input workload and FIFO depths (`D-14-research/08-hls-scheduler-mapping.md:25-27`; [[D-22]]).

## Report Calibration

D-08 treats HLS reports as post-tool authority and simulation as validation/mismatch triage, not the hot-loop DSE oracle (`D-08.md:49-67`). For calibration, that means the estimator should compare report latency and accepted II against the semantic level being claimed. A serial simulator mismatch against an HLS `DATAFLOW` report is expected evidence of mode divergence, not necessarily a bug. A parallel/back-pressure mismatch is more actionable: it can indicate missing channel depth, unmodeled memory dependence, wrong child masking, or report/parser attribution problems. The report-calibrated path should therefore keep the runtime tree equations as a fast analytic scaffold, keep serial simulation as value compatibility evidence, and use bounded parallel simulation as selective evidence about stalls and liveness rather than as an unqualified replacement for reports (`D-08-research/03-runtime-model-equations.md:25-27`; `D-08-research/05-new-estimator-calibration.md:19-29`).
