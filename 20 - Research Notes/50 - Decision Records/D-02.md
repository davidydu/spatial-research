# D-02 — Priority/Round-Robin Dequeue Grouping Contract

## Question

Define the grouping contract for priority and round-robin dequeues without relying on hash collisions.

Source: `src/spatial/lang/api/PriorityDeqAPI.scala:11,17,26,47,97`; `src/spatial/metadata/access/AccessData.scala:43`; `src/spatial/transform/unrolling/MemoryUnrolling.scala:303-304`.

Decision criteria: User decision: priority/round-robin dequeue grouping preserves hash-based compatibility or introduces an explicit grouped-dequeue IR construct.

## Research findings

- The surface API contract is call-scoped arbitration, not hash identity. All three helpers return one value, but stage one `FIFOPriorityDeq` per candidate FIFO and one `PriorityMux` over the candidate values. The fixed-priority overload enables FIFO `i` only after earlier FIFOs are empty; the conditional overload computes `shouldDequeue`; and `roundRobinDeq` rotates priority with `(iter + idx) % size`. The current grouped operation is therefore "select one FIFO from this API call and destructively read it," even though the IR is currently split across many nodes. See [[01-api-surface]].

- Grouping exists only as side metadata today. `PriorityDeqGroup(grp: Int)` is an optional tag on symbols, and the API assigns it from `fifo.head.toString.hashCode()` or `fifos.head.toString.hashCode`. Memory unrolling explicitly copies this tag from scalar priority dequeues to banked priority dequeues. Later consumers call `.get`, so missing metadata is not a graceful "ungrouped" case. See [[02-metadata-lifecycle]].

- Backend behavior makes the group semantically load-bearing. Control metadata excludes `FIFOBankedPriorityDeq` from ordinary read streams and groups priority inputs by `prDeqGrp`. Chisel forward pressure ORs FIFO readiness inside each priority group and ANDs across groups; codegen also adds the deferred `&& !fifo.empty` guard on banked priority dequeue reads. If two distinct logical groups collide, readiness can be weakened from "one ready in group A and one ready in group B" to "one ready anywhere in the merged group." See [[03-backend-consumers]].

- Scheduling and effects still need per-FIFO destructive-read visibility. `FIFOPriorityDeq` extends `Dequeuer`, and `DequeuerLike` writes the FIFO memory effect. The current split representation is useful because dependence analysis, access analysis, and unrolling can still see one mutating FIFO access per candidate. Any explicit grouped IR must preserve that multi-access/effect view; it cannot become a pure value mux. See [[04-scheduler-effects]].

- The usage corpus depends on semantics, not hashes. Dynamic tests use varargs priority for worker queues, sentinel/flush paths, and credit queues; conditional-list priority gates ID queues on matching payload queues; nested cases place two priority dequeues back-to-back; and the round-robin test uses a dynamic iteration expression. The tests stress liveness, no unintended dequeue, and no cross-group leakage. None suggest user dependence on the first-FIFO string hash. See [[05-usage-survey]].

- Hash compatibility is a brittle target. Normal FIFO symbols likely stringify as `xNN`, because `Ref.toString` prints node ids and staging allocates ids from `state.nextId()`. The stored group integer can be mirrored or copied, but recomputing it from a later textual symbol id would only accidentally match the original. Preserving exact hash behavior would mean preserving accidental collision behavior and JVM/Scala string hashing assumptions rather than a DSL contract. See [[06-hash-risk]].

- A first-class grouped-dequeue IR is feasible if it exposes the right axes: FIFO list, condition list, policy (`priority` or `round_robin(iter)`), empty-guard behavior, selected result, and per-candidate destructive effects. Existing Chisel/Scala-compatible lowering can still derive banked priority accesses and a priority mux, but analyses would read group structure directly instead of reverse-engineering `prDeqGrp`. See [[07-explicit-ir-option]].

- HLS lowering wants one explicit arbitration event. A safe HLS shape probes candidate emptiness/conditions, selects one eligible lane according to priority or rotated priority, and performs exactly one destructive stream read. The HLS contract should preserve OR-within-group forward pressure, codegen-time empty guarding, and the user-provided `iter` for round-robin. Reconstructing this from scattered nodes plus integer metadata is unnecessarily fragile. See [[08-hls-lowering]].

- Local hardware precedent points the same way. Fringe's `StreamArbiter` is a named arbiter module with structural inputs, a selected stream, and valid/ready routing; it does not infer arbitration identity from unrelated hashes. Spatial already has first-class `PriorityMux` for value selection; the missing piece is the equally first-class grouped destructive-read event. See [[09-precedent]].

- Cost analysis favors a two-layer plan. Preserving current hashes is cheap but carries silent-bug risk. Deterministic explicit group-id metadata is a good migration bridge because it keeps today's per-lane IR and backend consumers. Explicit grouped-dequeue IR is the best long-term Rust/HLS shape, though it requires updates to access discovery, unrolling, pressure analysis, diagnostics, and lowering. See [[10-cost-risk]].

## Recommendation

**Adopt explicit call-scoped grouped-dequeue IR as the Rust/HLS architectural contract, with deterministic metadata only as a compatibility bridge. Reject hash-based compatibility as a semantic promise.**

The contract should be:

1. A source `priorityDeq` or `roundRobinDeq` call creates exactly one logical grouped dequeue event.
2. The event owns an ordered list of candidate FIFOs, a same-length list of caller conditions, and a policy:
   - `Priority`: lowest source-order eligible FIFO wins.
   - `RoundRobin(iter)`: eligible FIFO with lowest `(iter + idx) % n` priority wins, matching current API behavior.
3. The event returns one value selected from the winning FIFO.
4. At most one candidate FIFO is destructively read for one event firing.
5. Each candidate remains visible as a potential FIFO write effect / destructive read for dependence, scheduling, and access analysis.
6. Forward pressure treats candidates in the same event as alternatives: OR within the group, AND across distinct groups.
7. Empty guarding is part of the grouped-dequeue lowering contract. The API may keep II-friendly enable expressions, but emitted HLS/RTL must not dequeue an empty candidate FIFO.

This recommendation chooses the explicit-IR side of the D-02 fork. Hash exactness should not be preserved except for importing or comparing legacy Scala/Chisel metadata during migration. In particular, the Rust/HLS implementation should not reproduce `fifo.head.toString.hashCode()` as the authoritative group id, and should not define hash collision merging as valid behavior.

### Why not pure hash compatibility?

It preserves the wrong thing. The hash is not in the user API, not visible in the type system, and not stable as an architectural concept. It is currently a workaround because the IR lacks a grouped destructive-read node. Preserving it would force the Rust/HLS rewrite to encode JVM symbol-printing and integer-hash accidents, while still leaving important behavior hidden behind metadata.

### Why not only deterministic metadata?

Deterministic group ids are a useful implementation bridge but too weak as the architectural answer. They still require downstream consumers to rediscover the group from many scalar access nodes, and the `.get`-based consumer path would remain a footgun if any lowering path forgets to attach the tag. Use deterministic metadata for compatibility, diagnostics, and staged migration; do not make it the core IR contract.

### Why explicit grouped IR?

It matches the source-level operation, the backend pressure semantics, and the HLS lowering shape. It also lets diagnostics name the actual operation: "priority dequeue group with N candidates," with policy, guards, and selected lanes, instead of printing an opaque integer bucket. It is the only option that eliminates accidental collision behavior while preserving the meaningful behavior current tests and codegen rely on.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Define a grouped-dequeue IR node or Rust HIR/MIR construct with `fifos`, `conditions`, `policy`, `iter`, and result type.
2. Specify its multi-access/effect view: one potential destructive read/write effect per candidate FIFO.
3. Lower current `priorityDeq` and `roundRobinDeq` APIs to the grouped construct rather than raw hash-grouped scalar dequeues.
4. Add a deterministic group-token bridge for any legacy pass that still expects `PriorityDeqGroup(Int)`.
5. Update unrolling / HLS lowering to produce banked accesses or stream reads from group structure, not from `toString.hashCode`.
6. Update forward-pressure computation to consume group structure directly while preserving OR-within-group and AND-across-groups semantics.
7. Add tests for fixed priority, conditional priority with payload guards, adjacent independent groups, shared first FIFO cases, round-robin `iter`, and no-empty-dequeue behavior.
