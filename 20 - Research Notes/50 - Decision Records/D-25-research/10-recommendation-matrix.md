# D-25 Angle 10: Recommendation Matrix

## Evidence Baseline

`OneHotMux` and `PriorityMux` are distinct APIs, so D-25 should not erase that distinction casually: `oneHotMux` stages `OneHotMux`, while `priorityMux` stages `PriorityMux` (`/Users/david/Documents/David_code/spatial/src/spatial/lang/api/MuxAPI.scala:26-31`). The IR does not enforce dynamic one-hotness. It unwraps a singleton, folds the first statically true selector, and merely warns when more than one literal selector is true (`/Users/david/Documents/David_code/spatial/src/spatial/node/Mux.scala:21-35`). Scalagen then gives dynamic multi-true inputs observable OR behavior by filtering true lanes and calling `reduce{_|_}` (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenBits.scala:30-37`). `PriorityMux` is already the ordered alternative, with an `if` / `else if` cascade and typed invalid fallback (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/scalagen/ScalaGenBits.scala:39-45`). Chisel preserves the split with `Mux1H` for `OneHotMux` and Chisel `PriorityMux` for priority selection (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenMath.scala:226-232`). The current test suite also encodes the split: `MuxTest` passes two runtime true selectors, expects one-hot OR, and expects priority first-enabled behavior separately (`/Users/david/Documents/David_code/spatial/test/spatial/tests/feature/unit/MuxTest.scala:6-40`).

## Test And Migration Focus

The key tests are not just hand-written muxes. Compiler producers must be covered because they rely on exclusivity without proving it locally: `MemoryDealiasing` returns `oneHotMux(conds, reads)` and also muxes addresses, dimensions, sizes, origins, par, and status reads (`/Users/david/Documents/David_code/spatial/src/spatial/transform/MemoryDealiasing.scala:22-35`, `:90-127`, `:150-159`). `SwitchOptimizer` and `FlatteningTransformer` turn motion-safe bit switches into `oneHotMux` when more than two cases remain (`/Users/david/Documents/David_code/spatial/src/spatial/transform/SwitchOptimizer.scala:65-79`; `/Users/david/Documents/David_code/spatial/src/spatial/transform/FlatteningTransformer.scala:85-96`). The one direct app use in `Inference` appears genuinely one-hot because `(col + j) % bufdim` is compared to every buffer column (`/Users/david/Documents/David_code/spatial/test/spatial/tests/apps/Inference.scala:481-483`).

## Decisive Recommendation And Rejections

Choose **dual-mode strict one-hot default plus Scalagen OR legacy**. Name the default `strict_onehot_v1`: statically reject known overlap or zero-true where possible, dynamically assert exactly one selector in simulator/debug HLS, and lower through any one-hot-equivalent network. Name the escape hatch `compat_scalagen_or_reduce_v1`: reproduce Scalagen OR-reduce, including `MuxTest` behavior, but label results non-strict and forbid one-hot-based optimizations.

Reject OR-reduce default because it preserves silent fixed-point corruption as policy. Reject one-hot-only because it breaks real legacy goldens unnecessarily. Reject priority default because `PriorityMux` already owns ordered arbitration. Reject invalid/poison as the main contract because it is a diagnostic mechanism, not the value semantics users expect from a mux.
