# D-25 Angle 3: Tests, App Usage, Call Sites

## Direct IR and backend meaning

The public helper is small and direct: `oneHotMux[A:Bits](sels, vals)` stages `OneHotMux`, while `priorityMux[A:Bits](sels, vals)` stages `PriorityMux` (`src/spatial/lang/api/MuxAPI.scala:26-31`). The IR rewrite for `OneHotMux` only handles one selector, statically literal-true selectors, and all-identical values; when more than one selector is literally true it warns, then returns the first true value (`src/spatial/node/Mux.scala:21-35`). Dynamic multi-true selectors are not checked there. `PriorityMux` deliberately lacks the analogous literal-true branch because that code is commented out (`src/spatial/node/Mux.scala:41-55`).

Backend behavior is divergent enough to matter for D-25. Scalagen emits `OneHotMux` as a list of true lanes followed by `.reduce{_|_}` (`src/spatial/codegen/scalagen/ScalaGenBits.scala:30-37`), while `PriorityMux` is an ordered `if` / `else if` cascade with an invalid fallback (`src/spatial/codegen/scalagen/ScalaGenBits.scala:39-45`). Chiselgen emits `OneHotMux` as Chisel `Mux1H` and `PriorityMux` as Chisel `PriorityMux` (`src/spatial/codegen/chiselgen/ChiselGenMath.scala:226-232`). PIR names one-hot muxes by arity, e.g. `OneHotMuxN` (`src/spatial/codegen/pirgen/PIRGenBits.scala:12-14`), and the area model records both mux families by selector count (`src/spatial/targets/NodeParams.scala:43-45`).

## Compiler-generated OneHotMux

The largest source-generated surface is `MemoryDealiasing`. Alias reads issue one read per candidate memory and return `oneHotMux(conds, reads)` (`src/spatial/transform/MemoryDealiasing.scala:22-35`); alias metadata queries also mux addresses, dimensions, sizes, origins, pars, and status reads (`src/spatial/transform/MemoryDealiasing.scala:90-127`, `src/spatial/transform/MemoryDealiasing.scala:150-159`). These sites rely on alias conditions being exclusive; no local assertion proves it.

Control lowering is the second major source. `SwitchTransformer` constructs ordinary if/else selectors as `cond2 & prevCond` and `!cond2 & prevCond`, so source if/else chains should be mutually exclusive by construction (`src/spatial/transform/SwitchTransformer.scala:59-70`, `src/spatial/transform/SwitchTransformer.scala:77-84`). Later, `SwitchOptimizer` converts motion-safe bit-valued switches with more than two selectors to `oneHotMux`, after warning only for statically multiple true selectors or no selectors (`src/spatial/transform/SwitchOptimizer.scala:45-79`). `FlatteningTransformer` has the same bit-switch-to-`oneHotMux` path when flattening inner controls (`src/spatial/transform/FlatteningTransformer.scala:85-96`). Chisel switch codegen also uses `Mux1H` for bit results (`src/spatial/codegen/chiselgen/ChiselGenController.scala:440-446`).

`HierarchicalToStream` builds a one-hot mux from dependency intake enables grouped by target memory (`src/spatial/transform/streamify/HierarchicalToStream.scala:193-203`). This is only dynamically one-hot if the stream dependency graph prevents two intakes for the same memory from firing together; the code does not assert that invariant.

## Tests and app usage

There is one direct unit fixture, and it explicitly covers dynamic multi-true `OneHotMux`. `MuxTest` passes runtime selectors with two enabled entries (`"0 0 1 0 1 ..."`), computes `mux1h := oneHotMux(...)`, and comments that it should take all enabled entries and OR them (`test/spatial/tests/feature/unit/MuxTest.scala:5-21`). The golden value for `OneHotMux` is exactly a bitwise OR of every enabled value, while the `PriorityMux` golden chooses the first enabled entry (`test/spatial/tests/feature/unit/MuxTest.scala:27-40`). That fixture makes Scalagen's OR semantics observable rather than merely accidental.

Application usage is much thinner. The only direct app hit is in `Inference`, where `sel = Seq.tabulate(bufdim){k => local_col == k}` selects among buffered columns and then calls `oneHotMux(sel, data_elements.map(_(i)))` (`test/spatial/tests/apps/Inference.scala:476-484`). Since `bufdim` is fixed at 8 and `local_col = (col + j) % bufdim`, this looks intended to be exactly one true selector dynamically (`test/spatial/tests/apps/Inference.scala:434-443`, `test/spatial/tests/apps/Inference.scala:481-483`). FSM tests mention bound `Mux1H`, but they come from mutually exclusive `if` / `else if` state tests, not overlapping selectors (`test/spatial/tests/feature/control/fsm/FSMBasicCond.scala:40-50`; `test/spatial/tests/ee109/Lab2Part3BasicCondFSM.scala:18-25`). A MachSuite comment proposes a future `Mux1H` refactor but is not active code (`test/spatial/tests/apps/MachSuite.scala:720-724`).

## PriorityMux and overlap by design

Several multi-true surfaces are intentionally `PriorityMux`, not `OneHotMux`. Plain `priorityDeq` selects over `fifo.map(!_.isEmpty)`, so multiple non-empty FIFOs are normal and first-true priority is the contract (`src/spatial/lang/api/PriorityDeqAPI.scala:10-21`). The conditional and round-robin overloads compute a single winner before the final `PriorityMux`, but still encode priority policy in ordered selectors (`src/spatial/lang/api/PriorityDeqAPI.scala:31-50`, `src/spatial/lang/api/PriorityDeqAPI.scala:71-104`). `EarlyUnroller` uses selectors `eA & eB`, `eA`, `eB`, and `true`, which are deliberately overlapping and therefore require priority behavior (`src/spatial/transform/streamify/EarlyUnroller.scala:169-176`). RewriteTransformer's Mersenne modulo corrections also use ordered boundary tests that can overlap (`src/spatial/transform/RewriteTransformer.scala:73-76`, `src/spatial/transform/RewriteTransformer.scala:104-116`).

## Missing fixtures

The missing coverage is not "any multi-true test"; `MuxTest` already has one. The gaps are compiler-generated overlap cases: alias conditions in `MemoryDealiasing`, dependency intakes in `HierarchicalToStream`, and switch lowering after transformations. Add fixtures that force dynamic overlap at those sites and compare Scalagen OR behavior against Chisel `Mux1H`. Also add zero-true and non-integer payload cases: Scalagen's `reduce{_|_}` has no empty case and assumes a valid `|`, while Chisel `Mux1H` has hardware-specific behavior that is not captured by the current integer-only `MuxTest`.
