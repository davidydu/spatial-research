# D-19 Research: Source and IR Scalagen Surface

## Question Surface

[[D-19]] is the Rust+HLS policy choice for FMA precision: keep Scalagen-compatible unfused multiply-add behavior, or move the simulator/backend contract toward Chisel/HLS fused FMA behavior. The decision queue frames it exactly that way (`20 - Research Notes/40 - Decision Queue.md:83-85`), and Q-146 records the immediate source of ambiguity: Scalagen emits FMA-shaped nodes as multiply then add, while Chisel hardware may preserve fused precision (`20 - Research Notes/20 - Open Questions.md:2078-2088`). This is adjacent to [[D-18]] because float FMA changes when `FloatPoint.clamp` is reached, and adjacent to [[D-10]] because both decisions ask whether Rust is a Scalagen oracle or a hardware-correlated oracle.

## Source and IR Entry Points

There is no obvious public `fma` surface in the inspected `Fix`/`Flt` language methods. Users write ordinary `*` and `+`: fixed point stages `FixAdd` and `FixMul` from infix methods, and floating point stages `FltAdd` and `FltMul` similarly (`argon/src/argon/lang/Fix.scala:51-53`; `argon/src/argon/lang/Flt.scala:33-37`). The explicit FMA surface appears after analysis/rewrite. The IR has a `FixFMA` node named "Fixed fused multiply add" and taking two multipliers plus an addend (`argon/src/argon/node/Fix.scala:130-131`). The float IR has the same shape, but `FltFMA` extends only `FltToFltOp`, unlike `FltAdd` and `FltMul`, which carry unstaged binary functions (`argon/src/argon/node/Flt.scala:22-26`). So the node name says "FMA", but its precise rounding semantics are not defined in the IR class itself.

## Rewrite Surface

The default compiler policy is to seek these nodes: `fuseAsFMA` defaults true, with `forceFuseFMA` available as a separate flag (`src/spatial/SpatialConfig.scala:75-76`). `RewriteAnalyzer` marks fixed and floating add nodes as `canFuseAsFMA` only in hardware scope, only when the add consumes a matching multiply pattern, only when inner-reduce metadata agrees, and only while `spatialConfig.fuseAsFMA` remains true (`src/spatial/traversal/RewriteAnalyzer.scala:42-58`; `src/spatial/metadata/rewrites/package.scala:7-9`). `RewriteTransformer` then rewrites `m1*m2 + add` or `add + m1*m2` into `FixFMA`/`FltFMA` (`src/spatial/transform/RewriteTransformer.scala:263-274`). Its specialization guard explicitly allows same-cycle cases, reduction-specialized cases, or forced fusion (`src/spatial/transform/RewriteTransformer.scala:28-39`). That means D-19 is not about syntax alone; it is about what a compiler-created IR node should mean after the original source has already been reshaped.

## ScalaGen Meaning

Scalagen is the key reference surface for [[20 - Numeric Reference Semantics]]. For both fixed and float FMA nodes, it emits ordinary source text:
`val lhs = ($m1 * $m2) + $add` (`src/spatial/codegen/scalagen/ScalaGenFixPt.scala:150`; `src/spatial/codegen/scalagen/ScalaGenFltPt.scala:120`). This is semantically unfused in the Scala emulator because the ordinary numeric operators have their own representation steps. `FixedPoint.*` shifts and clamps the product, and `FixedPoint.+` clamps the sum (`emul/src/emul/FixedPoint.scala:14-17`). `FloatPoint.*` and `FloatPoint.+` each call `FloatPoint.clamped` as separate operations (`emul/src/emul/FloatPoint.scala:153-157`). In contrast, Chiselgen lowers both `FixFMA` and `FltFMA` through `Math.fma`, with fixed point converting the result back to fixed (`src/spatial/codegen/chiselgen/ChiselGenMath.scala:50-51`). That is the concrete fused-versus-unfused split D-19 has to name.

## Reg, Counter, and Reduction Context

The reduction path makes this decision visible beyond scalar expressions. The accumulator IR includes `AccumFMA`, `AccumMarker.Reg.FMA`, `RegAccumFMA`, and ordinary `RegAccumOp` (`src/spatial/node/Accumulator.scala:9-16`; `src/spatial/node/Accumulator.scala:22-50`). `Reduce` builds a counter chain, binds counter iterators, stages map/load/reduce/store blocks, and writes through a `Reg` accumulator (`src/spatial/lang/control/ReduceClass.scala:38-49`; [[60 - Counters and Primitives]]). `AccumAnalyzer` recognizes `FixFMA(..., RegRead(reg))` and `FltFMA(..., RegRead(reg))` as register FMA cycles (`src/spatial/traversal/AccumAnalyzer.scala:187-190`), and `AccumTransformer` replaces those markers with `RegAccumFMA` (`src/spatial/transform/AccumTransformer.scala:110-120`). Scalagen then emits the specialized register path as `m0 * m1 + reg.value`, with the first iteration storing only `m0*m1` (`src/spatial/codegen/scalagen/ScalaGenReg.scala:57-64`). Its ordinary `RegAccumOp` path even throws if `AccumFMA` reaches it, so the pass invariant matters (`src/spatial/codegen/scalagen/ScalaGenReg.scala:41-50`). PIR similarly expands `RegAccumFMA` into a `FixMul` feeding `RegAccumOp("AccumAdd")` (`src/spatial/codegen/pirgen/PIRGenReg.scala:42-44`). For Rust+HLS, the minimum policy statement should therefore cover scalar `FixFMA`/`FltFMA` and reduction-specialized `RegAccumFMA`, not just user-written `a*b+c`.
