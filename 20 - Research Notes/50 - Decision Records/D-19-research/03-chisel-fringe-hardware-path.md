# Chisel, Fringe, and Hardware FMA Path

## Rewrite Entry

[[D-19]] is not just a backend spelling choice; the current compiler actively creates FMA IR in hardware regions. `RewriteAnalyzer` marks both fixed and floating `Add(Mul(a,b),c)` forms as FMA candidates only when the pattern is in hardware, has a single forward consumer or a recognized reduce-specialization consumer, stays in the same inner-reduce classification, and `spatialConfig.fuseAsFMA` is enabled (`/Users/david/Documents/David_code/spatial/src/spatial/traversal/RewriteAnalyzer.scala:42`, `/Users/david/Documents/David_code/spatial/src/spatial/traversal/RewriteAnalyzer.scala:45`, `/Users/david/Documents/David_code/spatial/src/spatial/traversal/RewriteAnalyzer.scala:51`, `/Users/david/Documents/David_code/spatial/src/spatial/traversal/RewriteAnalyzer.scala:58`). The default is to fuse (`/Users/david/Documents/David_code/spatial/src/spatial/SpatialConfig.scala:75`), with CLI escape hatches for `--noFuseFMA` and `--forceFuseFMA` (`/Users/david/Documents/David_code/spatial/src/spatial/Spatial.scala:455`, `/Users/david/Documents/David_code/spatial/src/spatial/Spatial.scala:456`). The transformer then rewrites fixed adds to `fixFMA` and float adds to `fltFMA` when the cycle/specialization test passes (`/Users/david/Documents/David_code/spatial/src/spatial/transform/RewriteTransformer.scala:263`, `/Users/david/Documents/David_code/spatial/src/spatial/transform/RewriteTransformer.scala:270`). This matters for [[20 - Numeric Reference Semantics]] because the source expression can become fused without the user naming an FMA.

## ChiselGen Boundary

The IR has explicit nodes: `FixFMA` is a fixed fused multiply-add node, while `FltFMA` is a float-to-float op without an unstaged Scala implementation (`/Users/david/Documents/David_code/spatial/argon/src/argon/node/Fix.scala:130`, `/Users/david/Documents/David_code/spatial/argon/src/argon/node/Flt.scala:26`). `ChiselGenMath` lowers both through `Math.fma`, but the fixed path immediately calls `.toFixed($lhs, "cast_$lhs")`, while the float path assigns the raw floating result (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenMath.scala:50`, `/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenMath.scala:51`). It also schedules them with separate latency keys, `FixFMA` and `FltFMA` (`/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenMath.scala:144`, `/Users/david/Documents/David_code/spatial/src/spatial/codegen/chiselgen/ChiselGenMath.scala:185`). [Inference] ChiselGen therefore preserves a semantic distinction: float FMA is one target primitive if the target supports it, while fixed FMA is followed by a normal fixed cast whose rounding/overflow policy is the default truncate/wrap path.

## Fringe Float FMA

Fringe's `Math` object says implementations should live in BigIP, and floating `Math.fma` follows that rule: it asserts identical formats, creates a `FloatingPoint` result, and assigns `globals.bigIP.ffma(...)` (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Math.scala:9`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Math.scala:671`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Math.scala:676`). The abstract BigIP hook exists but defaults to `throw Unimplemented("ffma")` (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/BigIP.scala:88`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/BigIP.scala:89`). Zynq/CXP-style targets override it with an `FFma` module carrying `flow`, `a`, `b`, and `c` into a Xilinx floating-point IP wrapper (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/zynq/BigIPZynq.scala:203`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/zynq/ZynqBlackBoxes.scala:1172`). That wrapper emits TCL for `Operation_Type {FMA}`, custom exponent/fraction widths, nonblocking flow control, and fixed latency 19 (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/zynq/ZynqBlackBoxes.scala:1217`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/zynq/ZynqBlackBoxes.scala:1219`). This is the strongest evidence that the hardware precedent for `FltFMA` is truly fused, not just syntactic `mul` plus `add`.

## HardFloat and Fixed Conversion

The sim-side HardFloat material is mixed. `BigIPSim` imports HardFloat and implements `fadd`, `fsub`, and `fmul` with `MulAddRecFN`, using a multiplier input of `1.0` for add/sub and `0.0` as the addend for multiply (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/BigIPSim.scala:6`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/BigIPSim.scala:136`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/BigIPSim.scala:155`). The underlying `MulAddRecFN` is a genuine fused recFN datapath: it multiplies `mulAddA * mulAddB`, adds aligned `mulAddC`, then rounds once through `RoundRawFNToRecFN` (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/hardfloat/MulAddRecFN.scala:327`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/hardfloat/MulAddRecFN.scala:339`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/hardfloat/MulAddRecFN.scala:344`). [Inference] Because the inspected `BigIPSim` class starts at line 12 and ends after conversion overrides without an `ffma` override, `FltFMA` simulation depends on a target with an override or falls back to the abstract throwing hook (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/BigIPSim.scala:12`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/targets/BigIPSim.scala:293`); this echoes prior [[D-18]] backend-divergence concerns.

Fixed `Math.fma` is weaker. Its own TODO says to use an IP core; the implementation builds a wrapper that computes `mul(m0,m1,... Truncate, Wrapping)` plus a retimed addend (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Math.scala:843`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Math.scala:857`). The ChiselGen post-cast calls `FixedPoint.toFixed`, which routes to `Math.fix2fix(... Truncate, Wrapping)` and then BigIP `fix2fix` (`/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/FixedPoint.scala:156`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Math.scala:703`, `/Users/david/Documents/David_code/spatial/fringe/src/fringe/templates/math/Math.scala:716`). Decision implication: for Rust+HLS, [[D-19]] should treat float FMA as hardware-fused precedent, but fixed FMA as a multiply-add template with explicit final fixed conversion unless a new policy intentionally upgrades it.
