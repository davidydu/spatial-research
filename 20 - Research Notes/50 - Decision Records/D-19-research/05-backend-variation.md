# D-19 Research: Backend Variation Beyond Scalagen and Chisel

## Scope

[[D-19]] should not assume that once `FixFMA` or `FltFMA` exists in IR, every backend has the same fused, constant-folded, or executable meaning. In Argon, `FixFMA` is a ternary `FixOp1` with no `unstaged` function or local rewrite override (`argon/src/argon/node/Fix.scala:130-131`). `FltFMA` is likewise a ternary `FltToFltOp`, while ordinary float add/mul are `BinaryFltToFltOp` nodes with `_ + _` and `_ * _` unstaged semantics (`argon/src/argon/node/Flt.scala:22-27`). So [[20 - Numeric Reference Semantics]] can use Scalagen as a reference path, but the non-Scalagen backends need a capability matrix: algebraic emission, simulator execution, resource reporting, and rewrite behavior are separate surfaces.

## Cppgen and Pirgen

[[10 - Cppgen]] treats both FMA nodes as source-level multiply-add expressions: `FixFMA(a,b,c)` and `FltFMA(a,b,c)` emit `$a * $b + $c`, not `std::fma`, not a rounded fused host intrinsic (`src/spatial/codegen/cppgen/CppGenMath.scala:34-36`). Its constants are also backend-native: fixed constants print as the literal plus `L` for wide fixed types, floating constants print as raw strings, and text/bit have special cases (`src/spatial/codegen/cppgen/CppGenCommon.scala:125-130`). This makes Cppgen useful as an algebraic host approximation, but not evidence of fused IEEE or custom-format behavior.

[[10 - Pirgen]] preserves FMA more structurally. `PIRGenFixPt` and `PIRGenFltPt` both route `FixFMA`/`FltFMA` through `genOp(lhs)` (`src/spatial/codegen/pirgen/PIRGenFixPt.scala:70-72`; `src/spatial/codegen/pirgen/PIRGenFltPt.scala:54-56`). `genOp` uses the operation class name and all product-iterator fields as inputs, then emits `OpDef(op=...).addInput(...)` with a type annotation (`src/spatial/codegen/pirgen/PIRGenHelper.scala:78-85`). Constants are typed PIR constants via `Const(...).tp(tp)` and vector constants get the same treatment (`src/spatial/codegen/pirgen/PIRCodegen.scala:81-93`). One caveat: `RegAccumFMA` lowers to a separate `FixMul` followed by `RegAccumOp("AccumAdd")`, so PIR has both a bare FMA representation and a split accumulation representation (`src/spatial/codegen/pirgen/PIRGenReg.scala:42-44`).

## Executor and Models

[[20 - Scala Executor]] is the sharpest divergence. `FixResolver` has an explicit `FixFMA` case, but it computes `(m0 * m1) * add`, not `(m0 * m1) + add` (`src/spatial/executor/scala/resolvers/FixResolver.scala:10-15`). `FltResolver` has no `FltFMA` case at all; it handles float predicates, conversions, text, and random, then defers (`src/spatial/executor/scala/resolvers/FltResolver.scala:8-42`). The generic `NaryResolver` only executes `Binary`, `Unary`, and `Comparison` nodes through their `unstaged` functions (`src/spatial/executor/scala/resolvers/NaryResolver.scala:22-47`), which excludes both ternary FMA nodes.

The reporting/model surface is also fixed-heavy. `ResourceReporter` estimates arithmetic area for `FixFMA` only, using `"FixFMA"` across LUT, FF, RAM, and DSP queries (`src/spatial/codegen/resourcegen/ResourceReporter.scala:157-163`). `ResourceCountReporter` maintains one `fixed_ops` counter and increments it for `FixFMA`, with no float-op counter or `FltFMA` case (`src/spatial/codegen/resourcegen/ResourceCountReporter.scala:32-57`, `src/spatial/codegen/resourcegen/ResourceCountReporter.scala:150-157`). The CSV latency surface does name both `FixFMA` and `FltFMA` for at least VCS (`resources/models/VCS_Latency.csv:84-127`), and `NodeParams` maps all `FltOp` nodes with no parameters (`src/spatial/targets/NodeParams.scala:37-39`), but area-report plumbing does not mirror that float coverage. [[10 - Area Model]] and [[20 - Latency Model]] should therefore be read as overlapping, not identical, contracts.

## Constant Rewriting

FMA constant rewriting is mostly around the node, not in the node. Ordinary fixed add/mul perform local constant folding and power-of-two rewrites before hardware FMA fusion can happen (`argon/src/argon/node/Fix.scala:69-80`, `argon/src/argon/node/Fix.scala:117-125`). Then `RewriteAnalyzer` marks add-of-mul patterns as FMA-fusable when hardware scope, consumer shape, reduce-op status, and `fuseAsFMA` allow it (`src/spatial/traversal/RewriteAnalyzer.scala:39-58`). `RewriteTransformer` finally rewrites `FixAdd(FixMul(...), add)` and `FltAdd(FltMul(...), add)` into `FixFMA`/`FltFMA` (`src/spatial/transform/RewriteTransformer.scala:263-274`).

After that point, there is no `FixFMA` or `FltFMA` op-local simplifier for all-constant FMA, multiply-by-zero/one, or add-zero. The fixed path only recognizes `FixFMA(Final(lin), iter, Final(ofs))` as an affine form for static modulo/vector constant propagation and residual metadata (`src/spatial/transform/RewriteTransformer.scala:214-229`, `src/spatial/transform/RewriteTransformer.scala:303-310`). There is no analogous `FltFMA` constant rewrite. D-19 should specify whether a Rust/HLS backend may add FMA-specific constant folding, and if so whether that is canonical or an explicitly declared backend optimization.
