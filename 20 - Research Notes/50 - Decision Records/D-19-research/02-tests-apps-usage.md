## Search scope

D-19 asks whether Rust+HLS should preserve Scalagen's unfused multiply-add precision or accept Chisel/HLS fused FMA precision (`20 - Research Notes/40 - Decision Queue.md:83-85`). I searched tests and apps for `FMA`, `fma(...)`, fused/multiply-add wording, dot products, GEMM/GEMV, reductions, convolution, SGD/SVRG, MLP/backprop, and sparse reductions. There are no direct `fma(...)` API calls in `test`, `apps`, or `fringe/test`; the observable surface is spelled as `x * y + z`, `Reduce(...){ a*b }{_+_}`, and FMA-specific accumulator rewrites. Relevant cross-links: [[D-19]], [[60 - Counters and Primitives]], [[B0 - Rewrites]], [[open-questions-semantics#Q-sem-11 - 2026-04-25 FMA fused versus Scalagen unfused semantics|Q-sem-11]].

## Direct FMA-like tests

The most explicit behavioral test is `FMAInLoop`: it marks `rf1(0,0) + temp` as "Should not fuse", `i + temp` into `rf2` as "Fuse is ok", and a scalar `Reg` accumulation as "Fuse is ok (because specialized reduce)" (`test/spatial/tests/feature/math/FMAInLoop.scala:17-29`). Its observable check is exact struct equality against staged host golds (`test/spatial/tests/feature/math/FMAInLoop.scala:33-42`). `ReduceSpecialization` is stronger on IR shape: it names `fma32`/`fma8`, builds product-sum reductions and piped variants (`test/spatial/tests/feature/memories/reg/ReduceSpecialization.scala:20-22`, `:43-47`, `:70-78`), checks exact values (`:183-195`), and requires `RegAccumFMA` writers in `checkIR` (`:215-229`). `InitiationIntervals` also labels a reduction `'IIFMA` over `i*i` and validates exact sum equality (`test/spatial/tests/compiler/InitiationIntervals.scala:114-119`, `:252-287`).

The scalar math suites provide broad but weaker coverage. `FloatBasics2`, `FixBasics2`, and `HalfBasics` each reserve operation slot 15 for `sram1(i) * sram2(i) + sram3(i)` and compare against the same host expression (`test/spatial/tests/feature/math/FloatBasics2.scala:55`, `:88`; `FixBasics2.scala:55`, `:88`; `HalfBasics.scala:55`, `:85`). `FloatBasics3` and `FixBasics` put `x*y+z` at index 3 of an enumerated operator vector (`test/spatial/tests/feature/math/FloatBasics3.scala:11-17`; `FixBasics.scala:11-17`). Regression visibility is uneven: mini runs `FloatBasics2` and `FixBasics2`; nightly names `FloatBasics2` plus `feature.memories.*` (`regressions/mini.list:1-2`; `regressions/nightly.list:1-2`).

## Reductions and apps

Dot/GEMM tests are the main app-facing multiply-add pressure. `DotProduct` and `DotProductStream` use exact fixed-point product reductions; `DotProductFlt` uses `Float` and accepts `abs(gold - result) < 0.3f` (`test/spatial/tests/feature/dense/DotProduct.scala:52-73`, `:127-144`, `:155`, `:188-208`). `GEMMLib*` exercises library `gemm(...)` with float/int/half and parallel variants, then compares each element with `<= .001` (`test/spatial/tests/feature/dense/GEMMLibTest.scala:41-52`, `:61-69`, `:73-94`). `MatMult_inner` and `MatMult_systolic` are risky as FMA sentinels because both compare only checksum sums, not elementwise matrices (`test/spatial/tests/feature/dense/MatMult_inner.scala:58-62`, `:91-102`; `MatMult_systolic.scala:55-60`, `:97-108`).

MachSuite adds realistic tolerances: `GEMM_NCubed` reduces `a*b` and allows `0.5`, `GEMM_Blocked` uses tiled `MemReduce` and also allows `0.5`, and `SPMV_CRS` allows `0.2` with a comment that Scala and Chisel differ on bounds (`test/spatial/tests/apps/MachSuite.scala:1402-1420`, `:1663-1692`, `:2081-2101`). ML kernels are threshold-style rather than precision-style: Backprop combines dense-layer reductions plus bias, then only requires more than one of six weight/bias groups to fall within `0.75` (`test/spatial/tests/apps/MachSuite.scala:2387-2421`, `:2604-2615`). SGD/LP_SGD/LP_SVRG reduce dot products and gradients but accept final Cartesian-distance thresholds (`test/spatial/tests/apps/SGD.scala:137-143`, `:160-198`; `LP_SGD.scala:201-207`, `:244-263`; `LP_SVRG.scala:203-210`, `:239-257`, `:302-323`). CNN inference uses exact integer equality for convolution sums, so it catches arithmetic rewrites only for integer/fixed paths (`test/spatial/tests/apps/Inference.scala:157-184`, `:713-740`).

## Gaps and fixtures needed

Current tests protect "some multiply-add-shaped program still passes", not the D-19 semantic choice. The direct tests either use exact integer/fixed reductions or compute the expected value through the same staged expression, so they do not pin a fused versus unfused floating bit pattern. The tolerance tests are intentionally broad enough that one-ulp FMA differences will pass, while checksum-only GEMM can hide elementwise cancellation.

D-19 needs deterministic paired fixtures: one `Float`, one `Half`, and one custom `FltPt` case where rounded product plus add differs from single-round fused FMA; a cancellation-to-zero/subnormal case; a dot product whose final sum crosses a tight threshold only under one mode; and a GEMM tile with elementwise asserts, not checksum asserts. Each fixture should record both `fusedExpectedBits` and `unfusedExpectedBits`, then select the expectation from the D-19 policy. Add explicit non-fusion fixtures for `FMAInLoop`'s RegFile case and explicit fusion fixtures for `RegAccumFMA`, so optimization legality and numeric precision are tested separately.
