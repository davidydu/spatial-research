# D-04 — HLS Memory-Resource Taxonomy And Catch-All Allocation

## Question

Define the HLS memory-resource taxonomy and allocator catch-all behavior.

Source: `src/spatial/targets/MemoryResource.scala:6-10`; `src/spatial/targets/HardwareTarget.scala:44-45`; `src/spatial/traversal/MemoryAllocator.scala:56-103`.

Decision criteria: User decision: define the HLS memory-resource list, min-depth rules, capacity fields, and final catch-all allocation behavior.

## Research findings

- The active allocator is simple but real. `MemoryAllocator.process` prints a visible `TODO: un-gut memory allocator`, then runs a greedy allocation pass. It treats `target.memoryResources` as an ordered list, tries every resource except the last with capacity and min-depth checks, then assigns all leftovers to the final resource without checks. See [[01-allocator-contract]].

- Current target taxonomies are not HLS-portable. Xilinx, Generic, Altera, and Euresys all share `URAM_RESOURCE`, `BRAM_RESOURCE`, `URAM_RESOURCE_OVERFLOW`, and `LUTs_RESOURCE`, with `LUTs_RESOURCE` last and `BRAM_RESOURCE` as the metadata default. ASIC and Plasticine expose a singleton `SRAM_RESOURCE`. Generic also defines an `SRAM_RESOURCE` that is not used in its `memoryResources` list. See [[02-current-target-taxonomy]].

- Capacity accounting is field-name sensitive. The allocator wraps each candidate cost as `Area(resource.name -> count)`, so any capacity-tested HLS resource must have a matching capacity field or an adapter that summarizes primitive fields into that key. DSE validates summarized area against `target.capacity`, and `MemoryReporter` exposes per-instance resource names. See [[03-capacity-reporting]].

- Allocation eligibility is narrower than "all memories." Only `SRAM`, `FIFO`, and `LIFO` are currently SRAM-able and enter capacity-tested allocation. Other local memories such as `Reg`, `FIFOReg`, `RegFile`, `LineBuffer`, `LUT`, `MergeBuffer`, `StreamIn`, and `StreamOut` receive the catch-all resource unless another path overrides them. Remote DRAM/Frame/LockDRAM are not local memory-resource candidates. See [[04-memory-eligibility]].

- The HLS taxonomy needs to separate source storage kind from backend binding. Useful HLS-facing categories include scalar/register storage, LUTRAM/distributed RAM, BRAM, URAM, HLS stream/FIFO, external DRAM/interface, and an automatic/inferred local class. Partitioning and binding pragmas should be modifiers, not separate storage classes. See [[05-hls-taxonomy]].

- Min-depth is a hard eligibility gate today. `MemoryResource.minDepth` defaults to `0`; Xilinx-style targets set primary URAM to `512`, BRAM to `32`, and overflow URAM to `32`. The checked depth is physical per-bank depth, so aggressive banking can push a large logical memory below BRAM/URAM eligibility. See [[06-min-depth-policy]].

- `defaultResource` and catch-all are different. `defaultResource` is used only when `Memory.resourceType` is unset. The allocator's fallback is `resources.last`. On Xilinx-like targets, allocated leftovers become `LUTs`, not the default `BRAM`; on singleton targets, everything becomes `SRAM`. See [[07-catch-all-semantics]].

- DSE and reports need pre-tool and post-tool manifests. Before HLS, record the compiler-requested resource, fallback status, physical bank depth, bank count, buffer depth, estimated fields, and emitted pragmas. After HLS/synthesis reports, reconcile actual BRAM/URAM/LUTRAM/register usage without rewriting the original allocation decision. See [[08-dse-report-integration]].

- HLS/DSL precedent supports explicit memory kind, explicit binding intent, and report validation. Spatial already has user-facing memory classes, banking metadata, resource classes, and capacity summaries; D-04 should preserve that layering rather than collapse everything into HLS inference. See [[09-precedent]].

- Cost analysis favors a hybrid policy. Reusing the current URAM/BRAM/LUT order is cheap but Xilinx-shaped and silently makes LUTs the fallback. A pure tool-driven policy loses early DSE feedback. A strict-capacity policy needs validated HLS estimate/report fields. The practical v1 answer is an HLS storage-class adapter backed by `MemoryResource` objects and an explicit final `AUTO` fallback. See [[10-cost-risk]].

## Recommendation

**Adopt an HLS-specific storage-resource taxonomy mapped onto `MemoryResource`, with ordered capacity-tested resources followed by an explicit `AUTO_LOCAL` catch-all.**

### HLS resource list

For v1, define these HLS storage resources:

1. `URAM`
   - Capacity field: `URAM`.
   - Default `minDepth = 512`.
   - Only enabled on targets that declare URAM capacity or explicitly allow URAM binding.

2. `BRAM`
   - Capacity field: `BRAM`.
   - Default `minDepth = 32`.
   - Primary hard-memory fallback for ordinary local arrays/FIFOs/LIFOs.

3. `URAM_OVERFLOW` (optional target feature)
   - Capacity field: `URAM`.
   - Default `minDepth = 32`.
   - Ordered after BRAM, matching the existing "use URAM for medium memories only if BRAM pressure forces it" pattern.

4. `LUTRAM`
   - Capacity field: `LUTRAM` or a target-provided summary into LUT/SLICEM fields.
   - Default `minDepth = 0`.
   - Capacity-tested when the target has a modeled LUTRAM budget; otherwise omit this resource from the checked list.

5. `AUTO_LOCAL`
   - Mandatory final list element.
   - Default `minDepth = 0`.
   - `fallback = true`.
   - Emits no hard storage binding; records "let HLS infer" as the compiler request.

`Regs`, stream endpoints, and external DRAM should be recorded in the HLS memory manifest, but they should not be smuggled into the SRAM/FIFO/LIFO allocator unless the allocator is deliberately widened. `Reg`/scalar storage is a scalarization/report category; `StreamIn`/`StreamOut` and DRAM are interface/lifetime categories.

### Allocation order

Use this default order when resources exist:

`URAM -> BRAM -> URAM_OVERFLOW -> LUTRAM -> AUTO_LOCAL`

Omit resources that the target cannot capacity-check. Always keep `AUTO_LOCAL` last.

The list order is architectural: earlier resources are capacity-tested and min-depth-tested; the final resource is the explicit catch-all. Do not reuse the current Xilinx list blindly, and do not make `LUTRAM` the mandatory final fallback.

### Min-depth policy

Use current Spatial values as defaults:

- `URAM.minDepth = 512`
- `BRAM.minDepth = 32`
- `URAM_OVERFLOW.minDepth = 32`
- `LUTRAM.minDepth = 0`
- `AUTO_LOCAL.minDepth = 0`

Make these target-configurable. Treat them as hard eligibility gates for automatic allocation to named physical resources, because an accepted assignment is also a capacity accounting claim. Explicit user bindings may override the default gate only if the backend can either emit a warning or reject unsupported bindings before HLS.

### Capacity fields

Every non-final checked resource must either:

- have `resource.name` as a capacity field, or
- provide a summary function from vendor primitive report fields into that resource name.

For HLS v1, use aggregate fields where possible:

- `URAM`
- `BRAM`
- `LUTRAM`
- `Regs`
- `DSPs`
- plus optional vendor/raw fields preserved for report reconciliation.

`Regs` and `DSPs` belong in the overall HLS capacity schema even when the memory allocator only consumes `URAM`/`BRAM`/`LUTRAM`, because DSE validity and post-HLS reconciliation need the full resource surface.

### Catch-all behavior

`AUTO_LOCAL` is the only valid final catch-all for HLS FPGA targets.

Its semantics:

1. It is assigned unconditionally to memories not placed in checked resources.
2. It does not mean free LUTRAM, BRAM, or registers.
3. It must be printed in the memory manifest and reports.
4. It must be reconciled against HLS/tool reports.
5. In exploration mode, `AUTO_LOCAL` is allowed but should warn.
6. In strict/final mode, unresolved or over-capacity `AUTO_LOCAL` after report reconciliation is an error.

Do not implement automatic spill to DRAM as the catch-all. That changes locality, latency, interface, banking, and transfer semantics; it is a separate memory hierarchy decision, not a fallback storage class.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Define `HlsStorageClass` and map each class to a `MemoryResource` adapter.
2. Add HLS target capacity schemas for `URAM`, `BRAM`, `LUTRAM`, `Regs`, and `DSPs`.
3. Require `AUTO_LOCAL` to be present and last in every HLS target's `memoryResources` list.
4. Add diagnostics for resources skipped by capacity, resources skipped by min-depth, and all `AUTO_LOCAL` assignments.
5. Emit a pre-HLS memory allocation manifest per physical memory duplicate.
6. Parse HLS/synthesis reports into normalized fields and reconcile `AUTO_LOCAL` outcomes.
7. Add allocator tests for list order, min-depth gates, missing capacity fields, capacity decrement, non-SRAM catch-all assignment, and strict-mode fallback failure.
8. Decide later whether to widen allocator eligibility beyond `SRAM`/`FIFO`/`LIFO` for `RegFile`, `LineBuffer`, `LUT`, `MergeBuffer`, or stream endpoints.
