# D-03 â€” HLS Frontend `while` / `return` Semantics

## Question

Decide whether the HLS frontend preserves Spatial unsupported `while`/`return` behavior.

Source: `src/spatial/lang/api/SpatialVirtualization.scala:103-114`.

Decision criteria: User decision: Rust frontend preserves Spatial unsupported `while`/`return` restrictions or defines new HLS-visible semantics for them.

## Research findings

- Spatial's virtualization surface is explicit: staged variables, assignment, use, and `if` are supported, while source `return`, `while`, and `do while` are hard frontend errors. `__throw` is also not staged into Spatial IR; it delegates to Forge's ordinary throw macro. The current language surface is therefore structured Spatial control, not arbitrary host-language control. See [[01-virtualization-surface]].

- The rejection is a normal compiler error path. `error(ctx, msg)` records one counted error and prints a source diagnostic; the follow-up `error(ctx)` prints source/caret context. Compilation then stops at normal staging/pass checkpoints when errors are present. A Rust frontend can preserve this behavior by rejecting these constructs before HLS emission, with better diagnostic text. See [[02-error-mechanics]].

- Spatial already has explicit alternatives for most intended loop patterns. Counted iteration is `Foreach` over `CounterChain`; forever/streaming iteration is `Stream.Foreach(*)`; state-dependent pre-test loops are `FSM(init)(notDone)(action)(next)`; local controller early exit is `breakWhen` / `stopWhen`; and staged conditionals are `IfThenElse` / `Switch`. These are not just syntax: they carry schedule, iterator, effect, and codegen structure. See [[03-supported-control-alternatives]].

- The checked-in app/test corpus does not contain working user-level `while`, `do while`, or early `return` that would be broken by preserving rejection. It does contain many FSM and `breakWhen` workarounds, especially in CHStone/MachSuite-style algorithms with data-dependent termination. That creates ergonomic pressure for future sugar, but not compatibility pressure to accept raw host loops now. See [[04-usage-survey]].

- The frontend mechanics explain why `if` is allowed and `while` / `return` are not. `if` can be staged as two typed blocks plus one `IfThenElse` result. General `while` and early return require loop-carried state, dominance/effect tracking, and conditional gating of later staged effects. Existing `VarRead` rewrite behavior assumes all statements in a block execute, which makes arbitrary loops and early exit a semantic change rather than a parser feature. See [[05-frontend-implications]].

- HLS C++ could express `while` and `return` syntactically, but that is not enough. Bounded `while` would need tripcount / bound policy and II/DSE implications. FSM-lowered `while` would need explicit loop-carried state and side-effect timing. Early `return` would need accelerator completion, stream/memory/ArgOut commit rules, simulator parity, and cleanup semantics. See [[06-hls-possibilities]].

- If Rust/HIR support is introduced, it needs a formal control-flow value model: block typing, return flag/value joins, loop-carried phi values, effect gating after possible returns, boundedness annotations, schedule selection, and parity diagnostics. `do while` has a separate first-iteration-before-test obligation. See [[07-rust-hir-semantics]].

- Preserving the restriction has good v1 properties: it keeps existing Spatial compatibility, avoids simulator/synthesis divergence, keeps unsupported constructs loud, and allows better replacement diagnostics. It does not prevent future `while` support because the frontend can later map restricted syntax to explicit controller nodes. See [[08-preserve-restriction]].

- Precedent favors structured control nodes for staged/HLS DSLs. Spatial itself admits host control only when it can become explicit IR (`IfThenElse`, controllers, FSMs, `breakWhen`). Dataflow and handshake-style control also argue against casual C-like early returns inside accelerator regions. See [[09-precedent]].

- Cost/risk analysis is decisive. Exact rejection has the lowest implementation and verification cost. Bounded `while` is the smallest possible future extension, but still requires bounds and tests. FSM-lowered `while`, early `return`, and full C-like semantics are high-risk because they can silently change side-effect ordering or simulator/HLS behavior. See [[10-cost-risk]].

## Recommendation

**Preserve Spatial's unsupported `while`, `do while`, and early `return` restrictions for the Rust/HLS v1 frontend.**

This should be an explicit language contract, not an accidental omission:

1. In HLS-visible Spatial regions, reject source `while`, `do while`, and early `return`.
2. Emit one hard frontend diagnostic with source span/caret.
3. Recommend the structured replacement:
   - `Foreach` for statically bounded counted loops.
   - `Stream.Foreach(*)` or `Accel(*)` for intentional forever/streaming loops.
   - `FSM` / `StateMachine` for data-dependent pre-test loops.
   - `breakWhen` / `stopWhen` for controller-local early termination.
   - Tail expressions / explicit accumulator values for block results instead of early return.
4. Do not lower these constructs directly to HLS C++ `while`, `do while`, or `return` in v1.
5. Treat `throw` consistently with this policy: no HLS-visible exception semantics unless a separate exception/abort design is accepted.

### Why this beats "HLS supports it, so allow it"

HLS syntax support is not the missing piece. The missing piece is Spatial semantics. A C++ `while` in emitted HLS has no automatic answer for Spatial schedule metadata, loop-carried staged variables, static bounds, pipeline II, DSE tripcount, stream back-pressure, break timing, or simulator parity. A C++ `return` has no automatic answer for accelerator completion, partially committed ArgOuts, outstanding stream transactions, memory side effects, or nested controller cleanup.

Allowing these constructs before defining those answers would create a new language, not preserve Spatial. Worse, it could produce HLS that compiles while diverging from the Rust simulator or from users' existing Spatial mental model.

### Why not add bounded `while` now?

Bounded `while` is the most plausible future extension, but it still needs a written model:

- Required max tripcount / bound annotation.
- Mapping to `Foreach` versus `FSM`.
- Zero-iteration behavior.
- Loop-carried state / phi values.
- Interaction with user II and compiler II.
- Diagnostics when bounds are unknown.
- Simulator/HLS parity tests.

The checked-in corpus already uses `FSM` and `breakWhen` for these patterns, so v1 does not need to take this risk to preserve working programs. A later D-03 follow-up can introduce "structured bounded while" as sugar over existing controllers once the model is written.

### Why not support early `return`?

Early return is higher risk than `while`. Spatial blocks are staged regions with effect summaries; returning from the middle of one would have to disable or guard all later effects. A return from inside nested controllers would also need to define which controller exits, which outputs are valid, and whether pending work is drained or abandoned. Current Spatial intentionally has controller-local `breakWhen`, not arbitrary function unwinding. For v1, reject early return and require explicit tail values or side-effecting outputs.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Add Rust/HLS frontend diagnostics for staged `while`, `do while`, and early `return`.
2. Make diagnostics replacement-oriented: suggest `Foreach`, `FSM`, `Stream.Foreach(*)`, or `breakWhen` based on the syntactic shape.
3. Document that final block expressions are allowed, but early return is not.
4. Treat HLS-visible `throw` / panic-like constructs as unsupported unless routed to an explicit accelerator-abort feature.
5. Add negative tests for `while`, `do while`, early `return`, nested early `return`, and `throw` in staged regions.
6. Add positive tests showing equivalent patterns with `Foreach`, `FSM`, and `breakWhen`.
7. Open a future extension note for bounded `while` sugar over `Foreach` or `FSM`; do not implement it under v1.
