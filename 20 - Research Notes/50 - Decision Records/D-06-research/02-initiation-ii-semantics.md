**Where the II inputs come from.**

`IterationDiffAnalyzer` runs before `InitiationAnalyzer` in the banking-analysis sequence, while `InitiationAnalyzer` is also rerun later at schedule finalization (`src/spatial/Spatial.scala:142`, `src/spatial/Spatial.scala:233`). The `iterDiff` fact itself is metadata with default value 1 and an optional-presence accessor, so `InitiationAnalyzer` can distinguish "no annotated diff" from the default getter (`src/spatial/metadata/memory/AccumulatorData.scala:82`, `src/spatial/metadata/memory/package.scala:26`). `IterationDiffAnalyzer` writes `iterDiff` onto reader, writer, and memory for affine local accumulation conflicts that have `ignoreAllConflicts`; special reduce forms hard-code `0` for touch-and-go mem-reduce and `1` for regular reduce (`src/spatial/traversal/IterationDiffAnalyzer.scala:26`, `src/spatial/traversal/IterationDiffAnalyzer.scala:115`, `src/spatial/traversal/IterationDiffAnalyzer.scala:195`, `src/spatial/traversal/IterationDiffAnalyzer.scala:201`). User II enters separately through control options: `CtrlOpt.set` writes `x.userII = ii.map(_.toDouble)`, and `Pipe.II(ii)` constructs that option (`src/spatial/lang/control/CtrlOpt.scala:18`, `src/spatial/lang/control/Control.scala:29`).

**Inner-control formula.**

For ordinary inner controls, the analyzer first computes each block's `(latency, interval)` by calling `latencyAndInterval`; that helper takes block latency as the max path latency and interval as the largest cycle length, with segmented WAR cycles summed per memory before taking the max (`src/spatial/traversal/InitiationAnalyzer.scala:25`, `src/spatial/util/modeling.scala:112`, `src/spatial/util/modeling.scala:116`, `src/spatial/util/modeling.scala:118`). `bodyLatency` is the sum of block latencies, not the max, while the raw `interval` is the max of every block interval and the slowest blackbox pipeline factor (`src/spatial/traversal/InitiationAnalyzer.scala:26`, `src/spatial/traversal/InitiationAnalyzer.scala:27`, `src/spatial/traversal/InitiationAnalyzer.scala:28`). A blackbox contributes `bboxInfo.pf`, a primitive Spatial blackbox's own `II`, or `1.0` otherwise (`src/spatial/metadata/blackbox/package.scala:10`, `src/spatial/metadata/blackbox/package.scala:13`).

The `iterDiff` part is a scalar collapse over statements: collect only statements with present `IterDiff`, sort them, set `forceII1` when the maximum annotated diff is nonpositive, and choose `minIterDiff` as the smallest positive diff (`src/spatial/traversal/InitiationAnalyzer.scala:29`, `src/spatial/traversal/InitiationAnalyzer.scala:30`, `src/spatial/traversal/InitiationAnalyzer.scala:31`). The compiler estimate is:

`compilerII = 1` if `forceII1`; otherwise `interval` if no positive diff or the smallest positive diff is 1; otherwise `ceil(interval / minIterDiff)` (`src/spatial/traversal/InitiationAnalyzer.scala:37`). The selected `II` is `bodyLatency` if the user explicitly requested `Sequenced`; otherwise it is `userII` if present; otherwise `min(compilerII, bodyLatency)` (`src/spatial/traversal/InitiationAnalyzer.scala:38`, `src/spatial/traversal/InitiationAnalyzer.scala:39`). That sequenced test uses `getUserSchedule`, not the finalized compiler/default schedule (`src/spatial/metadata/control/package.scala:796`).

**Outer and FSM cases.**

Outer controls do not reuse `latencyAndInterval` or `iterDiff`; they visit children and set interval to `max(1.0, child.II...)`, then choose `userII` or that interval and record the same value as `compilerII` (`src/spatial/traversal/InitiationAnalyzer.scala:14`, `src/spatial/traversal/InitiationAnalyzer.scala:17`, `src/spatial/traversal/InitiationAnalyzer.scala:19`). Inner FSMs are special: they compute not-done/next-state latencies and action cycles, detect action writes read by next state, and raise interval to action latency when such state dependencies exist (`src/spatial/traversal/InitiationAnalyzer.scala:55`, `src/spatial/traversal/InitiationAnalyzer.scala:59`, `src/spatial/traversal/InitiationAnalyzer.scala:73`, `src/spatial/traversal/InitiationAnalyzer.scala:75`). These FSM branches assign `II` and `bodyLatency`, but do not set `compilerII` in the shown code path (`src/spatial/traversal/InitiationAnalyzer.scala:79`, `src/spatial/traversal/InitiationAnalyzer.scala:80`, `src/spatial/traversal/InitiationAnalyzer.scala:90`, `src/spatial/traversal/InitiationAnalyzer.scala:91`). For inner switches, the parent switch's `bodyLatency` and `II` are copied to child cases (`src/spatial/traversal/InitiationAnalyzer.scala:93`, `src/spatial/traversal/InitiationAnalyzer.scala:95`).

**Metadata semantics and consumers.**

`BodyLatency`, `InitiationInterval`, `UserII`, and `CompilerII` are distinct metadata records; `UserII` is `SetBy.User`, while the others here are analysis-owned (`src/spatial/metadata/control/ControlData.scala:144`, `src/spatial/metadata/control/ControlData.scala:153`, `src/spatial/metadata/control/ControlData.scala:162`, `src/spatial/metadata/control/ControlData.scala:170`). Accessors default `bodyLatency` to `Nil`, `II` and `compilerII` to `1.0`, and `userII` to `None` (`src/spatial/metadata/control/package.scala:826`, `src/spatial/metadata/control/package.scala:832`, `src/spatial/metadata/control/package.scala:835`, `src/spatial/metadata/control/package.scala:838`). Schedules are separate: `Sequenced` is one `CtrlSchedule`, user schedules are stored in `UserScheduleDirective`, and flow rules default UnitPipe/Accel to Sequenced while most other controllers default to Pipelined, with later legality overrides (`src/spatial/metadata/control/ControlData.scala:7`, `src/spatial/metadata/control/ControlData.scala:67`, `src/spatial/flows/SpatialFlowRules.scala:321`, `src/spatial/flows/SpatialFlowRules.scala:350`, `src/spatial/flows/SpatialFlowRules.scala:364`).

Downstream consumers use the chosen `II` and `bodyLatency`, not the original dependence edge. Chisel controller generation passes `bodyLatency.sum` and `II` into the state-machine module and `IICounter`, forcing II to 1 for outer controls or when retiming is disabled (`src/spatial/codegen/chiselgen/ChiselGenController.scala:337`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:338`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:357`). Stream pressure logic delays FIFO/full-empty checks by `sym.s.get.II - 1` (`src/spatial/codegen/chiselgen/ChiselGenCommon.scala:252`, `src/spatial/codegen/chiselgen/ChiselGenCommon.scala:270`). Runtime models serialize latency and II into `ControllerModel`, the tree report highlights `compilerII != II`, and the Scala executor gates inner-pipeline launches by `ctrl.II` (`src/spatial/model/RuntimeModelGenerator.scala:282`, `src/spatial/model/RuntimeModelGenerator.scala:284`, `src/spatial/codegen/treegen/TreeGen.scala:126`, `src/spatial/codegen/treegen/TreeGen.scala:129`, `src/spatial/executor/scala/ControlExecutor.scala:307`, `src/spatial/executor/scala/ControlExecutor.scala:330`).

**D-06 implication.**

For HLS, the key local fact is that Spatial's `iterDiff` has already been collapsed into one positive minimum, or a nonpositive force-to-one case, before downstream scheduling/codegen sees it (`src/spatial/traversal/InitiationAnalyzer.scala:29`, `src/spatial/traversal/InitiationAnalyzer.scala:37`). Reusing it as the HLS scheduler authority would lose edge identity, memory/port context, and backend report feedback. HLS tools may choose an achieved II different from a requested pragma because of dependence, resource, or memory-port constraints (unverified). Best fit for D-06: keep Spatial `iterDiff` as diagnostic input to "Spatial expected II" and mismatch reporting, but replace it as the HLS scheduling authority with an HLS-facing dependence/schedule model.
