# D-06 cost and risk analysis

## Cost ranking

The cheapest policy is **diagnostic-only reuse**: keep Spatial-style `iterDiff` as an explanatory lint, but do not let it decide HLS legality. It can reuse the existing warning shape for unknown bounds, loose mode, and explicit user II (`src/spatial/traversal/IterationDiffAnalyzer.scala:37-56`) while preserving the `userII`/`compilerII`/`II` distinction already encoded in metadata (`src/spatial/metadata/control/ControlData.scala:153-177`). The next cheapest useful policy is the **hybrid requested-II plus backend-report authority**: compute a requested II with the Spatial heuristic, submit it to HLS, then treat tool-accepted II as authoritative for final reports. Exact reuse as scheduler authority and a full HLS-specific dependence/schedule model are both high-cost, but for different reasons: exact reuse must port many Spatial-specific assumptions, while replacement must build new dependence/resource/report plumbing.

## Exact reuse as authority

Exact reuse has the highest correctness risk because `IterationDiffAnalyzer` is not a general HLS scheduler. It only considers accumulation cycles where the memory is local, the reader and writer differ, conflicts have been waived, and both sides already have affine matrices (`src/spatial/traversal/IterationDiffAnalyzer.scala:21-29`; `src/spatial/metadata/memory/package.scala:335-346`). Those matrices come from access analysis and lane expansion: access analysis writes `affineMatrices`, and expansion enumerates `multiLoop(ps)`, scales iterator coefficients by `ctrParOr1`, and injects lane constants (`src/spatial/traversal/AccessAnalyzer.scala:223-225`; `src/spatial/traversal/AccessExpansion.scala:74-124`). Reusing the result as an HLS authority means either reproducing those contracts exactly or accepting scheduler decisions based on partial evidence.

The fragile cases are visible in comments and fallbacks. Unknown counter bounds use `Expect(start/end/step)` when possible, but otherwise warn, assume a huge value under `--looseIterDiffs`, defer to `userII`, or conservatively return `(1, step*p)` (`src/spatial/traversal/IterationDiffAnalyzer.scala:37-56`; `src/spatial/Spatial.scala:440-442`). The pass also marks unresolved assumptions about writer-leading-reader cases, different control hierarchy levels, and lane-distance ordering (`src/spatial/traversal/IterationDiffAnalyzer.scala:36`, `src/spatial/traversal/IterationDiffAnalyzer.scala:63-71`). If HLS scheduling treats this as truth, these warnings become potential miscompiles rather than diagnostics.

## Consumer blast radius

The implementation cost is not just the analyzer. `InitiationAnalyzer` collects statement `iterDiff`s, forces II=1 for nonpositive distances, computes `compilerII = ceil(interval/minIterDiff)`, then applies sequenced and `userII` overrides to final `II` (`src/spatial/traversal/InitiationAnalyzer.scala:29-40`). User `.II(...)` is source-level metadata via `CtrlOpt.set` (`src/spatial/lang/control/CtrlOpt.scala:18-21`), so an HLS port must decide whether it is a request, a hard constraint, or only a diagnostic.

`segmentMapping` is equally load-bearing. The analyzer writes lane-to-segment mappings when later read lanes depend on earlier write lanes (`src/spatial/traversal/IterationDiffAnalyzer.scala:119-168`), and the metadata explicitly means later lanes depend on earlier lanes (`src/spatial/metadata/memory/AccumulatorData.scala:89-93`). Banking groups accesses only when same-port conflicts do not overlap in the same segment, and it errors if group-unroll sees inter-segment dependencies (`src/spatial/traversal/banking/MemoryConfigurer.scala:351-362`, `src/spatial/traversal/banking/MemoryConfigurer.scala:370-373`). Unrolling fractures accesses by segment, writes the segment on replacement accesses, and copies `iterDiff` forward (`src/spatial/transform/unrolling/MemoryUnrolling.scala:240-255`, `src/spatial/transform/unrolling/MemoryUnrolling.scala:301-309`). The latency model also sums segmented WAR cycles and pushes later-segment reads by a hard-coded SRAM load latency (`src/spatial/util/modeling.scala:116-124`, `src/spatial/util/modeling.scala:433-453`). Exact reuse therefore inherits banking, unrolling, and timing obligations.

## Replacement and hybrid risks

An **HLS-specific dependence/schedule model** has high initial cost but the cleanest long-term semantics. It should model HLS-visible memory resources, dependence distances, pipeline pragmas, operator latencies, partition choices, and backend-reported II (unverified). That overlaps directly with D-07 banking-search/partition planning, D-08 latency source for HLS DSE, and D-21 accepted-II versus compiler-II metadata (`20 - Research Notes/40 - Decision Queue.md:35-41`, `20 - Research Notes/40 - Decision Queue.md:91-93`). It must also update runtime/DSE consumers: runtime-model generation serializes `lhs.II` into controller models, and cycle formulas use `II` for inner-control runtime (`src/spatial/model/RuntimeModelGenerator.scala:261-284`; `models/src/models/RuntimeModel.scala:192-200`, `models/src/models/RuntimeModel.scala:334-337`).

The **hybrid** is the best cost/risk staging point. Keep Spatial `iterDiff` to compute `spatial_compiler_ii`/`requested_ii`, preserve diagnostics for unknown bounds and `userII`, but make parsed HLS accepted II final for HLS performance and legality (report parsing is tool-specific, unverified). This avoids treating `ceil(interval/iterDiff)` as an oracle while retaining useful compatibility evidence in reports, similar to how TreeGen already highlights `II` versus `CompilerII` mismatches (`src/spatial/codegen/treegen/TreeGen.scala:126-131`). The residual risk is reconciliation: Chisel pressure and reports delay by final `II`, runtime/DSE consume final `II`, and D-21 must define exactly which number each consumer reads (`src/spatial/codegen/chiselgen/ChiselGenCommon.scala:248-263`; `src/spatial/model/RuntimeModelGenerator.scala:261-284`).
