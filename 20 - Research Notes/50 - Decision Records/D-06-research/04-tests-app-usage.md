# D-06 research angle 4: tests/app usage

## Scope

The test/app surface has only one literal `iterDiff` expectation, but many behaviors that depend on iteration-distance reasoning. The named `userII` metadata is source-level: `CtrlOpt.set` maps a DSL `.II(...)` option into `x.userII`, and `UserII` is defined as user-set controller metadata (`src/spatial/lang/control/CtrlOpt.scala:18`, `src/spatial/lang/control/CtrlOpt.scala:21`, `src/spatial/metadata/control/ControlData.scala:162`, `src/spatial/metadata/control/ControlData.scala:168`). In tests/apps, the exposed form is `.II(n)`, e.g. `ExplicitII` calls this the "user syntax for explicitly setting II" and uses `Pipe.II(1).Foreach` (`test/spatial/tests/feature/control/ExplicitII.scala:5`, `test/spatial/tests/feature/control/ExplicitII.scala:18`).

## Compiler Contracts

`FlatAffineAccess` is the hardest direct II assertion: after affine loop nests with `par 4`, `checkIR` collects `UnrolledForeach` nodes named `LOOP1D`, `LOOP2D`, and `LOOP3D`, then asserts every computed `x.II` is `1.0` (`test/spatial/tests/compiler/InitiationIntervals.scala:38`, `test/spatial/tests/compiler/InitiationIntervals.scala:44`, `test/spatial/tests/compiler/InitiationIntervals.scala:81`, `test/spatial/tests/compiler/InitiationIntervals.scala:83`). That is a diagnostic/scheduler parity expectation, not just functional correctness; an HLS replacement that computes different II while producing correct values would still fail this Spatial test shape.

`IterationDiffs` is broader but more indirect. Its helper writes `sram(i) = sram(i + rhsoffset) * X + X * numoffset` under configurable start/stop/step/par, and comments label the expected II as body latency, body latency divided by distance, or single-lane latency with segmentation (`test/spatial/tests/compiler/InitiationIntervals.scala:121`, `test/spatial/tests/compiler/InitiationIntervals.scala:124`, `test/spatial/tests/compiler/InitiationIntervals.scala:159`, `test/spatial/tests/compiler/InitiationIntervals.scala:165`, `test/spatial/tests/compiler/InitiationIntervals.scala:177`, `test/spatial/tests/compiler/InitiationIntervals.scala:190`). The test finally compares matrix rows and scalar reductions against gold values (`test/spatial/tests/compiler/InitiationIntervals.scala:252`, `test/spatial/tests/compiler/InitiationIntervals.scala:285`, `test/spatial/tests/compiler/InitiationIntervals.scala:287`). This implies correctness dependence on respecting loop-carried read/write distance, with comments as diagnostic expectations.

## IterDiff Edge Cases

`LoopInvariantCycle` gives the most explicit app-facing clue: a loop over `r,c,ib,b` says Spatial "Should choose iterDiff = Some(1) (maximally conservative)" and also says that if II of 1 is desired, `ib` must be moved outward (`test/spatial/tests/compiler/InitiationIntervals.scala:325`, `test/spatial/tests/compiler/InitiationIntervals.scala:326`). The update reads and writes `y(r,c,b)` across `ib` and `i`, then checks a tensor gold result (`test/spatial/tests/compiler/InitiationIntervals.scala:327`, `test/spatial/tests/compiler/InitiationIntervals.scala:330`, `test/spatial/tests/compiler/InitiationIntervals.scala:356`). `Blur` is similarly labeled "tricky" for the analyzer; its inner stencil says II is 1 at `par = 1` and increases for higher par, then asserts a gold row (`test/spatial/tests/compiler/InitiationIntervals.scala:403`, `test/spatial/tests/compiler/InitiationIntervals.scala:416`, `test/spatial/tests/compiler/InitiationIntervals.scala:440`). These are not just legacy comments: the computations are written so a too-aggressive scheduler can violate producer/consumer ordering.

## Reductions, Accumulations, Par Lanes

Reduction tests separate semantic correctness from Spatial-specific specialization. `AccumReplacements` enumerates valid and invalid accumulation idioms, then asserts both numerical results and IR counts of `RegWrite` versus `RegAccumOp` (`test/spatial/tests/compiler/AccumReplacements.scala:46`, `test/spatial/tests/compiler/AccumReplacements.scala:64`, `test/spatial/tests/compiler/AccumReplacements.scala:239`, `test/spatial/tests/compiler/AccumReplacements.scala:257`, `test/spatial/tests/compiler/AccumReplacements.scala:275`, `test/spatial/tests/compiler/AccumReplacements.scala:338`, `test/spatial/tests/compiler/AccumReplacements.scala:359`, `test/spatial/tests/compiler/AccumReplacements.scala:360`). For HLS parity, the values are correctness dependencies; the exact `RegAccumOp` count looks like legacy Spatial IR behavior unless the HLS path promises identical diagnostics.

`ParLaneMask` is a stronger correctness case for lane valids. It documents a prior bug where child metapipe enables were derived from stage-0 counters, then uses odd `x`, nested `Foreach(... par 2)`, `Reduce(... par 4)`, and `MemReduce(... par 2)`, finally asserting scalar and matrix golds (`test/spatial/tests/feature/control/ParLaneMask.scala:9`, `test/spatial/tests/feature/control/ParLaneMask.scala:21`, `test/spatial/tests/feature/control/ParLaneMask.scala:26`, `test/spatial/tests/feature/control/ParLaneMask.scala:27`, `test/spatial/tests/feature/control/ParLaneMask.scala:41`, `test/spatial/tests/feature/control/ParLaneMask.scala:60`, `test/spatial/tests/feature/control/ParLaneMask.scala:62`). HLS parity needs per-lane masks and reduction tails, not merely total loop trip counts.

## Application Signals

Applications use `.II` mostly as a performance/user-intent hint. `FilterStream1D` explicitly says II=1 is forced to squeeze a shift-register write/read into one cycle (`test/spatial/tests/feature/streaming/FilterStream1D.scala:87`, `test/spatial/tests/feature/streaming/FilterStream1D.scala:195`), `DigitRecognition` combines `par_factor = 40` with `Pipe.II(1)` over the training loop (`test/spatial/tests/Rosetta/DigitRecognition.scala:35`, `test/spatial/tests/Rosetta/DigitRecognition.scala:363`), and MachSuite merge sort requests `Pipe.II(6)` for FIFO merge logic (`test/spatial/tests/apps/MachSuite.scala:1761`). By contrast, Smith-Waterman/Needleman-Wunsch-style apps encode true dependencies with row parallelism and delayed row starts: "ensure a later row does not scan an element until previous row has populated it" (`test/spatial/tests/apps/MachSuite.scala:866`, `test/spatial/tests/apps/MachSuite.scala:868`, `test/spatial/tests/apps/MachSuite.scala:872`; also `test/spatial/tests/apps/SW.scala:97`, `test/spatial/tests/apps/SW.scala:100`, `test/spatial/tests/apps/SW.scala:105`). Matrix multiply, SPMV, and MLP use parameterized `MemReduce`/`Reduce` lanes as throughput structure (`test/spatial/tests/apps/MachSuite.scala:1655`, `test/spatial/tests/apps/MachSuite.scala:1663`, `test/spatial/tests/apps/MachSuite.scala:2081`, `test/spatial/tests/apps/MachSuite.scala:2390`, `test/spatial/tests/apps/MachSuite.scala:2434`).

Implication: HLS should not blindly reuse Spatial `iterDiff` as the whole scheduler model. It should preserve its correctness lessons for loop-carried accumulation distances, par-lane masks, and conservative unknown-distance cases; preserve user `.II` as a constraint or explicit diagnostic; and treat exact Spatial IR specialization counts as legacy diagnostic parity rather than semantic parity.
