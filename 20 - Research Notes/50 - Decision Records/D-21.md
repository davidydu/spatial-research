# D-21 - HLS Accepted-II Reconciliation Contract

## Question

Define the contract for HLS tool-accepted II versus compiler II metadata.

Source: `src/spatial/traversal/InitiationAnalyzer.scala:14-41`; [[C0 - Retiming]]

Decision criteria: User decision: define where the flow records and reconciles HLS tool-accepted II versus Spatial `compilerII`/requested II.

## Research findings

- Spatial already has three pre-HLS II meanings. `userII` records explicit DSL intent, `compilerII` records the initiation-analysis estimate, and selected `II` is the interval consumed by codegen, runtime models, and simulators. `StateMachine` is a migration hazard because the special branch writes selected `II` and `bodyLatency` but not `compilerII`. See [[01-source-metadata-initiation-analyzer]].

- Accepted II is a post-tool fact. HLS reports should be normalized into a schedule/synthesis result carrying accepted II, latency, resource context, schedule warnings, report source, parse status, and failure reason. Co-simulation can validate behavior and triage mismatches, but it should not be the ordinary accepted-II source. See [[02-hls-report-sources]].

- Existing consumers mostly read selected `II`, not `compilerII`, and none read an HLS-accepted value. Chisel controller generation, stream pressure logic, runtime models, DSE, and the Scala executor all depend on selected `II`; TreeGen is the clearest existing mismatch UI because it prints `CompilerII` when it differs from `II`. See [[03-current-consumers]].

- D-21 is the timing ledger that D-08 and D-20 need. D-08 can use reports as post-tool latency authority only if accepted II remains separate from estimates. D-20 cycle-aware delay-line traces can only be meaningful when they declare whether they assumed compiler II, user request, estimated HLS II, or parsed accepted II. See [[04-overlap-d08-d20]].

- The proposed schema is a per-loop/controller schedule record, not one more scalar on `Control`. It needs stable compiler identity plus backend aliases, then fields for `user_requested_ii`, `compiler_ii`, `selected_ii`, `hls_requested_ii`, `hls_accepted_ii`, `latency_source`, `schedule_status`, and diagnostics. See [[05-ii-reconciliation-schema]].

- The main risk is false authority. Compiler-II-only preserves migration comfort while lying after a tool relaxes or improves a schedule. HLS-overwrites catches final reports but erases why the compiler emitted a request. Backend-default/no-record makes user `.II` violations and DSE regressions invisible. See [[06-risk-failure-modes]].

- Tests should distinguish compiler fixtures from report fixtures. Existing tests assert selected II or final values; Rust/HLS needs fixtures for explicit user II, compiler-estimated II, emitted request II, matched accepted II, accepted-higher, accepted-lower, missing report, unparseable report, and ambiguous loop mapping. See [[07-tests-validation-fixtures]].

- Migration cost is medium-high but bounded. Add sidecar schedule records first, emit stable loop labels and requested II next, parse reports into accepted/status fields without changing old scalar consumers, then update DSE and timing summaries to prefer report-backed values. See [[08-implementation-cost-migration]].

- Backend policy should emit a requested II pragma when the compiler can produce a valid request, parse accepted II after tool runs, warn by default on mismatch, and fail only under strict policy or hard report/tool errors. Backend-default scheduling is an escape hatch, not the normal path. See [[09-hls-backend-policy-options]].

- The recommendation matrix decisively selects dual-record reconciliation. It is the only policy that scores well on correctness, provenance, DSE compatibility, cycle-model compatibility, diagnostics, and future backend flexibility. See [[10-recommendation-matrix]].

## Recommendation

Adopt **Dual-Record II Contract With Report-Backed Reconciliation**.

The Rust/HLS rewrite should introduce a per-loop/controller schedule ledger with four separate II claims:

1. `user_requested_ii`: optional DSL request from `.II(n)`.
2. `spatial_compiler_ii`: Spatial-style initiation-analysis estimate.
3. `selected_ii`: the pre-HLS effective interval used for Spatial compatibility, compiler diagnostics, and legacy model parity.
4. `hls_requested_ii` and `hls_accepted_ii`: the emitted HLS request and the parsed post-tool result.

These fields must not overwrite each other. Before an HLS run, `hls_accepted_ii` is unknown and consumers may use selected/compiler/requested II only under a labelled pre-report source such as `spatial_parity_fallback` or `hls_estimate`. After report parsing succeeds, `hls_accepted_ii` becomes post-tool evidence for timing summaries, DSE calibration, and artifact diagnostics. It still does not mutate `compilerII` or selected `II`; those remain the causal record explaining what the compiler believed and what it asked the tool to do.

Every record should be keyed by stable compiler identity rather than report strings alone. Use a deterministic `controller_uid` derived from kernel, IR symbol, parent path, block index, source context, and sibling ordinal. Store backend aliases separately: emitted loop label, report loop id, report path, tool name/version, target profile, and parser version. This is necessary because HLS tools can rename, inline, flatten, or duplicate loops, and a report-backed accepted II is only meaningful if it maps back to the Spatial controller that produced the request.

The reconciliation result should be explicit:

- `not_run`
- `request_only`
- `matched`
- `accepted_higher`
- `accepted_lower`
- `tool_rejected`
- `report_missing`
- `report_unparseable`
- `loop_mapping_ambiguous`
- `stale_report`

Default mismatch behavior should be diagnostic, not fatal. If the user requested II=1 and the tool accepted II=2, the flow should record `accepted_higher`, surface a warning, and make report-backed DSE/timing use the accepted value. It should fail only when the user or CI selects a strict-II mode, when a hard contract is violated, when synthesis is rejected, or when the run requires a final report and no trustworthy report can be parsed.

This decision composes with earlier records. D-08 gets a clean source hierarchy: fast estimates before reports, parsed accepted II and report latency after tools run, simulation only for validation and triage. D-20 gets a named II source for any cycle-aware delay-line trace. D-06 keeps `IterationDiffAnalyzer` as request/explanation evidence rather than final backend authority.

### Rejected alternatives

Reject **compiler-II-only authority**. It preserves current scalar consumers but converts a Spatial estimate into HLS truth. That will over-rank designs when the tool relaxes II, under-rank designs when the tool improves II, and hide user-II violations.

Reject **HLS-accepted-overwrites**. Accepted II is the right post-tool fact, but overwriting `compilerII` or selected `II` destroys the evidence trail. A higher accepted II should become `accepted_higher`, not a silent mutation of compiler metadata.

Reject **backend-native/no-contract**. Letting the tool choose without recording request and acceptance avoids schema work by removing the contract D-21 exists to define. It leaves DSE, diagnostics, simulator traces, and cross-backend comparison without a shared schedule ledger.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Define the schedule-ledger schema with stable `controller_uid`, backend aliases, requested/compiler/selected/accepted II fields, latency source, schedule status, report provenance, and diagnostics.
2. Preserve existing Spatial scalar meanings in compatibility layers; do not repurpose `compilerII` as accepted II.
3. Emit `hls_requested_ii` from explicit user request when valid, otherwise from the selected compiler policy, and record request source/status.
4. Add HLS report parsers that populate `hls_accepted_ii`, latency, report source, tool version, parse status, and failure reason.
5. Add loop/controller mapping manifests and stale-report detection keyed by design point, backend, target profile, partition plan, and tool version.
6. Update DSE and timing summaries to prefer report-backed accepted II when available while preserving labelled pre-report estimates.
7. Add diagnostics for accepted-higher/lower, missing report, unparseable report, ambiguous loop mapping, tool rejection, and strict-II failures.
8. Add fixtures for selected-II parity, explicit user II, compiler estimate mismatch, matched accepted II, accepted-higher, accepted-lower, missing/unparseable report, and stale report.
9. Require D-20 cycle-debug traces and any future queue/backpressure trace to declare the II source used.
