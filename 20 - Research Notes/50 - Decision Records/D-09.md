# D-09 — Rust Simulator Unbiased-Rounding Determinism

## Question

Choose deterministic or nondeterministic unbiased rounding for the Rust simulator.

Source: `spatial/emul/src/emul/FixedPoint.scala:232-241`; `spatial/src/spatial/codegen/scalagen/ScalaGenFixPt.scala:111-114`.

Decision criteria: User decision: unbiased rounding matches JVM RNG nondeterminism, uses a seedable deterministic RNG, or becomes deterministic rounding.

## Research findings

- The current algorithm is stochastic and seedless at the call site. `FixedPoint.unbiased` receives a value with four guard fractional bits, computes `biased = bits >> 4`, extracts a 1/16-granularity remainder, draws `scala.util.Random.nextFloat()`, and rounds farther from zero when `rand + remainder >= 1`; it then either wraps or saturates. `*&`, `/&`, `*&!`, `/&!`, and unbiased fixed casts reach this path, though constant rewrites can bypass it. See [[01-current-algorithm-call-surface]].

- Scalagen and the Scala executor are the only complete host-reference paths. Scalagen emits unbiased arithmetic and casts through emul; the executor also calls emul for unbiased casts and generic fixed binary nodes. Cppgen and Roguegen either lack unbiased arithmetic cases or lower unbiased casts as plain casts/saturation, while PIR preserves node identity and resourcegen only counts fixed ops. Chisel/Fringe already treats `Unbiased` as PRNG/LFSR stochastic rounding rather than deterministic arithmetic rounding. See [[02-codegen-backend-surface]].

- The specs currently say Scalagen plus `emul` is numeric ground truth, but they also acknowledge this exact policy is unresolved. D-09 is framed as a Rust simulator choice, while D-17/Q-144 repeats the same alternatives as the canonical unbiased-rounding policy. D-09 should therefore decide simulator replay/seed behavior now and leave any global semantic redefinition of `Unbiased` to D-17. See [[03-spec-open-question-survey]].

- Existing checked usage does not depend on a JVM RNG sequence. `SpecialMath` tests stochastic multiplication by averaging 256 samples and comparing the mean against the exact product within one LSB; saturating unbiased multiplication is checked through endpoint equality. Application tests use `*&!` or `/&` in convergence-style checks rather than exact bit sequences. Coverage gaps remain for division, saturating division, and unbiased casts. See [[04-tests-app-usage]].

- HLS/hardware implications argue against JVM-sequence compatibility. Current Fringe hardware uses seeded PRNG/LFSR-like logic for `Unbiased`, and synthesized HLS cannot depend on ambient JVM randomness. Deterministic rounding would be cheaper for HLS, but it would replace Spatial's stochastic intent rather than simply make it reproducible. See [[05-hls-hardware-implications]].

- A seedable Rust simulator policy should use explicit seed, policy, and algorithm fields. The most robust design derives per-event thresholds from stable operation/dynamic/lane identity rather than consuming one shared sequential stream, because a stream can become order-, thread-, or scheduling-sensitive. Sequential legacy-stream mode may still be useful for debugging, but should not be the reference mode. See [[06-seedable-rng-policy]].

- Deterministic arithmetic alternatives are real semantic changes. Round-to-nearest-even would be the cleanest deterministic replacement if D-17 chooses to redefine unbiased rounding, but it would not preserve current stochastic behavior. Truncation or nearest-away variants are simpler but discard the mean-preserving intent tested today. Stable hashed threshold rounding preserves the probability law while making replay deterministic. See [[07-deterministic-rounding-alternatives]].

- Reproducibility and CI favor seeded stochastic rounding. JVM-like nondeterminism makes failures hard to replay and can freeze incidental traversal order into semantics. Deterministic arithmetic gives perfect replay but changes the contract. Seeded stable stochastic rounding preserves the current probability rule while giving fixed-seed tests and explicit auto-seed/fuzz modes. See [[08-repro-debug-test-risk]].

- Migration cost analysis favors a stable per-site seeded design. Exact JVM nondeterminism has high compatibility cost and poor HLS fit. Sequential RNG is simpler but fragile under reordering. Stable per-operation/per-site derivation costs more metadata but gives better replay, stronger tests, and a clean manifest/config story. See [[09-cost-migration-matrix]].

- The final recommendation matrix names the preferred D-09 answer: **Seeded Stable Stochastic Rounding**. It gives statistical Spatial parity with controlled replay, no promise of JVM global RNG sequence compatibility, and a clear handoff to D-17 for any future canonical redefinition. See [[10-recommendation-matrix-d17-handoff]].

## Recommendation

**Adopt Seeded Stable Stochastic Rounding for the Rust simulator.**

Rust should preserve the stochastic unbiased-rounding probability law, including the four-guard-bit threshold structure and saturating-versus-wrapping split, but it should not attempt to reproduce the JVM process-global `scala.util.Random` sequence. The compatibility label should be:

`rng_compatibility = statistical-spatial, not-jvm-sequence`

This keeps Rust simulator behavior reproducible for CI and debugging, while still matching what existing tests and current hardware intent actually exercise: stochastic rounding, not ambient host nondeterminism.

### Simulator contract

Add an explicit Rust simulator rounding context with at least:

1. `sim_rng_policy`
   - `seeded_stable` by default for reference/CI.
   - `legacy_stream` as a debugging mode if useful.
   - `auto_seeded` for fuzz/statistical runs, with the resolved seed printed and recorded.
   - deterministic modes only as named non-default policies pending D-17.

2. `sim_rng_seed`
   - A user-specified `u64`.
   - Default fixed value for CI and reproducible local runs.

3. `sim_rng_algorithm`
   - Versioned, for example `spatial-unbiased-v1`.
   - Included in run metadata so future algorithm changes are explicit.

4. `signed_negative_behavior`
   - For D-09 v1, preserve current legacy behavior: when the stochastic threshold fires and the shifted base is negative, step to `biased - 1`.
   - D-17 may later decide whether that behavior is canonical or should be replaced by a mathematically symmetric candidate rule.

In `seeded_stable` mode, derive each rounding threshold from stable event identity, not from a shared mutable global stream. A reasonable identity tuple is:

`(seed, algorithm_version, static_op_id, dynamic_instance_id, lane_or_element_id)`

The exact encoding can be an implementation detail, but the contract should say that changing simulator thread count or evaluation order must not change results when the IR identity and inputs are the same.

### What D-09 decides vs D-17

D-09 decides the Rust simulator should be reproducible and seed-controlled while preserving stochastic Spatial behavior. It does not decide that `Unbiased` is permanently stochastic across all backends.

D-17 should later decide the canonical language-wide policy:

- stochastic rounding with standardized seed/event derivation across Scala, Rust, and HLS,
- seedable-but-stochastic Rust simulator with HLS-specific stochastic hardware,
- deterministic round-to-nearest-even or another deterministic replacement,
- deprecation or renaming if "unbiased" no longer means stochastic rounding.

If D-17 changes the canonical policy, it should update the numeric spec, Rust simulator, JVM `emul`, Scalagen tests, and HLS lowering together. D-09's seeded design should be treated as the safe v1 simulator policy, not as a unilateral language redefinition.

### Rejected alternatives

Reject **JVM RNG sequence compatibility**. It preserves the wrong thing: an implicit process-global Scala RNG stream, not user-visible semantics. It also makes CI failures difficult to replay and does not map cleanly to HLS/hardware.

Reject **ambient nondeterminism as the Rust default**. A simulator reference should be replayable. Random/fuzz modes are useful only if they record the resolved seed and policy.

Reject **deterministic arithmetic rounding as a Rust-only shortcut**. Round-to-nearest-even may be a good D-17 canonical replacement, but choosing it only in Rust would split reference semantics from Scalagen/emul and current stochastic hardware intent.

Reject **truncation or nearest-away defaults**. They are cheaper, but they are not statistically unbiased and would undermine existing mean-based tests.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:
1. Define Rust simulator RNG metadata: `sim_rng_policy`, `sim_rng_seed`, `sim_rng_algorithm`, `rng_compatibility`, and `signed_negative_behavior`.
2. Implement `seeded_stable` threshold generation from stable unbiased-rounding event IDs.
3. Preserve current `FixedPoint.unbiased` guard-bit arithmetic, clamped/saturating split, divide-by-zero invalid propagation, and legacy negative threshold behavior in v1.
4. Emit or store stable IDs for unbiased arithmetic and cast sites in generated Rust.
5. Print and persist resolved seed/policy/algorithm metadata in simulator runs and mismatch reports.
6. Add tests for exact replay with fixed seed, changed-output allowance with changed seed, and order/thread-count independence.
7. Add property/statistical tests for `*&`, `/&`, `*&!`, `/&!`, `FixToFixUnb`, and `FixToFixUnbSat`.
8. Keep `legacy_stream` and deterministic policies as explicit non-default modes, not silent behavior.
9. Cross-reference D-17 so canonical rounding policy can later ratify or replace this simulator policy.
10. If D-17 chooses deterministic rounding, migrate Rust, Scala emul, Scalagen tests, and HLS lowering together rather than leaving D-09 as a Rust-only divergence.
