# D-13 — Pipe Holder Observability In HLS

## Question

Choose whether HLS lowering preserves pipe holders as observable state.

Source: `src/spatial/transform/PipeInserter.scala:170-209`; `src/spatial/transform/PipeInserter.scala:256-305`; [[50 - Pipe Insertion]].

Decision criteria: User decision: HLS lowering must preserve `Reg`/`FIFOReg`/`Var` pipe holders as observable state or may optimize them to SSA where legal.

## Research findings

- `PipeInserter` does not lower escaping inner-stage values as ordinary SSA edges. It emits a generated `Pipe`, writes each escaping value to a holder inside the pipe, reads the holder afterward, and substitutes downstream uses to that read. Holder choice is deliberate: bit values under stream control become `FIFOReg`, other bit values become `Reg`, and non-bit values become `Var`. See [[01-pipe-inserter-mechanics]].

- The holder nodes have real effects in the current IR and backends. `Reg` is mutable single-slot state with read, write, reset, and init behavior. `FIFOReg` is a one-element queue-like holder with enqueue/dequeue, destructive read, full/empty/valid state, and pressure participation. `Var` is mutable Argon state with assignment/read effects and a local rewrite rule, even though it is weaker as hardware state. See [[02-holder-node-semantics-codegen]].

- Downstream compiler passes consume these holders as stateful nodes. Access analysis, memory analysis, unrolling, streamify, FIFO initialization, modeling, resource reporting, naming, sanity checks, Scalagen, and Chiselgen all observe `Reg`, `FIFOReg`, or `Var` opcodes or their effects. Replacing holders with SSA before those consumers would remove memory identity and effect-ordering anchors. See [[03-downstream-consumers]].

- Corpus evidence is indirect but important. Tests use FIFOReg as an ordering token between pipes, and comments in dynamic/stream tests describe FIFOReg-style timing and dependency hazards. `ParallelPipeInsertion` and `PropUses` exercise values crossing primitive/control boundaries, although not all are strong passing goldens. See [[04-tests-apps-evidence]].

- HLS offers better physical forms than literal Spatial holders, but only after preserving the semantic boundary. Private one-write/one-read scalar escapes may become SSA or local registers. Stream-controlled `FIFOReg` escapes map more naturally to one-token channels/FIFOs or dataflow channels. Delay-line-like cases may map to shift registers, but that is not generic FIFOReg semantics. See [[05-hls-alternatives-optimization]].

- The written spec already exposes the ambiguity. The Pipe Insertion entry documents explicit holder allocation/read/write behavior, but its HLS note says tools often infer registers and Rust could emit raw SSA. Q-139 exists to resolve that tension by defining when the holders are observable and when they are optimization artifacts. See [[06-spec-open-question-contract]].

- Observability is class-specific. `FIFOReg` is observable by default because it carries stream/back-pressure and destructive-dequeue semantics. `Reg` is observable when reset/init, enable, multiple writers, loop lifetime, timing, or scheduling effects can be observed. `Var` is the best SSA candidate, but only when its assignment/read pair is generated, local, unconditional, and equivalent under Argon's rewrite/effect model. See [[07-observability-classification]].

- The implementation-cost analysis rejects "optimize everything to SSA" as a deceptively expensive cleanup. Literal preservation is safest but can overconstrain HLS. The practical migration path is HLS-native preservation first, then legality-proven elimination for private generated holders. See [[08-implementation-migration-cost]] and [[09-risk-failure-modes]].

- The recommendation matrix converges on **holders are semantic until classified and proven otherwise**. Preserve the IR-level holder boundary, map surviving holders to HLS-native registers/channels, and eliminate only with a recorded proof. See [[10-recommendation-matrix]].

## Recommendation

Adopt **Semantic Holder Boundaries With Proof-Based Elimination**.

The Rust IR should treat `PipeInserter` holders as observable compiler state by default. The observable contract is the boundary: a value escapes an inner primitive stage by being written under the generated `Pipe`'s enable/control context, then read in the parent context, with the holder's read/write/dequeue effects available to scheduling, analysis, diagnostics, and lowering.

This does not require final HLS code to contain literal Spatial `Reg`, `FIFOReg`, or `Var` objects. The HLS backend may map a surviving holder to a target-native representation:

- `FIFOReg`: a one-token FIFO/channel or equivalent ready/valid/dataflow object when destructive read, stream ordering, full/empty, or pressure semantics can matter.
- `Reg`: a scalar register, local state object, or scheduled variable when init/reset/enable/lifetime/order semantics must be preserved.
- `Var`: local mutable state for non-bit escapes, or SSA only when the Argon `VarAssign`/`VarRead` behavior is proved equivalent.

Then allow proof-based elimination. A holder may be lowered to SSA only if all of these are proven and recorded:

1. It was introduced by pipe insertion, not user-authored state.
2. It has one dynamic producer and one compatible consumer per firing, or all producers/consumers are proven mutually exclusive with equivalent enable conditions.
3. The read is dominated by the write under the same control/enable path, so init/reset/read-before-write behavior is unobservable.
4. The parent is not stream control, or the stream/pressure/destructive-dequeue semantics have an equivalent explicit representation.
5. No downstream pass or report still requires the holder's identity for access analysis, unrolling, retiming, resource accounting, naming, debug, or diagnostics.
6. Anti-dependencies and scheduling barriers represented by the holder's effects are preserved in the HLS schedule or emitted code.

Default classifications:

- `FIFOReg`: preserve by default; SSA elimination is exceptional.
- `Reg`: preserve across reset/init, enable, loop lifetime, multiple-writer, retiming, or schedule boundaries; SSA is allowed for private one-write/one-read pipe escapes after proof.
- `Var`: preserve in the Rust IR long enough to respect assignment/read effects; lower to SSA for generated, local, unconditional non-bit handoffs when the proof is straightforward.

Add provenance to every generated holder and every elimination:

- `holder_origin`: `pipe_inserter`
- `escape_sym`, source span, original stage id, parent controller, parent schedule
- `holder_kind`: `reg | fiforeg | var`
- write op id, read op id, enable path, reset/init path
- HLS mapping: `ssa | scalar_register | fifo_channel | dataflow_channel | mutable_local | rejected`
- if eliminated, a proof summary and blocking facts if not eliminated

Reject **optimize all to SSA** as the base HLS contract. It erases effect boundaries, stream pressure, destructive-dequeue behavior, reset/init behavior, and compiler-consumer identity before the backend has proved equivalence.

Reject **literal preservation forever** as the final-code requirement. It is safe, but too Chisel-shaped and can block HLS from doing legal scalar/register optimization. Preserve semantics and provenance, not necessarily the exact Spatial node spelling in emitted HLS.

Reject **Var-only blanket SSA**. `Var` is the easiest holder to flatten, but it still has assignment/read effects and a rewrite rule that assumes execution conditions. It needs the same dominance/enable proof, just with a lower default burden than `FIFOReg`.

## Decision

Status: awaiting-user-confirmation.

If confirmed, follow-up implementation tasks:

1. Add pipe-holder provenance metadata for generated holders, including origin, stage, parent control, holder kind, write/read ids, enables, and init/reset paths.
2. Preserve `Reg`, `FIFOReg`, and `Var` holder nodes through Rust IR analyses until HLS mapping or proof-based elimination runs.
3. Implement HLS-native mappings for `FIFOReg` channels, `Reg` scalar/register state, and `Var` mutable-local or SSA forms.
4. Implement a holder-elimination proof pass with explicit dominance, enable-equivalence, stream-boundary, reset/init, effect-ordering, and consumer-identity checks.
5. Emit diagnostics explaining why each generated holder was preserved, mapped, eliminated, or rejected.
6. Add positive tests for private one-write/one-read SSA elimination and negative tests for FIFOReg stream tokens, disabled writes, switch/case escapes, reset/init observation, multiple producers, and loop-carried holders.
7. Keep this decision cross-referenced with D-14 and D-22, since controller schedule semantics and FIFO/stream boundedness affect when holder erasure is safe.
