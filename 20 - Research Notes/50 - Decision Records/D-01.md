# D-01 — BigIP Arithmetic Policy

## Question

What policy should the Rust+HLS port use for arithmetic operations the BigIP abstract base does not implement?

Three candidate policies:

- **A. Reject** — compile error if user invokes an op the target hasn't lowered.
- **B. Preserve placeholders** — ops compile but emit zero/identity (matches BigIPSim simulator behavior).
- **C. Lower to named vendor IP** — registry mapping each op to vendor-specific IP or HLS macros.

Source: `fringe/src/fringe/BigIP.scala:22-105`, `fringe/src/fringe/targets/BigIPSim.scala:82-96`.

## Research findings (from 8/10 angles; angles 5 and 8 detached without writing)

### Surface

`BigIP.scala` exposes **38 method signatures**, **34 unique op names** (signed/unsigned divide, mod, multiply collapse).

- **16 abstract** (all targets must implement): `divide`, `mod`, `multiply` (signed+unsigned), `fadd`, `fsub`, `fmul`, `fdiv`, six float comparisons (`flt`, `fgt`, `fge`, `fle`, `fne`, `feq`).
- **21 concrete-with-default** that throw `Unimplemented(op)` by default: `sqrt`, `sin`, `cos`, `atan`, `sinh`, `cosh`, `log2`, `fabs`, `fexp`, `ftanh`, `fsigmoid`, `fln`, `frec`, `fsqrt`, `frsqrt`, `ffma`, `fix2flt`, `fix2fix`, `flt2fix`, `flt2flt`, `fltaccum`. Default error text: `"<op> is not implemented for the given target."`
- **1 concrete helper**: `getConst`.

→ See [[01-api-survey]].

### Per-target reality

- **Zynq family** (Zynq, AWS, ZedBoard, KCU1500, ZCU, VCU1525, CXP): instantiate Vivado LogiCORE IP (`div_gen`, `mult_gen`, `cordic`, `floating_point`).
- **ASIC**: DesignWare blackboxes for integer + Vivado float IP.
- **Sim family** (Sim, AWS_Sim, VCS, Arria10, DE1SoC): HardFloat (`MulAddRecFN`, `DivSqrtRecFN_small`, `CompareRecFN`).
- **Most transcendentals** (sin/cos/sinh/cosh/atan) only have implementations in `BigIPSim` and they are **identity-pass-through TODO stubs**, not real math.
- Zynq/CXP/ASIC inherit the throwing default for sin/cos/etc. → these operations **already fail at synthesis time today** if a user calls them on hardware targets.

→ See [[02-bigipsim-behavior]], [[03-per-target-overrides]].

### Math API contract

`Math` (the public arithmetic facade) returns plain `UInt`/`SInt`/`Bool` with **no validity/support sideband**. Callers (FloatingPoint, FixedPoint operators, Accumulator templates, MemPrimitives bank/address arithmetic) immediately use results in `Mux`/address logic. → A runtime "unsupported" signal would require a new result type or sideband channel pervading the entire codegen layer. Elaboration-time rejection is dramatically cheaper.

→ See [[04-math-dispatch]].

### App categories that depend on which ops

| Category | Mandatory ops |
|---|---|
| Dense BLAS/GEMM/GEMV | `fadd`, `fmul`, `fdiv` |
| ML/CNN inference (LogReg, batchNorm) | `fadd`, `fmul`, `fdiv`, `fsigmoid`, `fexp`, `fsqrt`, `fabs`, `fmax`, `pow` |
| FFT/DSP | `fmul`, `fadd`, `fsub`, `sin`, `cos` |
| Image/stencil/CNN | `fmul`, `fadd`, `fsub`, `fabs`, `fmax`; sometimes `fsqrt` |
| Sparse SpMV/PageRank | `fmul`, `fadd`, `fdiv` |
| Black-Scholes/MD | `fdiv`, `fexp` (or Taylor), `fln` (or Taylor), `fsqrt` (or approx), `fabs`, `pow` |
| Sorting | integer ops only |

Reject-on-unsupported would **exclude visible app categories**, not just edge cases — unless transcendentals have an HLS-native lowering path.

→ See [[06-app-categories]].

### HLS native math support

Vivado/Vitis HLS and Intel HLS provide direct equivalents for **most** Spatial BigIP ops via `hls_math.h` / `HLS/math.h` headers. Native operators handle `+`, `-`, `*`, `/`, integer `%`, `fadd`, `fmul`, `fdiv`, `fix2flt`, `flt2fix`, `fltaccum` (via `+=`).

Header-level math covers: `sqrt`, `1/sqrt` (`rsqrt`), `exp`, `log`, `sin`, `cos`, `tan`, `tanh`, `pow`, `fma`, `recip`, `abs`.

**AMD/Vitis gaps:** `log2` (use `log(x)/log(2)`), `sigmoid` (use `1/(1+exp(-x))`).
**Intel gaps:** narrower fixed-point math header.

→ Most BigIP ops do **not** need explicit vendor IP — they have direct HLS-tool-native lowerings. → See [[07-hls-native-math]].

### Precedent in DSL→HLS compilers

Across LegUp, Bambu, Calyx, Halide HLS, HeteroCL, hls4ml/FINN: **dominant pattern is reject early** with diagnostics, restrict source, or pluggable-backend lowering with a primitive library. Silent fallback is uncommon because it hides material precision/area/latency divergence.

→ See [[09-dsl-precedent]].

### Implementation cost analysis

- **Option A (Reject):** cheapest. Static unsupported set; existing error message has the right shape. Risk: too conservative for transcendentals.
- **Option B (Placeholder):** smallest LOC change. Silent wrong-answer risk severe for conversions/accumulators. Not recommended as default.
- **Option C (Lower to IP):** largest. 34 op names × parameter adaptation × per-target maintenance.
- **Hybrid: Reject + opt-in IP**: sensible balance.

→ See [[10-implementation-cost]].

## Recommendation

**Adopt a 3-tier capability model with reject-by-default for unsupported ops.**

### Tier 0 — Mandatory (HLS port MUST implement)

The 16 abstract BigIP ops, with no opt-out:
- Integer `divide`, `mod`, `multiply` (signed+unsigned)
- Float `fadd`, `fsub`, `fmul`, `fdiv`
- Float comparisons `flt`, `fgt`, `fge`, `fle`, `fne`, `feq`

These have native HLS equivalents (`+`, `-`, `*`, `/`, `<`, `==`, etc.), so the cost is trivial.

### Tier 1 — HLS-native (opt-in, default-on for AMD/Intel HLS targets)

Lower to `hls_math.h` / `HLS/math.h` headers:
- `sqrt`, `fsqrt`, `frsqrt` → `hls::sqrt`, `hls::rsqrt`
- `fexp`, `fln` → `hls::exp`, `hls::log`
- `sin`, `cos`, `tan`, `cosh`, `sinh`, `tanh` → respective `hls::*`
- `ffma` → `hls::fma` or `a*b+c` (HLS infers FMA via DSP binding)
- `fabs` → `hls::abs`
- `fix2flt`, `flt2fix`, `fix2fix`, `flt2flt` → C++ casts / `to_float()`

This converts ~13 of the 21 default-unimplemented ops into "just works" via the HLS toolchain. Effort: small (one lowering rule per op).

### Tier 2 — Custom IP / explicit opt-in

For ops without HLS-native support, require explicit per-target opt-in:
- `fsigmoid` → user provides `1/(1+exp(-x))` macro OR opts into a vendor IP / LUT-approximation
- `log2` (AMD) → user provides `log(x)/log(2)` OR opts in
- `fltaccum` → may need vendor accumulator IP for II=1
- `atan` → CORDIC or hls::atan if available

Default behavior for Tier 2: **Reject** with clear diagnostic naming the op, target, and listing alternatives:
```
error: BigIP op `fsigmoid` is not supported for target `Zynq` in v1.
  Alternatives: provide manual lowering as `1/(1+exp(-x))`, or enable target.bigIP.opt_in("fsigmoid", "vendor_sigmoid_ip").
  See: docs/hls/op-support.md
```

### Why this beats pure (A), (B), or (C)

- **vs pure Reject (A):** Tier 1 HLS-native lowering avoids excluding entire app categories (FFT, ML, batchNorm, Black-Scholes) that need transcendentals.
- **vs Placeholder (B):** No silent wrong-answer bugs. The HLS backend never emits zero-instead-of-result.
- **vs full IP map (C):** Most ops covered by free HLS-native lowering; only a small Tier-2 set requires vendor-IP wiring, and that wiring is opt-in per app/target.

The recommendation aligns with DSL→HLS precedent (Calyx, Bambu) and matches Spatial's existing reality (Zynq targets already silently throw on transcendentals — Tier 0+1 actually expands the supported surface).
