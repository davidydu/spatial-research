# HLS Possibilities

## Baseline: Unsupported Is Semantic

Spatial does not merely lack codegen for Scala `while`, `do while`, or `return`; the application-facing API includes `SpatialVirtualization`, and those hooks issue hard frontend diagnostics for all three constructs (`src/spatial/lang/api/StaticAPI.scala:7`, `src/spatial/lang/api/SpatialVirtualization.scala:103`, `src/spatial/lang/api/SpatialVirtualization.scala:107`, `src/spatial/lang/api/SpatialVirtualization.scala:111`). Forge's default virtualized hooks would otherwise macro-expand them back to native Scala control flow (`forge/src/forge/EmbeddedControls.scala:48`, `forge/src/forge/EmbeddedControls.scala:131`, `forge/src/forge/EmbeddedControls.scala:136`, `forge/src/forge/EmbeddedControls.scala:141`). Therefore, lowering these constructs to HLS C++ would be new HLS-visible semantics, not compatibility preservation.

## Possibility A: Preserve And Reject

The lowest-risk rule is to keep rejecting source `while`, `do while`, and early `return` in the Rust/HLS frontend. This aligns with Spatial's supported control surface: `Foreach` stages an `OpForeach` from explicit counters and optional `stopWhen`; `FSM` stages a `StateMachine` from explicit `notDone`, `action`, and `nextState` lambdas (`src/spatial/lang/control/ForeachClass.scala:26`, `src/spatial/lang/control/ForeachClass.scala:30`, `src/spatial/lang/control/FSM.scala:18`, `src/spatial/lang/control/FSM.scala:23`). The IR model is controller-centric: `OpForeach` carries a `CounterChain` and iterators, while `StateMachine` carries `notDone`, `action`, and `nextState` blocks (`src/spatial/node/Control.scala:45`, `src/spatial/node/Control.scala:103`). Rejecting arbitrary source loops avoids inventing a hidden conversion from imperative control to these explicit controllers.

## Possibility B: Lower To Bounded Loops

An HLS frontend could lower `while` to `while (cond) body` and `do while` to a first-iteration body followed by a guarded loop in generated C++ (unverified). This is tempting for host-like parity, but it collides with Spatial's bounded-loop assumptions. Counters are first-class values with `start`, `end`, `step`, and `par`; zero-step counters are rejected as accidental infinite iteration, with `*` reserved for intentional forever counters (`src/spatial/lang/Counter.scala:15`, `src/spatial/node/Control.scala:9`, `src/spatial/node/Control.scala:13`, `src/spatial/traversal/UserSanityChecks.scala:64`). Several analyses expect known iterator structure: iteration-difference analysis asks users to set static bounds, accept conservative II, enable loose diffs, or explicitly set II when bounds are not known (`src/spatial/traversal/IterationDiffAnalyzer.scala:37`, `src/spatial/traversal/IterationDiffAnalyzer.scala:45`, `src/spatial/traversal/IterationDiffAnalyzer.scala:49`). General HLS tools often need static or asserted tripcount bounds for scheduling, reporting, and pipelining quality (unverified). Thus, if this option is chosen, the contract should require a syntactic or inferred max tripcount and define how unknown bounds affect II, diagnostics, and DSE.

## Possibility C: Lower To FSM

Data-dependent loops fit Spatial's existing `FSM` shape better than counter loops. Hardware codegen already treats `StateMachine` specially: it marks the app as having an FSM, wires `state`, `nextState`, `initState`, and `doneCondition`, and uses `~notDone` as the done condition (`src/spatial/codegen/chiselgen/ChiselGenController.scala:469`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:474`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:482`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:484`). The Fringe controller stores and advances FSM state only under controller enables, and gates datapath execution on not-done conditions (`fringe/src/fringe/templates/Controllers.scala:221`, `fringe/src/fringe/templates/Controllers.scala:227`, `fringe/src/fringe/templates/Controllers.scala:233`, `fringe/src/fringe/templates/Controllers.scala:237`). This suggests an HLS `while` could be specified as sugar for an FSM when all loop-carried state is explicit. For `do while`, the initial state would need a "body-before-test" phase (unverified). The cost is frontend complexity: mutation, loop-carried values, memory effects, and side-effect ordering must become explicit SSA/FSM state rather than accidental C++ locals.

## Return And Parity Constraints

Native C++ `return` from inside generated HLS could skip following statements and resource cleanup in the C++ function (unverified). Spatial's existing early-exit-like machinery is different: `ExitIf` and `BreakpointIf` are explicit global-effect nodes (`argon/src/argon/node/Debugging.scala:17`, `argon/src/argon/node/Debugging.scala:21`); Chisel lowers them to breakpoint wires and completes the accelerator through the top-level done latch (`src/spatial/codegen/chiselgen/ChiselGenDebug.scala:22`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:398`, `src/spatial/codegen/chiselgen/ChiselGenController.scala:404`); the C++ host wrapper later reads early-exit args after `c1->run()` (`src/spatial/codegen/cppgen/CppGenAccel.scala:29`, `src/spatial/codegen/cppgen/CppGenAccel.scala:38`, `src/spatial/codegen/cppgen/CppGenAccel.scala:42`). Scala simulation also warns that `breakWhen` timing differs between Scala and synthesis: Scala breaks at the loop end, while synth breaks immediately (`src/spatial/codegen/scalagen/ScalaGenController.scala:75`). Therefore a new `return` cannot simply mean "C++ return" unless simulator, HLS, cleanup, ArgOut validity, stream closure, and side-effect visibility are all specified. A safer new-semantics option is an explicit accelerator-exit effect with defined commit points; otherwise preserve-and-reject remains the clean D-03 answer.
